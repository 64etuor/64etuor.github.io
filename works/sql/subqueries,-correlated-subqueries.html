Subqueries, Correlated subqueries

Understanding Subqueries and Correlated Subqueries in SQL

Introduction

  • Subqueries (also known as inner queries) are queries nested inside other SQL statements, such as SELECT, INSERT, UPDATE, or DELETE. They allow you to dynamically use the results of one query as input or conditions for another.
  • Correlated subqueries are a special type of subquery that references one or more columns from the outer query. Unlike a regular subquery, a correlated subquery is evaluated once for each row processed by the outer query, making it more dynamic but potentially less performant.

Regular Subqueries

Definition

  • A subquery is a query enclosed in parentheses and embedded inside a larger query. It is often used in WHERE clauses, FROM clauses (as a derived table), or SELECT clauses.
  • Generally, a subquery can be executed independently, and its result is then passed to the outer query.

Common Uses of Subqueries

  1. Filtering Results in a WHERE Clause:

    SELECT EmployeeID, FirstName, LastName
FROM Employees
WHERE DepartmentID = (
  SELECT DepartmentID
  FROM Departments
  WHERE DepartmentName = 'Sales'
);
    • Explanation: The subquery finds the DepartmentID for the "Sales" department. The outer query then returns all employees in that department.

  2. Using Subqueries in the FROM Clause (Derived Tables):

    SELECT d.DepartmentName, e.EmployeeCount
FROM (
  SELECT DepartmentID, COUNT(*) AS EmployeeCount
  FROM Employees
  GROUP BY DepartmentID
) e
INNER JOIN Departments d ON d.DepartmentID = e.DepartmentID;
    • Explanation: The subquery e aggregates employees by department. The outer query joins this result to the Departments table to retrieve department names alongside employee counts.

  3. Subqueries in the SELECT List:

    SELECT 
  FirstName,
  LastName,
  (SELECT COUNT(*) FROM Orders WHERE Orders.EmployeeID = Employees.EmployeeID) AS OrderCount
FROM Employees;
    • Explanation: For each employee, the subquery counts how many orders they have processed.

Characteristics of Regular Subqueries

  • Typically executed once, and the result is reused by the outer query.
  • Does not necessarily depend on the outer query’s current row. In other words, the subquery can stand alone and run independently.

Correlated Subqueries

Definition

  • A correlated subquery is a subquery that references columns from the outer (parent) query. It means the subquery depends on the current row of the outer query to be evaluated.
  • As a result, a correlated subquery is executed once for each row in the outer query, making it more computationally expensive in many cases.

Example of a Correlated Subquery

SELECT e.EmployeeID, e.FirstName, e.LastName
FROM Employees e
WHERE e.Salary > (
  SELECT AVG(Salary) 
  FROM Employees e2
  WHERE e2.DepartmentID = e.DepartmentID
);
  • Explanation:
    • For each employee e in the outer query, the correlated subquery (SELECT AVG(Salary) FROM Employees e2 WHERE e2.DepartmentID = e.DepartmentID) calculates the average salary for that employee’s department.
    • Thus, the subquery depends on e.DepartmentID, a value in the outer query, changing the subquery’s criteria as the outer query progresses through its rows.

How Correlated Subqueries Work

  • The outer query selects a row.
  • The correlated subquery runs, using values from that row, and returns a result.
  • The outer query uses that result to determine if the row meets the conditions.
  • This process repeats for each row of the outer query’s result set.

Performance Considerations

  • Because a correlated subquery runs once per outer row, it can be much slower on large datasets.
  • In many cases, it can be optimized by rewriting the query using joins, temporary tables, or window functions.

Comparing Regular Subqueries and Correlated Subqueries

Aspect Regular Subquery Correlated Subquery
Dependency Can often run independently of the outer query. Depends on values from the outer query’s current row.
Execution Frequency Typically runs once, and the result is reused. Runs once for each row in the outer query result set.
Complexity Generally simpler and more straightforward. More complex, may be harder to understand and maintain.
Performance Can be efficient if well-indexed and properly structured. Potentially slower due to repeated execution per row.
Use Cases Aggregation filters, derived tables, conditions on sets. Row-by-row comparisons, dynamic conditions based on outer rows.

Practical Tips

  1. Identify When Correlated Subqueries Are Necessary:

    • If your logic requires a per-row lookup that depends on the outer row, a correlated subquery might be needed. For example, checking if an employee’s salary is greater than the average salary of their department (calculated at runtime).

  2. Optimize Correlated Subqueries:

    • Consider rewriting the query using JOINs, COMMON TABLE EXPRESSIONS (CTEs), or window functions if possible.
    • Example transformation using a CTE and a join: 
      WITH DeptAvgSalary AS (
  SELECT DepartmentID, AVG(Salary) AS AvgDeptSalary
  FROM Employees
  GROUP BY DepartmentID
)
SELECT e.EmployeeID, e.FirstName, e.LastName
FROM Employees e
JOIN DeptAvgSalary das ON e.DepartmentID = das.DepartmentID
WHERE e.Salary > das.AvgDeptSalary;
      • This avoids the per-row computation and might run faster.

  3. Indexing for Performance:

    • If you cannot avoid correlated subqueries, ensure that the columns used in the subquery’s conditions are indexed, which can mitigate performance issues.

Conclusion

  • Subqueries and correlated subqueries are powerful SQL techniques that allow for sophisticated queries without overly complex joins or intermediate tables.
  • Regular subqueries are independent and can be used to filter, aggregate, or create derived tables for the outer query.
  • Correlated subqueries depend on the outer query and are evaluated once per row, making them more flexible but potentially less performant.
  • When properly understood and applied, these constructs can simplify SQL logic and help you write more dynamic and powerful queries.
