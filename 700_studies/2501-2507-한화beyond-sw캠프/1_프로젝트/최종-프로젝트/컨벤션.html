1. 코드 및 Git 컨벤션

1.1. 코드 포맷팅

코드 포맷팅 pre-commit git hook은 프론트엔드 루트 폴더에서 npm install을 하면 자동 설정됩니다.

🔗 관련 문서: 개발 환경 설정 #Pre-commit

Java

  • pre-commit hook으로 Spotless 컨벤션을 적용하여 코드 스타일을 강제합니다.
  • 적용 규칙: Java 
    spotless {
    java {
        // import 순서: java, javax, org, com, 그 외
        importOrder('java', 'javax', 'org', 'com', '')

        // 사용하지 않는 import 문 자동 제거
        removeUnusedImports()

        // Google Java Format 적용 (버전 명시)
        googleJavaFormat('1.27.0')

        // 어노테이션 포맷팅
        formatAnnotations()
    }
}

Vue 3 / JavaScript

  • Prettier: 일관된 코드 포맷팅을 자동화합니다.
  • ESLint: 문법 및 스타일을 검사합니다 (eslint-config-standard, eslint-plugin-vue, @typescript-eslint 플러그인 사용).
  • eslint-plugin-unused-imports: 사용되지 않는 import를 자동으로 감지하고 제거합니다.
  • Husky + lint-staged: Git Hook을 사용하여 커밋 전 Prettier 포맷팅, ESLint --fix, 미사용 import 정리를 자동으로 실행합니다.

1.3. Git 워크플로우 & 브랜치 전략

  • 브랜치 생성 정책: GitHub 이슈당 하나의 브랜치를 생성합니다.
    • 브랜치명에는 반드시 이슈 번호를 포함합니다. (예: feat/user/로그인-기능#90)
  • 워크플로우: Gitflow 전략을 따릅니다.
  • 브랜치 유형:
    • feat/: 신규 기능 개발
    • fix/: 버그 및 기타 수정
    • release/: 릴리즈 준비
    • hotfix/: 긴급 패치
    • chore/: 빌드, 인프라, 도구, 문서 등 비즈니스 로직과 무관한 작업
    • refactor/: 코드 리팩토링
  • PR 규칙: 최소 1명 이상의 코드 리뷰를 거치며, 머지 전 CI 빌드가 반드시 통과해야 합니다.

1.4. 이슈 및 PR 네이밍 컨벤션

이슈 네이밍

[영문도메인명] 간단한 설명 - 상세 설명
  • 예시:
    • [user] 로그인 API - JWT 토큰 발급 기능 구현
    • [notification] 상품 목록 - 무한 스크롤 및 필터링 기능 추가
    • [reservation] 버그 수정 - 주문 취소 시 재고 복구 오류

PR 네이밍

타입: [영문도메인명] 간단한 설명 #<이슈번호>
  • 타입: feat, fix, refactor, chore, docs, test
  • 예시:
    • feat: [user] 로그인 API 구현 #123
    • fix: [sales] 주문 취소 버그 수정 #456
    • refactor: [user] 사용자 서비스 리팩토링 #789

중요: PR 제목에 #이슈번호를 포함해야 GitHub Actions가 이슈 내용을 PR에 자동으로 연결합니다.

브랜치 네이밍

타입/도메인/간단설명#<이슈번호>
  • 예시:
    • feat/user/로그인-기능#1
    • fix/sales/주문취소-버그#2
    • chore/backend/의존성-업데이트#6

1.6. 프로젝트 공통 가이드라인

API 설계 가이드라인 (REST)

  • 엔드포인트 경로: kebab-case를 사용하며, 리소스를 나타내는 명사를 사용합니다. (예: GET /users, POST /orders)
  • HTTP 메서드: CRUD(Create, Read, Update, Delete) 오퍼레이션에 맞게 HTTP 메서드를 매핑합니다. (GET, POST, PUT/PATCH, DELETE)
  • 상태 코드: 표준 HTTP 상태 코드를 명확히 활용합니다. (예: 200 OK, 201 Created, 400 Bad Request, 404 Not Found, 500 Internal Server Error)
  • 페이징/필터링/정렬: 쿼리 파라미터를 통해 구현합니다. (예: ?page=1&size=20&sort=createdAt,desc)
  • 버전 관리: API 경로에 버전을 포함합니다. (예: /api/v1/users)
  • 응답 래핑: 일관된 JSON 구조로 응답을 감싸서 반환합니다.

  • 자세한 내용은 본 문서의 3. REST API 설계 가이드를 참조하십시오.

예외 처리 & 로깅

  • 예외 처리: 글로벌 예외 핸들러(@ControllerAdvice)를 사용하여 예외를 중앙에서 처리합니다.
  • 커스텀 예외: ErrorCode enum과 ApiException 커스텀 클래스를 조합하여 비즈니스 예외를 정의합니다.
  • 로깅: SLF4J + Logback 조합을 사용합니다.
    • 로그 레벨: ERROR, WARN, INFO, DEBUG, TRACE를 명확히 구분하여 사용합니다.
    • 민감 정보(개인정보, 비밀번호 등)는 로그에 남기지 않으며, 구조화된 JSON 로그 포맷을 권장합니다.

테스트 가이드라인

  • 단위 테스트: Java는 JUnit5 + Mockito, Vue는 Jest + Vue Test Utils를 사용합니다.
  • 통합 테스트: Spring Boot Test를 활용하여 계층 간 통합을 검증합니다.
  • E2E 테스트: Cypress 또는 Playwright 중 선택하여 사용합니다. CI에서 npx playwright test 커맨드로 자동화합니다.
  • 커버리지: 전체 코드 커버리지 최소 80%를 목표로 하며, JaCoCo로 측정합니다.
  • 테스트 네이밍: given_when_then 패턴을 사용하여 테스트의 의도를 명확하게 표현합니다.

문서화

  • API 명세: OpenAPI (Swagger)를 통해 API 스펙을 자동으로 생성하고 UI를 제공합니다.
  • 커버리지 테스트 및 문서화: JaCoCo를 사용합니다.
  • 프로젝트 문서: README.md에 프로젝트 개요, 기술 스택, 실행 방법을 명시하고, 아키텍처 다이어그램을 포함합니다.

보안 및 비밀 관리

  • 민감 정보: Github Gist에서 중앙 관리합니다. (담당자: 이채은)
  • 환경별 설정: application.yml, .env, 기타 Github Secrets를 통해 환경별 구성을 분리합니다.
  • 의존성 검토: OWASP Dependency-Check
  • 인증·인가: OAuth2 / JWT 표준을 준수하여 구현합니다.

의존성 및 버전 관리

  • Gradle/Maven: BOM(Bill of Materials)을 활용하여 의존성 버전 충돌을 방지합니다. 🔗 참고자료

2. 데이터베이스 명명 규칙

프로젝트 팀원은 모두 다음 사항을 이해하고 준수합니다.

2.1. 명명 규칙이 중요한 이유

  • 데이터 구조의 긴 수명: 데이터베이스 스키마는 특정 애플리케이션보다 훨씬 오랫동안 유지되며 여러 서비스에서 재사용될 수 있습니다.
  • 이름은 계약이다: 객체 이름은 API 계약과 같아서 한 번 배포되면 변경 비용이 매우 큽니다.
  • 개발자 편의성: 일관된 규칙은 테이블과 컬럼을 찾는 시간을 줄여 생산성을 높입니다.

2.2. 기본 명명 규칙
규칙 설명 예시
따옴표 금지 이름에 공백이나 대문자를 사용하지 않아 쿼리 작성 시 따옴표가 불필요하게 만듭니다. 잘못된 예: "FirstName", "All Employees"
모두 소문자 대소문자를 혼용하면 DB에 따라 인용부호가 필요해지므로 모두 소문자로 통일합니다. first_name
데이터 타입 미포함 컬럼명에 _tx, _dt 등 데이터 타입을 포함하지 않습니다. 타입 변경 시 이름까지 바꿔야 하는 문제를 방지합니다. X: name_tx → O: name
snake_case 사용 단어 구분은 언더스코어(_)를 사용합니다. word_count
약어 금지 의미가 명확하도록 완전한 단어(full word) 사용을 원칙으로 합니다. X: mid_nm → O: middle_name
통용 약어 허용 i18n(Internationalization), l10n(Localization) 등 널리 쓰이는 약어는 예외적으로 허용합니다. i18n_code
예약어 회피 DB 예약어(user, table, order, group)는 테이블/컬럼명으로 사용하지 않습니다. X: user → O: app_user

2.3. 테이블·컬럼 명명

  1. 단수 명사 사용: 테이블명은 리소스를 표현하는 단수 명사를 사용합니다. (예: person)
    • 예외 ①: like 등 예약어와 혼동을 피하기 위해 복수형 허용 (예: likes)
    • 예외 ②: 단·복수 동일 명사 (예: series, equipment)
    • 예외 ③: 배열·리스트 타입의 컬럼은 복수형 사용 가능 (예: tags)
  2. PK (Primary Key): <테이블명>_id 또는 간단히 id를 사용합니다.
  3. FK (Foreign Key): <참조테이블_단수형>_id 패턴을 사용합니다. (예: team_id, person_id, admin_id)
  4. Composite-FK/PK: 참조 테이블 이름과 컬럼 조합으로 명확히 표현합니다.

2.4. 인덱스·제약 조건 명명

인덱스와 제약 조건에 명시적인 이름을 지정하면 추적 및 디버깅이 용이해집니다.
객체 패턴 예시
INDEX ix_<table>_<column(s)> ix_person_first_name_last_name

2.5. 접두사·접미사 지양

  • 관계 타입 접두사: tb_, vw_ 등 객체 유형을 나타내는 접두사는 사용하지 않습니다.
  • 애플리케이션 이름: 스키마(네임스페이스)로 분리하고 테이블 이름에 앱 이름을 포함하지 않습니다.
  • 데이터 타입 접미사: _tx, _dt 등은 타입 변경 시 유지보수 비용을 증가시키므로 금지합니다.

2.6. 감사·상태 컬럼 (Auditing & Status)
컬럼 의미
created_at 레코드 생성 시각 (Timezone 포함)
modified_at 레코드 최종 수정 시각 (Timezone 포함)
deleted_at Soft-delete 시각 (삭제되지 않은 경우 NULL)
status 업무 상태 코드 (예: active, inactive, approved, rejected)

2.7. 코드 예시 (MariaDB/MySQL)

SQL

CREATE TABLE person (
    id BIGINT UNSIGNED AUTO_INCREMENT,
    email VARCHAR(255) NOT NULL,
    first_name VARCHAR(100) NOT NULL,
    last_name VARCHAR(100) NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT pk_person PRIMARY KEY (id),
    CONSTRAINT uk_person_email UNIQUE (email),
    CONSTRAINT ck_person_email_lower_case CHECK (email = LOWER(email))
) ENGINE=InnoDB COMMENT '사용자 정보';

CREATE INDEX ix_person_first_name_last_name
    ON person (first_name, last_name);

CREATE TABLE team (
    id BIGINT UNSIGNED AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    CONSTRAINT pk_team PRIMARY KEY (id)
) ENGINE=InnoDB COMMENT '팀 정보';

CREATE TABLE team_member (
    team_id BIGINT UNSIGNED NOT NULL,
    person_id BIGINT UNSIGNED NOT NULL,
    CONSTRAINT pk_team_member PRIMARY KEY (team_id, person_id),
    CONSTRAINT fk_team_member_to_team FOREIGN KEY (team_id) REFERENCES team(id),
    CONSTRAINT fk_team_member_to_person FOREIGN KEY (person_id) REFERENCES person(id)
) ENGINE=InnoDB COMMENT '팀 멤버 관계';

2.10. Best Practice
# 규칙 예시/설명
1 Boolean 컬럼 접두사 is_active, has_thumbnail, can_edit (상태, 소유, 가능 여부 명확화)
2 날짜·시간 컬럼 통일 특정 시점은 _at (created_at), 특정 날짜나 기간은 _date 또는 _start/_end
3 복합 컬럼명 순서 “관계 → 속성 → 단위” 순으로 조합 (예: order_total_price)
4 ENUM·코드 컬럼 코드 값 컬럼은 _code 접미사 (status_code)를 붙이고, 의미를 담은 룩업 테이블을 생성
5 스키마 명명 기능 또는 Bounded Context 단위로 스키마 분리 (예: auth, chat, billing)
6 다중 컬럼 인덱스 순서 필터링(선택도 높은 순) 우선, 그 다음 정렬 순 (예: (status, created_at))
7 특수 목적 접두사 임시 테이블 tmp_, 백업 테이블 bk_20250520_ 등은 목적을 명확히
8 객체 주석 관리 COMMENT ON TABLE person IS '개인 정보'; 로 테이블과 컬럼에 설명을 추가

3. REST API 네이밍 컨벤션

범위와 목적

본 가이드는 DevEagles 팀이 개발·운영하는 모든 내부‧외부 REST 엔드포인트의 URI 스키마와 쿼리스트링 정책을 통일합니다.

일관된 규칙은 서비스 간 탐색성·테스트 용이성·자동화 수준을 높이고 최종 프로젝트에서 마이크로서비스가 독립적으로 진화할 수 있도록 돕습니다.

3.1. 일반 URI 스타일
규칙 지침 예시
소문자 사용 경로 전체를 소문자로 작성합니다. GET /user-profiles
케밥-케이스 단어 구분은 하이픈(-)을 사용합니다. 스네이크(_)·카멜 표기 금지. POST /order-items
복수 명사 컬렉션 리소스는 복수형으로 작성합니다. GET /users, GET /projects
확장자 없음 응답 포맷은 Content-Type/Accept 헤더로 협상합니다. .json·.xml 등 금지. GET /users/some-id
트레일링 슬래시 없음 URI 끝에 /를 붙이지 않습니다. /users (O), /users/ (X)
리소스 ID 추측 불가능한 고유 식별자(UUIDv4 권장)를 사용합니다. /users/9f2d3b-…

3.2. CRUD 매핑
HTTP 컬렉션 URI (/tasks) 단일 항목 URI (/tasks/{taskId}) 의미
GET 목록 또는 단일 조회 특정 항목 조회 Read: 리소스 조회
POST 새 항목 생성 — Create: 리소스 생성
PUT — 전체 교체(멱등성 보장) Update: 리소스 전체 수정
PATCH — 부분 수정 Update: 리소스 부분 수정
DELETE — 삭제 Delete: 리소스 삭제

3.3. 계층 구조 & 관계

  • 두 단계 규칙: 가독성을 위해 중첩 깊이를 2단계 이하로 제한합니다.
    • /projects/{projectId}/issues (컬렉션)
    • /projects/{projectId}/issues/{issueId} (항목)
  • 서브리소스 vs. 단일 속성:
    • 독립적으로 존재할 수 있는 엔티티는 복수형 서브리소스로 표현합니다. (/users/{id}/posts)
    • 1:1 종속 관계이며 속성에 가까운 표현은 단수형 경로를 사용합니다. (/users/{id}/avatar)

3.4. CRUD 외 액션

  • 상태 중심 리소스 선호: 동사 대신 '결과적 리소스'를 생성하는 방식으로 접근합니다.
    • 예: POST /payments/{id}/refunds (환불이라는 리소스를 생성)
  • 액션 접미사(:verb) 사용: 리소스로 표현하기 힘든 동사형 작업은 :verb 패턴을 사용합니다.
    • POST /invoices/{id}:send – 인보이스 이메일 발송
    • POST /users/{id}:activate – 계정 활성화

3.5. 쿼리 파라미터 규칙
목적 패턴 예시
필터링 ?{field}={value}&... ?status=active&priority=high
전문검색 ?q={keyword} ?q=deadline
정렬 ?sort={field1},-{field2} ?sort=created-at,-priority ('-'는 내림차순)
페이지네이션 ?page={num}&limit={num} ?page=3&limit=50 또는 ?offset=100&limit=50
희소 필드 ?fields={field1},{field2} ?fields=title,assignee,created-at (응답 필드 제한)

3.6. 버전 관리 전략

  1. 헤더 기반(권장 방식): Accept: application/vnd.goody.v1+json
  2. URI 접두사: 모든 API 경로에 버전을 명시합니다. /v1/users

리소스 뒤에 버전(/users/v1)을 두지 않습니다. 구버전 API는 Sunset 헤더로 폐지 일정을 공지합니다.

3.7. 별칭 경로

  • /users/me – 현재 인증된 사용자를 가리키는 별칭.
  • /organizations/current – JWT 클레임 등 컨텍스트에서 현재 조직을 해석.

별칭은 정확히 하나의 리소스만 가리키며 UUID 같은 ID 패턴과 충돌하지 않아야 합니다.

3.8. 예약 쿼리 키워드

page, limit, offset, sort, q, fields

위 키워드는 시스템에서 예약되었으므로, 도메인 필드 이름으로 재사용하는 것을 금지합니다.

3.9. 오류·성공 응답 포맷

  • Location 헤더: 201 Created 응답 시, Location 헤더에 생성된 리소스의 URI를 반드시 포함해야 합니다. (Location: /tasks/{newId})
  • Problem + JSON (RFC 7807): 오류 응답은 표준화된 Problem Details 포맷을 사용합니다. JSON 
    {
  "type": "https://docs.goody.dev/errors/validation-error",
  "title": "Validation failed",
  "status": 400,
  "detail": "priority는 [low, medium, high] 중 하나여야 합니다.",
  "instance": "/tasks/..."
}

3.10. 네이밍 체크리스트

  1. 모든 명사는 복수·소문자인가?
  2. 하이픈(-)만 사용했는가? (스네이크/카멜 불가)
  3. 중첩 깊이가 2단계를 넘지 않는가?
  4. 동사형 액션은 :verb 접미사를 사용했는가?
  5. ID는 추측 불가능한가? (UUIDv4 권장)
  6. URI 끝에 트레일링 슬래시나 파일 확장자가 없는가?

3.11. 전체 흐름 예시

HTTP

4. 로그 컨벤션

4.1. 개요

이 문서는 DevEagles 프로젝트의 백엔드와 프론트엔드에서 로그를 일관성 있게 작성하고 효율적으로 관리하기 위한 규칙을 정의합니다. 표준화된 로그는 시스템 디버깅, 사용자 행동 분석, 성능 모니터링 및 잠재적 문제 파악을 용이하게 합니다.

4.2. 공통 로그 원칙

로그 레벨
레벨 설명 사용 사례
ERROR 시스템 동작에 영향을 주는 심각한 오류 또는 예외 상황. 즉시 조치가 필요. API 호출 실패, DB 연결 오류, 처리되지 않은 예외, 비즈니스 로직 실패
WARN 시스템은 정상 동작하지만 주의가 필요한 잠재적 문제 상황. Deprecated 기능 사용, 예상치 못한 데이터, 로그인 실패 횟수 초과
INFO 시스템의 주요 동작 흐름 및 상태 변경 추적. 사용자 로그인/회원가입, 주문 생성, 페이지 이동 등 핵심 기능의 성공적 완료
DEBUG 개발 및 디버깅 과정에서만 필요한 상세 정보. 운영에서는 비활성화. 메서드 진입/종료, 변수 값 확인, API 요청/응답 상세 데이터

로그 메시지 형식

  • 구조: [동작] - [대상] - [핵심 정보] 형식으로 작성하여 로그의 목적을 명확히 합니다.
    • 좋은 예시: 사용자 로그인 성공 - 사용자 ID: {}, IP: {}
    • 나쁜 예시: 로그인 성공, 에러 발생
  • 변수: 문자열 연결(+) 대신 플레이스홀더({}) 또는 템플릿 리터럴을 사용하여 가독성과 성능을 확보합니다.

민감 정보 처리

  • 제외 대상: 비밀번호, 주민등록번호, 신용카드 정보, 연락처, 주소, API 키, 토큰 등
  • 처리 방법: 불가피하게 관련 데이터를 기록해야 할 경우, 반드시 마스킹 처리하여 원본 값을 식별할 수 없도록 합니다. JavaScript 
    // 이메일 마스킹 예시
'dev-eagles@example.com' -> 'de***@example.com'
    Java 
    // 전화번호 마스킹 예시
"010-1234-5678" -> "010-****-5678"

성능 고려사항

  • 조건부 로깅: DEBUG 레벨과 같이 상세한 로그는 로깅 프레임워크의 활성화 여부를 확인하는 조건문 안에서 실행하여, 운영 환경에서의 불필요한 연산을 방지합니다. (if (log.isDebugEnabled()) { ... })
  • 대용량 데이터 로깅 지양: 전체 목록이나 큰 객체를 직접 로그로 남기기보다, 데이터의 크기(size)나 핵심 식별자 등 요약된 정보를 기록합니다.

4.3. 프론트엔드 로그 컨벤션 (Vue.js)

환경별 로그 관리

자동 환경 감지 시스템:

  • 개발 환경: 자동으로 debug 레벨 활성화 (모든 로그 출력)
  • 운영 환경: 자동으로 error 레벨만 활성화 (에러만 출력)
  • 수동 오버라이드: 환경변수로 로그 레벨 직접 설정 가능

환경 변수 설정 (.env 파일)

Bash

# .env.development (개발 환경) - 자동으로 debug 레벨 적용
VITE_DEV_MODE=true
# VITE_LOG_LEVEL=debug  # 생략 가능 (자동 설정)

# .env.production (운영 환경) - 자동으로 error 레벨 적용  
VITE_DEV_MODE=false
# VITE_LOG_LEVEL=error  # 생략 가능 (자동 설정)

# .env.local (수동 오버라이드 예시)
VITE_LOG_LEVEL=info  # 자동 설정 무시하고 info 레벨 사용

실제 구현된 로거 특징

JavaScript

// 환경에 따른 자동 로그 레벨 결정 로직
const isDev = import.meta.env.DEV || import.meta.env.VITE_DEV_MODE === "true";
const autoLogLevel = isDev ? "debug" : "error";  // 자동 결정
const logLevel = import.meta.env.VITE_LOG_LEVEL || autoLogLevel;  // 수동 오버라이드 가능

// 성능 측정 메서드는 개발 모드에서만 동작
time(label) {
  if (isDev) {  // 개발 모드에서만 실행
    console.time(`${this.prefix} ${label}`);
  }
}

timeEnd(label) {
  if (isDev) {  // 개발 모드에서만 실행
    console.timeEnd(`${this.prefix} ${label}`);
  }
}

로거 사용법

JavaScript

import { logger, createLogger } from "@/plugins/logger";

// 기본 로거 사용
logger.debug("디버그 메시지", { data: "test" });  // 개발환경에서만 출력
logger.info("정보 메시지");                       // info 레벨 이상에서 출력
logger.warn("경고 메시지");                       // warn 레벨 이상에서 출력
logger.error("에러 메시지");                      // 모든 환경에서 출력

// 개발 전용 로그 (운영환경에서 자동 차단)
logger.devLog("개발 모드에서만 출력");

// 성능 측정 (개발환경에서만 동작)
logger.time("API 호출");   // 개발환경에서만 시작
// ... API 호출
logger.timeEnd("API 호출"); // 개발환경에서만 종료

// 모듈별 로거
const apiLogger = createLogger("API");
apiLogger.info("API 호출 시작"); // [INFO] [API] API 호출 시작

영역별 로그 작성 가이드

  • 컴포넌트: onMounted, onUnmounted 등 생명주기 시점에 debug 로그를, 사용자 주요 액션(폼 제출 등)에는 info 또는 error 로그를 기록합니다.
  • API 요청 (Axios Interceptor): 요청 시작/성공 시 debug 로그를, 요청 설정 오류나 응답 오류 발생 시 error 로그를 기록하여 모든 API 통신을 추적합니다.
  • 상태 관리 (Pinia): Store의 action 내부에서 비동기 작업의 시작, 성공, 실패 시점에 info와 error 로그를 기록합니다.
  • 라우터 (Vue Router): router.beforeEach, afterEach 훅을 사용해 페이지 이동을 info 레벨로 추적하고, router.onError를 통해 라우팅 오류를 error 레벨로 기록합니다.

에러 추적 및 리포팅

  • 전역 에러 핸들러: app.config.errorHandler (Vue), window.addEventListener('error') (전역), window.addEventListener('unhandledrejection') (Promise)를 설정하여 처리되지 않은 모든 에러를 포착하고 logger.error로 기록합니다.

성능 및 분석

  • 성능 모니터링: Performance API를 활용하여 페이지 로드 시간(LCP, FCP), API 응답 시간 등을 측정하고 info 또는 warn 로그로 기록합니다.
  • 로그 분석: 사용자 행동 패턴(버튼 클릭, 페이지 뷰 등)을 info 레벨로 기록하여 데이터 분석에 활용합니다.

4.4. 백엔드 로그 컨벤션 (Java/Spring)

환경별 설정 (Spring Profile & Logback)

현재 구현은 dev와 prod 프로파일별로 완전히 다른 로그 정책을 적용합니다:

  • dev 프로파일: 컬러풀한 콘솔 출력 + SQL 로그 별도 출력 + 모든 레벨 로그
  • prod 프로파일: 파일 출력만 + 비동기 로깅 + 에러 로그 분리 + SQL 로그 차단

1환경변수 설정

Bash

# 로그 디렉토리 설정 (기본값: ./logs)
export LOG_DIR=/var/log/deveagles

# 로그 파일명 설정 (기본값: deveagles)  
export LOG_FILE_NAME=myapp

# Spring Profile 설정
export SPRING_PROFILES_ACTIVE=dev  # 또는 prod

로그 파일 관리 정책

실제 파일 구조

logs/
├── deveagles.log                     # 현재 일반 로그
├── deveagles-error.log               # 현재 에러 로그  
├── deveagles.2024-01-15.1.log.gz     # 압축된 백업 파일
├── deveagles-error.2024-01-15.1.log.gz # 압축된 에러 백업 파일
└── ...

롤링 정책

  • 파일 크기: 개별 파일 최대 100MB
  • 보관 기간: 일반 로그 30일, 에러 로그 90일
  • 전체 용량: 일반 로그 10GB, 에러 로그 5GB
  • 압축: 롤링된 파일은 자동으로 .gz 압축

계층별 로그 작성 가이드

  • Controller 계층: API 요청의 시작과 최종 성공/실패를 info와 error 레벨로 기록합니다. 클라이언트로부터 받은 요청 데이터(주요 파라미터)를 명시합니다.
  • Service 계층: 비즈니스 로직의 핵심적인 분기(성공, 실패, 예외적 상황)를 info, warn, error 레벨로 기록합니다. debug 레벨로 메서드 진입/종료를 기록할 수 있습니다.
  • Repository 계층: DB 접근 시도 및 결과(성공/실패)를 debug 또는 error 레벨로 기록합니다.

SQL 로그 처리

  • 개발환경: SQL 로그를 [SQL] 태그로 별도 콘솔에 출력하여 가독성을 높입니다.
  • 운영환경: SQL 로그는 성능에 영향을 줄 수 있으므로 완전히 차단합니다.
  • 대상: Hibernate SQL, MyBatis 매퍼 등 모든 SQL 실행 로그

4.5. 모니터링 및 운영

로그 모니터링

  • 에러 알림: 운영 환경에서 ERROR 레벨 로그 발생 시 즉시 디스코드 팀 채널, 이메일 등으로 알림을 전송합니다.
  • 성능 모니터링: API 응답 시간, 페이지 로드 시간 등 성능 지표를 로그를 통해 추적합니다.
  • 로그 분석: ELK Stack 또는 클라우드 로그 분석 서비스(AWS CloudWatch)를 활용하여 로그를 중앙에서 수집, 검색, 시각화합니다.

보안 고려사항

  • 로그 파일 접근 권한 제한: 로그 파일에 대한 읽기 권한을 필요한 시스템/운영 계정으로만 제한합니다.
  • 로그 전송 암호화: 외부 로그 수집 서비스로 전송 시 HTTPS/TLS를 사용하여 데이터를 보호합니다.
  • 민감 정보 스캔: 정기적으로 로그 파일에서 민감 정보 누출 여부를 점검하는 프로세스를 도입합니다.

성능 최적화

  • 비동기 로깅: 운영환경에서 AsyncAppender를 사용하여 로깅으로 인한 애플리케이션 스레드의 블로킹을 최소화합니다.
  • 로그 레벨 동적 변경: 운영 중 이슈 발생 시, 서버 재시작 없이 로그 레벨을 동적으로 조정할 수 있는 기능을 구현합니다 (예: Spring Boot Actuator).
  • 로그 샘플링: 대용량 트래픽 발생 시, 모든 요청을 로깅하는 대신 특정 비율만 샘플링하여 기록하는 전략을 고려합니다.
