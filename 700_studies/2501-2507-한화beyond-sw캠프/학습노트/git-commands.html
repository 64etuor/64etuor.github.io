git commands

📚 Git 명령어 총정리

📌 개요

Git은 분산 버전 관리 시스템(Distributed Version Control System)으로, 소스 코드나 문서 등의 파일 변경 이력을 효율적으로 관리하고 협업을 돕습니다. 로컬 저장소(Local Repository)와 원격 저장소(Remote Repository)를 함께 사용함으로써 협업 시 충돌 관리, 히스토리 추적, 백업 및 복원이 용이합니다.

🧱 Git 기본 설정

  1. 사용자 정보 설정

    • Git은 모든 커밋에 작성자 정보를 담습니다.
    • 전역 설정(모든 저장소에 적용) 
      git config --global user.name "사용자이름"
git config --global user.email "사용자이메일@example.com"
    • 특정 저장소 내에서만 설정 
      git config user.name "사용자이름"
git config user.email "사용자이메일@example.com"

  2. 기본 에디터 설정

    • Git 명령어 실행 중 편집기 호출 시 사용할 에디터 지정 
      git config --global core.editor "vim"

  3. 환경 설정 확인

    • 현재 Git 설정값 확인 
      git config --list

🔧 저장소(Repository) 관련 명령어

  1. 저장소 초기화

    • 현재 디렉토리를 로컬 Git 저장소로 초기화합니다. 
      git init
    • 실행 후 .git 디렉토리가 생성되며, 여기에는 Git 히스토리와 설정파일이 들어있습니다.

  2. 원격 저장소 복제 (Clone)

    • 원격 저장소(remote repository)를 로컬로 복제 
      git clone <원격저장소URL> [로컬디렉토리명]
    • 옵션
      • --depth 1: 히스토리를 가장 최근 커밋까지만 가져오는 "shallow clone"
      • 예제 
        git clone https://github.com/username/repo.git

🗂 파일 상태 관리 명령어

  1. 상태 확인 (Status)

    • 현재 작업 디렉토리의 변경 사항 추적 상황을 표시합니다. 
      git status
    • 추적되지 않은 파일, 수정되었으나 스테이징하지 않은 파일, 스테이징된 파일을 구분해서 보여줍니다.

  2. 변경사항 비교 (Diff)

    • 작업 디렉토리와 스테이징 영역의 차이 비교 
      git diff
    • 스테이징 영역과 마지막 커밋의 차이 비교 
      git diff --staged
    • 특정 파일 변경사항 비교 
      git diff <파일명>

  3. 파일 추가 (Add)

    • 작업 디렉토리의 변경 사항(추가/수정/삭제)을 스테이징 영역으로 올립니다. 
      git add <파일명 또는 디렉토리명>
    • 모든 변경사항 스테이징 
      git add .
    • 특정 확장자 파일 전체 스테이징 
      git add *.txt

  4. 커밋 (Commit)

    • 스테이징 영역에 있는 변경사항을 기록(커밋)합니다. 
      git commit -m "커밋 메시지"
    • 마지막 커밋 메시지 수정 
      git commit --amend -m "수정된 커밋 메시지"
    • 수정된 커밋은 새로운 커밋 해시를 가지며, 원격에 이미 푸시된 경우 주의가 필요합니다.2

🍃 브랜치(Branch) 관리 명령어

  1. 브랜치 조회

    git branch
    • 현재 로컬에 존재하는 브랜치를 보여주며, 현재 체크아웃된 브랜치는 앞에 * 표시가 있습니다.

  2. 브랜치 생성

    git branch <브랜치명>
    • 현재 HEAD 기준으로 새로운 브랜치를 생성합니다.

  3. 브랜치 변경(체크아웃, Checkout)

    git checkout <브랜치명>
    • 다른 브랜치로 이동(HEAD 포인터를 해당 브랜치로 변경)합니다.
    • 새 브랜치 생성과 동시에 체크아웃 
      git checkout -b <브랜치명>

  4. 브랜치 삭제

    git branch -d <브랜치명>
    • 해당 브랜치를 삭제합니다.
    • 병합되지 않은 변경사항이 있으면 안전을 위해 -D 옵션을 사용해야 합니다.

  5. 브랜치 병합(Merge)

    git merge <합칠브랜치명>
    • 현재 체크아웃된 브랜치에 <합칠브랜치명>의 변경사항을 병합합니다.

🔀 Merge 전략 & Rebase

  1. Merge

    • 두 브랜치의 히스토리를 통합하는 명령어입니다.
    • 별도의 병합 커밋(Merge commit)을 생성하는 경우, 히스토리가 분기 후 다시 합쳐지는 형태를 남깁니다.

  2. Rebase

    git rebase <기준브랜치>
    • 히스토리를 재정렬하여, 마치 한 줄로 이어진 것처럼 만듭니다.
    • 병합 커밋 없이 깔끔한 히스토리를 유지하지만, 이미 공유된 커밋을 재베이스하는 것은 충돌 및 혼선을 야기할 수 있으므로 주의가 필요합니다.3

🌐 원격 저장소(Remote) 관련 명령어

  1. 원격 저장소 추가 (Remote Add)

    git remote add <원격이름> <원격URL>
    • 보통 origin이라는 이름으로 원격 저장소를 추가합니다.

  2. 원격 저장소 조회 (Remote -v)

    git remote -v
    • 현재 로컬 저장소와 연결된 원격 저장소 목록 및 URL을 확인합니다.

  3. 원격 저장소로 푸시 (Push)

    git push <원격이름> <브랜치명>
    • 로컬의 변경사항을 원격 저장소로 업로드합니다.
    • 예: 
      git push origin main

  4. 원격 저장소로부터 가져오기 (Fetch, Pull)

    • 원격 저장소의 변경사항을 로컬에 가져오기 
      git fetch <원격이름>
      • 이 명령어는 원격의 변경사항을 로컬 저장소의 원격 추적 브랜치(remote tracking branch)에 반영하지만, 현재 작업 브랜치에는 합치지 않습니다.
    • 원격 변경사항을 로컬 브랜치에 병합(Fetch + Merge) 
      git pull <원격이름> <브랜치명>
      • 예: 
        git pull origin main
      • git pull은 내부적으로 git fetch 후 git merge를 자동으로 수행합니다.
    • 충돌(Conflict) 발생 시 충돌 파일을 수정한 뒤 스테이징, 커밋을 통해 해결합니다.

  5. 원격 브랜치 추적하기

    git checkout -b <로컬브랜치명> <원격이름>/<원격브랜치명>
    • 원격 브랜치를 로컬 브랜치로 체크아웃하여 추적 브랜치를 생성합니다.

💾 로그(Log) & 히스토리 관리

  1. 로그 조회 (Log)

    git log
    • 커밋 히스토리를 표시합니다.
    • 옵션
      • --oneline: 한 줄로 간단히 표시
      • --graph: 그래픽 형태(텍스트 기반)로 브랜치 구조 표시
      • --decorate: 레퍼런스(브랜치, 태그) 표시
      • 예: 
        git log --oneline --graph --decorate

  2. 특정 파일 히스토리 조회

    git log -- <파일명>
    • 해당 파일의 변경 히스토리만 추적합니다.

  3. 커밋간 차이 비교

    git diff <커밋해시1> <커밋해시2>
    • 두 특정 커밋 간의 변경사항을 비교합니다.

♻️ 리셋(Reset), 리버트(Revert), 체크아웃(Checkout)으로 히스토리 변경

  1. Reset

    git reset <옵션> <커밋해시>
    • 로컬 히스토리를 특정 커밋 시점으로 되돌립니다.
    • 옵션
      • --soft: 워킹 디렉토리와 스테이징 영역은 유지, HEAD만 옮김
      • --mixed: 스테이징 영역은 초기화, 워킹 디렉토리는 유지
      • --hard: 워킹 디렉토리, 스테이징 영역 모두 해당 커밋 상태로 돌아감
    • 주의: 이미 원격에 푸시한 커밋을 reset하는 것은 협업에 혼선을 줄 수 있습니다.

  2. Revert

    git revert <커밋해시>
    • 지정한 커밋을 취소하는 "새로운 커밋"을 추가합니다.
    • 커밋해시는 7자만 적어도 가능!
    • 기존 히스토리는 유지하면서, 특정 변경사항을 되돌리는 안전한 방식입니다.4

  3. Checkout (커밋 혹은 파일 상태로 돌아가기)

    git checkout <커밋해시> -- <파일명>
    • 특정 커밋 시점의 파일 상태로 돌아가고, 그 상태를 현재 워킹 디렉토리에 반영합니다.
    • 이후 git add와 git commit으로 변경사항을 반영할 수 있습니다.

📦 Stash (임시 저장)

  1. Stash 생성

    git stash
    • 현재 워킹 디렉토리의 변경사항을 스택처럼 쌓아두어 임시 보관합니다.
    • 워킹 디렉토리를 깨끗한 상태로 만들며, 추후 다시 복원할 수 있습니다.

  2. Stash 목록 확인

    git stash list
    • 쌓여있는 stash들의 목록을 표시합니다.

  3. Stash 복원 (적용)

    git stash apply [stash@{n}]
    • stash에 임시 보관했던 변경사항을 다시 워킹 디렉토리에 반영합니다.
    • stash@{n}을 지정하지 않으면 가장 최근 stash를 적용합니다.

  4. Stash 삭제

    git stash drop [stash@{n}]
    • 지정한 stash 항목을 삭제합니다.

  5. Stash 팝

    git stash pop
    • 가장 최근 stash를 적용하고 해당 stash를 목록에서 제거합니다.

🏷 태그(Tag)

  1. 태그 생성

    git tag <태그명> [커밋해시]
    • 현재 커밋(HEAD)이나 지정한 커밋에 태그를 추가합니다.
    • 예: 
      git tag v1.0.0

  2. 태그 목록 확인

    git tag

  3. 태그 푸시

    git push <원격이름> <태그명>
    • 특정 태그만 원격으로 푸시합니다.
    • 모든 태그 푸시 
      git push <원격이름> --tags

  4. 태그 삭제

    git tag -d <태그명>
    • 로컬 태그 삭제 후 원격 태그도 삭제하려면 
      git push <원격이름> :refs/tags/<태그명>

🛠 기타 유용한 명령어

  1. 브랜치 리네임(Rename)

    git branch -m <기존브랜치명> <새브랜치명>

  2. 커밋 메시지 변경 (Amend)

    git commit --amend -m "새로운 메시지"
    • 스테이징된 변경사항을 마지막 커밋에 포함시키고 메시지도 수정합니다.

  3. 원격 브랜치 삭제

    git push <원격이름> --delete <브랜치명>
    • 원격 저장소에서 특정 브랜치를 삭제합니다.

  4. Git 무시 파일(.gitignore)

    • .gitignore 파일을 생성하여 Git이 추적하지 않을 파일 패턴을 지정할 수 있습니다.
    • 예: *.log, node_modules/ 등을 .gitignore에 추가하면 해당 파일/디렉토리는 Git이 추적하지 않습니다.

Footnotes

  1. Chacon, S., & Straub, B. (2014). Pro Git. Apress. ↩
  2. 이미 원격에 푸시한 커밋을 변경하는 것은 git push --force와 함께 사용할 수 있으나 협업 시 주의가 필요합니다. ↩
  3. rebase는 팀원들과의 협의 후 사용하는 것이 좋습니다. 이미 공유된 히스토리를 재작성하면 충돌과 혼선을 야기할 수 있습니다. ↩
  4. revert는 기존 히스토리를 파괴하지 않기 때문에 협업 시 안전한 선택입니다. ↩
