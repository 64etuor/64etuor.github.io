Tag System

Tag System

Summary

This document defines the Obsidian vault's tag system.
Folders, properties, tags, and links form 4 layers, each with a distinct role, maximizing search efficiency and knowledge connectivity.
Design principle: MECE (Mutually Exclusive, Collectively Exhaustive) — dimensions don't overlap and collectively cover everything.

1. Design Principles

1.1 Four-Layer Separation

Layer Role Link Strength Example
Folder Document's structural location (where it lives) Medium 700_Studies/2501-2502 Challenge/
Property Document's metadata (what it is) - type: literature, status: done
Tag Document's topic/theme (what it's about) Loose #data/sql, #src/lecture
Link Contextual relationship between documents (why they connect) Strong [[Normalization Theory]], [[SQL JOIN Guide]]
Key Rules
  • Don't repeat with tags what the folder already says.
    • #Books tag inside 200_Books/ folder -> unnecessary
  • Structured values go in properties; free classification goes in tags.
    • status: done (property) -> #done (tag)
  • Tags express only "what topic is this note about."

1.2 Tag Formatting Rules

Rule Description Example
English only AI auto-tagging compatible, global standard #data/sql ✅ / #데이터/SQL
Lowercase Prevents case confusion #dev/java ✅ / #Dev/Java
Nested tags #parent/child hierarchy #ai/genai, #dev/python
Max 2 levels Avoid 3+ levels #data/sql ✅ / #data/sql/join
3-8 per note Prevent over-tagging domain 1-3 + source 1 + action 0-1
Frontmatter only Use tags: property, not inline tags YAML array format

2. Property Standard (Frontmatter)

Properties vs Tags

Properties are structured fields where you select from predefined values.
Optimized for Dataview queries; they serve as the note's "ID card."

2.1 Common Properties

Required (7 fields):

---
title: "Document Title"
created: 2026-03-23
modified: 2026-03-23
type: permanent
status: active
author: Yangha Park
tags: []
---

Optional (add as needed):

description: ""   # One-line summary (~100 chars) for AI indexing & Dataview listing
source: ""        # URL or origin of the content
aliases: []       # Alternative names for Obsidian search
description 활용
  • AI가 노트를 탐색할 때 본문 파싱 없이 관련성을 판단하는 첫 번째 단서
  • Dataview 테이블에서 description 컬럼으로 노트 목록 한눈에 파악
  • 한 줄(~100자) 이내로 작성. 비워 둬도 무방 — AI 처리 시 자동 생성 가능

2.2 type Values

type ≠ folder

type indicates the nature of the document (how it was written), not necessarily matching its folder location.
Example: a permanent note can exist in 700_Studies/, and a reference can exist in 300_Projects/.
The "typical locations" below are guidelines, not rules.

Value Description Purpose Typical location (reference only)
fleeting Temporary memo Quick capture, organize later 100_Inbox/
literature Literature note Note summarizing an external source — others' ideas rewritten in own words 200_Books/, 700_Studies/
permanent Permanent note Self-authored knowledge — synthesized from multiple sources or original insight 600_Tech Stack/, anywhere
reference Reference material Preserved external source (minimal processing) 500_Resources/, Clippings/
project Project document Project deliverables, plans, technical docs 300_Projects/
log Log/journal Diary, retrospective, meeting minutes 900_Writings/Diaries/
output Creative output Finished work: essays, creative writing 900_Writings/Essays/
Difference between type and src/* tags

These answer different questions:

Question Answer type Example
type How was this note written? (format) Summarizing others vs own thinking literature, permanent
src/* What medium was the source? (channel) Book vs lecture vs video vs paper src/book, src/lecture

The same literature note can be src/book (reading) or src/lecture (lecture).
The two dimensions are orthogonal — combine them for precise classification like "lecture-based literature note" (type: literature + #src/lecture).

2.3 status Values

Value Description When to use
inbox Unprocessed Just created, not yet classified
draft Draft Started writing, incomplete
active Active Currently in progress or frequently referenced
review Needs review Content needs verification or update
done Done Writing complete, no further edits needed
archive Archived Inactive, reference only when needed

3. MECE Tag System

Tags are composed of 3 independent dimensions. Each dimension is orthogonal; combine them as needed.

Tag = Domain(A) + Source(B) + Action(C)
e.g.: #dev/python + #src/book + #action/practice

3.1 Dimension A — Domain

"What field is this note about?"

Usage

Assign 1-3 domain tags per note.
If the parent tag is sufficient, don't use a subtag.
e.g.: Python in general -> #dev/python / Software development in general -> #dev

3.2 Dimension B — Source

"What is the origin of this note?"

Tag Description Example
src/book Book Reading notes, reviews
src/article Web article, blog Clippings, scraps
src/lecture Lecture, class Bootcamp lecture notes
src/video Video YouTube, talks
src/course Course, program Bootcamp, challenge, reading group
src/paper Academic paper Research papers
src/doc Official documentation API docs, tech references
src/experience Direct experience Project work, practice, work tasks
Assign only 1 source tag per note. Pick the primary source.

3.3 Dimension C — Action

"Is there something to do with this note?" (optional)

Tag Description When to use
action/todo Has pending tasks Incomplete organization, needs writing
action/review Needs review Study note due for review
action/practice Needs practice Code exercises, problem solving
action/followup Needs follow-up Further research, contact needed
Action tags are temporary.

Remove them once the action is complete.
For persistent states, use the status property instead.

  • Update frontmatter template (tags -> YAML array, add type/status properties)

Phase 2: Gradual migration (existing notes)

  • Unify property keys: tag -> tags, modification date -> modified
  • Convert old tags to new system on each edit (gradual, not batch)
