MECE 태그 체계 설계

MECE 태그 체계 설계

요약

이 문서는 Obsidian 볼트의 MECE(Mutually Exclusive, Collectively Exhaustive) 태그 체계를 정의합니다.
폴더·프로퍼티·태그·링크 4계층이 각자의 역할을 갖도록 설계하여, 검색 효율과 지식 연결성을 극대화합니다.

1. 설계 원칙

1.1 4계층 분리 원칙

계층 역할 연결 강도 예시
폴더 문서의 구조적 위치 (어디에 사는가) 중간 700_Studies/2501-2502 한달 노트 챌린지/
프로퍼티 문서의 메타데이터 (무엇인가) - type: literature, status: done
태그 문서의 주제·테마 (무엇에 관한 것인가) 느슨 #data/sql, #src/lecture
링크 문서 간 맥락적 관계 (왜 연결되는가) 강함 [[정규화 이론]], [[SQL JOIN 정리]]
핵심 규칙
  • 폴더가 이미 말해주는 것을 태그로 반복하지 않는다.
    • 200_Books 폴더 안에서 #Books 태그 → 불필요
  • 구조화된 값은 프로퍼티로, 자유로운 분류는 태그로.
    • status: done (프로퍼티) ✅ / #done (태그) ❌
  • 태그는 "이 노트가 어떤 주제에 관한 것인지"만 표현한다.

1.2 태그 작성 규칙

규칙 설명 예시
영어 통일 AI 자동 태깅 호환, 글로벌 표준 #data/sql ✅ / #데이터/SQL ❌
소문자 대소문자 혼란 방지 #dev/java ✅ / #Dev/Java ❌
중첩 태그 #상위/하위 형태로 계층화 #ai/genai, #dev/python
최대 2단계 3단계 이상은 지양 #data/sql ✅ / #data/sql/join ❌
3~8개/노트 과다 태깅 방지, 핵심만 도메인 1~3 + 출처 1 + 액션 0~1
프론트매터 배치 인라인 태그 대신 tags: 프로퍼티 사용 YAML 배열 형태

2. 프로퍼티 표준 (Frontmatter)

프로퍼티 vs 태그

프로퍼티는 정해진 값 중 하나를 선택하는 구조화된 필드입니다.
Dataview 쿼리에 최적화되며, 노트의 "신분증" 역할을 합니다.

2.2 type 값 정의

type ≠ 폴더

type은 문서의 성격(어떻게 작성했는가)을 나타내며, 폴더 위치와 반드시 1:1 대응하지 않습니다.
예: 700_Studies/ 안에 permanent 노트가 있을 수 있고, 300_Projects/ 안에 reference가 있을 수 있습니다.
아래 "주로 발견되는 위치"는 참고용이며, 폴더 배치를 강제하지 않습니다.

값 설명 용도 주로 발견되는 위치 (참고용)
fleeting 일시적 메모 빠른 기록, 나중에 정리 +/
literature 문헌 노트 외부 원천을 바탕으로 정리한 노트 — 남의 생각을 자신의 표현으로 요약·재구성 200_Books/, 700_Studies/
permanent 영구 노트 자신의 사고로 만든 지식 — 여러 원천을 종합하거나 독자적 통찰을 기록 600_Tech Stack/, 어디서든
reference 참고 자료 외부 자료 원문 보존 (가공 최소화) 500_Resources/, Clippings/
project 프로젝트 문서 프로젝트 산출물·기획·기술 문서 300_Projects/
log 일지·기록 일기, 회고, 회의록 900_Writings/Diaries/
output 산출물 에세이, 창작물 등 완성된 결과물 900_Writings/Essays/
type과 src/* 태그의 차이

이 두 가지는 서로 다른 질문에 답합니다.

질문 답의 종류 예시
type 이 노트를 어떻게 썼는가? (작성 형식) 남의 생각 정리 vs 내 생각 정리 literature, permanent
src/* 원천의 매체가 무엇인가? (정보 채널) 책 vs 강의 vs 영상 vs 논문 src/book, src/lecture

같은 literature 노트라도 src/book(독서)일 수 있고 src/lecture(강의)일 수 있습니다.
두 차원은 직교하므로, 조합하여 "강의 기반 문헌 노트"(type: literature + #src/lecture)처럼 정밀하게 분류할 수 있습니다.

2.3 status 값 정의

값 설명 사용 시점
inbox 미정리 방금 생성, 아직 분류 안 됨
draft 초안 작성 시작, 미완성
active 활성 현재 작업 중 또는 자주 참조
review 검토 필요 내용 확인·업데이트 필요
done 완료 작성 완료, 더 이상 수정 불요
archive 보관 비활성, 필요 시 참조

3. MECE 태그 체계

태그는 3개의 독립 차원으로 구성됩니다. 각 차원은 서로 직교(orthogonal)하며, 조합하여 사용합니다.

태그 = 도메인(A) + 출처(B) + 액션(C)
예: #dev/python + #src/book + #action/practice

3.1 차원 A — 도메인 (Domain)

"이 노트는 어떤 분야에 관한 것인가?"

사용법

한 노트에 1~3개의 도메인 태그를 부여합니다.
상위 태그만으로 충분하면 하위 태그를 쓰지 않습니다.
예: 파이썬 전반 → #dev/python / 개발 일반론 → #dev

dev/ — 소프트웨어 개발

태그 범위
dev 개발 일반 (상위 태그 단독 사용 가능)
dev/java Java 언어, JDK, Spring 포함
dev/python Python 언어, Flask, Django 포함
dev/js JavaScript, TypeScript, Vue, React 포함
dev/sql SQL 구문, 쿼리 작성 (DB 설계는 dev/db)
dev/db 데이터베이스 설계, 정규화, 인덱싱
dev/oop 객체지향 설계, SOLID, 디자인 패턴
dev/web 웹 개발 일반, HTML/CSS, API
dev/mobile 모바일 앱 개발, Flutter, Android
dev/test 테스트, QA, 디버깅

data/ — 데이터 & 분석

태그 범위
data 데이터 일반
data/analysis 데이터 분석 방법론, EDA, 코호트 분석
data/stats 통계학, 확률, 가설검정
data/viz 시각화, 대시보드, Power BI, plotly
data/ml 머신러닝, 딥러닝, 모델링
data/engineering 데이터 엔지니어링, ETL, BigQuery, 파이프라인
data/preprocessing 데이터 전처리, 클리닝, 크롤링

ai/ — 인공지능

태그 범위
ai AI 일반
ai/genai 생성형 AI, LLM, ChatGPT, Claude
ai/prompt 프롬프트 엔지니어링
ai/agent AI 에이전트, 자율 시스템
ai/mcp MCP(Model Context Protocol), 도구 연동

infra/ — 인프라 & DevOps

태그 범위
infra 인프라 일반
infra/docker Docker, 컨테이너, 가상화
infra/aws AWS 서비스 전반
infra/linux Linux, 셸 스크립트
infra/git Git, GitHub, 버전관리
infra/cicd CI/CD, GitHub Actions
infra/obsidian Obsidian 설정, 플러그인, 볼트 관리

biz/ — 비즈니스 & 경영

태그 범위
biz 비즈니스 일반
biz/marketing 마케팅, CRM, 고객 분석, AARRR
biz/strategy 경영 전략, BCG 매트릭스, 사업 기획
biz/pm 프로젝트 관리, PMBOK, Agile, Scrum
biz/hr 인사관리, 역량 평가, 조직 관리
biz/startup 창업, 정부지원사업, 사업계획서

career/ — 커리어 & 전문성

태그 범위
career 커리어 일반
career/resume 이력서, 자기소개서, 지원서
career/interview 면접 준비, 기술 면접
career/cert 자격증 (SQLD, 정처기 등)
career/portfolio 포트폴리오 구성

lang/ — 언어 학습

태그 범위
lang 언어 학습 일반
lang/english 영어 학습, 회화, 문법

write/ — 글쓰기 & 문학

태그 범위
write 글쓰기 일반
write/essay 에세이, 칼럼, 서평
write/poem 시, 운문
write/fiction 소설, 단편, 창작
write/diary 일기, 개인 기록
write/review 독서 감상, 회고록

finance/ — 재무 & 투자

태그 범위
finance 재무 일반
finance/tax 세금, 종합소득세
finance/invest 투자, 부동산, 주식
finance/insurance 보험, 청구

life/ — 생활 & 자기관리

태그 범위
life 생활 일반
life/habit 습관, 루틴, 자기계발
life/health 건강, 운동, 심리
life/hobby 취미, 맛집, 사진(미러리스) 등

3.2 차원 B — 출처 (Source)

"이 노트의 원천은 무엇인가?"

태그 설명 예시
src/book 도서 독서 노트, 서평
src/article 웹 아티클, 블로그 Clippings, 스크랩
src/lecture 강의, 수업 부트캠프 강의 노트
src/video 영상 유튜브, 강연 영상
src/course 코스, 프로그램 부트캠프, 챌린지, 독서모임
src/paper 논문 학술 논문, 연구 자료
src/doc 공식 문서 API 문서, 기술 레퍼런스
src/experience 직접 경험 프로젝트 경험, 실습, 업무
한 노트에 출처 태그는 1개만 부여합니다.

가장 주된 출처를 선택하세요.

3.3 차원 C — 액션 (Action)

"이 노트에 대해 해야 할 일이 있는가?" (선택적)

태그 설명 사용 시점
action/todo 할 일이 있음 정리 미완, 추가 작성 필요
action/review 복습·검토 필요 학습 노트 재검토
action/practice 실습 필요 코드 실습, 문제 풀이
action/followup 후속 조치 추가 조사, 연락 필요
액션 태그는 임시 태그입니다.

해당 액션을 완료하면 반드시 제거합니다.
장기적으로 남아 있어야 하는 상태는 status 프로퍼티를 사용하세요.

5.6 기타

8. AI 자동 태깅 프롬프트 가이드

AI 태깅 프롬프트 (업데이트용)
You are a tag classification expert. Analyze the document and assign tags following these rules:

## Tag System (3 dimensions, all in English lowercase)

### Dimension A - Domain (1~3 tags, pick the most relevant):
dev/, data/, ai/, infra/, biz/, career/, lang/, write/, finance/, life/

Subtags:
- dev: java, python, js, sql, db, oop, web, mobile, test
- data: analysis, stats, viz, ml, engineering, preprocessing
- ai: genai, prompt, agent, mcp
- infra: docker, aws, linux, git, cicd, obsidian
- biz: marketing, strategy, pm, hr, startup
- career: resume, interview, cert, portfolio
- lang: english
- write: essay, poem, fiction, diary, review
- finance: tax, invest, insurance
- life: habit, health, hobby

### Dimension B - Source (exactly 1 tag):
src/book, src/article, src/lecture, src/video, src/course, src/paper, src/doc, src/experience

### Dimension C - Action (0~1 tag, only if applicable):
action/todo, action/review, action/practice, action/followup

## Rules:
- Use ONLY tags from the list above. Do not invent new tags.
- Use the most specific subtag that applies (e.g., dev/python not just dev).
- Output format: comma-separated tags without # prefix.
- Total tags: 3~8 per note.

9. 마이그레이션 로드맵

Phase 1: 즉시 적용 (새 노트)

  • 이 문서의 태그 체계를 새 노트에 즉시 적용
  • 프론트매터 표준 템플릿 업데이트 (tags → YAML 배열, type/status 프로퍼티 추가)
  • AI 태깅 템플릿 프롬프트 업데이트

Phase 2: 점진적 마이그레이션 (기존 노트)

  • 프로퍼티 키 통일: tag → tags, modification date → modified
  • 기존 태그를 편집 시마다 새 체계로 변환 (일괄 변환보다 점진적 접근)
  • Tag Wrangler 플러그인으로 대량 리네임 가능한 것부터 처리

Phase 3: 자동화 & 검증

  • Dataview 대시보드로 미분류 노트 모니터링
  • 비표준 태그 탐지 쿼리 작성
  • 주간 리뷰 시 태그 정합성 점검
