Governance and Tech Debt Review: Definitive Recommendation

Date: 2026-03-12

Corpus reviewed: 27 markdown documents in /Users/davidpark/Projects/Mirrorless_Mono/docs/plans/

Initiatives represented: 13 workstreams spanning product UX, platform architecture, incident response, and reliability hardening

External guidance consulted: OpenAI, Microsoft, Google ADK, AWS, NIST AI RMF, Atlassian, Thoughtworks, Microsoft Well-Architected

Executive Recommendation

Do not keep governance review and tech debt review as one undifferentiated process.

Also do not split them into fully separate document systems or separate boards.

The best fit for this corpus is a unified intake with split review tracks:

  1. Feature / governance track -- for user-facing behavior, UX, and product scope decisions

  2. Reliability / tech debt track -- for defects, regressions, operational safety, and engineering friction

  3. Hybrid architecture track -- for work that changes both product behavior and system reliability

This recommendation is driven by three converging lines of evidence:

  • The shape of the documents themselves (Section 1): the plans share a common structure but differ in triggers, reviewers, urgency, and evidence expectations

  • Eight concrete governance gaps (Section 2): the corpus already shows where undifferentiated review fails

  • External best practices (Section 3): industry guidance from NIST, AWS, Microsoft, Thoughtworks, and others confirms that governance must be cross-cutting but review must be fit-for-purpose

The concrete answer: divide the review tracks, not the documentation system.

1. What the Corpus Shows

Across 27 documents, the work clusters into three practical groups.

A. Feature-heavy / governance-heavy initiatives (5)

Primarily about user-facing behavior, UX choices, or product-level scope decisions:

  • header-profile-badge-* -- UI placement, animation, visibility

  • experimental-feature-flags-* -- user feedback-driven (MIR-KSK-294), approved status, UI placement decisions

  • retake-warning-popup-* -- preventing accidental user data loss through confirmation UX

  • touch-control-drawer-* -- touch targets, visual specs, guarded interactions (long-press session completion)

  • face-blur-image-* -- privacy-sensitive media processing with user-visible output

These need reviewers who can judge product fit, UX consistency, and scope discipline.

B. Reliability / tech debt initiatives (5)

Primarily about defects, regressions, operational safety, or engineering friction:

  • crm-fastapi-live-regression-* -- audit-oriented package: approved constraints, timebox, verification commands, execution log

  • crash-recovery-hardening-* -- 22 potential crash recovery vulnerabilities, atomic writes, re-queuing, startup ordering

  • cicd-redesign-* -- broken CI topology and deploy gating

  • sharp-thumbnail-* -- silent correctness bug

  • lut-pipeline-race-condition-* -- surgical fix for two root causes

These need reviewers who can judge failure modes, rollback posture, regression coverage, and operational impact.

C. Hybrid architecture initiatives (3)

The reason a hard split is the wrong answer:

  • kiosk-auth-offline-* -- security policy + offline resilience + queue architecture + token scope + user-visible QR login

  • qr-login-persistence-* -- reliability gap (refresh/crash state loss) that shows up as a user-facing badge/profile regression

  • auto-update-architecture-* -- operational safety + rollout policy + rollback strategy + admin-facing UI/API surface

These need both governance-style and reliability-style review. If the two tracks were fully separated, these initiatives would either be reviewed twice with duplicate overhead or reviewed once through the wrong lens.

Why one process is not enough

| Dimension | Feature track | Reliability track |

|-----------|--------------|-------------------|

| Trigger | Proactive: roadmap, user feedback, planned extension | Reactive: regression, code review finding, CI failure, production defect |

| Reviewer fit | Product/UX reviewer: animation, affordances, scope | Infra/reliability reviewer: CI topology, rollback flows, race conditions |

| Urgency | Planned cadence, paired design/plan cycles | May need immediate action (e.g., CRM regression 12-hour plan) |

| Evidence | Smoke tests, UX validation | Root-cause analysis, verification commands, execution logs, schema notes |

A single review checklist either under-specifies evidence for debt work or over-burdens routine feature work.

Why a full split also fails

  • Artifact shape is shared. Most initiatives follow: design doc, implementation plan, file-change scope, verification expectations. Breaking this into separate repositories or templates destroys useful consistency.

  • Hybrid work is common, not exceptional. Three of the most architecturally important initiatives are hybrid -- this is a real category, not edge-case noise.

  • Low-friction documentation is a strength. The corpus is co-located in one directory with consistent naming. Splitting too hard adds admin work faster than it adds clarity.

2. Eight Governance Gaps as Evidence

The following gaps were identified by cataloging all 27 documents against governance best practices. They provide the concrete "why" for differentiating review tracks and adding operating mechanisms.

Gap 1: Ownership is mostly implicit

Most docs do not name a responsible team, DRI, approver, or operational owner. The few owner references are implementation settings, not accountability metadata.

Implication: Without explicit ownership, review assignments default to whoever is available rather than whoever is qualified. This is especially costly for hybrid work where both product and reliability perspectives are needed.

Gap 2: Documentation standards are inconsistent

Some design docs include Date, Status, Issue, Target, or explicit constraints; many do not. The corpus mixes design docs, plans, execution logs, and schema notes without a required template or common metadata standard.

Implication: Inconsistent metadata makes track routing manual and error-prone. A reviewer cannot quickly determine whether a document needs feature review, reliability review, or both.

Gap 3: Rollout policy is underdefined

Outside the auto-update work, most features describe implementation but not activation policy, blast-radius control, phased rollout, canary strategy, or approval gates. Even the auto-update design chooses full simultaneous rollout.

Implication: This is the gap where governance and tech debt most visibly collide. A feature with no rollout policy creates tech debt (cleanup, rollback complexity) when it fails.

Gap 4: Auditability is uneven

The CRM regression documents are the strongest example of audit trail discipline (execution logs, schema notes, verification commands), but that standard is not carried across most other feature docs.

Implication: Reliability review needs audit artifacts. Feature review needs scope artifacts. A unified process that does not distinguish these produces inconsistent quality.

Gap 5: Observability governance is thin

Logging appears in several plans, but there is little definition of metrics, alert thresholds, dashboards, incident response, or runbooks. Recovery behavior is described more often than operational response.

Implication: Tech debt review should explicitly assess observability posture. Feature review should not be burdened with this.

Gap 6: Security and privacy process is under-specified

Kiosk auth, QR persistence, and face blur add sensitive token/media handling, but the docs rarely define key-rotation policy, encryption-at-rest expectations, incident procedures, consent/privacy classification, or deletion SLAs.

Implication: Security review is a cross-cutting concern that should be triggered by track routing metadata, not left to chance.

Gap 7: Data lifecycle controls are local, not policy-driven

Some docs mention cleanup, TTLs, or file deletion, but there is no consistent cross-cutting policy for retention windows, secure deletion, orphan cleanup, or backfill/migration validation.

Implication: Data lifecycle gaps create tech debt that accumulates invisibly until it causes an incident. The reliability track should explicitly assess this.

Gap 8: Experiment governance is incomplete

Experimental feature flags get a registry and UI, but there is no documented graduation, sunset, approval, or telemetry policy for deciding whether a flag stays experimental, becomes default, or is removed.

Implication: Flags without graduation policy are tech debt by definition. Feature review should cover the experiment lifecycle; reliability review should assess cleanup cost.

3. External Best Practices Grounding

Each element of the recommendation is grounded in published industry guidance.

Cross-cutting governance (NIST)

NIST AI RMF defines Govern as a cross-cutting function infused throughout risk management, not a one-time checkpoint. (NIST AI RMF Core)

Application: Governance metadata (track routing, ownership, verification type) should be embedded in every design doc, not maintained in a separate system.

Centralized governance with federation path (AWS)

AWS describes a centralized generative AI operating model as a hub where reusable components, orchestration, and monitoring live in one place, promoting consistency and control. It also describes a federated model as decentralized development with centralized integration and monitoring. (AWS Bedrock operating models)

Application: Start centralized (one review process, one document location, one template family). Evolve to federated only when multiple domain teams genuinely need autonomy.

Architecture review board with debt ownership (AWS, Microsoft)

AWS recommends an ARB with executive sponsorship, defined stakeholders, and a single source of truth for guidance. (AWS ARB guidance) Microsoft Well-Architected recommends version-controlled ADRs and treating tech debt as a regularly recurring backlog item. (Microsoft Well-Architected)

Application: One lightweight architecture and debt council with two jobs: (1) review important architecture and policy changes, (2) continuously prioritize debt items created by those decisions.

Debt is multi-dimensional (Thoughtworks)

Thoughtworks emphasizes that debt is not only code debt; it includes architectural, technical decision, test, documentation, infrastructure, process, culture, UX, and security debt. (Thoughtworks technical debt guide)
