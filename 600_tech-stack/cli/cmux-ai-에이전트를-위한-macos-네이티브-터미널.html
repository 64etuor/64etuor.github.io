cmux - AI 에이전트를 위한 macOS 네이티브 터미널

한 줄 요약

cmux는 AI 코딩 에이전트를 병렬 실행·모니터링·제어하기 위해 설계된 macOS 네이티브 터미널로, 내장 브라우저·사이드바 알림·Unix Socket API를 제공한다.

도입: 무엇을, 왜

Claude Code, Codex, Gemini CLI 등 AI 에이전트를 터미널에서 병렬로 돌리면 tmux만으로는 한계에 부딪힌다. 브라우저를 따로 띄워야 하고, 에이전트 간 상태 공유가 안 되고, 사이드바에 진행률을 표시할 방법이 없다.

cmux는 manaflow-ai가 개발한 AI 에이전트 전용 macOS 네이티브 터미널이다. Ghostty의 렌더링 엔진(libghostty)을 사용하며 Swift + AppKit으로 구축되었다. 2026년 1월 최초 공개 후 2월 정식 출시, v0.63.1(2026-03-28 기준), GitHub 12.6k+ Stars.

핵심 철학은 "a primitive, not a solution" — 완성된 솔루션이 아니라 read-screen, send, notify 같은 원자적 빌딩 블록을 제공하고 조합은 사용자와 에이전트에 맡긴다.

핵심 구조

4단계 계층

Window                          (앱 윈도우)
  └── Workspace                 (사이드바 탭 — git 브랜치, PR 상태 표시)
        └── Pane                (분할 영역)
              └── Surface       (터미널 탭 또는 브라우저 패널)
계층 참조 형식 비유
Window window:1 건물
Workspace workspace:1 층(Floor)
Pane pane:1 방(Room)
Surface surface:1 / UUID 책상(Desk)

자동 환경변수

변수 설명
CMUX_WORKSPACE_ID 현재 워크스페이스 ID → --workspace 생략 가능
CMUX_SURFACE_ID 현재 서피스 ID → --surface 생략 가능
CMUX_SOCKET_PATH Unix 소켓 경로 (기본: ~/Library/Application Support/cmux/cmux.sock)

tmux vs cmux

구분 tmux cmux
플랫폼 Linux/macOS/Windows(WSL) macOS 14+ 전용
UI TUI (텍스트 기반) 네이티브 GUI (사이드바, 탭)
내장 브라우저 없음 WebKit 내장 (DOM 조작, 스크린샷)
알림 시스템 없음 사이드바 뱃지, 진행률, 알림
세션 지속성 detach/reattach (SSH 생존) 레이아웃 복원 (프로세스 미복원)
프로그래밍 API 제한적 (control mode) Unix socket JSON-RPC + CLI
AI 에이전트 지원 범용 Claude/Codex/Gemini 병렬 최적화
원격 서버 SSH에서 동작 v0.63.0부터 SSH 원격 워크스페이스 지원

tmux는 SSH 원격 서버, CI/CD, Linux에서 여전히 필수. cmux는 로컬 macOS에서 AI 에이전트를 운용하는 시나리오에 특화. 두 도구는 보완 관계.

작동 원리

설치

# Homebrew (권장)
brew tap manaflow-ai/cmux
brew install --cask cmux

# CLI 전역 사용 (선택)
sudo ln -sf "/Applications/cmux.app/Contents/Resources/bin/cmux" /usr/local/bin/cmux

macOS Gatekeeper 경고 시: 시스템 설정 > 보안 > "확인 없이 열기" 또는 xattr -cr /Applications/cmux.app

# 설치 확인
cmux version         # cmux 0.63.1 (...)
cmux identify        # 현재 workspace/pane/surface (JSON)

CLI 핵심 명령어

워크스페이스 관리

cmux new-workspace [--cwd /path] [--command "npm run dev"]
cmux rename-workspace "my-project"
cmux list-workspaces
cmux tree [--all]                  # 전체 계층 트리
cmux select-workspace --workspace workspace:2
cmux close-workspace --workspace workspace:2

패인 분할과 서피스

cmux new-split right               # 우측 분할
cmux new-split down                # 하단 분할
cmux new-surface                   # 터미널 탭 추가
cmux new-surface --type browser --url https://localhost:3000  # 브라우저 탭
cmux list-panes
cmux list-pane-surfaces

텍스트 전송과 출력 캡처 (AI 에이전트 핵심)

# 전송
cmux send "npm run dev\n"                        # 현재 서피스 (\n = Enter)
cmux send --surface surface:2 "npm test\n"       # 특정 서피스
cmux send-key ctrl+c                              # 키 전송

# 캡처 (에이전트가 다른 패인 출력을 읽는 핵심 기능)
cmux read-screen --lines 30                       # 최근 30줄
cmux read-screen --scrollback                     # 전체 스크롤백
cmux capture-pane --surface surface:2 --lines 50  # tmux 호환 별칭

동기화

# 시그널 기반
cmux wait-for build-done --timeout 60   # 대기 측
cmux wait-for -S build-done             # 시그널 측

# 폴링 기반
while ! cmux read-screen --surface surface:2 --lines 10 | grep -q "ready"; do
	sleep 1
done

# 내용으로 서피스 검색
cmux find-window --content "error" --select

내장 브라우저 (WebKit)

Playwright/Selenium 없이 터미널에서 웹 페이지를 직접 조작할 수 있다.

# 네비게이션
cmux browser open http://localhost:3000
cmux browser open-split http://localhost:3000    # 분할하여 열기
cmux browser goto http://localhost:3000/login
cmux browser back / forward / reload / url

# DOM 조작
cmux browser snapshot -i                          # 접근성 트리 기반 스냅샷
cmux browser click "button#submit" --snapshot-after
cmux browser fill "input[name='email']" "user@example.com" --snapshot-after
cmux browser type "#search" "검색어"
cmux browser get text "h1"
cmux browser get html ".content"

# 대기와 JavaScript
cmux browser wait --selector ".loaded"
cmux browser wait --text "Welcome" --timeout-ms 5000
cmux browser wait --url-contains "/dashboard"
cmux browser eval "document.title"
cmux browser screenshot --out ./capture.png

사이드바: 알림·진행률·상태

# 데스크톱 알림
cmux notify --title "빌드 완료" --body "3분 소요" --subtitle "frontend"

# 사이드바 로그
cmux log --level success --source "test" "42/42 테스트 통과"
cmux log --level error --source "build" "TypeScript 컴파일 실패"

# 진행률
cmux set-progress 0.75 --label "빌드 진행 중..."
cmux clear-progress

# 상태 배지
cmux set-status "phase" "testing" --icon "sparkle" --color "#28a745"
cmux clear-status "phase"

# 마크다운 뷰어
cmux markdown ./README.md

설계 판단

왜 tmux가 아닌 새로운 터미널인가

tmux는 수십 년간 표준이었지만, AI 에이전트 운용 관점에서 구조적 한계가 있다:

  1. 브라우저 자동화 불가 — 개발 서버 UI 확인에 별도 Playwright 설정 필요
  2. 사이드바 없음 — 에이전트 진행 상황을 한눈에 파악 불가
  3. 프로그래밍 API 부재 — capture-pane 파싱이 유일한 통신 수단
  4. 상태 연동 없음 — 워크스페이스별 브랜치·PR 번호 표시 불가

cmux는 이 문제들을 네이티브 레벨에서 해결한다. GPU 가속 렌더링(libghostty), WebKit 내장 브라우저, Unix Socket API를 하나의 앱에 통합했다.

"Primitive, Not Solution" 철학

Claude Squad나 Amux가 완성된 솔루션을 지향한다면, cmux는 원자적 빌딩 블록을 제공하고 조합은 사용자에게 맡긴다. Unix 철학의 "한 가지 일을 잘 하는 도구"와 맞닿아 있다.

라이선스

GPL-3.0-or-later 오픈소스. 별도 "Founder's Edition" 유료 구독(우선 지원, AI 기능 조기 접근, iOS 앱, 클라우드 VM)이 존재한다.

실전 적용

Claude Code + cmux 연동

# 왼쪽 패인: Claude Code, 오른쪽 패인: 서버
cmux new-split right
cmux send --surface surface:2 "python3 -m http.server 8000\n"

# Claude에게 요청:
# "오른쪽 패인(surface:2)에서 서버 로그를 cmux read-screen으로 읽어서 알려줘"

AI 에이전트가 cmux를 tmux로 오인하는 경우가 있다. CLAUDE.md에 "이 환경은 cmux입니다. tmux가 아닌 cmux CLI를 사용하세요."를 추가하면 해결된다.

멀티에이전트 병렬 실행

# 에이전트별 워크스페이스 생성
cmux new-workspace --command "claude 'React 19 새 기능 조사해줘'"
cmux rename-workspace "agent-react"

cmux new-workspace --command "claude 'Next.js 15 마이그레이션 가이드 작성해줘'"
cmux rename-workspace "agent-nextjs"

# 모니터링
cmux read-screen --workspace workspace:2 --lines 5
cmux find-window --content "완료" --select

cmux claude-teams — CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 환경변수와 tmux 호환 심을 자동 주입하여 Agent Teams 모드를 활성화하는 통합 명령.

개발서버 + 브라우저 E2E 검증

cmux new-split right
cmux send --surface surface:2 "cd my-app && npm run dev\n"

# 서버 준비 대기
while ! cmux read-screen --surface surface:2 --lines 10 | grep -q "ready"; do
	sleep 1
done

# 브라우저 검증
cmux browser open http://localhost:3000
cmux browser wait --text "Welcome" --timeout-ms 10000
cmux browser fill "input[name='email']" "test@example.com"
cmux browser click "button[type='submit']" --snapshot-after
cmux browser wait --url-contains "/dashboard" --timeout-ms 5000
cmux browser get text "h1"    # "Dashboard"면 성공

tmux → cmux 마이그레이션 매핑

tmux 명령 cmux 명령
tmux new-session -d -s name cmux new-workspace + cmux rename-workspace name
tmux split-window -v / -h cmux new-split down / right
tmux send-keys -t %1 "cmd" C-m cmux send --surface surface:1 "cmd\n"
tmux capture-pane -t %1 -p cmux capture-pane --surface surface:1
tmux kill-session -t name cmux close-workspace --workspace workspace:1
tmux wait-for channel cmux wait-for channel --timeout 30
(없음) cmux browser open URL
(없음) cmux identify
(없음) cmux notify / log / set-progress

CLAUDE.md / SKILL.md 설정 팁

# .claude/skills/cmux/SKILL.md에 다음 내용 배치:
# 1. 계층 구조: Window > Workspace > Pane > Surface
# 2. 환경변수: CMUX_WORKSPACE_ID, CMUX_SURFACE_ID (자동)
# 3. 핵심 명령어: identify, read-screen, send, browser snapshot
# 4. tmux → cmux 변환 규칙
# 5. SSH/CI에서는 cmux 불가 → tmux 폴백

# cmux 감지 + tmux 폴백 패턴
if command -v cmux &>/dev/null && [ -n "$CMUX_WORKSPACE_ID" ]; then
	cmux new-split right
	cmux send --surface surface:2 "npm run dev\n"
else
	tmux split-window -h
	tmux send-keys "npm run dev" C-m
fi

제한사항

  • macOS 전용 — Swift + AppKit 기반 (커뮤니티 Linux/Windows 포트 존재)
  • 세션 복원 — 레이아웃·디렉토리·URL은 복원되지만 실행 중이던 프로세스는 미복원
  • 초기 단계 — v0.63.1, 1.0 이전이라 breaking change 가능
  • WKWebView 기반 — CDP 전용 기능(CPU/메모리 프로파일링)은 불가
  • 화면 스크래핑 방식 — read-screen은 렌더링된 텍스트를 읽는 RPA 방식

핵심 인사이트

cmux는 도구 하나가 아니라 "개념의 신호탄"이다. "터미널끼리 말하게 하라"는 방향 — AI 에이전트가 다른 터미널의 출력을 읽고, 명령을 보내고, 브라우저를 조작하는 것 — 은 에이전트 시대의 인터페이스가 어떤 형태가 될지를 보여준다.

한 줄 요약

AI 에이전트 병렬 운용이 필요하다면 cmux의 read-screen + send + browser 조합이 tmux 단독보다 구조적으로 우위다. 단, macOS 전용이고 초기 단계이므로 원격 서버 작업에는 tmux와 병행 사용이 현실적이다.
