Understanding Covariance and Invariance in Java

1. Introduction to Covariance and Invariance

In object-oriented programming and type systems, covariance and invariance describe how types relate to each other in generic and collection-based contexts. These concepts are crucial for understanding Java’s type system, especially in the context of generics and arrays.

Covariance

A type system is covariant if a type parameter relationship follows the same direction as the original subtype relationship. That is, if A is a subtype of B, then T<A> is also a subtype of T<B>.

Invariance

A type system is invariant if a type parameter relationship does not follow the original subtype relationship. That is, even if A is a subtype of B, T<A> and T<B> are unrelated types.

2. Covariance in Java: Arrays

In Java, arrays are covariant, meaning an array of a subtype can be assigned to an array of its supertype.

Example of Covariance in Arrays

@Test
void arrayCovariance() {
    Integer[] integers = new Integer[]{1, 2, 3};
    printArray(integers);
}

void printArray(Object[] arr) {
    for (Object e : arr) {
        System.out.println(e);
    }
}

✔ This works because Integer[] is a subtype of Object[]. Arrays in Java support covariance.

Potential Issue with Covariant Arrays

However, covariance in arrays can lead to runtime exceptions (ArrayStoreException) due to type mismatch.

Object[] objArray = new Integer[5];
objArray[0] = "Hello"; // Throws ArrayStoreException

🚫 This fails at runtime because Java performs array type checking dynamically.

3. Invariance in Java: Generics

Unlike arrays, generics in Java are invariant. That means List<Integer> is not a subtype of List<Object>.

Example of Invariant Generics

@Test
void genericInvariance() {
    List<Integer> list = Arrays.asList(1, 2, 3);
    printCollection(list); // Compilation error
}

void printCollection(Collection<Object> c) {
    for (Object e : c) {
        System.out.println(e);
    }
}

🚫 This fails with a compilation error:

java: incompatible types: java.util.List<java.lang.Integer> cannot be converted to java.util.Collection<java.lang.Object>

Even though Integer is a subtype of Object, List<Integer> and List<Object> are completely unrelated types.

4. Why Generics Are Invariant

The reason generics are invariant is to ensure type safety. If Java allowed List<Integer> to be treated as List<Object>, it would introduce unsafe operations.

For example:

List<Integer> intList = new ArrayList<>();
List<Object> objList = intList; // Hypothetically allowed
objList.add("Hello"); // This would corrupt type safety!

🚫 This would break type safety since an Integer list now contains a String.

5. Using Wildcards to Handle Invariance

To allow methods to work with multiple types without breaking type safety, Java introduces wildcards (?).

Wildcard Example

void printCollection(Collection<?> c) {
    for (Object e : c) {
        System.out.println(e);
    }
}

✔ Now, Collection<Integer>, Collection<String>, etc., can be passed without issues.

6. Covariance with Bounded Wildcards (? extends T)

Java allows bounded wildcards to limit acceptable types.

void printNumbers(List<? extends Number> list) {
    for (Number num : list) {
        System.out.println(num);
    }
}

✔ This allows List<Integer>, List<Double>, etc.

🚫 But prevents adding elements (except null) because Java does not know the exact type.

list.add(10); // Compilation error

7. Contravariance with Lower Bounded Wildcards (? super T)

If we need to allow superclasses instead of subclasses, we use ? super T.

void addElements(List<? super Integer> list) {
    list.add(10); // Allowed
    list.add(20);
}

✔ Now, List<Integer>, List<Number>, List<Object> are acceptable.

🚫 However, retrieving elements is restricted to Object.

Integer num = list.get(0); // Compilation error
Object obj = list.get(0); // Works fine

8. PECS (Producer-Extends, Consumer-Super) Rule

Joshua Bloch introduced PECS:

  • Use ? extends T for reading (producing values).
  • Use ? super T for writing (consuming values).
void processList(List<? extends Number> producer) { // Read-only
    for (Number n : producer) {
        System.out.println(n);
    }
}

void addValues(List<? super Integer> consumer) { // Write-only
    consumer.add(10);
}
