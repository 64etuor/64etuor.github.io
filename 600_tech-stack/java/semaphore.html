Semaphore

Semaphore

데이터베이스 커넥션 풀은 보통 10개다. 스레드가 50개 동시에 커넥션을 요청하면 어떻게 되나? 40개는 기다린다. 이 "동시에 10개까지만 허용"하는 메커니즘이 세마포어다.

synchronized는 한 번에 하나만 통과시키는 문이라면, 세마포어는 동시에 N명까지 입장 가능한 대기열이다.

신호기에서 온 이름

철도 세마포어 신호기:
  팔이 올라감 → 열차 통과 가능 (permit 획득)
  팔이 내려감 → 열차 정지 (permit 없음, 대기)

1965년 에츠허르 다익스트라(Dijkstra)가 이 철도 신호 체계에서 이름을 따왔다. 원래 연산 이름도 네덜란드어 P(proberen, 시도) / V(verhogen, 증가)였다. 지금은 acquire / release로 쓴다.

핵심은 단순하다:

  • permit(허가증) 이라는 정수 카운터가 있다
  • acquire() → permit > 0이면 1 감소하고 통과, 0이면 대기
  • release() → permit 1 증가, 대기 중인 스레드 깨움

Java 코드로 보는 동작

import java.util.concurrent.Semaphore;

// 동시에 3개 스레드만 허용
Semaphore semaphore = new Semaphore(3);

Runnable task = () -> {
	try {
		semaphore.acquire();  // permit 획득 (없으면 대기)
		System.out.println(Thread.currentThread().getName() + " 작업 중");
		Thread.sleep(2000);
	} catch (InterruptedException e) {
		Thread.currentThread().interrupt();
	} finally {
		semaphore.release();  // permit 반환
	}
};

// 10개 스레드가 동시에 시도
for (int i = 0; i < 10; i++) {
	new Thread(task, "Thread-" + i).start();
}

실행하면 항상 3개씩 끊어서 출력된다:

Thread-0 작업 중
Thread-1 작업 중
Thread-2 작업 중
(2초 대기)
Thread-3 작업 중
Thread-4 작업 중
Thread-5 작업 중
...

뮤텍스와 세마포어는 다른 물건이다

Mutex (synchronized) Semaphore
동시 접근 수 1 N
소유권 잠근 스레드만 풀 수 있음 아무 스레드나 release 가능
용도 상호 배제 (mutual exclusion) 리소스 풀, 속도 제한
Java 구현 synchronized, ReentrantLock java.util.concurrent.Semaphore

permit이 1인 세마포어(new Semaphore(1))는 뮤텍스처럼 동작하지만, 소유권 개념이 없다는 점이 결정적으로 다르다. 뮤텍스는 lock을 건 스레드만 unlock할 수 있지만, 세마포어는 A가 acquire하고 B가 release할 수 있다.

공정성(Fairness)

// 기본: 비공정 (Non-fair) — 성능 우선
Semaphore unfair = new Semaphore(3);

// 공정 모드 — FIFO 보장
Semaphore fair = new Semaphore(3, true);

fair=true로 설정하면 먼저 대기한 스레드가 먼저 permit을 받는다. 대신 내부적으로 FIFO 큐를 유지해야 하므로 처리량이 떨어진다. 대부분의 경우 기본값(비공정)으로 충분하다.

내부 구조: AQS 위에서 돌아간다

Semaphore는 AbstractQueuedSynchronizer(AQS)를 상속한 내부 클래스 Sync로 구현된다.

Semaphore
  └── Sync extends AQS
        ├── NonfairSync   (기본)
        └── FairSync       (fair=true)

AQS의 state 필드가 곧 permit 카운터다:

// acquire 내부 (간략화)
for (;;) {
	int available = getState();           // 현재 permit 수
	int remaining = available - 1;
	if (remaining < 0 || compareAndSetState(available, remaining)) {
		return remaining;                 // 음수면 큐에서 대기
	}
}

CAS(Compare-And-Swap) 연산으로 permit을 원자적으로 감소시킨다. 락 없이 동작하므로 경합이 낮을 때 매우 빠르다.

실전에서 쓰는 3가지 패턴

커넥션 풀 제한

public class ConnectionPool {
	private final Semaphore semaphore;
	private final Queue<Connection> pool;

	public ConnectionPool(int size) {
		this.semaphore = new Semaphore(size);
		this.pool = new ConcurrentLinkedQueue<>();
		for (int i = 0; i < size; i++) {
			pool.add(createConnection());
		}
	}

	public Connection acquire() throws InterruptedException {
		semaphore.acquire();
		return pool.poll();
	}

	public void release(Connection conn) {
		pool.offer(conn);
		semaphore.release();
	}
}

API 호출 속도 제한 (Rate Limiting)

// 초당 최대 5회 호출
Semaphore rateLimiter = new Semaphore(5);

ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);
scheduler.scheduleAtFixedRate(() -> {
	int permitsToRelease = 5 - rateLimiter.availablePermits();
	if (permitsToRelease > 0) {
		rateLimiter.release(permitsToRelease);
	}
}, 1, 1, TimeUnit.SECONDS);

// 호출 시
public String callApi() throws InterruptedException {
	rateLimiter.acquire();
	return httpClient.get("https://api.example.com/data");
}

tryAcquire로 타임아웃 처리

if (semaphore.tryAcquire(500, TimeUnit.MILLISECONDS)) {
	try {
		// 리소스 사용
	} finally {
		semaphore.release();
	}
} else {
	// 500ms 내 permit 못 얻음 → 대체 로직
	return fallbackResponse();
}

tryAcquire는 대기 시간 제한이 필요한 곳에서 필수다. 무한 대기는 장애 전파의 시작점이다.

흔한 실수

release를 빠뜨리면 permit이 영원히 줄어든다. 예외가 발생해도 반드시 반환해야 하므로 finally 블록은 필수다.

// ❌ 예외 시 permit 누수
semaphore.acquire();
doSomething();  // 여기서 예외 발생하면?
semaphore.release();

// ✅ finally로 보장
semaphore.acquire();
try {
	doSomething();
} finally {
	semaphore.release();
}

acquire 없이 release를 호출하면 permit이 원래보다 늘어난다. Semaphore는 소유권을 추적하지 않으므로, 이 버그는 조용히 동시 접근 수를 초과시킨다.

핵심 정리

  • 세마포어는 "동시에 N개까지"를 제어하는 카운터 기반 동기화 도구다
  • 뮤텍스(1개 제한, 소유권 있음)와는 용도가 다르다
  • Java의 Semaphore는 AQS + CAS로 구현되어 락 없이 빠르게 동작한다
  • finally에서 release()를 빠뜨리면 시스템이 천천히 멈춘다
