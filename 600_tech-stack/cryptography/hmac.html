HMAC — 해시 함수를 메시지 인증에 쓰는 안전한 방법

2009년, Flickr API에서 인증 우회 취약점이 발견됐다. 공격자는 API 키를 몰라도 유효한 서명을 만들 수 있었다. 원인은 단순했다 — 서명 생성에 MD5(secret || message) 구조를 썼고, MD5의 length extension attack이 이를 뚫었다.

hash(key || message)가 깨지는 이유

SHA-256이나 MD5 같은 Merkle-Damgard 해시 함수는 내부 상태를 블록 단위로 갱신한다. H(key || message)의 최종 해시값을 알면, key를 모르더라도 그 해시 상태에서 이어 붙여 H(key || message || padding || attacker_data)의 유효한 해시를 계산할 수 있다.

원본: H(secret || "amount=100")  → abc123...
공격: H(secret || "amount=100" || padding || "&amount=999")  → def456...

공격자는 secret을 모른다. 하지만 abc123...이라는 해시 출력이 곧 해시 함수의 내부 상태이므로, 거기서부터 추가 데이터를 먹여 새 해시를 만들 수 있다. 서버는 이 조작된 해시를 정상 서명으로 받아들인다.

이 문제는 해시 함수의 결함이 아니라 사용 방식의 결함이다. 해시 함수는 무결성 검증용이지, 키 기반 인증용으로 설계되지 않았다. HMAC(RFC 2104, 1997)은 해시 함수를 안전하게 MAC으로 변환하는 구성법이다.

HMAC의 이중 해싱 구조

HMAC의 정의는 한 줄이다:

HMAC(K, m) = H((K' ⊕ opad) || H((K' ⊕ ipad) || m))
  • K': 키를 블록 크기에 맞춘 값. 키가 블록보다 길면 해시로 줄이고, 짧으면 0으로 패딩
  • ipad: 0x36을 블록 크기만큼 반복 (inner padding)
  • opad: 0x5c를 블록 크기만큼 반복 (outer padding)

안쪽에서 바깥으로 풀어보면:

  1. K'와 ipad를 XOR → 내부 키 생성
  2. 내부 키 뒤에 메시지를 붙이고 해싱 → 내부 해시
  3. K'와 opad를 XOR → 외부 키 생성
  4. 외부 키 뒤에 내부 해시를 붙이고 해싱 → 최종 HMAC
import hmac
import hashlib

# Python 표준 라이브러리
tag = hmac.new(b'my-secret-key', b'hello world', hashlib.sha256).hexdigest()
print(tag)  # 734cc62f32841568f45f6ab251...

ipad와 opad는 왜 0x36과 0x5c인가

두 상수의 핵심 역할은 내부 해시와 외부 해시가 서로 다른 키로 동작하게 만드는 것이다.

  • K' ⊕ 0x36...과 K' ⊕ 0x5c...는 항상 다른 값이다 (0x36 ⊕ 0x5c = 0x6a ≠ 0)
  • 두 값 사이의 해밍 거리가 충분하므로, 내부와 외부 해시가 암호학적으로 독립된 함수처럼 동작한다

0x36과 0x5c 자체에 수학적 마법은 없다. Bellare, Canetti, Krawczyk의 원 논문(1996)에서 "충분히 다른 두 상수"로 선택한 것이며, 보안 증명은 이 두 값이 다르기만 하면 성립한다. 이후 RFC 2104로 표준화되면서 이 값이 고정됐다.

length extension attack이 통하지 않는 이유도 여기에 있다. 공격자가 내부 해시 H((K' ⊕ ipad) || m)를 확장하더라도, 외부 해시 단계에서 K' ⊕ opad라는 알 수 없는 키가 다시 개입한다. 확장 공격은 외부 해시 경계를 넘지 못한다.

Python으로 직접 구현해 보기

표준 라이브러리의 hmac 모듈이 내부에서 하는 일을 풀어쓰면:

import hashlib

def hmac_sha256(key: bytes, message: bytes) -> bytes:
	block_size = 64  # SHA-256의 블록 크기
	hash_func = hashlib.sha256

	# 1. 키 정규화
	if len(key) > block_size:
		key = hash_func(key).digest()
	key = key.ljust(block_size, b'\x00')

	# 2. ipad / opad 생성
	ipad = bytes(k ^ 0x36 for k in key)
	opad = bytes(k ^ 0x5c for k in key)

	# 3. 내부 해시
	inner = hash_func(ipad + message).digest()

	# 4. 외부 해시
	return hash_func(opad + inner).digest()

# 검증: 표준 라이브러리와 동일한 결과
import hmac as hmac_lib
key = b'secret'
msg = b'hello'
assert hmac_sha256(key, msg) == hmac_lib.new(key, msg, hashlib.sha256).digest()

16줄로 끝난다. HMAC의 강점은 기존 해시 함수를 그대로 가져다 쓰면서도 보안 증명이 가능하다는 점이다.

HMAC이 쓰이는 곳

프로토콜 / 시스템 HMAC 사용 방식
JWT (HS256) HMAC-SHA256(secret, header.payload)로 토큰 서명. 서버가 secret을 알아야 검증 가능
AWS Signature V4 날짜 → 리전 → 서비스 → 요청 순으로 HMAC을 체이닝하여 서명 키 도출
HKDF Extract와 Expand 양쪽 모두 HMAC을 기반 연산으로 사용
TLS 1.3 PRF(Pseudorandom Function)로 HMAC 기반 HKDF 사용. Finished 메시지 검증에도 HMAC 적용
TOTP (Google Authenticator) HMAC-SHA1(secret, timestamp_counter)로 6자리 OTP 생성

AWS Signature V4의 키 체이닝은 HMAC의 실전 활용을 잘 보여준다:

def aws_signature_key(secret: str, date: str, region: str, service: str) -> bytes:
	k_date = hmac_sha256(("AWS4" + secret).encode(), date.encode())
	k_region = hmac_sha256(k_date, region.encode())
	k_service = hmac_sha256(k_region, service.encode())
	k_signing = hmac_sha256(k_service, b"aws4_request")
	return k_signing

각 단계의 HMAC 출력이 다음 단계의 키가 된다. 날짜, 리전, 서비스가 모두 서명 키에 바인딩되므로, 한 리전의 서명 키로 다른 리전의 API를 호출할 수 없다.

HMAC vs 다른 MAC 알고리즘

MAC 기반 장점 한계
HMAC 해시 함수 (SHA-2, SHA-3 등) 해시만 있으면 구현 가능, 보안 증명 존재, 가장 넓은 호환성 블록 암호 기반보다 느릴 수 있음
CMAC 블록 암호 (AES) AES 하드웨어 가속 활용 가능 해시 함수 없는 환경 전용, 가변 길이 메시지에서 HMAC보다 복잡
Poly1305 다항식 연산 극도로 빠름, ChaCha20과 조합(AEAD)에 최적 단독 MAC으로 부적합 — 일회용 키(nonce-derived) 필수
KMAC SHA-3 (KECCAK) SHA-3의 스펀지 구조가 자체적으로 키 입력 지원, HMAC의 이중 해싱 불필요 SHA-3 생태계 한정

HMAC이 여전히 지배적인 이유는 실용적이다. SHA-2 구현체는 어디에나 있고, TLS·JWT·AWS 등 기존 인프라가 전부 HMAC 위에 올라가 있다. SHA-3 환경이라면 KMAC이 더 깔끔하지만, 이행 비용이 정당화될 만큼 HMAC에 약점이 발견되지 않았다.

보안은 해시 함수에서 물려받는다

HMAC-SHA256의 보안은 SHA-256의 성질에 의존한다. 정확히는:

  • 위조 저항: 공격자가 키 없이 유효한 MAC을 만들려면 SHA-256의 PRF(Pseudorandom Function) 성질을 깨야 한다
  • 충돌 저항 상속: 해시 함수의 충돌 저항이 살아 있으면 HMAC의 위조 저항도 유지된다
  • 키 복구 저항: HMAC 태그를 여러 개 관찰해도 키를 역산하는 것은 해시 함수의 일방향성만큼 어렵다

MD5의 충돌 저항은 2004년에 깨졌지만, HMAC-MD5는 2025년 현재까지 실질적인 위조 공격이 알려지지 않았다. HMAC의 이중 해싱 구조가 추가 방어층을 제공하기 때문이다. 그러나 이것은 "HMAC-MD5가 안전하다"는 뜻이 아니라, "HMAC 구조가 해시 함수의 약점을 어느 정도 완화한다"는 뜻이다. 새 시스템에서는 SHA-256 이상을 써야 한다.

테이크어웨이

hash(key || message)를 MAC으로 쓰고 싶은 유혹은 직관적이지만 구조적으로 깨진다. HMAC은 해시 함수를 두 번 호출하고, 각 호출에 서로 다른 키를 먹이는 것만으로 이 문제를 해결했다. 구현은 16줄, 보안 증명은 수학적으로 견고하고, 지난 30년간 실전에서 검증됐다. 메시지 인증이 필요한 자리에 해시 함수를 직접 넣으려 한다면, 그 자리는 이미 HMAC의 자리다.
