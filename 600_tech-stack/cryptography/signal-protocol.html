Signal Protocol — 메시지 하나마다 키를 바꾸는 종단간 암호화

2016년 4월, WhatsApp이 10억 사용자 전원에게 종단간 암호화(E2EE)를 일괄 적용했다. 서버 운영자조차 메시지를 읽을 수 없는 암호화를 수십억 단위로 배포한 건 이때가 처음이었다. 그 기반이 된 프로토콜이 Signal Protocol이고, 핵심 메커니즘은 Double Ratchet Algorithm이다.

오프라인 상대와 비밀을 만드는 법: X3DH

암호화 채팅의 첫 번째 난관은 "상대방이 오프라인일 때 어떻게 공유 비밀(shared secret)을 합의하느냐"다. Signal Protocol은 X3DH(Extended Triple Diffie-Hellman)로 이를 해결한다.

각 사용자는 서버에 세 종류의 공개키를 등록해 둔다:

  • Identity Key (IK): 장기 신원 키. 계정 생성 시 1회 생성.
  • Signed Pre Key (SPK): 중기 키. 주기적으로 교체하고, Identity Key로 서명.
  • One-Time Pre Key (OPK): 일회용 키. 한 번 사용되면 서버에서 삭제.

Alice가 Bob에게 첫 메시지를 보내려 한다고 하자. Bob은 오프라인이다.

Alice 쪽 계산 (Bob의 공개키 번들을 서버에서 가져옴):
  DH1 = DH(Alice_IK_private, Bob_SPK_public)
  DH2 = DH(Alice_EK_private, Bob_IK_public)
  DH3 = DH(Alice_EK_private, Bob_SPK_public)
  DH4 = DH(Alice_EK_private, Bob_OPK_public)   ← OPK가 있을 때만

  SK = HKDF(DH1 || DH2 || DH3 || DH4)

EK는 Alice가 이 세션을 위해 즉석 생성한 Ephemeral Key다. 네 번의 DH 연산 결과를 이어붙인 뒤 HKDF로 하나의 초기 공유 비밀(SK)을 유도한다. Bob이 나중에 온라인에 돌아오면, 같은 DH 쌍을 역방향으로 계산해서 동일한 SK를 얻는다.

OPK가 일회용인 이유가 중요하다. DH1~DH3만으로도 프로토콜은 작동하지만, 만약 Bob의 SPK 비밀키가 유출되면 과거 세션을 복호화할 수 있다. OPK는 세션마다 다른 키를 섞어 넣어서 이 위험을 차단한다.

메시지마다 키를 바꾸는 Double Ratchet

X3DH로 만든 SK는 시작점일 뿐이다. 이후 메시지마다 키가 바뀌는 구조가 Double Ratchet이다. "래칫(ratchet)"이란 한 방향으로만 돌아가는 톱니바퀴를 뜻한다 — 키를 앞으로 전진시킬 수는 있지만, 뒤로 되돌릴 수는 없다.

Double Ratchet은 두 가지 래칫의 결합이다:

  • DH Ratchet: 메시지를 주고받을 때마다 새로운 DH 키 쌍을 생성하고 교환. 비대칭 키 갱신.
  • Symmetric-key Ratchet: 같은 방향으로 연속 메시지를 보낼 때 KDF 체인으로 대칭키를 한 단계씩 전진.
[Root Chain]
    │
    ├── DH Ratchet step (새 DH 키 교환)
    │        │
    │        ├──→ [Sending Chain] ──KDF──→ MK1, MK2, MK3 ...
    │        │
    │        └──→ [Receiving Chain] ──KDF──→ MK1, MK2, MK3 ...
    │
    ├── DH Ratchet step (다시 새 DH 키 교환)
    │        │
    │        ├──→ [Sending Chain'] ...
    │        └──→ [Receiving Chain'] ...
    ...

세 개의 KDF 체인이 하는 일

HKDF 기반의 KDF(Key Derivation Function)가 체인마다 다른 역할을 수행한다.

  • Root Chain: DH Ratchet의 출력을 입력으로 받아 새로운 chain key를 유도. Root Key는 전체 세션의 "줄기"다.
  • Sending Chain: Root Chain에서 파생된 chain key로부터 메시지 키(MK)를 하나씩 생성. 각 MK는 정확히 한 메시지의 암호화에 사용된다.
  • Receiving Chain: 상대방의 Sending Chain과 대칭. 수신된 메시지를 복호화할 MK를 생성.
KDF 체인 한 단계의 구조:

  (chain_key_n, input) ──HKDF──→ (chain_key_n+1, message_key_n)

chain_key_n은 다음 단계의 입력이 되고, message_key_n은 해당 메시지 암·복호화에 쓰인 뒤 폐기된다. 체인 키만 가지고는 이전 메시지 키를 역산할 수 없다 — HMAC 기반 KDF의 일방향성 때문이다.

Forward Secrecy: 오늘의 키가 어제를 열지 못하는 이유

전통적 TLS(세션 키 하나로 전체 통신)와 달리, Double Ratchet은 메시지 키를 사용 후 즉시 삭제한다. 공격자가 현재 디바이스의 메모리를 탈취해도 이미 폐기된 과거 메시지 키는 존재하지 않는다.

Forward secrecy가 성립하는 두 가지 층위:

  • Symmetric ratchet 수준: 체인 키 n+1에서 n을 역산할 수 없음. 한 메시지 키 유출이 같은 체인의 이전 메시지에 영향을 주지 않음.
  • DH ratchet 수준: 새 DH 교환이 일어나면 이전 DH 비밀키는 삭제됨. 설령 현재 체인 키 전체가 유출되어도 이전 DH ratchet 구간의 메시지는 안전함.

Post-Compromise Security: 키가 뚫려도 스스로 회복한다

Forward secrecy만으로는 부족한 시나리오가 있다. 공격자가 현재 세션 상태(chain key + DH private key)를 통째로 탈취하면, 이후 메시지도 계속 읽을 수 있지 않은가?

여기서 DH Ratchet의 자가 치유(self-healing) 특성이 작동한다. 상대방이 다음 메시지를 보내면서 새 DH 공개키를 포함시키는 순간, 새로운 DH 비밀이 계산된다. 공격자가 모르는 상대방의 새 ephemeral 비밀키가 개입하므로, 공격자는 이 시점부터 키 유도를 따라갈 수 없게 된다.

이것이 post-compromise security(또는 future secrecy)다. 프로토콜이 손상된 상태에서도, 정상적인 메시지 교환이 한 번만 일어나면 보안이 자동 복구된다.

누가 Signal Protocol을 쓰는가

서비스 적용 범위 비고
Signal 전체 메시지 + 음성/영상 통화 레퍼런스 구현
WhatsApp 전체 메시지 (2016~) 10억+ 사용자 일괄 적용

Signal Protocol이 사실상 1:1 메시징 E2EE의 산업 표준이 된 이유는 단순하다. 오프라인 키 교환(X3DH), 메시지별 키 갱신(Double Ratchet), forward secrecy, post-compromise security를 하나의 프로토콜에 담았고, libsignal이라는 잘 정리된 라이브러리로 통합 비용을 낮췄다.

다른 E2EE 프로토콜과 비교

Signal Protocol은 1:1 대화에 최적화되어 있다. 그룹 채팅으로 가면 구조적 한계가 드러난다.

  • MLS (Messaging Layer Security): IETF RFC 9420으로 표준화. 트리 기반 키 관리(TreeKEM)로 그룹 멤버 추가·제거 시 비용이 O(log n). Signal Protocol의 그룹 방식(Sender Keys)은 멤버 변경마다 O(n) 비용이 든다.
  • Matrix/Olm + Megolm: Matrix 프로토콜이 사용. Olm은 Double Ratchet의 변형(1:1용), Megolm은 그룹 대화용 래칫. Megolm은 sender ratchet만 사용하기 때문에 post-compromise security가 제한적이다 — 세션을 명시적으로 갱신해야 회복된다.

그룹 확장성 O(n) — Sender Keys O(log n) — TreeKEM O(1) — Megolm

핵심 테이크어웨이

Signal Protocol의 설계 철학은 하나로 요약된다: 키의 수명을 극단적으로 짧게 유지한다. X3DH는 일회용 키로 세션을 시작하고, Double Ratchet은 메시지마다 키를 폐기한다. 이 구조 덕분에 키 유출의 피해 범위가 단일 메시지로 제한되고, 정상적 통신이 재개되면 보안이 자동 회복된다. "완벽한 방어"가 아니라 "피해를 최소화하고 빠르게 회복하는 설계" — 이것이 현실 세계에서 수십억 대의 디바이스가 채택한 이유다.
