Age — GPG 없이 파일을 암호화하는 가장 단순한 방법

GPG로 파일 하나 암호화하려면 키링 생성, 신뢰 모델 설정, 서브키 관리, 만료일 지정까지 거쳐야 한다. 2019년 Google의 암호학 엔지니어 Filippo Valsorda는 "I'm giving up on PGP"라는 글을 쓴 뒤, 직접 대안을 만들었다. 그 결과물이 Age(Actually Good Encryption)다.

30초 만에 암호화하기

# 키 생성
$ age-keygen -o key.txt
Public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p

# 파일 암호화 (수신자 공개키 지정)
$ age -r age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p \
	-o secret.txt.age plaintext.txt

# 복호화
$ age -d -i key.txt -o plaintext.txt secret.txt.age

이게 전부다. 키링 없고, 설정 파일 없고, 신뢰 모델 없다. age-keygen이 X25519 키 쌍을 만들고, age -r이 암호화하고, age -d가 복호화한다.

설계 철학: 빼는 것이 설계다

Age의 설계 문서는 명시적으로 다음을 거부한다:

  • 설정 옵션 (알고리즘 선택 없음 — ChaCha20-Poly1305 고정)
  • 키 서버, 신뢰 웹, 인증서 체인
  • 서명/검증 (암호화 전용)
  • 압축 (별도 도구에 위임)
  • 하위 호환성을 위한 레거시 알고리즘

GPG가 20가지 암호화 알고리즘, 4가지 압축 방식, 복잡한 패킷 포맷을 지원하는 것과 정반대다. Age는 "파일 암호화"라는 단일 목적만 수행하고, 나머지는 유닉스 철학에 따라 다른 도구에 맡긴다.

세 가지 수신자 타입

X25519 네이티브 키

$ age-keygen -o key.txt
# key.txt 내용:
# created: 2026-04-07T10:30:00+09:00
# public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
AGE-SECRET-KEY-1QMPZWYRC55E7GFREW0EJ2A8FKG6UL4NF5P0S...

공개키는 `age1`로 시작하는 Bech32 인코딩, 비밀키는 `AGE-SECRET-KEY-1`로 시작한다. 포맷이 하나뿐이니 "어떤 키를 써야 하지?"라는 질문 자체가 사라진다.

SSH 키 재사용

# 이미 가지고 있는 SSH 키로 암호화
$ age -R ~/.ssh/id_ed25519.pub -o secret.age plaintext.txt

# SSH 비밀키로 복호화
$ age -d -i ~/.ssh/id_ed25519 secret.age

별도 키를 만들 필요 없이 기존 `ed25519`나 `rsa` SSH 키를 그대로 쓸 수 있다. GitHub에 등록된 공개키로 암호화하면 해당 사용자만 복호화 가능하다:

$ curl https://github.com/FiloSottile.keys | age -R - -o secret.age plaintext.txt

패스프레이즈 (대칭 암호화)

$ age -p -o secret.age plaintext.txt
Enter passphrase:

패스프레이즈 모드에서는 [[Argon2]]id로 키를 도출한다. 메모리-하드 함수이므로 GPU 기반 브루트포스에 강하다.

내부 구조: HKDF가 키를 분리한다

Age의 암호화 과정은 다음 단계를 거친다:

  1. 임시 X25519 키 쌍 생성 (에페메랄 키)

  2. 수신자 공개키와 ECDH → shared secret 도출

  3. HKDF-SHA-256으로 shared secret에서 파일 키(file key) 도출

  4. 파일 키에서 다시 HKDF로 nonce와 payload 키 도출

  5. ChaCha20-Poly1305로 페이로드 암호화

    HKDF가 핵심 접착제 역할을 한다. 하나의 DH 결과에서 용도별로 독립된 키를 뽑아내고, 각 키가 서로의 보안에 영향을 주지 않도록 분리한다. 수신자가 여러 명일 때는 동일한 파일 키를 각 수신자의 공개키로 개별 래핑(wrapping)하여 헤더에 포함한다.

GPG와의 트레이드오프

기준 Age GPG
목적 파일 암호화 전용 암호화 + 서명 + 인증
알고리즘 선택 없음 (X25519 + ChaCha20 고정) 20+ 알고리즘
키 포맷 1종 (또는 SSH 키 재사용) 주키 + 서브키 + UID + 신뢰
설정 파일 없음 gpg.conf, 키링, 에이전트
서명 미지원 (범위 밖) 지원
이메일 통합 미지원 OpenPGP/MIME
패키지 서명 미지원 apt, rpm 등에 사용 
Age는 GPG를 대체하는 게 아니라, GPG의 기능 중 "파일 암호화"만 떼어내서 제대로 만든 도구다. 서명이 필요하면 [minisign](https://jedisct1.github.io/minisign/)이나 [signify](https://man.openbsd.org/signify)를 쓰면 된다.

실전 채택 사례

Age는 단독 CLI 도구를 넘어 인프라 생태계에 통합되고 있다:

  • SOPS — Mozilla가 만든 시크릿 관리 도구. v3.7부터 Age를 백엔드로 지원한다. sops --age age1... secrets.yaml로 YAML/JSON의 값만 선택적 암호화 가능.

  • restic / borgmatic — 백업 도구들이 Age 기반 암호화를 플러그인 또는 파이프로 지원. tar | age -r ... | restic 패턴.

  • chezmoi — dotfile 관리자. Age로 .env, SSH 키 등 민감 파일을 암호화한 채 Git에 커밋.

  • CI/CD 시크릿 — GitHub Actions에서 age -d -i $KEY 한 줄로 시크릿 복호화. 별도 시크릿 매니저 없이 Git 저장소 안에 암호화된 파일을 둘 수 있다.

    공통 패턴이 보인다. 전부 "GPG로 할 수는 있지만 너무 복잡했던" 자리를 Age가 채우고 있다.

가져갈 것

암호화 도구의 가치는 알고리즘의 강도가 아니라 실제로 사용되느냐에 달려 있다. GPG를 제대로 쓰는 사람은 드물고, 잘못 쓰는 사람은 많다. Age는 잘못 쓸 수 있는 선택지를 제거함으로써 보안을 높였다. 파일 암호화가 필요한 곳에 GPG를 쓰고 있다면, brew install age 한 줄이 복잡성을 제거하는 가장 빠른 방법이다.
