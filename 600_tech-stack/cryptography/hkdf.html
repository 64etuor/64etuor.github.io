HKDF — 하나의 비밀에서 여러 키를 안전하게 뽑아내는 방법

TLS 1.3 핸드셰이크가 끝나면 클라이언트와 서버는 단 하나의 shared secret을 공유한다. 그런데 실제 통신에 필요한 키는 client write key, server write key, client write IV, server write IV 등 최소 4개다. 하나의 비밀에서 서로 독립적인 여러 키를 뽑아내는 함수가 HKDF(HMAC-based Extract-and-Expand Key Derivation Function, RFC 5869)다.

날것의 비밀은 키가 아니다

Diffie-Hellman이나 ECDH로 생성한 shared secret은 균일한 랜덤이 아니다. 특정 비트 패턴에 편향이 있고, 엔트로피가 전체 비트 길이보다 작을 수 있다. 이런 값을 AES 키로 직접 쓰면 통계적 공격에 노출된다.

키 도출 함수(KDF)는 이 문제를 해결한다:

  • 편향된 입력에서 균일한 의사난수(pseudorandom)를 추출
  • 하나의 소스에서 용도별로 독립된 여러 키를 생성
  • 키 길이를 자유롭게 조절

HKDF는 Hugo Krawczyk이 2010년 설계한 KDF로, "extract-then-expand" 패러다임을 HMAC 위에 구현했다.

Extract: 엔트로피 농축

HKDF-Extract(salt, IKM) → PRK
  • IKM(Input Keying Material): 원본 비밀. DH 공유 비밀, 패스워드 해시, 하드웨어 엔트로피 등
  • salt: 선택적 비랜덤 값. HMAC의 키 자리에 들어간다
  • PRK(Pseudorandom Key): 고정 길이의 균일한 의사난수

내부 동작은 단순하다:

PRK = HMAC-Hash(salt, IKM)

salt가 없으면 해시 길이만큼의 0 바이트가 대신 들어간다. salt가 있으면 IKM의 엔트로피 분포가 불균일하더라도 PRK의 품질이 보장된다. RFC 5869는 "salt 값은 비밀일 필요는 없지만, 가능하면 사용하라"고 권고한다.

Python으로 보면:

import hmac, hashlib

def hkdf_extract(salt: bytes, ikm: bytes, hash_func=hashlib.sha256) -> bytes:
    if not salt:
        salt = b'\x00' * hash_func().digest_size
    return hmac.new(salt, ikm, hash_func).digest()

Expand: 용도별 키 파생

HKDF-Expand(PRK, info, L) → OKM
  • PRK: Extract 단계의 출력
  • info: 컨텍스트 구분자. 같은 PRK에서 다른 용도의 키를 뽑을 때 사용
  • L: 원하는 출력 길이(바이트)
  • OKM(Output Keying Material): 최종 키

Expand는 HMAC을 카운터 모드로 반복 호출한다:

T(0) = empty string
T(1) = HMAC-Hash(PRK, T(0) || info || 0x01)
T(2) = HMAC-Hash(PRK, T(1) || info || 0x02)
T(3) = HMAC-Hash(PRK, T(2) || info || 0x03)
...
OKM = T(1) || T(2) || T(3) || ... 의 첫 L바이트

최대 출력 길이는 255 × HashLen이다. SHA-256 기준 8,160바이트.

import math

def hkdf_expand(prk: bytes, info: bytes, length: int,
                hash_func=hashlib.sha256) -> bytes:
    hash_len = hash_func().digest_size
    n = math.ceil(length / hash_len)
    okm = b''
    t = b''
    for i in range(1, n + 1):
        t = hmac.new(prk, t + info + bytes([i]), hash_func).digest()
        okm += t
    return okm[:length]

info 파라미터가 만드는 키 독립성

HKDF의 실용적 강점은 info 필드에 있다. 같은 PRK에서 info만 바꾸면 암호학적으로 독립된 키가 나온다:

prk = hkdf_extract(salt=b'my-salt', ikm=shared_secret)

# TLS 1.3 스타일 — 각 키가 서로 독립
client_key = hkdf_expand(prk, b'client_write_key', 32)
server_key = hkdf_expand(prk, b'server_write_key', 32)
client_iv  = hkdf_expand(prk, b'client_write_iv',  12)
server_iv  = hkdf_expand(prk, b'server_write_iv',  12)

info에 프로토콜 버전, 세션 ID, 용도 레이블 등을 인코딩하면 키 분리(key separation)가 자동으로 보장된다. 한 용도의 키가 유출되어도 다른 용도의 키를 역산할 수 없다.

실전에서 HKDF를 쓰는 프로토콜

프로토콜 HKDF 사용 방식
TLS 1.3 핸드셰이크 Secret → traffic key, IV, finished key 등 전체 키 스케줄을 HKDF로 도출
Signal Protocol Double Ratchet에서 체인 키 → 메시지 키 파생에 HKDF 사용
WireGuard Noise 프레임워크 기반, 핸드셰이크 체이닝에 HKDF 적용
Age (암호화 도구) 파일 암호화 키 도출에 HKDF-SHA-256 사용
AWS KMS 데이터 키 파생 과정에서 내부적으로 HKDF 변형 적용

TLS 1.3의 키 스케줄은 HKDF의 교과서적 활용 사례다. RFC 8446 §7.1을 보면, Early Secret → Handshake Secret → Master Secret으로 이어지는 체인 전체가 HKDF-Extract와 Derive-Secret(HKDF-Expand-Label의 래퍼)으로 구성되어 있다.

HKDF가 아닌 것: 패스워드 해싱과의 차이

HKDF는 이미 충분한 엔트로피를 가진 입력을 전제한다. 사람이 입력한 패스워드처럼 엔트로피가 낮은 값에는 적합하지 않다.

HKDF PBKDF2 / Argon2 / scrypt
입력 가정 높은 엔트로피 (DH 공유비밀, 하드웨어 난수 등) 낮은 엔트로피 (패스워드)
연산 비용 HMAC 1-수백 회 (의도적으로 빠름) 수만~수백만 반복 (의도적으로 느림)
목적 키 파생, 키 확장 브루트포스 저항
메모리 사용 무시할 수준 Argon2/scrypt는 의도적으로 메모리 소비

패스워드 → 암호화 키가 필요한 상황이라면 Argon2id로 먼저 높은 엔트로피의 중간 키를 만들고, 그 중간 키를 HKDF에 넣는 2단계 설계가 안전하다.

테이크어웨이

HKDF의 설계 철학은 "관심사의 분리"다. Extract는 엔트로피 품질을 정규화하고, Expand는 용도별 키를 찍어낸다. 두 단계가 분리되어 있기 때문에, 이미 균일한 입력이라면 Extract를 건너뛰고 Expand만 써도 안전하다(RFC 5869 §3.3).

암호 프로토콜을 설계하거나 읽을 때, "이 키는 어디서 왔는가"를 추적하면 거의 항상 HKDF가 나온다. 키 관리의 기반 인프라이면서 동시에 구현이 HMAC 호출 몇 번으로 끝나는 단순함 — 그 조합이 HKDF가 사실상 표준이 된 이유다.
