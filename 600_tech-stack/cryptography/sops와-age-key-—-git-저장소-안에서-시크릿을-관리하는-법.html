SOPS와 Age Key — Git 저장소 안에서 시크릿을 관리하는 법

.env 파일을 .gitignore에 넣고, Slack DM으로 동료에게 보내고, Notion 비공개 페이지에 붙여넣는다. 대부분의 팀에서 시크릿 관리라고 부르는 행위의 실체다. 문제는 이 방식이 "관리"가 아니라 "분산"이라는 점이다 — 시크릿이 어디에 몇 개나 있는지 아무도 모르고, 한 사람이 퇴사하면 키 로테이션 범위조차 특정할 수 없다.

SOPS와 Age는 이 문제를 정면으로 해결한다. 시크릿을 Git 저장소 안에 넣되, 값만 암호화한다.

핵심 아이디어: 키는 보이고, 값만 숨긴다

# .sops.yaml로 암호화된 secrets.yaml
db:
	host: ENC[AES256_GCM,data:dGVzdA==,iv:...,tag:...,type:str]
	port: 5432                    # ← 이건 평문
	password: ENC[AES256_GCM,data:c2VjcmV0,iv:...,tag:...,type:str]
app:
	api_key: ENC[AES256_GCM,data:YWJjMTIz,iv:...,tag:...,type:str]
	debug: false                  # ← 이것도 평문

SOPS(Secrets OPerationS)는 Mozilla가 만든 시크릿 관리 도구로, YAML·JSON·ENV·INI 파일에서 값(value)만 선택적으로 암호화한다. 키(key)와 구조는 그대로 노출되므로 git diff로 "어떤 설정이 바뀌었는지"를 확인할 수 있고, 코드 리뷰에서 구조 변경을 추적할 수 있다.

파일 전체를 암호화하는 age -e나 gpg -e와의 결정적 차이다.

Age 키 생성부터 SOPS 암호화까지

1. Age 키 생성

$ age-keygen -o ~/.config/sops/age/keys.txt
Public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p

Age 키는 한 줄짜리 텍스트다. 키링도 없고, 설정 파일도 없다. 공개키는 age1로 시작하는 Bech32 문자열이고, 이것만 있으면 누구든 나를 위해 암호화할 수 있다.

2. SOPS 설정 (.sops.yaml)

프로젝트 루트에 .sops.yaml을 만들어 암호화 규칙을 선언한다:

creation_rules:
	- path_regex: \.enc\.yaml$
	  age: >-
	    age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p,
	    age1abc...second_recipient_public_key
  • path_regex: 어떤 파일을 암호화 대상으로 삼을지
  • age: 복호화 가능한 수신자 목록 (쉼표로 여러 명 지정)

팀원이 추가되면 공개키를 한 줄 추가하고 sops updatekeys를 실행하면 된다. 퇴사하면 한 줄 삭제 후 같은 명령.

3. 암호화 / 복호화

# 새 시크릿 파일 생성 (에디터가 열림)
$ sops secrets.enc.yaml

# 기존 평문 파일을 암호화
$ sops -e plaintext.yaml > secrets.enc.yaml

# 복호화해서 stdout으로 출력 (파일 안 건드림)
$ sops -d secrets.enc.yaml

# 특정 값만 추출
$ sops -d --extract '["db"]["password"]' secrets.enc.yaml

SOPS는 Age 비밀키를 SOPS_AGE_KEY_FILE 환경변수 또는 ~/.config/sops/age/keys.txt에서 자동으로 찾는다. 별도 에이전트나 데몬이 필요 없다.

왜 Age인가 — GPG, AWS KMS와의 비교

SOPS는 Age 이전에도 GPG, AWS KMS, GCP KMS, Azure Key Vault, HashiCorp Vault를 백엔드로 지원했다. Age가 v3.7에서 추가된 이후 빠르게 기본 선택지가 된 이유는 단순함이다.

기준 Age GPG AWS KMS
키 생성 age-keygen 1줄 키링 생성 + 신뢰 설정 + 서브키 IAM 권한 + KMS 키 정책
키 배포 공개키 문자열 공유 키서버 또는 파일 교환 IAM Role 연동
로컬 실행 가능 가능 AWS 인증 필요
CI/CD 설정 비밀키 1줄 → 환경변수 GPG 키링 import 필요 OIDC 또는 Access Key
의존성 age 바이너리 하나 gpg + gpg-agent aws-cli + 네트워크

KMS 계열은 프로덕션 인프라에서 강력하지만, 개인 프로젝트나 소규모 팀에서는 오버헤드가 크다. Age는 "인프라 없이 즉시 시작"이라는 간극을 정확히 메운다.

실전 확장: Git을 넘어서

chezmoi — dotfile 속 시크릿

# chezmoi가 Age로 암호화된 .env를 관리
$ chezmoi add --encrypt ~/.env
$ chezmoi cd && git add . && git commit -m "add encrypted .env"

Age와 chezmoi를 연동하면, dotfile 저장소를 퍼블릭으로 공개하면서도 .env, SSH 키, API 토큰 등은 암호화된 채 커밋할 수 있다. 새 머신에서 chezmoi apply 한 줄이면 모든 설정이 복원된다.

GitHub Actions — CI/CD 시크릿

# .github/workflows/deploy.yml
jobs:
	deploy:
	  steps:
	    - name: Decrypt secrets
	      env:
	        SOPS_AGE_KEY: ${{ secrets.SOPS_AGE_KEY }}
	      run: |
	        sops -d config/secrets.enc.yaml > .env
	        source .env
	        ./deploy.sh

GitHub Secrets에 Age 비밀키 하나만 등록하면 된다. KMS 연동이나 OIDC 설정 없이, 저장소 안의 암호화된 파일을 그 자리에서 복호화한다. GitHub Actions 워크플로우에서 별도 시크릿 매니저 없이 동작하므로 인프라 비용이 0이다.

Kubernetes — Helm values 암호화

# Helm values 파일의 시크릿만 암호화
$ sops -e values-secret.yaml > values-secret.enc.yaml

# 배포 시 복호화 후 Helm에 전달
$ sops -d values-secret.enc.yaml | helm upgrade myapp ./chart -f -

helm-secrets 플러그인을 쓰면 helm install 명령에 SOPS 복호화를 투명하게 통합할 수 있다. 클러스터에 Sealed Secrets나 External Secrets Operator를 설치하지 않아도 되는 경량 대안이다.

백업 암호화 — restic / borgmatic

# 백업 파이프라인: tar → age 암호화 → restic 저장
$ tar cf - /important/data \
    | age -r age1ql3z7hjy... \
    | restic backup --stdin --stdin-filename backup.tar.age

백업 데이터를 클라우드 스토리지에 올릴 때, Age로 파이프 암호화하면 스토리지 제공자조차 내용을 볼 수 없다. 복호화 키만 안전하게 보관하면 된다.

키 관리 실무: 잃어버리면 끝이다

Age 키의 최대 약점은 곧 최대 강점이기도 한 단순함이다. 키를 잃어버리면 복구 메커니즘이 없다. KMS처럼 클라우드가 키를 보관해주지 않는다.

실전에서 검증된 키 관리 패턴:

  • 다중 수신자: .sops.yaml에 팀원 전원의 공개키를 등록한다. 한 명의 키가 유실되어도 다른 멤버가 복호화 가능
  • 비상 키(break-glass key): 팀과 별도로 관리하는 마스터 키를 오프라인 저장소(USB, 금고)에 보관
  • 키 로테이션: 새 키 생성 → .sops.yaml 업데이트 → sops updatekeys → 전체 파일 재암호화
  • 키 파일 권한: chmod 600 keys.txt — 소유자만 읽기 가능
# 키 로테이션: 새 키 추가 후 전체 재암호화
$ sops updatekeys secrets.enc.yaml
# → .sops.yaml의 최신 수신자 목록으로 파일 키를 재래핑

.env를 Slack으로 보내던 시절과의 차이

기존 방식 SOPS + Age
.env를 .gitignore에 추가 .enc.yaml을 Git에 커밋
Slack/Notion으로 시크릿 공유 공개키만 공유, 암호화 파일은 Git
퇴사자 키 로테이션 범위 불명 .sops.yaml에서 키 삭제 → updatekeys
CI/CD에서 vault 연동 필요 Age 비밀키 1줄이면 복호화
로컬-스테이징-프로덕션 설정 분리 불명확 path_regex로 환경별 규칙 지정
"누가 마지막으로 DB 비번 바꿨지?" git log secrets.enc.yaml로 추적

가져갈 것

시크릿 관리의 핵심 문제는 암호학이 아니다. 시크릿이 어디에 있는지 모르는 것이다. SOPS + Age는 시크릿을 코드와 같은 저장소에 두면서, 값만 암호화하여 Git의 버전 관리·코드 리뷰·접근 제어를 그대로 활용하게 한다.

시작하는 데 필요한 것은 brew install sops age 두 줄과, age-keygen 한 번이다. 별도 서버도, 클라우드 설정도, 팀 온보딩 문서 10페이지도 필요 없다. 30초 만에 첫 번째 시크릿을 암호화할 수 있고, 그 파일을 당장 git push 할 수 있다.
