Argon2 — GPU와 ASIC을 무력화하는 패스워드 해싱

2012년 6월, LinkedIn에서 유출된 650만 개의 패스워드 해시가 온라인에 공개됐다. SHA-1 단일 해싱이었고, 공격자들은 수 시간 만에 대부분을 평문으로 복원했다. 이 사건은 "빠른 해시 함수는 방어가 아니라 공격자의 무기"라는 사실을 증명한 대표적 사례다.

SHA-1이 무너진 이유

# SHA-1 해싱 — 단 한 번의 연산
echo -n "password123" | sha1sum
# -> cbfdac6008f9cab4083784cbd1874f76618d2a97

# GPU(RTX 4090) 기준 SHA-1 처리 속도: ~26 billion hashes/sec

SHA-1, MD5 같은 범용 해시 함수는 **속도를 목적으로 설계**됐다. 파일 무결성 검증, 체크섬 계산에는 빠를수록 좋다. 하지만 패스워드 해싱에서 속도는 곧 취약점이다. GPU 한 장으로 초당 수백억 개의 후보를 시도할 수 있기 때문이다.

bcrypt와 PBKDF2도 부족한 이유

bcrypt(1999)와 PBKDF2(2000)는 의도적으로 느린 해시 함수로, 반복 횟수(iteration count)를 높여 연산 비용을 올린다. 수년간 표준으로 쓰였고 지금도 많은 시스템에서 동작 중이다. 문제는 이들이 **CPU-bound**라는 점이다.

  • bcrypt: 4KB 내부 상태만 사용 — GPU 코어 수천 개에 각각 올릴 수 있음

  • PBKDF2: 메모리 요구량이 사실상 0 — ASIC 설계가 용이

  • 2024년 기준 ASIC 한 대가 bcrypt cost=12를 초당 수만 회 처리 가능

    연산 시간만 늘리는 전략은 병렬 하드웨어 앞에서 한계에 도달한다. 필요한 것은 시간뿐 아니라 메모리를 강제하는 함수다.

Argon2가 메모리를 무기로 쓰는 방법

2015년 Password Hashing Competition(PHC) 우승 알고리즘인 Argon2는 의도적으로 대량의 메모리를 할당한 뒤, 그 메모리 블록들을 반복적으로 참조한다. 핵심 아이디어는 단순하다.

[블록 0] -> [블록 1] -> [블록 2] -> ... -> [블록 N]
   ^            |            ^                 |
   +--- 이전 블록 참조 ---+--- 비순차 참조 ---+

  • 각 1KB 블록은 이전 블록과 데이터 의존적 인덱스의 블록을 입력으로 생성

  • 메모리를 줄이면(trade-off attack) 같은 블록을 재계산해야 하므로 시간이 기하급수적으로 증가

  • GPU는 코어당 사용 가능한 메모리가 제한적 — 64MB를 요구하면 RTX 4090(24GB)에서 동시 인스턴스가 ~375개로 제한

    이것이 memory-hardness다. 연산만 빠르게 돌려서는 공격할 수 없다.

세 가지 변형: Argon2d, Argon2i, Argon2id

변형 메모리 접근 패턴 사이드채널 내성 주요 용도
Argon2d 데이터 의존적 (data-dependent) 낮음 암호화폐 채굴, 서버 백엔드
Argon2i 데이터 독립적 (data-independent) 높음 로그인 서버 (타이밍 공격 우려 시)
Argon2id 첫 패스 Argon2i + 이후 Argon2d 하이브리드 RFC 9106 권장 기본값

  • Argon2d: 비밀 값에 따라 메모리 접근 순서가 바뀜 — GPU 공격에 가장 강하지만 사이드채널(타이밍 분석)에 취약

  • Argon2i: 접근 순서가 입력과 무관 — 사이드채널에 강하지만 GPU 내성이 Argon2d보다 약간 낮음

  • Argon2id: 첫 번째 반복에서 Argon2i 패턴으로 사이드채널을 방어하고, 이후 Argon2d로 전환해 GPU 내성 확보

    실무에서는 Argon2id를 선택하면 된다. RFC 9106이 명시적으로 권장하는 변형이다.

파라미터 튜닝: 시간, 메모리, 병렬성

Argon2는 세 개의 파라미터로 비용을 조절한다.
파라미터 의미 OWASP 최소 권장값
time_cost (t) 반복 횟수 1 (메모리가 충분할 때)
memory_cost (m) 사용 메모리 (KiB) 47,104 (46 MiB)
parallelism (p) 스레드 수 1 
OWASP 2024 가이드라인이 제시하는 설정 조합:

  • 1순위: m=47104 (46 MiB), t=1, p=1 — 메모리 우선

  • 2순위: m=19456 (19 MiB), t=2, p=1 — 메모리 제약 시

  • 3순위: m=12288 (12 MiB), t=3, p=1 — 더 적은 메모리 환경

    튜닝 원칙은 명확하다: 메모리를 먼저 최대한 올리고, 그래도 응답 시간이 남으면 time_cost를 올린다. 로그인 요청 기준 해싱에 0.5~1초가 소요되도록 맞추는 것이 일반적이다.

Python 구현 예제

from argon2 import PasswordHasher

# OWASP 권장 설정으로 초기화
ph = PasswordHasher(
	time_cost=1,
	memory_cost=47104,   # 46 MiB
	parallelism=1,
	hash_len=32,
	type=argon2.Type.ID  # Argon2id
)

# 해싱
password = "correct-horse-battery-staple"
hashed = ph.hash(password)
print(hashed)
# $argon2id$v=19$m=47104,t=1,p=1$c29tZXNhbHQ$...

# 검증
try:
	ph.verify(hashed, password)
	print("인증 성공")
except argon2.exceptions.VerifyMismatchError:
	print("패스워드 불일치")

# 파라미터 변경 감지 (rehash 필요 여부)
if ph.check_needs_rehash(hashed):
	new_hash = ph.hash(password)
	# DB 업데이트

`argon2-cffi`는 해시 문자열 안에 알고리즘, 버전, 파라미터, salt를 모두 포함한다. 별도의 salt 컬럼이 필요 없고, `check_needs_rehash()`로 파라미터 업그레이드도 자동 감지한다.

해시 함수 비교: MD5에서 Argon2까지

함수 연도 메모리 사용 GPU 내성 현재 상태
MD5 1992 없음 없음 완전 폐기 — 충돌 공격 가능
SHA-1 1995 없음 없음 폐기 — 2017 SHAttered 공격
bcrypt 1999 4 KB 낮음 레거시 — ASIC 공격 가능
scrypt 2009 설정 가능 중간 사용 가능 — 파라미터 튜닝 복잡
Argon2id 2015 설정 가능 (MB~GB) 높음 RFC 9106 표준, OWASP 1순위 권장

Argon2와 HKDF의 역할 분담

패스워드 기반 암호화 시스템에서 Argon2와 [[HKDF]]는 서로 다른 단계를 담당한다.

사용자 패스워드
    │
    ▼
 [Argon2id] ──── 느리고 메모리 집약적 → intermediate key (IKM)
    │
    ▼
 [HKDF-Extract] ─ salt + IKM → PRK (pseudorandom key)
    │
    ▼
 [HKDF-Expand] ── PRK → encryption key, MAC key, IV 등

  • Argon2: 저엔트로피 패스워드를 공격 비용이 높은 중간 키로 변환 (rate-limiting)

  • HKDF: 중간 키에서 용도별 키를 안전하게 파생 (key separation)

    Age 암호화 도구가 이 패턴을 사용한다 — scrypt/Argon2로 패스워드를 키로 변환한 뒤, HKDF로 파일 키를 파생한다.

도입 시 체크리스트

  • Argon2id를 기본으로 선택한다 (RFC 9106 Section 4)

  • 서버 메모리 예산 내에서 memory_cost를 최대한 높인다

  • 해싱 시간이 로그인 UX에 영향을 주지 않는 선에서 time_cost를 조절한다

  • 해시 문자열을 그대로 저장한다 — salt와 파라미터가 내장되어 있다

  • check_needs_rehash()로 기존 해시의 파라미터 업그레이드를 자동화한다

  • 패스워드에서 암호화 키를 파생해야 한다면, Argon2 출력을 HKDF에 넘긴다

    패스워드 해싱의 목표는 해시를 깨지 못하게 만드는 것이 아니다. 공격 비용을 공격자의 예산보다 높게 만드는 것이다. Argon2id는 그 비용의 단위를 '시간'에서 '시간 x 메모리'로 바꿨고, 그것이 GPU 수천 장을 무력화하는 이유다.
