0. 프롤로그 — 왜 지금 Harness인가

0-1. 프롬프트만으로는 부족하다

AI와의 협업은 짧은 기간에 세 레벨을 거쳐 왔다.


<프롬프트 → 컨텍스트 → 하네스. AI와 일하는 방법의 추상화 레벨이 올라간다.>

레벨 패러다임 질문 대표 도구
1 Prompt Engineering "이렇게 말해봐" 프롬프트 템플릿
2 Context Engineering "이런 배경지식을 줘봐" RAG, 지식베이스
3 Harness Engineering "환경 자체를 설계하자" 스캐폴딩·규칙·검증 루프

프롬프트가 무엇을 말할 것인가의 문제라면, 컨텍스트는 무엇을 알게 할 것인가의 문제다. 하네스는 그 둘을 품고 "에이전트가 어떤 공간 안에서 움직이게 할 것인가" 를 다룬다.

0-2. 빠른 테크 회사들의 데이터 포인트


<같은 모델, 다른 하네스 — 이미 빠른 회사들은 하네스에 베팅하고 있다.>

  • LangChain — 모델 고정, 하네스만 바꿔서 TerminalBench 52.8→66.5%, 벤치마크 순위 30위 → Top 5
  • OpenAI Codex 팀 — 엔지니어 3~7명이 5개월 동안 단 한 줄도 수동 코드 없이 100만 줄 프로덕션 코드 를 구축. 쓴 시간은 "더 좋은 모델 찾기"가 아니라 전부 환경 설계였다
  • Anthropic — 싱글 에이전트 $9 실패 vs 3-에이전트 하네스 6h/$200 완성. 모델이 아니라 하네스가 차이를 만든다
  • Stripe — 매주 1,000 PR 머지. 전문화된 소형 에이전트 + 6계층 하네스(격리 VM → 린터 → CI → 상한 → 하이브리드 → 인간 리뷰)

모델 교체로 5% 개선하는 것보다, 하네스 설계로 15% 개선하는 것이 현실적이다.

0-3. 정의 — "작업 환경을 만들어주는 것"


<Harness Engineering의 정의 — 맥락·제한·작업 흐름·검증을 포괄하는 환경 설계.>

Harness Engineering: AI가 혼자서도 잘 일할 수 있는 작업 환경을 만들어주는 것.

  • OpenAI: "사람이 방향을 잡고, 에이전트가 실행한다."
  • Anthropic: "규율은 코드가 아니라 스캐폴딩에서 드러난다."

좁은 의미에서 Harness는 가드레일·툴셋·플러그인이지만, 이 강의에서 말하는 Harness는 그 모두를 품는 "작업 환경 자체의 설계" 다. 특정 스킬 하나, 특정 에이전트 조합 하나가 아니라 맥락·제한·흐름·검증을 관통하는 개념이다.

Agenda — Harness의 6개 축


<Harness의 6개 축 — 순환 구조. 한 바퀴 돌고 나면 다음 바퀴가 더 좋아진다.>

축 영문 질문 본문
구조 Scaffolding 뭘 깔아두는가 §1
맥락 Context AI가 뭘 아는가 §2
계획 Planning 뭘 할지 정하는가 §3
실행 Execution 어떻게 시키는가 §4
검증 Verification 어떻게 믿는가 §5
개선 Compounding 어떻게 나아지는가 §6

이 6개 축은 선형(線形)이 아니라 순환 루프다. 검증 결과가 맥락에 돌아가고, 개선이 구조를 다시 다듬는다. 한 사이클을 돌수록 팀 생산성의 저점이 올라간다.

1. 구조(Scaffolding) — 한 번 깔아두면 계속 쓰는 것

1-1. 프로젝트 구조가 품질을 결정한다


<폴더링 & 아키텍처 — 3개월 후, 구조 없는 프로젝트는 30%, 클린 아키텍처는 85%의 AI 품질 유지율을 보인다.>

핵심 원리 3가지:

  1. Monorepo로 묶기 — 소스코드·문서·테스트·설정을 한 프로젝트에서 관리해 AI가 전체 맥락을 한 눈에 본다
  2. 역할별 폴더링 — 목적이 명확한 폴더 구조. AI는 "테스트 작성해줘"라는 말을 듣자마자 tests/ 로 간다
  3. 아키텍처가 품질을 결정한다 — 레이어별 책임이 분리된 클린 아키텍처일수록 시간이 갈수록 AI 산출물의 품질이 올라간다. 구조가 없으면 파일 100개가 평면적으로 쌓이면서 품질이 떨어진다

표준 스캐폴딩:

my-project/
├── src/        # 비즈니스 로직
├── docs/       # AI가 참고할 문서
├── tests/      # 검증 인프라
├── .dev/       # 개발 도구·스크립트
├── .claude/    # AI 설정
├── out/        # 빌드 산출물
└── CLAUDE.md   # 프로젝트 지도

1-2. 사람의 문서 vs AI의 문서 — 분리해서 관리


<docs/는 사람의 책임, .dev/는 AI의 흔적. 설정 파일은 그 둘을 참조하고 지탱한다.>

위치 담당 역할
docs/ 사람 비즈니스 룰·도메인 정의·체크리스트·ADR·API 스펙
.dev/ AI learnings, troubleshooting 기록, 작업 로그, 디버깅 히스토리, 실험 결과
CLAUDE.md 사람 프로젝트 지도(~100줄) — docs/ 와 .claude/rules/ 로 포인팅
.claude/rules/ 사람 코딩 규칙·테스트 컨벤션. glob 패턴으로 조건부 로드
.claude/skills/ 사람+AI 반복 작업 레시피 (/commit, /review 등)
.claude/* 외 사람 hooks, agents, MCP, plugins

한 줄 원칙: 사람의 관리가 멈추는 순간 AI도 엉뚱한 맥락으로 일한다. docs/ 는 사람의 책임이다.

1-3. AI 도구 배치하기


<구조에 맞춰 도구를 배치 — Skills · Hooks · Agents · MCP · Plugins.>

컴포넌트 역할
Skills 반복 작업의 레시피화. /commit, /review 같은 슬래시 명령
Hooks 자동 안전장치. Pre(차단) / Post(검사) / Stop(일지) / Notification(알림)
Agents 전문가 팀원. 서브에이전트 파견 또는 팀 구성
MCP 외부 시스템 연결. DB · Slack · Linear 등 연동
Plugins 위 컴포넌트를 한 패키지로 묶어 배포·공유

1-4. 경계 설정 — 뭘 알려줄지, 어디까지 허용할지, 뭘 막을지


<3단 경계 — 알리는 층(CLAUDE.md) · 허용 층(Permission Mode) · 차단 층(Hooks).>

층 도구 예시
뭘 알려줄까 CLAUDE.md · .claude/rules/ 프로젝트 규칙·코딩 스타일·금기사항
어디까지 허용할까 settings.json(permission mode: plan/auto/bypass) "읽기는 자동, 쓰기는 승인"
뭘 막을까 .claude/hooks/ 위험 명령을 실행 전에 자동 차단

이중 안전장치 예: CLAUDE.md에 "직접 main push 금지" 작성 + Hook으로 git push --force 차단 → 사람이 잊어도 시스템이 기억한다.

1-5. Try It — 구조 설계를 도와주는 스킬

  • /scaffold (Harness Plugin) — 프로젝트 초기 구성을 자동으로 잡아준다. CLAUDE.md·rules·폴더 구조까지 한 번에 생성
  • /check-harness (Harness Plugin) — 현재 프로젝트의 Harness 상태를 체크리스트로 진단. 빠진 설정, 개선 포인트를 알려준다
  • /skill-creator (Anthropic 공식) — 스킬 생성 + 성능 점검 도구

2. 맥락(Context) — AI가 뭘 알고 일하는가

2-1. 설정 파일 한눈에 보기


<아래로 갈수록 범위가 좁아지고 우선순위가 높아진다.>

~/.claude/CLAUDE.md         ← 나만 적용 · 모든 프로젝트
└── my-app/
    ├── CLAUDE.md           ← 팀 공유 · Git 커밋
    ├── .claude/
    │   └── rules/          ← 주제별 분리 · glob으로 조건부 적용
    │       ├── code-style.md    (*.ts, *.tsx)
    │       ├── testing.md       (*.test.*, __tests__/**)
    │       └── security.md      (**/auth/**, *.sql)
    └── src/auth/
        └── CLAUDE.md       ← 이 폴더 작업 시에만 자동 로드

2-2. 상속 구조 — 하위가 상위를 덮어쓴다


<User → Project → Folder. 좁은 범위가 우선한다.>

스코프 위치 용도 덮어쓰기
User ~/.claude/CLAUDE.md 모든 프로젝트 공통 (내 습관, 스타일) 기본
Project my-app/CLAUDE.md 이 프로젝트 전용 (스택, 컨벤션) User를 오버라이드
Folder src/auth/CLAUDE.md 특정 모듈의 특수 규칙 Project를 오버라이드
  • User에 "camelCase 사용" → 모든 프로젝트 적용
  • Project에 "snake_case 사용" → 이 프로젝트만 snake_case
  • Folder에 "JWT만 사용" → auth/ 작업 시만 적용

2-3. CLAUDE.md 실전 가이드

스코프 담는 것
User 내 작업 습관, 선호 코딩 스타일, 공통 규칙
Project 이 프로젝트의 기술 스택, 컨벤션, 중요한 제약
Folder 특정 모듈의 특수 규칙

팁: 최대 200줄 정도를 유지하며 계속 업데이트하라. 너무 길어지면 AI 성능이 급격히 저하된다.

2-4. 맥락 관리의 핵심 원칙


<Progressive Disclosure · .claude/rules/ · Scope 계층. 한꺼번에 다 주면 AI도 헷갈린다.>

  1. Progressive Disclosure — 필요한 것만 필요할 때 보여주기
  2. .claude/rules/ — 상황별로 세분화된 규칙. CLAUDE.md가 길어지면 분리
  3. Scope 계층 — User / Project / Folder 각 레벨에 맞는 정보 배치

2-5. Progressive Disclosure — 덜 주고 정확히 가리키기


<다 때려넣지 말고, "이런 상황에서는 이걸 참고해"라고 가이드한다.>

핵심 아이디어: 프롬프트에 "이 상황에서는 이 문서를 읽어" 라고 가이드를 주면, AI가 동적으로 필요한 문서만 읽는다.

SKILL.md 또는 CLAUDE.md 안에서
	코드 작성 시 → references/code-style.md 참고
	테스트 시   → references/testing-guide.md 참고
	API 설계 시 → references/api-convention.md 참고
	배포 시     → references/deploy-checklist.md 참고

my-skill/
	SKILL.md
	references/
		code-style.md
		testing-guide.md
		api-convention.md
		deploy-checklist.md

CLAUDE.md (핵심만 30줄)
docs/
	architecture.md
	conventions.md

결과: SKILL/CLAUDE.md에 다 넣는 게 아니라 폴더로 분리하고 상황별로 참조시킨다 → 컨텍스트 절약. 이 패턴은 260406 AI 에이전트 문서화와 하네스 설계의 "토큰 경제학" 논의와 맞닿는다.

2-6. 세션 맥락 관리 — 쌓이면 비워라


<컨텍스트는 채우는 것만큼 비우는 것도 중요하다.>

사용률 상태 행동
~20% 쾌적 그대로 진행
~50% 주의 /compact
~80% 위험 /clear 또는 새 세션
명령 용도
/clear 컨텍스트 완전 초기화. 다른 주제로 전환할 때
/compact 오래된 대화를 요약·압축. 같은 주제를 이어갈 때
handoff 현재 세션의 맥락을 파일로 저장 → 새 세션에서 이어받기. 맥락 손실 없이 전환

이호연 기준: 20~30% 되면 새로 시작. 아예 다른 맥락의 작업을 하면 /clear, 이어가야 하면 handoff 로 맥락을 넘긴다.

2-7. .claude/rules/ 가이드

작동 방식: .claude/rules/ 폴더에 .md 파일을 놓으면 Claude Code가 자동으로 읽는다. 파일명에 glob 패턴을 넣으면 해당 파일 작업 시에만 로드된다.

# .claude/rules/ 구조
code-style.md   # 항상 로드
testing.md      # 항상 로드
react-*.md      # *.tsx 작업 시만
api-design.md   # routes/ 작업 시만

규칙 파일 상단에 glob 패턴을 지정하면 조건부 로드가 된다.

security.md
glob: **/*.sql, **/auth/**
→ .env 직접 수정 금지, SQL은 반드시 parameterized query

testing.md
glob: **/*.test.*, **/__tests__/**
→ 커버리지 80%+, mock 최소화, 통합테스트 우선

code-style.md
(항상 로드)
→ 함수 20줄 이내, camelCase, 주석은 영어

2-8. 주기적으로 점검하기


<Skill · MCP는 Context를 먹는다. 안 쓰는 것은 주기적으로 정리하자.>

  • Skill · MCP는 Context를 먹는다 — 생각보다 context를 많이 차지하고 pollution이 생길 수 있다. /context 로 현재 사용량을 확인하자
  • CLAUDE.md가 비대해질 때 — 관리 안 되면 중복·대치되는 내용이 쌓인다. AI한테 직접 점검시킨다:

    "CLAUDE.md, .claude/rules를 분석해서 논리적으로 문제가 없는지 + 중복되거나 대치되는 내용은 없는지 분석해줘"

쌓는 것만큼 점검하고 비우는 것 도 Context Engineering이다.

3. 계획(Planning) — AI에게 일 시키기 전에, 무엇을 할지 함께 정하기

3-1. 핵심 흐름: 계획 → 실행 → 검증


<한 번에 완벽하게가 아니라, 반복해서 수렴하는 구조.>

3-2. "해줘"의 함정


<"해줘" 안티패턴 vs "같이 계획부터" — 시간만 날리기 vs 높은 성공률.>

안티패턴: "해줘" 권장: "같이 계획부터"
"이거 만들어줘" "~~할 건데 같이 계획 세워보자"
AI가 알아서 만듦 AI가 계획 작성
검수 → "아닌데..." 사람이 검토 → 수정 → 승인
다시 시킴 → 검수 → 다시... 승인된 계획대로 실행
시간만 날림 높은 성공률

계획과 실행을 분리하면 검수 횟수가 줄고, 결과의 예측 가능성이 올라간다.

3-3. AskUserQuestion — 나에게 질문하게 만들기


<"해줘"가 아니라 "물어봐" — AI가 스스로 맥락을 채우게 한다.>

왜 질문이 중요한가: 사람은 자기 머릿속의 전제를 프롬프트에 다 담지 못한다. AI가 질문을 해야 빠진 맥락이 드러난다.

커스텀 Plan 스킬에서: 인터뷰 단계를 넣으면 AI가 자동으로 AskUserQuestion을 사용해 모호한 점을 물어본다. 스킬 프롬프트에 다음 문장을 넣는다:

"이해한 것을 미러링하고, 모호한 점을 질문해서 명확하게 해줘"

예시 대화:

USER: 나 결제 시스템을 리팩토링하려고 해. 네가 지금 이해한 게 뭔지
      정리해주고, 모호한 점이 없도록 내게 계속 질문해서 명확하게 해줘

AI:   이해한 것: 결제 모듈의 기존 구조를 개선하려는 것
      질문 1: 현재 PG사 연동이 단일인가요, 멀티인가요?
      질문 2: 구독 결제도 포함되나요?
      질문 3: DB 스키마 변경도 허용 범위인가요?

USER: 멀티 PG, 구독 포함, DB 스키마는 건드리지 마

AI:   명확해졌습니다. 그러면 계획을 세우겠습니다...

3-4. 계획의 진화 — 커스텀 Plan 스킬


<기본 Plan → 커스텀 Plan 스킬. 인터뷰 + 요구사항 도출 + 플랜 파일까지 자동화.>

  1. Plan Mode (기본) — 계획과 실행 분리. 하지만 사람이 모든 걸 프롬프트에 담을 수 없다
  2. 커스텀 Plan 스킬 — 인터뷰 + 요구사항 도출 + 플랜 파일까지 자동화하는 전용 스킬을 만든다

예시: /specify 스킬의 내부 흐름

1. 목표 확인  — 사용자의 의도를 미러링
2. 인터뷰   — 모호한 지점을 질문으로 끌어내기
3. 요구사항 + 태스크 도출
4. 플랜 파일로 떨구기 → /execute 로 실행

핵심 전환: implicit(머릿속) → explicit(플랜 파일).

3-5. Try It — 계획을 더 잘 세우기 위한 스킬

  • /specify — 스펙 파일을 더 잘 얻기 위한 스킬. 인터뷰 → 요구사항 도출 → 플랜 파일까지 자동화
  • /deep-interview — 깊이 있는 인터뷰로 unknown-unknown을 줄인다. 미처 생각 못한 엣지 케이스까지 끌어낸다
  • /clarify (plugins-for-claude-natives) — 요구사항을 명확하게 하고 싶을 때. 모호한 지시를 구체적인 스펙으로 변환

계획 단계에서 모호함을 줄이는 것 이 실행 품질을 결정한다.

4. 실행(Execution · Orchestration) — 혼자 vs 부하 파견 vs 팀

4-1. 실행 패턴 3종


<Single · Subagent · Team Mode. 90%는 단일/서브에이전트로 충분하고, Team Mode는 토큰이 ~7배 든다.>

패턴 구조 적합한 상황
혼자(Single) AI 1명 → 결과물 단순 작업. 대부분의 일상 작업
부하 파견(Subagent) 메인 → [조사·작업·검증] 병렬 → 메인이 종합 병렬·전문화가 필요한 작업
팀 협업(Team Mode) PM·개발·디자인·QA가 상호 소통 다관점·복잡. 에이전트 간 통신 필요

비용 감각: 90%는 단일/서브에이전트로 충분하다. Team Mode는 토큰이 약 7배 든다 — 정말 필요한 복잡도에만 쓴다.

4-2. 상황별 오케스트레이션 패턴


<순차 파이프라인 · 병렬 Subagent · Team Mode — 상황에 따라 다른 패턴을 쓴다.>

패턴 상황 명령 / 동작
순차 파이프라인 블로그 글: 조사 → 초안 → 퇴고 → 발행 순서가 중요 TaskCreate 로 체크박스 생성, 하나씩 직렬 수행
병렬 Subagent 경쟁사 3곳의 랜딩 페이지를 동시에 분석(서로 독립) Agent 3개 spawn, 병렬 분석 후 메인이 취합
Team Mode 새 기능을 설계자·구현자·리뷰어가 동시에 협업 TeamCreate 로 팀 생성, 에이전트 간 직접 소통

4-3. Ralph Loop — 될 때까지 반복


<완료 기준을 정하고, 충족할 때까지 AI가 알아서 돈다.>

1. 완료 기준 합의  ←  사람이 한 일
2. AI가 작업
3. 기준 충족? — NO → 2번으로
          — YES → 끝

예시: "랜딩 페이지 만들어줘"

  • 완료 기준: 모바일 반응형 + Lighthouse 90점 이상 + 카피 3번 이상 퇴고
  • 1차: 페이지 완성, Lighthouse 72점 → 실패
  • 2차: 92점, 모바일 레이아웃 깨짐 → 실패
  • 3차: 레이아웃 수정, 카피 퇴고 2회만 → 실패
  • 4차: 전항목 충족 → PASS

사람이 한 일: 기준 정하기 1번. 나머지는 AI가 알아서.

4-4. Auto Research — 자율 실험 루프


<karpathy/autoresearch — 사람은 방향만 정하고, AI가 밤새 실험을 돌린다.>

핵심 패턴: 코드 수정(train.py) → 실행(5분 학습) → 평가(성능 비교) → 판단(유지/폐기) → 다음 실험.

  • program.md 에 연구 방향 작성 (스킬과 같은 역할)
  • AI가 train.py 만 수정 — 아키텍처, 하이퍼파라미터 변경
  • 시간당 ~12개 실험 자율 수행, 밤새 무인 운영
  • 성능 개선되면 유지, 아니면 폐기 → 다음 실험

Ralph Loop의 확장판: 사람이 일정 시간 손을 떼도 성과가 수렴한다.

4-5. Try It — 실행을 도와주는 도구들

  • /agent-orchestrate (Harness Plugin) — 현재 문제에 최적화된 오케스트레이션 패턴을 자동 적용. 단일/병렬/파이프라인 선택
  • ralph (Claude Code 공식 Plugin) — 실행 결과를 자동 검증하는 공식 플러그인
  • autoresearch (karpathy) — 자율 실험 루프

5. 검증(Verification) — 결과물을 어떻게 믿을 것인가

5-1. 원칙 01 — 기준이 있어야 검증이 가능하다


<Sprint Contract — 작업 전에 "뭘 만들고 어떻게 검증할지" 합의한다.>

  1. 완료 조건을 먼저 정한다 — "이 3가지가 되면 끝" 을 시작 전에 합의
  2. 조건을 측정 가능하게 쓴다 — "잘 되게" 말고 "테스트 통과 + 빌드 성공"
  3. 미달이면 다시 돌린다 — 기준이 있으니 자동 반복이 가능해진다(Ralph Loop)

5-2. 원칙 02 — 컨텍스트를 나누고, 관점을 분리한다


<Generator vs Evaluator — 같은 컨텍스트에서 만들고 평가하면 안 된다.>

Anthropic: "만든 AI와 확인하는 AI를 분리하는 것이 가장 강력한 레버다."
"Evaluator를 회의적으로 튜닝하는 게, Generator를 자기비판적으로 만드는 것보다 훨씬 쉽다."

자기 작업을 평가하는 AI는 quality가 mediocre해도 자신 있게 칭찬한다. 관점을 물리적으로 분리해야 한다. 이는 하네스 엔지니어링(Harness Engineering) 이 다루는 "아키텍처 제약으로서의 검증 레이어"와 같은 원리다.

5-3. 원칙 03 — 모델도 나누고, 역할도 나눈다


<Codex · Gemini · Opus/Sonnet — 다른 모델, 다른 역할로 교차 검증.>

모델 역할 강점
Codex 코드 리뷰 로직 오류, 보안 취약점, 테스트 누락 검출
Gemini 문서 리뷰 일관성, 정확성, 구조 검증
Opus / Sonnet 성능별 분업 Opus: 복잡한 판단·아키텍처 / Sonnet: 빠른 확인·반복 검증

"Out of the box, Claude is a poor QA agent." — 검증 에이전트도 튜닝이 필요하다. 기준을 구체적으로, 회의적으로 설정해야 쓸 수 있다.

5-4. 원칙 04 — 에이전트에게 눈을 달아주기


<Browser Agent · Computer Use · 시각 검증 루프 — 코드뿐 아니라 화면도 본다.>

도구 용도
Browser Agent (chrome-cdp / agent-browser) 실제 Chrome 제어. DOM 탐색, 클릭, 스크린샷, 네비게이션. 웹앱 UX 직접 검증
Computer Use (built-in MCP) 스크린샷 + 마우스/키보드. 네이티브 앱·디자인 도구도 검증
시각 검증 루프 generate → screenshot → evaluate → 수정 루프. 사람이 눈으로 하는 것을 AI가 대신

5-5. 안전장치


<되돌릴 수 있는 환경 · 위험한 건 사람이 확인 · Dry-run 먼저.>

  1. 되돌릴 수 있는 환경 — 브랜치/Worktree 격리에서 작업. 실수해도 메인은 안전 (git worktree add)
  2. 위험한 건 사람이 확인 — 삭제·배포·외부 발송은 승인 후 실행 (Runtime Gate)
  3. Dry-run 먼저 — "이렇게 할 건데 맞나?" 미리보기 후 실행 (--dry-run)

"실수해도 괜찮은 구조" 를 만드는 것이 핵심.

5-6. Try It — 검증을 도와주는 도구들

  • /qa (Harness Plugin) — Browser Agent·Computer Use 도구를 활용해 QA를 자동화하는 스킬. 화면을 직접 보고, 클릭하고, 검증
  • verify 레퍼런스 (team-attention/hoyeon) — 검증 스킬을 더 깊이 이해하고 싶다면. 검증 전략·패턴·실제 적용 사례 정리

6. 개선(Compounding) — 어떻게 계속 나아질 것인가

6-1. 관측 → 패턴 발견 → Skill/Rule로 승격


<작업 → 관측(세션+사용 패턴) → 패턴 발견 → Skill 또는 Rule → 더 나은 작업.>

관측하기 개선하기
세션 분석 — 프롬프트 패턴 + Skill/Agent 호출 빈도 분석. 어디서 시간을 쓰는지, 어디서 실패하는지 3번 반복 → Skill로 — 같은 작업을 3번 반복하면 자동화할 타이밍. 스킬로 만들어서 재사용
AI Slop 감지 — 작업하다 보면 불필요한 코드, 중복 설정, 안 쓰는 규칙이 쌓인다. 이게 AI slop 3번 틀리면 → Rule 또는 CLAUDE.md에 — 같은 실수 반복 → .claude/rules/ 에 규칙 추가하거나 CLAUDE.md에 명시
Skill 개선 루프 — 만들기 → 사용 → 세션 분석 → 병목 → 개선. 스킬도 계속 다듬는다

6-2. 단순화하기


<-1 · 안 쓰는 건 치운다. ↑ 모델이 좋아지면 Harness를 재평가. ! 과설계 신호를 인식하기.>

  1. 안 쓰는 건 치운다 — 필요 없어진 Skill·MCP·Rule은 바로 삭제. 쌓이면 AI slop
  2. 모델이 좋아지면 Harness를 재평가 — 예전에 필요했던 가드레일이 지금은 불필요할 수 있다
  3. 과설계 신호를 인식하기 — 설정이 너무 복잡하면 뭔가 잘못된 것. 점점 단순해져야 정상

Anthropic: "Harness의 공간은 모델이 좋아져도 줄어들지 않는다. 이동할 뿐이다."

6-3. 자가 진단 — 잘 가고 있나


<잘 가고 있다는 신호 vs 실패하고 있다는 징후.>

+ 잘 가고 있다 − 실패 징후
같은 말을 두 번 하지 않는다 (맥락 전달이 잘 되고 있다) 검수에 시간이 더 오래 걸린다 (검증 자동화 필요 신호)
실수가 규칙이 된다 (개선 루프가 돌고 있다) 시켰는데 원하는 결과가 안 나온다 (맥락 전달 또는 계획 단계 부족)
차단 장치가 뭔가를 막고 있다 (사고가 구조적으로 예방되고 있다) 스킬·에이전트가 많은데 잘 안 쓴다 (context pollution + 제대로 동작 안 함)
불필요한 것이 줄어든다 (복잡해지는 게 아니라 단순해지고 있다) 가이드 파일이 길어지고 관리 안 된다 (CLAUDE.md·docs 분리/정리 없이 쌓임)

좋은 Harness는 점점 단순해진다. 복잡해지고 있다면 뭔가 잘못된 것.

6-4. Try It — session-wrap


<Claude Code 세션 종료 후 세션을 분석해 패턴·실수·자동화 기회를 자동으로 뽑아낸다.>

session-wrap (plugins-for-claude-natives): Claude Code 세션이 끝난 뒤, 세션 내용을 분석해 인사이트를 추출하는 플러그인.

  • 패턴 발견 — 반복 작업 패턴을 감지해서 Skill 후보로 제안
  • 실수 추출 — 세션 중 실수를 정리해서 Rule 후보로 제안
  • 문서 업데이트 — CLAUDE.md·docs 업데이트 필요 항목 감지

세션 분석 → 패턴 발견 → Skill/Rule 추가 — 이것이 개선 루프의 핵심.

7. 통합 관점 — 배경 이론과 기업 사례

7-1. 진화 흐름과 개념의 계보

이호연이 강의 서두에서 언급한 "Prompt → Context → Harness" 3-레벨은 하네스 엔지니어링(Harness Engineering) 이 정리한 세 패러다임 진화표와 동일한 축에 놓인다.

패러다임 시기 작업 대상 핵심 도구 한계
프롬프트 엔지니어링 2022~ 에이전트 입력 문구 프롬프트 템플릿 비결정론적, 확장 불가
컨텍스트 엔지니어링 2024~ 에이전트가 참조하는 정보 RAG, 지식베이스 환경 제약이 없어 일관성 부재
하네스 엔지니어링 2026~ 에이전트를 둘러싼 전체 환경 린터·CI·GC 에이전트·아키텍처 규칙 초기 설계 비용 높음

7-2. Martin Fowler가 정리한 하네스의 3대 구성

OpenAI 블로그를 Martin Fowler / Birgitta Böckeler 팀이 재해석한 분류는 이호연 강의의 6개 축을 더 상위에서 관통한다.

구성 요소 이호연 강의 대응 축 대표 장치
Context Engineering 구조(§1) + 맥락(§2) AGENTS.md·CLAUDE.md, ADR, 실행 계획 artifact, 동적 컨텍스트
Architectural Constraints 계획(§3) + 검증(§5) 안전장치 커스텀 린터, 구조적 테스트, 피드백 루프 설계
Garbage Collection 개선(§6) 문서-코드 불일치 탐지, 미사용 코드 제거, 아키텍처 위반 스캔

특히 린터 에러 메시지 자체를 "에이전트에게 다음 행동을 지시하는 프롬프트"로 재설계하는 것 이 가장 큰 레버리지 포인트라는 OpenAI의 관찰은, 이호연의 "3번 틀리면 Rule로 승격" 루프의 메커니즘적 설명이다.

7-3. 역설 — 제약이 자율성을 확장한다

"에이전트가 능력이 없어서 실패하는 게 아니라, 환경이 충분히 명세되지 않아서 실패한다." — OpenAI Codex 팀

이호연이 마무리 슬라이드에서 "너무 옥죄지 말 것, 자유도를 높게 주고 안 되는 것만 제한하기" 라고 경고하는 이유가 여기에 있다. 경계는 에이전트를 가두는 벽이 아니라, 그 안에서 자유롭게 움직일 수 있게 해주는 스캐폴딩이다.

7-4. 기업 사례 속의 6개 축 매핑

7-5. 역할의 재정의

이전 역할 하네스 시대의 역할
코드 타이피스트 환경 설계자
버그 수정자 피드백 루프 설계자
아키텍처 구현자 아키텍처 제약 인코더
코드 리뷰어 에이전트 의도 명세자
기술 부채 상환자 GC 에이전트 설계자

토스는 이를 "조직 생산성의 저점을 높이는 작업" 이라고 표현한다. 개인 역량 차이가 아니라, 팀 전체가 기본 이상의 생산성을 발휘하도록 시스템을 만드는 것 이 핵심 엔지니어링 과제다.

7-6. 한계와 주의사항

하네스는 만능이 아니다.

  • 초기 비용 — OpenAI는 하네스 구축에만 5개월을 썼다. 소규모 실험·단기 프로토타입에는 과잉 설계가 된다
  • 과도한 제약 리스크 — 너무 좁은 해결 공간은 창의적 해결을 억제한다. 점진적으로 추가하며 관찰할 것
  • 하네스 자체의 유지보수 부담 — 하네스도 코드다. 시스템이 진화하면 하네스도 업데이트해야 한다
  • GC 에이전트 오작동 — 유효한 코드를 잘못 삭제할 수 있다. PR 기반 인간 승인은 유지할 것
  • AGENTS.md의 효과 논란 — ETH 취리히 연구에서 AGENTS.md가 기대만큼 성능에 기여하지 않는다는 결과가 나왔다. 모든 하네스 요소가 동등한 효과를 갖지는 않는다
