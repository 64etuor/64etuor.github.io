하네스 엔지니어링 세미나 — 모델이 아니라 환경이 성능을 결정한다

하네스 엔지니어링 세미나

"에이전트에게 자유를 주려면, 먼저 경계를 설계하라."

0. 훅 — 같은 모델, 다른 성능

2026년 상반기의 세 장면.

  • LangChain — 모델은 그대로 둔 채 하네스만 바꿔 TerminalBench 52.8→66.5%, 벤치마크 순위 30위 → Top 5
  • OpenAI Codex 팀 — 엔지니어 3~7명이 5개월 동안 수동 코드 한 줄 없이 100만 줄의 프로덕션 코드를 구축. 쓴 시간은 전부 "환경 설계"
  • Anthropic — 싱글 에이전트 $9 실패 vs 3-에이전트 하네스 6h/$200 완성

한 문장 요약:

모델 교체로 5% 개선하는 것보다, 하네스 설계로 15% 개선하는 것이 현실적이다.

1. 정의 — 하네스란 무엇인가

Harness Engineering: AI가 혼자서도 잘 일할 수 있는 작업 환경을 설계하는 엔지니어링 분야.

"하네스(Harness)"는 말의 마구, 안전벨트처럼 대상의 에너지를 통제·인도하는 장치다. 에이전트의 능력을 억압하는 것이 아니라, 그 능력이 올바른 방향으로 신뢰 가능하게 발현되도록 구조적으로 인도한다.

AI 엔지니어링의 3계층

<모델은 안쪽 핵. 컨텍스트가 감싸고, 가장 바깥을 하네스가 두른다. 지난 3년간 레버의 위치가 안→밖으로 이동해 왔다.>

레벨 패러다임 핵심 질문 한계
1 Prompt Engineering (2022~) "이렇게 말해봐" 비결정론적, 확장 불가
2 Context Engineering (2024~) "이런 배경지식을 줘봐" 환경 제약이 없어 일관성 부재
3 Harness Engineering (2026~) "환경 자체를 설계하자" 초기 설계 비용 큼

2. 두 개의 렌즈 — 6축 프레임과 4요소 프레임

2-1. 팀 워크플로우 관점

<순환 루프. 한 바퀴 돌 때마다 팀 생산성의 저점이 올라간다.>

축 질문 대표 장치
구조 뭘 깔아두는가 Monorepo, 역할별 폴더, CLAUDE.md
맥락 AI가 뭘 아는가 User/Project/Folder 스코프, .claude/rules/, Progressive Disclosure
계획 뭘 할지 정하는가 Plan Mode, /specify, AskUserQuestion, Plan → MD → Execute
실행 어떻게 시키는가 Single / Subagent / Team Mode, Ralph Loop
검증 어떻게 믿는가 Generator≠Evaluator, 교차 모델, 시각 검증, Dry-run
개선 어떻게 나아지는가 3번 반복→Skill, 3번 틀림→Rule, session-wrap

2-2. Anthropic 4요소 — 단일 에이전트 내부 관점

Anthropic이 공식 하네스 스킬에서 제시한 모델은 더 압축적이다. 에이전트 한 개의 산출물 품질은 다음 네 가지로 결정된다.

요소 내용 나쁘면 생기는 일
Action Space 툴 이름·입력 스키마·반환 형태 겹치는 툴 난립, 에이전트가 선택 못 함
Observation 툴 결과의 구조 (status / summary / next_actions / artifacts) "에러 발생" 한 줄 — 다음 행동 불가
Recovery 에러 경로의 루트 원인 힌트·안전 재시도·명시적 정지 조건 무한 재시도 loop, 또는 침묵 실패
Context Budget 시스템 프롬프트 불변, 스킬 온디맨드, 참조 선호, 경계 시점에 compact 컨텍스트 오염, 주의력 희석

2-3. 두 프레임의 매핑

6축 (팀·워크플로우) 4요소 (에이전트 내부)
구조 · 맥락 Context Budget (+ Observation의 반절)
계획 · 실행 Action Space
검증 Observation + Recovery
개선 전 요소 튜닝 루프

<외부(팀이 보는 워크플로우)와 내부(에이전트 한 개의 실행)에서 같은 문제를 다르게 자른 것. 두 렌즈를 동시에 쓴다.>

3. 핵심 원리 5개 — 가장 먼저 내재화할 것

원리 1. 제약이 자율성을 확장한다

OpenAI Codex 팀의 관찰:

"에이전트가 능력이 없어서 실패하는 게 아니라, 환경이 충분히 명세되지 않아서 실패한다."

에이전트가 막히면 에이전트에게 더 좋은 프롬프트를 주려 하지 말고, 하네스의 빈 구멍을 메워라. 해결 공간을 구조적으로 좁힐수록 그 안에서의 신뢰 가능한 자율성은 커진다.

원리 2. 만든 AI와 확인하는 AI를 분리하라

"Evaluator를 회의적으로 튜닝하는 게, Generator를 자기비판적으로 만드는 것보다 훨씬 쉽다." — Anthropic

자기 작업을 평가하는 AI는 quality가 평범해도 자신 있게 칭찬한다. 관점을 물리적으로 분리한다. Codex는 코드 리뷰, Gemini는 문서 리뷰, Opus는 복잡 판단, Sonnet은 빠른 반복 검증 — 이렇게 모델·역할을 교차시킨다. "Out of the box, Claude is a poor QA agent" 를 기본값으로 깔고 설계하라.

원리 3. 에러 메시지를 다음 행동 지시로 재설계하라

4요소 중 Recovery가 가장 큰 레버다. 린터·테스트·타입체커·CI가 실패했을 때 단순히 "FAIL"이 아니라, 에이전트가 다음 턴에 바로 고칠 수 있는 수정 지시 프롬프트 형태로 출력하도록 재설계하는 것이 OpenAI 하네스의 결정적 레버리지 포인트였다.

status: error
summary: auth module imports payment layer (violates dependency direction)
next_actions:
	- move shared types to `src/shared/types.ts`
	- or invert dependency via interface in `src/auth/ports.ts`
stop_condition: if fix causes >3 layer touches, escalate to human

원리 4. Progressive Disclosure — 덜 주고 정확히 가리키기

모든 지식을 CLAUDE.md에 때려넣으면 컨텍스트 오염이 시작된다. 핵심은:

  • CLAUDE.md 는 ~100줄 지도
  • 상세 규칙은 .claude/rules/에 glob 조건부 로드 (*.tsx 작업 시만 react 규칙)
  • 도메인 지식은 references/ 폴더에 두고, 스킬 프롬프트에 "이 상황에서는 references/X.md 참고" 라고 안내
  • 에이전트가 동적으로 필요한 문서만 읽는다

이것이 4요소의 Context Budget 규칙 "스킬을 온디맨드로 로드, 긴 문서는 참조로 대체"의 구체화다.

원리 5. 3번 반복 → Skill, 3번 틀림 → Rule

<개선 루프의 최소 단위. 관측 → 승격 → 다음 세션 품질 상승.>

반복이 승격 신호다. 팀의 "암묵지"를 코드화된 스킬과 규칙으로 옮기는 것이 하네스가 시간이 갈수록 날카로워지는 메커니즘이다.

4. 실전 요령 — 5개 포인트

4-1. Single Source of Truth 원칙

리포지토리가 외부 컨텍스트 없이도 에이전트가 작업할 수 있는 단일 진실 공급원이 되어야 한다. 글로벌 스킬과 로컬 스킬의 책임 범위가 겹치지 않게 한다 — 글로벌은 "어떻게(how)", 로컬은 "여기서 뭘(what here)".

4-2. 구버전 문서는 없는 것보다 나쁘다

코드로 검증 가능한 정보(함수 시그니처 등)가 틀리면 LLM이 코드를 보고 정정한다. 하지만 코드만 봐서는 검증할 수 없는 맥락(아키텍처 이유, 폐기 사유, 조직 소유권)이 구버전이면 할루시네이션이 누적된다. 유지 못 할 문서는 삭제하거나 DEPRECATED 표시, modified 날짜를 반드시 남긴다.

4-3. 실행 패턴은 비용 감각으로 고른다

패턴 적합 상황 비용
Single 대부분의 일상 작업 1x
Subagent (병렬) 독립 조사·전문 분석 ~2~3x
Team Mode 다관점·에이전트 간 통신 필요 ~7x

90%는 Single/Subagent로 끝난다. Team Mode는 정말 필요한 복잡도에만 쓴다.

4-4. 완료 조건을 먼저, 측정 가능하게

Ralph Loop("될 때까지 반복")가 작동하려면 완료 조건이 측정 가능해야 한다. "잘 되게" ❌ → "테스트 통과 + Lighthouse ≥ 90 + 모바일 반응형" ✅. 기준이 있어야 자동 반복이 가능하다.

4-5. 되돌릴 수 있는 환경에서 일한다

  • Git worktree로 브랜치 격리 (git worktree add)
  • 삭제·배포·외부 발송은 Runtime Gate로 사람 확인
  • --dry-run 먼저, 실제 실행은 그 다음

"실수해도 괜찮은 구조" 가 자율성을 감당할 수 있게 해준다.

5. 안티패턴 케이스북

실전에서 반복되는 실패 패턴들. 각각 한 줄 증상 + 처방.

# 안티패턴 증상 처방
1 "해줘" 함정 의도를 프롬프트에 못 담고 "이거 만들어줘" → AI가 상상으로 만듦 → 검수 → 다시 → 무한 반복 Plan 분리, AskUserQuestion으로 AI가 먼저 묻게 한다
2 CLAUDE.md 비만증 모든 규칙을 한 파일에 축적 → 200줄 넘어가면 AI 정확도 급락 .claude/rules/ 분리 + glob 조건부 로드 + 주기적 정리 프롬프트
3 Self-evaluation 블라인드 Generator가 자기 결과물을 칭찬 모델·역할·컨텍스트 3중 분리
4 Tool name 증식 의미가 겹치는 툴 10개. 에이전트가 선택 실패, 랜덤하게 고름 툴을 통합·재명명. "catch-all tool" 금지
5 불투명한 에러 출력 Error: failed — 다음 행동 불가 Observation 스키마(status/summary/next_actions/artifacts) 강제
6 Doom Loop 같은 툴을 무한 재시도, 같은 에러 반복 명시적 stop_condition + 회수 제한 + 루프 감지 훅
7 구버전 하네스의 거짓 안전감 "하네스 있으니 괜찮지" 하며 리뷰 생략 → 90% 맞지만 10%가 낡음 규칙마다 추가 날짜·근거 기록. 하네스 자체에 대한 주기적 lint
8 과설계 Skill·MCP·Rule이 너무 많아 context 대부분이 가이드라인 안 쓰는 건 바로 치운다. 좋은 하네스는 점점 단순해진다
9 Plan → Execute 단방향 Plan이 틀렸는데 실행이 끝까지 감. 사람이 다시 다 뜯어본다 Plan ↔ Execute 양방향 루프. 실행 중 Plan 자체를 수정 가능하게
10 AGENTS.md 신봉 "이 파일만 쓰면 된다"는 맹신 ETH Zurich 연구 등에서 AGENTS.md의 성능 기여가 기대보다 작다는 결과. 모든 하네스 요소가 동등하지 않음을 인지

6. 기업 사례 — 6축/4요소로 읽기

사례 구조·맥락 (Context Budget) 계획·실행 (Action) 검증 (Obs+Recovery) 개선

<모두 "모델 교체 없이 하네스만 변주"해서 성능을 끌어올렸다.>

7. 엔지니어 역할의 재정의

이전 역할 하네스 시대의 역할
코드 타이피스트 환경 설계자 (Environment Designer)
버그 수정자 피드백 루프 설계자
아키텍처 구현자 아키텍처 제약 인코더
코드 리뷰어 에이전트 의도 명세자
기술 부채 상환자 GC 에이전트 설계자

토스의 표현을 빌리면 "조직 생산성의 저점을 높이는 작업". 개인 역량의 격차가 아닌, 팀 전체가 기본 이상의 생산성을 내도록 시스템을 만드는 것이 핵심 과제가 된다.

8. 1페이지 체크리스트

구조 · 맥락 (Context Budget)

  • Monorepo, 역할별 폴더, docs/·.dev/ 분리
  • CLAUDE.md ≤ 100줄, 지도 역할에 한정
  • .claude/rules/ glob 조건부 로드 구성
  • Progressive Disclosure: "이 상황에서 이 문서" 안내
  • 세션 사용률 20~30% 유지 (/compact, /clear, handoff)

계획 · 실행 (Action Space)

  • "해줘" 대신 "같이 계획" — Plan을 파일로 떨군다
  • 패턴 선택 (Single / Subagent / Team) 비용 감각 확인
  • 완료 조건이 측정 가능한가
  • 툴 이름·스키마 안정성 (catch-all 없음)

검증 (Observation + Recovery)

  • Generator ≠ Evaluator (모델·역할 분리)
  • 모든 툴 응답: status / summary / next_actions / artifacts
  • 에러 경로: 루트 원인 힌트 + 안전 재시도 + 명시적 stop
  • 되돌릴 수 있는 환경 (worktree + Runtime Gate + dry-run)

개선 (Compounding)

  • 3번 반복 → Skill, 3번 틀림 → Rule로 승격
  • 안 쓰는 Skill/MCP/Rule 주기적 제거
  • 하네스 자체에 대한 lint (규칙 최신성 검증)
  • 모델이 좋아지면 하네스 재평가

마무리 점검

  • 복잡해지고 있는가, 단순해지고 있는가? (단순해지고 있어야 정상)
  • 같은 말을 두 번 하지 않는가?
  • 차단 장치가 실제로 뭔가를 막고 있는가?

9. 마무리 — 남길 한 문장

AI가 코드를 쓰는 시대, 사람의 역할은 "잘 짜기"에서 "잘 일하는 환경을 만들기"로 바뀐다.

조심할 점: 하네스를 설계할 때 너무 옥죄지 말 것. 자유도를 높게 주고, 안 되는 것만 제한하기. 생각보다 에이전트는 예측 가능하게 잘 움직인다.
