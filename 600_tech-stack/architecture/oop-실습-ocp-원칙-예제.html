OOP 실습 - OCP 원칙 예제

Open-Closed Principle 적용 예제

1. 개념 소개

객체지향 프로그래밍(OOP)은 코드의 재사용성과 확장성을 높이는 중요한 개념이다. 이를 위해 추상화(abstraction), 다형성(polymorphism), 캡슐화(encapsulation), 상속(inheritance) 등의 개념을 활용한다.
이번 예제에서는 결제 시스템을 객체지향적으로 설계하여 확장성과 유지보수성을 높이는 방법을 학습한다.

2. 클래스 다이어그램

먼저, 설계된 클래스 다이어그램을 확인해보자.

📌 클래스 다이어그램 해석

  • PaymentProcessor 클래스는 결제를 수행하는 역할을 한다.
  • Payment 인터페이스는 결제 방식의 상위 개념으로, 모든 결제 방식이 process() 메서드를 구현하도록 강제한다.
  • CreditPayment와 KakaoPayment는 각각 신용카드와 카카오페이 결제 방식을 담당하는 구체적인 클래스이다.
  • PaymentProcessor는 Payment 인터페이스를 통해 결제 객체를 사용하기 때문에, 새로운 결제 방식을 추가해도 기존 코드에 영향을 주지 않는다. (OCP 원칙: 개방-폐쇄 원칙 준수)

3. 코드 구현

3.1 Payment 인터페이스 (상위 타입)

/* 모든 결제 방식이 따라야 할 인터페이스 */
public interface Payment {
    /* 결제 과정을 진행하는 메서드 */
    void process();
}

📌 설명:

  • Payment 인터페이스는 결제 방식의 공통 규약을 정의한다.
  • process() 메서드는 모든 결제 방식에서 반드시 구현해야 할 메서드이다.

3.2 PaymentProcessor (결제 수행 클래스)

/* 결제 요청을 처리하는 클래스 */
public class PaymentProcessor {  
    public void processPayment(Payment payment) {  
        payment.process();  // 결제 방식에 따라 다르게 동작
    }  
}

📌 설명:

  • PaymentProcessor는 Payment 인터페이스를 사용하여 결제 요청을 처리한다.
  • processPayment() 메서드는 다형성을 이용하여 어떤 결제 방식이든 동일한 방식으로 호출할 수 있다.

3.3 신용카드 결제 (구체적인 결제 방식)

/* 신용카드 결제 방식 */
public class CreditPayment implements Payment {
    @Override
    public void process() {
        System.out.println("신용카드로 결제합니다.");
    }
}

📌 설명:

  • CreditPayment 클래스는 Payment 인터페이스를 구현하여 신용카드 결제 로직을 정의한다.
  • process() 메서드를 오버라이딩하여 신용카드 결제 방식에 맞게 동작하도록 한다.

3.4 카카오페이 결제 (구체적인 결제 방식)

/* 카카오페이 결제 방식 */
public class KakaoPayment implements Payment {
    @Override
    public void process() {
        System.out.println("카카오페이로 결제합니다.");
    }
}

📌 설명:

  • KakaoPayment 클래스는 Payment 인터페이스를 구현하여 카카오페이 결제 로직을 정의한다.
  • process() 메서드를 오버라이딩하여 카카오페이 결제 방식에 맞게 동작하도록 한다.

4. 실행 코드 (테스트)

public class PaymentTest {
    public static void main(String[] args) {
        PaymentProcessor processor = new PaymentProcessor();

        // 신용카드 결제
        Payment creditPayment = new CreditPayment();
        processor.processPayment(creditPayment);

        // 카카오페이 결제
        Payment kakaoPayment = new KakaoPayment();
        processor.processPayment(kakaoPayment);
    }
}

📌 실행 결과

신용카드로 결제합니다.
카카오페이로 결제합니다.

5. 확장성 높은 설계의 장점

✅ OCP 원칙 (개방-폐쇄 원칙) 준수

  • 기존 코드(PaymentProcessor)를 수정하지 않고, 새로운 결제 방식을 추가할 수 있다.
  • 예를 들어, PayPal, 삼성페이, 네이버페이 등의 결제 방식을 추가할 경우, 새로운 클래스를 만들고 Payment 인터페이스만 구현하면 된다.

✅ 유지보수성 향상

  • 각 결제 방식이 독립적인 클래스로 존재하므로, 특정 결제 방식의 변경이 다른 코드에 영향을 주지 않는다.

✅ 다형성을 활용한 유연한 코드

  • Payment 인터페이스를 통해 다양한 결제 방식을 동일한 방식으로 처리할 수 있다.

6. 새로운 결제 방식 추가 예제

예를 들어, PayPal 결제 방식을 추가한다고 가정하자.

public class PayPalPayment implements Payment {
    @Override
    public void process() {
        System.out.println("PayPal로 결제합니다.");
    }
}

그리고 PaymentTest에서 사용하면 된다.

// PayPal 결제 방식 추가
Payment paypalPayment = new PayPalPayment();
processor.processPayment(paypalPayment);

👉 기존 코드(PaymentProcessor)를 수정하지 않고 새로운 결제 방식 추가 가능!

💡 핵심 정리

  1. 인터페이스(Payment)를 활용해 결제 방식의 공통 규약을 정의
  2. 구체적인 결제 방식(CreditPayment, KakaoPayment)을 인터페이스 구현
  3. 결제 처리 클래스(PaymentProcessor)는 인터페이스를 사용하여 결제 수행
  4. OCP 원칙을 준수하여 새로운 결제 방식 추가가 용이
