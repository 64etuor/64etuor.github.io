하네스 엔지니어링(Harness Engineering)

"에이전트에게 자유를 주려면, 먼저 경계를 설계하라."
— AI 에이전트 시대의 핵심 엔지니어링 원칙 [1]

1. 정의: 하네스 엔지니어링이란 무엇인가

하네스 엔지니어링이란 AI 에이전트가 신뢰성 있게 대규모 소프트웨어를 자율 구축할 수 있도록, 에이전트 내부(프롬프트·모델)가 아닌 에이전트를 둘러싼 외부 환경 전체를 설계하는 엔지니어링 분야 다 [2].

"하네스(Harness)"는 본래 말의 마구(馬具)나 안전벨트처럼 대상의 에너지를 안전하게 통제·인도하는 장치를 의미한다. AI에서의 하네스도 마찬가지다. 에이전트의 능력을 억압하는 것이 아니라, 그 능력이 올바른 방향으로 신뢰 가능하게 발현되도록 구조적으로 인도하는 시스템이다 [3].

공식화 배경

2026년 2월, OpenAI의 Codex 팀이 5개월에 걸쳐 단 한 줄의 수동 코드 없이 100만 줄 이상의 프로덕션 코드베이스를 구축한 경험을 공개하면서 이 개념이 공식화되었다 [2-1]. 당시 엔지니어들의 역할은 코드를 직접 쓰는 것이 아니라 "환경을 설계하고, 의도를 명세하고, 피드백 루프를 구축하는 것" 이었다 [4]. Martin Fowler의 ThoughtWorks 동료 Birgitta Böckeler가 이를 'AI 지원 소프트웨어 개발의 핵심 프레이밍'으로 분석해 주목도를 높였다 [5].

2. 개념의 계보: 세 패러다임의 진화

하네스 엔지니어링은 갑자기 등장한 개념이 아니라, AI 엔지니어링 추상화 레벨이 점진적으로 상향 이동해온 결과다 [6].

패러다임 시기 작업 대상 핵심 도구 한계
프롬프트 엔지니어링 2022~ 에이전트 입력 문구 프롬프트 템플릿 비결정론적, 확장 불가
컨텍스트 엔지니어링 2024~ 에이전트가 참조하는 정보 RAG, 지식베이스 환경 제약 없어 일관성 부재
하네스 엔지니어링 2026~ 에이전트를 둘러싼 전체 환경 린터, CI, GC 에이전트, 아키텍처 규칙 초기 설계 비용 높음

프롬프트가 "무엇을 말할 것인가"의 문제라면, 컨텍스트는 "무엇을 알게 할 것인가"의 문제다. 하네스는 그 둘을 포함하면서 "에이전트가 어떤 공간 안에서 움직이게 할 것인가"의 문제를 다룬다 [7].

3. 하네스의 3대 구성 요소

Martin Fowler는 OpenAI의 하네스를 세 범주로 분류했다 [5-1]. 이 분류는 현재 업계의 표준 해석으로 자리잡고 있다.

3-1. 컨텍스트 엔지니어링 (Context Engineering)

에이전트가 참조할 수 있는 지식 베이스를 코드베이스 안에 지속적으로 구축하고 갱신하는 작업 이다 [5-2].

  • AGENTS.md: 코드베이스의 목차 역할. 에이전트가 처음 작업을 시작할 때 전체 구조를 파악할 수 있는 약 100줄의 안내 문서 [8]
  • 아키텍처 결정 기록(ADR): 팀이 왜 특정 기술적 결정을 내렸는지에 대한 맥락
  • 실행 계획(Execution Plans): 단순 TODO가 아닌, 진행 상황과 결정 로그가 포함된 1급 아티팩트(first-class artifact)로 리포지토리에 버전 관리됨 [2-2]
  • 동적 컨텍스트: 관측 데이터(observability data), 브라우저 DOM 스냅샷, 로그 등 실시간 환경 정보에 에이전트가 접근하는 구조 [2-3]

핵심 원칙: 리포지토리가 외부 컨텍스트 없이도 에이전트가 작업을 수행할 수 있는 단일 진실 공급원(Single Source of Truth) 이 되어야 한다 [4-1].

3-2. 아키텍처 제약 (Architectural Constraints)

에이전트가 생성하는 코드가 시스템 아키텍처의 경계를 벗어나지 못하도록 결정론적·기계적으로 강제하는 구조 다 [2-4].

  • 커스텀 린터: 의존성 방향, 레이어 경계, 네이밍 컨벤션 등을 강제. OpenAI 사례에서는 이 린터 자체도 Codex가 생성했다 [2-5]
  • 구조적 테스트(Structural Tests): 비즈니스 도메인을 고정된 레이어 세트로 분리하고, 허용된 의존 방향을 자동 검사
  • 피드백 루프 설계: 린터 실패 시 단순 에러 출력이 아닌, 에이전트 컨텍스트에 수정 지시를 자동 주입 하도록 에러 메시지 자체를 설계하는 것이 핵심 레버리지 포인트 [9]

Stripe의 사례에서는 이를 6계층 하네스로 발전시켰다: 격리 VM → 로컬 린터 → 선택적 CI → 실용적 상한 → 하이브리드 워크플로우 → 인간 리뷰 [8-1].

3-3. 가비지 컬렉션 (Garbage Collection)

주기적으로 실행되는 에이전트가 코드베이스의 엔트로피(entropy) 를 자동 탐지하고 수정하는 메커니즘이다 [5-3].

  • 문서와 코드의 불일치 탐지
  • 아키텍처 위반 사항 스캔 및 수정
  • 사용되지 않는 코드 제거
  • 기술 부채의 능동적 관리

왜 중요한가: 에이전트가 자율적으로 코드를 생성할수록 엔트로피는 필연적으로 증가한다. GC 에이전트 없이는 시간이 지날수록 코드베이스가 에이전트 자신도 이해하기 어려운 상태로 붕괴한다 [4-2].

4. 핵심 원리와 철학

역설 1: 제약이 자율성을 확장한다

하네스 엔지니어링의 가장 중요한 반직관적 원리다. 에이전트의 해결 공간을 구조적으로 좁힐수록, 그 안에서의 신뢰 가능한 자율성은 확장된다 [1-1].

OpenAI 팀이 발견한 것은 "에이전트가 능력이 없어서 실패하는 것이 아니라, 환경이 충분히 명세되지 않아서 실패한다"는 사실이었다 [2-6]. 에이전트가 어려움을 겪을 때 직접 코드를 수정하는 것이 아니라, 하네스의 빈 구멍을 메워야 한다는 원칙이 여기서 나온다 [1-2].

원리 2: 인간의 주의(Attention)가 병목이다

AI 에이전트의 처리량이 인간의 리뷰 속도를 초과하는 시대에 핵심 문제는 에이전트 성능이 아니라 인간 주의(Human Attention)의 배분 이다 [10].

"에이전트 처리량이 인간 주의를 훨씬 초과하는 시스템에서, 수정은 싸고 기다림은 비싸다." — OpenAI Codex 팀 [10-1]

이에 따라 머지 게이트를 낮추고, 인간 피드백을 비동기로 전환하며, 에이전트가 독립적으로 완결할 수 있는 작업 단위를 최대화하는 방향으로 하네스를 설계해야 한다.

원리 3: 모델이 아니라 하네스가 성능을 결정한다

LangChain은 모델을 교체하지 않고 하네스만 개선함으로써 벤치마크 순위를 Top 30에서 Top 5로 끌어올렸다 [9-1]. Manus는 6개월 동안 동일한 모델에 대해 하네스를 5번 재설계하여 신뢰성과 태스크 완료율을 대폭 향상시켰다 [11].

이는 "모델은 상품, 하네스가 해자(Moat)" 라는 전략적 함의를 낳는다 [12]. 잘 설계된 하네스는 조직 고유의 아키텍처 철학, 도메인 지식, 운영 경험이 인코딩된 복제 불가능한 자산이다.

5. 다양한 관점에서의 해석

소프트웨어 공학 관점 (Martin Fowler / ThoughtWorks)

하네스는 새로운 '서비스 템플릿' 표준 이 된다. 기존의 서비스 템플릿이 팀 온보딩과 기술 스택 표준화를 위한 출발점이었다면, 하네스는 AI 에이전트를 포함한 팀 전체의 작업 방식을 정의하는 새로운 기반이 된다 [13].

"하네스 프레이밍이 유용한 이유는 대화를 프롬프트 기술에서 벗어나 집행 표면(enforcement surfaces)으로 돌리기 때문이다." — Martin Fowler [14]

시스템 다이나믹스 관점

하네스는 코드베이스의 엔트로피를 상쇄하는 균형 피드백 루프(Balancing Feedback Loop) 다. 에이전트 자율성이 증가하면 엔트로피가 증가하고, GC 에이전트와 아키텍처 제약이 이를 지속적으로 상쇄하는 구조다 [5-4]. 이 균형이 유지될 때만 대규모 코드베이스가 장기적으로 에이전트 작업 가능한 상태를 유지한다.

조직/전략 관점

하네스는 "팀의 엔지니어링 철학을 실행 가능한 코드로 구현한 것" 이다 [15]. 어떤 아키텍처 규칙, 도메인 경계, 의존성 방향을 하네스에 인코딩하느냐가 에이전트가 생산하는 코드의 품질과 방향성을 결정한다. 이 관점에서 하네스 설계는 기술적 작업인 동시에 조직의 기술 전략을 물리적으로 표현하는 행위다 [12-1].

AI 거버넌스 관점

"Governing AI Agents through Architectural Rigor"가 시사하듯, 하네스는 AI 안전 및 거버넌스의 실용적 구현체 이기도 하다 [16]. 에이전트의 행동 범위를 기술적으로 제약함으로써, 윤리·법적 요구사항을 프롬프트 지침이 아닌 구조적 강제로 보장할 수 있다.

6. 실전 구축 방법론: 성숙도 3단계

Level 1: 컨텍스트 기반 구축

작업 목적
AGENTS.md 작성 에이전트 온보딩 및 코드베이스 구조 이해
ADR 문서화 아키텍처 결정 맥락 보존
실행 계획 템플릿 표준화 에이전트 작업 단위 명확화
동적 컨텍스트 접근 설정 로그, DOM, 관측 데이터 연결

Level 2: 아키텍처 제약 구현

작업 목적
커스텀 린터 도입 의존성 방향, 레이어 경계 기계적 강제
린터 에러 메시지 재설계 단순 알림 → 에이전트 수정 지시 형식으로
구조적 테스트 추가 아키텍처 위반 자동 탐지
CI 파이프라인 통합 모든 에이전트 생성 코드의 자동 검증

Level 3: 가비지 컬렉션(GC: Garbage Collection) 에이전트 배포

작업 목적
GC 에이전트 프롬프트 설계 문서-코드 불일치, 미사용 코드 탐지
실행 스케줄링 설정 주기적 엔트로피 방지
GC 결과 PR 기반 반영 인간 승인 후 적용으로 오작동 방지
자가 검증 루프 추가 에이전트 제출 전 체크리스트 자동 실행 [15-1]

7. 실제 기업 적용 사례

OpenAI Codex 팀: Zero Manual Code

  • 기간: 5개월 (2025년 8월 ~ 2026년 2월) [7-1]
  • 규모: 엔지니어 3명 시작 → 7명으로 확대
  • 결과: 100만 줄 이상 프로덕션 코드, 수동 작성 코드 0줄 [8-2]
  • 생산성: 수동 작성 대비 약 10배 속도 향상, 엔지니어 1인당 하루 3.5개 PR 처리 [8-3] [2-7]
  • 핵심 하네스: AGENTS.md + 레이어드 의존성 아키텍처 + GC 에이전트

Stripe: Minions at Scale

  • 매주 1,000개 이상 PR 머지 [8-4]
  • MCP 서버를 통해 400개 이상 내부 도구에 에이전트 접근
  • 6계층 하네스: 격리 VM → 로컬 린터 → 선택적 CI → 실용적 상한 → 하이브리드 워크플로우 → 인간 리뷰 [8-5]

Manus: 하네스 반복 최적화

  • 6개월 동안 동일 모델에 대해 하네스를 5번 재설계 [11-1]
  • 모델 교체 없이 신뢰성과 태스크 완료율 대폭 향상
  • 도입한 개선: 자가 검증 루프, 루프 감지(doom loop 방지), 추론 샌드위치(계획/검증에는 고추론, 구현에는 중간 추론) [15-2]

8. 하네스가 재정의하는 엔지니어의 역할

하네스 엔지니어링 시대에 소프트웨어 엔지니어의 핵심 역량은 다음과 같이 재편된다 [2-8] [4-3].

이전 역할 하네스 시대의 역할
코드 타이피스트 (Syntax Writer) 환경 설계자 (Environment Designer)
버그 수정자 피드백 루프 설계자
아키텍처 구현자 아키텍처 제약 인코더
코드 리뷰어 에이전트 의도 명세자
기술 부채 상환자 GC 에이전트 설계자

토스는 이를 "조직 생산성의 저점을 높이는 작업"으로 표현했다 [17]. 개인 역량의 차이가 아닌, 팀 전체가 기본 이상의 생산성을 발휘할 수 있는 시스템을 만드는 것 이 핵심 엔지니어링 과제가 된다.

9. 한계와 주의사항

하네스 엔지니어링은 강력한 프레임워크지만, 맹목적 적용은 오히려 역효과를 낳을 수 있다 [10-2].

  • 초기 비용: OpenAI는 하네스 구축에 5개월을 투자했다 [13-1]. 소규모 실험적 프로젝트나 단기 프로토타입에는 과잉 설계가 될 수 있다
  • 과도한 제약 리스크: 너무 좁은 해결 공간은 에이전트의 창의적 문제 해결을 억제한다. 아키텍처 경계를 점진적으로 추가하며 에이전트 반응을 관찰하는 것이 필요하다
  • 하네스 자체의 유지보수 부담: 하네스도 코드다. 시스템이 진화하면 하네스도 업데이트해야 하며, 이는 새로운 기술 부채의 원천이 될 수 있다
  • GC 에이전트 오작동: GC가 유효한 코드를 잘못 삭제하는 리스크가 있다. PR 기반 반영으로 인간 승인 단계를 유지하는 것이 안전하다
  • AGENTS.md 효과 논란: ETH 취리히 연구팀은 AGENTS.md가 에이전트 성능에 기대만큼 기여하지 않는다는 연구를 발표했다 [10-3]. 하네스의 모든 요소가 동등한 효과를 갖는 것은 아니다

10. 미래 전망

2026년은 '에이전트가 코드를 쓸 수 있다'는 것이 증명된 2025년의 다음 단계로, 진짜 문제가 에이전트 자체가 아니라 하네스임을 업계가 공식 인식한 해 다 [15-3].

앞으로의 방향은 다음과 같이 예측된다:

  • 하네스 벤치마킹 표준화: Harness Bench처럼 git 커밋 기반 실세계 평가 프레임워크가 AI 코딩 어시스턴트를 블랙박스로 평가하는 표준이 될 것이다 [11-2]
  • Human-Aware SRE로 확장: 하네스가 소프트웨어 개발을 넘어 SRE(Site Reliability Engineering) 영역으로 확장되고 있다 [11-3]
  • 오픈소스 하네스 생태계: 팀별 하네스 컴포넌트가 마켓플레이스로 공유·재사용되는 구조가 대규모 조직에서 확산될 것이다 [17-1]
  • 메타 하네스: 하네스 설계 자체를 AI가 보조하거나 자동화하는 계층의 출현 가능성

결국 하네스 엔지니어링은 단순한 기술 도구가 아니라, "인간이 AI와 협업하는 방식 자체를 설계하는 새로운 공학 철학" 이다 [2-9]. 어떤 모델을 사용하느냐보다 어떤 하네스 위에서 그 모델을 운용하느냐가 앞으로의 기술 경쟁력을 결정할 것이다 [12-2].
