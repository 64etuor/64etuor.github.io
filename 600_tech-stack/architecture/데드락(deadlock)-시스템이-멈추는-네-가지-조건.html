데드락(Deadlock) - 시스템이 멈추는 네 가지 조건

두 스레드, 두 락, 영원한 대기

// Thread A
synchronized(lockA) {
	synchronized(lockB) { /* work */ }
}

// Thread B
synchronized(lockB) {
	synchronized(lockA) { /* work */ }
}

Thread A는 lockA를 쥔 채 lockB를 기다리고, Thread B는 lockB를 쥔 채 lockA를 기다린다. 둘 다 상대가 먼저 놓기를 기다리므로 프로그램은 영원히 멈춘다. 이것이 데드락이다.

데드락은 OS 교과서의 개념이 아니라 프로덕션에서 실제로 서비스를 중단시키는 장애 유형이다. MySQL의 SHOW ENGINE INNODB STATUS에서 LATEST DETECTED DEADLOCK 섹션을 한 번이라도 본 적 있다면, 이 글이 다루는 범위가 어디까지인지 감이 올 것이다.

Coffman의 네 가지 필요조건

1968년 Edward Coffman이 정의한 네 조건이 동시에 성립할 때 데드락이 발생한다.

조건 의미 깨뜨리는 전략
Mutual Exclusion 자원을 동시에 공유할 수 없음 Read-only 자원은 공유 허용
Hold and Wait 자원을 보유한 채 추가 자원을 요청 필요한 자원을 한 번에 모두 확보
No Preemption 보유 중인 자원을 강제로 빼앗을 수 없음 타임아웃으로 강제 회수
Circular Wait 프로세스 간 자원 요청이 순환 구조 자원에 순서를 부여하고 순서대로만 요청

네 조건 중 하나라도 깨지면 데드락은 발생하지 않는다. 실무에서 가장 효과적인 타겟은 보통 Circular Wait이다 — 자원 획득 순서를 고정하는 것만으로 대부분의 데드락을 예방할 수 있다.

OS 레벨: 스레드와 뮤텍스

위의 Java 예제를 수정해보자. lockA와 lockB에 순서를 부여하면 순환 대기가 깨진다.

// 두 스레드 모두 lockA → lockB 순서로 획득
synchronized(lockA) {
	synchronized(lockB) { /* work */ }
}

단순하지만 강력하다. 락이 많아질수록 순서를 관리하기 어려워지는데, 이때 흔히 쓰는 패턴은 락 객체의 해시값이나 ID 기준으로 항상 작은 쪽부터 획득하는 것이다.

def transfer(account_a, account_b, amount):
	first, second = sorted([account_a, account_b], key=id)
	with first.lock:
		with second.lock:
			account_a.balance -= amount
			account_b.balance += amount

tryLock과 타임아웃을 조합하면 No Preemption 조건도 깰 수 있다.

if (lockA.tryLock(100, TimeUnit.MILLISECONDS)) {
	try {
		if (lockB.tryLock(100, TimeUnit.MILLISECONDS)) {
			try { /* work */ }
			finally { lockB.unlock(); }
		}
	} finally { lockA.unlock(); }
}

획득에 실패하면 보유한 락을 전부 해제하고 재시도한다. 다만 모든 스레드가 동시에 재시도하면 라이브락(livelock)이 되므로, 랜덤 백오프를 넣는 것이 일반적이다.

DB 레벨: 트랜잭션과 Row Lock

MySQL InnoDB에서 가장 흔한 데드락 시나리오:

-- Transaction 1
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- row 1 잠금
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- row 2 대기

-- Transaction 2
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;   -- row 2 잠금
UPDATE accounts SET balance = balance + 50 WHERE id = 1;   -- row 1 대기 → 데드락

InnoDB는 데드락을 자동 탐지한다. Wait-for graph를 유지하면서 순환이 감지되면 비용이 적은 트랜잭션을 롤백시킨다. 개발자가 해야 할 일은:

  • 탐지 확인: SHOW ENGINE INNODB STATUS의 LATEST DETECTED DEADLOCK 섹션
  • 예방: 같은 테이블의 row를 여러 개 수정할 때 항상 PK 오름차순으로 접근
  • 대응: 애플리케이션에서 데드락 에러(MySQL 1213)를 잡아 재시도 로직 구현
@Retryable(value = DeadlockLoserDataAccessException.class, maxAttempts = 3)
public void transfer(Long fromId, Long toId, BigDecimal amount) {
	long first = Math.min(fromId, toId);
	long second = Math.max(fromId, toId);
	accountRepository.debit(first, amount);
	accountRepository.credit(second, amount);
}

PK 순서 고정 + 재시도. OS 레벨과 동일한 원리가 DB에서도 작동한다.

분산 시스템: 보이지 않는 순환

마이크로서비스 환경에서는 데드락의 형태가 달라진다. 뮤텍스나 row lock 대신 서비스 간 동기 호출 체인이 순환을 만든다.

Service A → (HTTP call) → Service B
Service B → (HTTP call) → Service A  // 순환

두 서비스가 서로를 동기 호출하면서 커넥션 풀이 고갈되면, 양쪽 모두 응답을 받지 못하고 멈춘다. 전통적인 데드락 탐지기가 없으므로 설계 단계에서 차단해야 한다.

  • 비동기 메시징: 동기 순환 호출을 이벤트 기반으로 전환
  • 타임아웃 + 서킷 브레이커: 응답이 없으면 빠르게 실패
  • 의존 방향 단방향화: 서비스 간 호출 그래프에서 순환을 제거

분산 락(Redis SETNX, ZooKeeper)을 쓸 때도 동일한 원칙이 적용된다. 락 획득 순서를 전역적으로 고정하거나, TTL을 설정해서 No Preemption을 깨야 한다.

하나의 원칙, 세 개의 계층

OS, DB, 분산 시스템 — 추상화 수준은 다르지만 데드락의 구조는 동일하다. Coffman의 네 조건이 동시에 성립하면 멈추고, 하나를 깨면 풀린다. 실무에서 가장 비용 대비 효과가 좋은 전략은 둘이다:

  • 자원 획득 순서 고정 (Circular Wait 차단)
  • 타임아웃 (No Preemption 차단)

데드락은 예방이 탐지보다 싸다. 코드 리뷰에서 "이 두 자원의 획득 순서가 모든 경로에서 동일한가?"라는 질문 하나가, 새벽 3시의 장애 대응보다 낫다.
