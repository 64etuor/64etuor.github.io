Git 브랜치 전략 및 규칙

Git Workflow 가이드

1. Git Flow

Git Flow는 Vincent Driessen이 제안한 브랜칭 모델로, 명확한 브랜치 전략을 제공하여 협업과 배포 프로세스를 체계적으로 관리할 수 있도록 한다. 주로 라이브러리, 프레임워크, 대규모 애플리케이션 개발에 적합하다.

Git Flow의 주요 브랜치

  1. master (main): 배포 가능한 안정적인 코드가 위치하는 브랜치
  2. develop: 개발을 위한 기본 브랜치로, 모든 기능 브랜치는 이곳에서 시작하고 완료 후 병합된다.
  3. feature: 개별 기능 개발을 위한 브랜치 (예: feature/login-page)
  4. release: 배포 준비를 위한 브랜치. 새로운 버전 출시 전 테스트 및 버그 수정을 진행한다.
  5. hotfix: 배포된 버전에서 발생한 긴급한 버그를 수정하기 위한 브랜치

Git Flow의 Workflow

  1. develop 브랜치에서 새로운 feature 브랜치를 생성하여 개발을 진행한다.
  2. 개발이 완료되면 develop 브랜치로 병합(Merge)하고, 필요하면 코드 리뷰를 진행한다.
  3. 배포가 필요하면 release 브랜치를 생성하여 추가적인 QA 및 버그 수정을 진행한 후 master 브랜치에 병합한다.
  4. release가 완료되면 develop에도 병합하여 변경 사항을 유지한다.
  5. 배포 후 긴급 수정이 필요한 경우 hotfix 브랜치를 master에서 생성하여 버그를 수정한 후 master 및 develop에 병합한다.
Git Flow는 명시적 버전 관리가 필요한 프로젝트(Library, Framework 등)에 적합하다. 하지만 수시로 배포되는 웹 애플리케이션 등의 프로젝트에는 부적합할 수 있다.

2. GitHub Flow

GitHub Flow는 Git Flow보다 단순한 브랜칭 모델로, 빠른 배포가 필요한 프로젝트(예: 웹 애플리케이션, SaaS 등)에 적합하다.

GitHub Flow의 주요 원칙

  1. main 브랜치는 항상 배포 가능한 상태로 유지한다.
  2. 새로운 작업이 필요하면 main에서 새로운 브랜치를 생성한다.
  3. 브랜치에서 개발을 진행하며, 완료되면 Pull Request(PR)를 생성한다.
  4. PR을 통해 코드 리뷰를 진행하고, 문제가 없다면 main 브랜치에 병합한다.
  5. 병합된 후에는 곧바로 배포한다.

GitHub Flow의 Workflow

  1. 새로운 기능이나 버그 수정을 위해 main 브랜치에서 새로운 브랜치를 생성한다.
  2. 개발 및 테스트를 진행한 후, 원격 저장소에 푸시한다.
  3. GitHub에서 Pull Request(PR)를 생성하여 코드 리뷰를 진행한다.
  4. 리뷰가 승인되면 main 브랜치에 병합하고, 배포한다.
  5. 필요하면 배포 후 브랜치를 삭제한다.
GitHub Flow는 단순하고 빠른 개발 및 배포가 필요한 프로젝트(웹 애플리케이션, SaaS, 스타트업 프로젝트 등)에 적합하다.

3. 브랜치 생성 및 네이밍 규칙

브랜치 네이밍 규칙을 정하면 팀원들이 쉽게 이해하고 협업할 수 있다. 일반적인 네이밍 규칙은 다음과 같다:

✅ 기본 원칙

  • main 브랜치에 직접 커밋하지 않는다.
  • 새로운 기능을 개발하기 전에는 main 또는 develop 브랜치를 기준으로 새로운 브랜치를 생성한다.
  • 브랜치 이름은 기능을 명확히 나타낼 수 있도록 한다.

📌 브랜치 네이밍 규칙 표

브랜치 유형 네이밍 규칙 설명
Feature (기능 추가) feature/기능이름 새로운 기능을 개발할 때 사용 (예: feature/login-page)
Bugfix (버그 수정) bugfix/이슈번호 또는 bugfix/버그설명 버그를 수정할 때 사용 (예: bugfix/1234, bugfix/fix-login-error)
Hotfix (긴급 수정) hotfix/이슈설명 배포 후 발견된 치명적인 버그를 긴급 수정 (예: hotfix/critical-bug)
Release (배포 준비) release/버전번호 배포 준비를 위한 브랜치 (예: release/1.0.0)
Chore (기타 작업) chore/작업내용 문서 업데이트, 빌드 설정, 코드 스타일 변경 등의 작업 (예: chore/update-eslint)
Refactor (리팩토링) refactor/변경내용 코드 구조 개선 또는 성능 최적화 작업 (예: refactor/improve-db-queries)
Docs (문서 변경) docs/문서설명 README나 개발 문서를 수정할 때 사용 (예: docs/api-docs-update)
Test (테스트 관련) test/테스트설명 테스트 코드 추가 또는 변경 시 사용 (예: test/add-unit-tests)
Experiment (실험적 기능) experiment/실험설명 실험적인 기능 개발 (예: experiment/new-ui-design)
CI/CD (배포 자동화) ci/설명, cd/설명 CI/CD 파이프라인 관련 작업 (예: ci/github-actions, cd/docker-deployment)

📌 브랜치 관리 방식

보통 Git Flow 또는 GitHub Flow 방식을 따라 브랜치를 관리합니다.

  1. main 또는 master → 최종 배포 브랜치
  2. develop → 개발 진행 중인 브랜치 (여러 기능이 머지될 예정)
  3. feature/* → 새로운 기능 개발
  4. bugfix/* → 버그 수정 브랜치 (배포 전 수정)
  5. hotfix/* → 긴급 버그 수정 (배포 후 발생한 이슈)
  6. release/* → 배포 준비 브랜치 (QA 테스트 등)
  7. chore/*, docs/*, refactor/* → 코드 개선, 문서 작업 등

📌 브랜치 예시

# Feature 개발 (로그인 페이지)
git checkout -b feature/login-page

# 버그 수정 (이슈 1234)
git checkout -b bugfix/1234

# 긴급 버그 수정 (로그인 오류)
git checkout -b hotfix/login-critical-error

# 코드 리팩토링 (쿼리 성능 개선)
git checkout -b refactor/improve-db-queries

# 문서 업데이트 (API 문서)
git checkout -b docs/update-api-docs

# CI/CD 설정 변경 (GitHub Actions 추가)
git checkout -b ci/add-github-actions

📌 브랜치 네이밍 규칙 정리

상황 브랜치 네이밍 규칙 예시
새로운 기능 개발 feature/기능이름 feature/login-auth
버그 수정 bugfix/이슈번호 bugfix/1234
긴급 핫픽스 hotfix/이슈설명 hotfix/critical-error
배포 준비 release/버전번호 release/1.0.0
코드 리팩토링 refactor/변경내용 refactor/optimize-db
문서 변경 docs/문서설명 docs/update-readme
테스트 관련 test/테스트설명 test/add-unit-tests
CI/CD 관련 ci/설명, cd/설명 ci/github-actions

4. Git Commit 메시지 규칙

일관된 커밋 메시지 스타일을 유지하면 프로젝트 관리가 쉬워진다.

✅ 기본 원칙

  1. 커밋 메시지는 명확하고 간결하게 작성한다.
  2. 첫 줄에는 요약을 작성하고, 필요하면 추가 설명을 포함한다.
  3. 동사 원형을 사용한다 (예: Fix, Add, Update).

✅ 커밋 메시지 스타일 예시

feat: 사용자 로그인 기능 추가
fix: 로그인 페이지에서 비밀번호 검증 오류 수정
refactor: 사용자 인증 로직 리팩토링
chore: 불필요한 콘솔 로그 제거
태그 설명
feat 새로운 기능 추가
fix 버그 수정
refactor 코드 리팩토링 (기능 변경 없음)
chore 코드 변경 없이 설정 변경 (예: 패키지 업데이트)
docs 문서 수정
test 테스트 코드 추가

5. PR(Pull Request) 작성 가이드

PR 작성 시 다음 사항을 포함하면 리뷰 과정이 효율적이다.

✅ PR 템플릿 예시

## 변경 사항
- 로그인 기능 추가
- JWT 인증 구현

## 테스트 방법
1. 로그인 페이지로 이동
2. 이메일과 비밀번호 입력 후 로그인 시도
3. 성공적으로 로그인되면 토큰이 반환되는지 확인

## 관련 이슈
- Closes #123
