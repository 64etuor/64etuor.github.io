프런트엔드 상태에서의 SSOT

프런트엔드 상태에서의 SSOT

헤더의 장바구니 배지는 3, 장바구니 페이지의 아이템 수는 4. QA가 올린 스크린샷을 보고 5분쯤 의심했습니다. 브라우저 네트워크 탭을 열어 보니 GET /cart가 두 번 호출되고 있었습니다. 헤더의 <CartBadge />와 페이지의 <CartList />가 각각 독립적으로 fetch를 하고, 각자 useState에 결과를 박아 두고 있었던 겁니다. 사용자가 담기 버튼을 누른 뒤 헤더만 리렌더되는 순간, 두 상태는 영원히 어긋났습니다.

같은 데이터가 두 곳에 살면 둘은 결국 다른 말을 한다

문제의 코드는 이렇게 생겼습니다.

function CartBadge() {
	const [count, setCount] = useState(0)
	useEffect(() => {
		fetch("/api/cart").then(r => r.json()).then(c => setCount(c.items.length))
	}, [])
	return <span>{count}</span>
}

function CartList() {
	const [items, setItems] = useState<Item[]>([])
	useEffect(() => {
		fetch("/api/cart").then(r => r.json()).then(c => setItems(c.items))
	}, [])
	// ...
}

서버에 있는 장바구니라는 하나의 사실을 클라이언트 두 군데가 각자 스냅샷으로 복사하고 있습니다. 한쪽이 갱신되어도 다른 쪽은 모릅니다. 이건 단순한 버그가 아니라 진실 공급원이 두 개라서 생긴 구조적 모순입니다. 컴포넌트 로컬 useState가 원격 데이터를 붙잡는 순간, 그 상태는 서버의 복제본이 되고 복제본은 반드시 드리프트합니다.

저장하지 말고 파생하라

비슷한 질병이 계산 가능한 값에도 나타납니다. 다음 코드를 봅시다.

function CartSummary({ items }: { items: Item[] }) {
	const [totalPrice, setTotalPrice] = useState(0)
	const [itemCount, setItemCount] = useState(0)

	useEffect(() => {
		setTotalPrice(items.reduce((s, i) => s + i.price * i.qty, 0))
		setItemCount(items.reduce((s, i) => s + i.qty, 0))
	}, [items])

	return <div>{itemCount}개 · {totalPrice}원</div>
}

totalPrice와 itemCount는 items에서 계산되는 값입니다. 그런데 useState에 넣는 순간 독립된 상태가 되었고, useEffect라는 동기화 코드가 필요해졌습니다. items가 바뀌었는데 useEffect가 아직 돌기 전 순간에 렌더가 끼어들면, 합계는 한 프레임 늦게 나타납니다. 할인 쿠폰이나 묶음 배송 로직이 추가되는 순간 이 어긋남은 UI 버그로 드러납니다.

파생 가능한 값은 저장하지 않고 매 렌더마다 다시 계산하는 게 원칙입니다.

function CartSummary({ items }: { items: Item[] }) {
	const totalPrice = useMemo(
		() => items.reduce((s, i) => s + i.price * i.qty, 0),
		[items]
	)
	const itemCount = useMemo(
		() => items.reduce((s, i) => s + i.qty, 0),
		[items]
	)

	return <div>{itemCount}개 · {totalPrice}원</div>
}

useMemo는 메모이제이션 도구로 알려져 있지만, 더 중요한 역할은 파생값을 상태로 착각하지 않게 만드는 문법적 신호입니다. items가 SSOT, totalPrice는 그 함수. 동기화 코드는 사라집니다.

판단 기준은 단순합니다. "이 값을 다른 상태에서 계산할 수 있는가?" 가능하면 useState에 넣지 않습니다. 성능이 문제라면 그때 useMemo로 감쌉니다. 계산부터가 아니라 저장부터 하는 습관이 드리프트의 대부분을 만듭니다.

서버 상태는 서버가 SSOT, 클라이언트는 캐시일 뿐이다

장바구니 버그의 진짜 해법은 "하나의 컴포넌트로 fetch를 올린다"가 아닙니다. 서버 데이터를 다루는 전담 레이어를 두고, 모든 컴포넌트가 같은 캐시 키를 바라보게 만드는 것입니다. React Query(TanStack Query)나 SWR이 이 역할을 합니다.

import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query"

const cartQuery = {
	queryKey: ["cart"],
	queryFn: () => fetch("/api/cart").then(r => r.json()),
}

function CartBadge() {
	const { data } = useQuery(cartQuery)
	return <span>{data?.items.length ?? 0}</span>
}

function CartList() {
	const { data } = useQuery(cartQuery)
	return <ul>{data?.items.map(i => <li key={i.id}>{i.name}</li>)}</ul>
}

function AddToCartButton({ productId }: { productId: string }) {
	const qc = useQueryClient()
	const { mutate } = useMutation({
		mutationFn: () => fetch("/api/cart", {
			method: "POST",
			body: JSON.stringify({ productId }),
		}),
		onSuccess: () => qc.invalidateQueries({ queryKey: ["cart"] }),
	})
	return <button onClick={() => mutate()}>담기</button>
}

queryKey: ["cart"]가 SSOT 역할을 합니다. 두 컴포넌트가 같은 키를 구독하기 때문에 캐시가 갱신되면 둘 다 같은 프레임에 새 값을 봅니다. invalidateQueries는 "서버의 진실이 바뀌었으니 캐시를 버려라"는 신호고, 다음 렌더에서 자동으로 재요청이 일어납니다. 로컬 useState는 이 그림에 없습니다.

여기서 중요한 개념 전환이 있습니다. 브라우저가 들고 있는 것은 서버 상태의 사본(cache)이지 상태 그 자체가 아니다. 네트워크 실패 시 재시도, 오래된 데이터 재검증, 포커스 복귀 시 갱신 같은 일이 클라이언트 상태 라이브러리의 책임이 아니라 서버 상태 라이브러리의 책임인 이유입니다. Zustand나 Redux에 fetch 결과를 쌓아 두는 패턴이 어색한 것도 이 때문입니다. 그 라이브러리들은 클라이언트에서 태어나 클라이언트에서 죽는 상태(예: 모달 열림 여부, 위저드 단계)를 위한 도구입니다.

상태는 네 종류다, 사는 곳이 다르다

상태를 한 덩어리로 보면 SSOT가 흐려집니다. 카테고리를 분리해야 어디에 둘지가 자명해집니다.

카테고리 SSOT 위치 예시 도구
서버 상태 원격 서버, 클라이언트는 캐시 장바구니, 주문 목록, 유저 프로필 React Query, SWR, RTK Query
URL 상태 브라우저 주소창 검색어, 필터, 정렬, 페이지 번호, 탭 useSearchParams, Next.js router
폼 상태 입력 중인 컴포넌트 트리 제출 전 입력값, 유효성 오류 React Hook Form, Formik
UI 일시 상태 컴포넌트 로컬 모달 열림, 드롭다운 포커스, 호버 useState, Zustand

각 카테고리를 다른 카테고리의 도구로 다루는 순간 버그의 저수지가 생깁니다. 서버 상태를 useState로 복사하면 앞서 본 드리프트. URL에 있어야 할 검색 필터를 컴포넌트 로컬 상태에 넣으면 사용자가 링크를 공유해도 필터가 따라가지 않습니다.

URL은 공짜로 주어진 공유 가능한 SSOT다

검색 페이지에서 사용자가 "서울"을 입력하고 가격 필터를 "10만원 이하"로 고른 뒤 URL을 복사해 동료에게 보냅니다. 동료가 링크를 열면 같은 결과가 보여야 합니다. 이게 가능하려면 필터 상태가 URL에 살아야 합니다.

function SearchPage() {
	const [params, setParams] = useSearchParams()
	const keyword = params.get("q") ?? ""
	const maxPrice = Number(params.get("max") ?? 0)

	const { data } = useQuery({
		queryKey: ["search", keyword, maxPrice],
		queryFn: () => fetch(`/api/search?q=${keyword}&max=${maxPrice}`)
			.then(r => r.json()),
		enabled: keyword.length > 0,
	})

	return (
		<>
			<input
				value={keyword}
				onChange={e => setParams({ q: e.target.value, max: String(maxPrice) })}
			/>
			{/* ... */}
		</>
	)
}

keyword를 useState로 다룬 뒤 useEffect로 URL에 동기화하는 코드를 본 적이 있을 겁니다. 그 순간 또다시 진실이 두 곳에 살게 됩니다. 뒤로 가기 한 번이면 두 상태가 어긋납니다. URL as state 패턴의 핵심은 params에서 직접 읽고 직접 쓰는 것입니다. 브라우저의 주소창이 이미 상태 저장소이자 히스토리 스택입니다. 별도로 관리하지 않아도 됩니다.

URL이 SSOT가 되어야 할 상태의 기준은 "새로고침해도 살아 있어야 하는가, 공유 가능해야 하는가"입니다. 검색, 필터, 정렬, 페이지 번호, 상세 뷰 탭, 모달의 딥링크 열기 — 이런 것들이 해당합니다. 반대로 "방금 클릭한 드롭다운이 열려 있는가"는 URL에 넣지 않습니다.

옵티미스틱 업데이트는 SSOT를 의도적으로, 잠깐 깨뜨린다

좋아요 버튼을 누를 때마다 서버 응답을 200ms 기다려야 한다면 UX는 무너집니다. 그래서 우리는 UI를 먼저 바꾸고 서버 호출은 뒤에 보냅니다. 이 순간 클라이언트 캐시는 잠깐 서버보다 앞서 나가고, SSOT가 일시적으로 깨집니다. 통제된 거짓말입니다. 통제 지점은 두 가지 — 낙관 값을 캐시에 주입할 때, 실패 시 원래대로 되돌릴 때.

const { mutate } = useMutation({
	mutationFn: (id: string) =>
		fetch(`/api/posts/${id}/like`, { method: "POST" }),
	onMutate: async (id) => {
		await qc.cancelQueries({ queryKey: ["post", id] })
		const prev = qc.getQueryData<Post>(["post", id])
		qc.setQueryData<Post>(["post", id], old =>
			old ? { ...old, likes: old.likes + 1, likedByMe: true } : old
		)
		return { prev }
	},
	onError: (_e, id, ctx) => {
		if (ctx?.prev) qc.setQueryData(["post", id], ctx.prev)
	},
	onSettled: (_d, _e, id) => {
		qc.invalidateQueries({ queryKey: ["post", id] })
	},
})

onMutate에서 현재 값을 스냅샷으로 쥐고, UI를 먼저 바꿉니다. 실패하면 onError에서 스냅샷으로 롤백합니다. onSettled에서 서버의 진짜 값으로 다시 동기화합니다. 세 단계가 모두 있어야 "잠깐의 거짓말"이 "영구적인 거짓말"로 타락하지 않습니다. 롤백 없는 옵티미스틱 업데이트는 옵티미스틱이 아니라 그냥 버그입니다.

판단 흐름 한 장

<상태 한 조각이 태어날 때마다 이 다섯 질문을 순서대로 던지면 SSOT가 자동으로 정해진다>

카테고리 규율이 설계를 결정한다

장바구니 배지 버그를 고치는 PR은 20줄짜리 작은 커밋이었습니다. 하지만 그 버그가 알려 준 건 코드 한 줄이 아니라 팀이 상태를 분류하지 않고 다루고 있다는 사실이었습니다. useState는 도구가 중립적이라 어떤 상태든 받아 주지만, 받아 줬다고 그게 올바른 자리는 아닙니다. 서버 상태, URL 상태, 폼 상태, UI 일시 상태는 수명주기도, 공유 범위도, 재시도 정책도 다릅니다. 같은 변수처럼 생겼지만 같은 종이 아닙니다.

새 상태를 만들 때 가장 먼저 할 질문은 "어떻게 저장할까"가 아니라 "이 값의 SSOT는 어디에 있어야 하는가"입니다. 서버인지, 주소창인지, 컴포넌트 트리인지. 저장소를 먼저 고르고 도구는 그다음에 고르면 useEffect로 복사해 붙이는 동기화 코드의 90퍼센트가 사라집니다. 남은 10퍼센트는 옵티미스틱 업데이트처럼 일부러 깨뜨리고 일부러 복구하는 의식적인 선택의 영역입니다.
