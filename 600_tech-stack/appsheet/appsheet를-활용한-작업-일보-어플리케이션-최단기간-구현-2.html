Appsheet를 활용한 작업 일보 어플리케이션 최단기간 구현 - 2

Appsheet를 활용한 작업 일보 어플리케이션 최단기간 구현 - 1에서 이어지는 내용입니다.

5단계: Views와 Slice 만들기

a) 이제 필요한 Views 즉, 앱의 스크린을 필요한 만큼 구현한다. 나의 경우, 필요 기능 명세에 언급했듯이 작업 이력 입력 기능, Rework 작업 입력 기능, 이력 조회 기능이 필요하기 때문에 3가지 Views를 만들었다. 그런데 작업한 내역을 생각해 보면 작업 이력, Rework 이력 등 모든 정보를 테이블 하나에 작성했다는 사실을 알 것이다. 그러므로 폼에 불필요한 데이터가 로드되지 않도록 Slice를 만들어 준다.

b) Slice columns에서 불필요한 열은 제거해준다. 나의 경우 Rework와 관련된 모든 열을 Slice columns에서 제거했다. 마찬가지로 Rework 정보를 입력하기 위한 Slice를 만들고 열을 제거해준다.

c) Rework Slice의 경우, 추가적으로 필요한 내용이 있다. Rework의 View를 만들 때 목록에 표시될 항목들은 불량 발생 여부를 체크하는 [isDefect] 값이 TRUE이고, Rework 완료 상황을 체크하는 [isReworked] 값이 TRUE가 아니어야 한다.(FALSE로 지정하지 않은 이유는 혹시라도 NULL 값이 발생했을 경우도 표시되게 하기 위해서다.) 따라서, Row filter condition에 목적에 맞게 수식을 입력해 준다.

d) 이제 View를 수정한다. 우선 작업 일보의 View type을 Form으로 하고 Column Order를 엔드유저가 입력하는 순서대로 정렬해 준다. 그리고 작업 일보는 연속적으로 작성하기 때문에 Auto re-open 옵션을 체크해서 활성화 한다.

e) 기본적인 생산 정보외에 불량 정보의 경우 불량이 아닌 양품 생산인 경우에는 작성할 필요가 없다. 따라서 불량 정보에 관계된 열의 Show if? 값에 수식을 넣어 불량 여부? 가 TRUE일 때만 관련 열이 표시되도록 바꿔준다. 마찬가지로 다른 불량 관련 열도 수정해준다. 그럼 isDefect 값에 따라 표시되는 항목의 개수가 달라진다.

불량 발생?이 Yes로 체크될 경우에만 아래의 불량 정보 입력 칸이 노출된다.

f) ModuleKey의 경우, 중복을 허용하지 않는 Unique 값이다. 따라서 Valid if? 에 수식을 추가해서 중복값이 입력되지 않도록 해준다.

IF(IN([_THISROW].[ID],
weldhisotry[ID]),TRUE,
NOT(IN([_THIS],weldHistory[ModuleKey])))
현재 행 ID가 테이블 내에 존재하는지 확인하고,
중복된 ID가 없을 경우 ModuleKey 중복 여부를 검사한다.

g) 그리고 모든 데이터 입력 방법을 엔드유저에게 친화적이게 바꿔준다. 가급적이면 숫자나 텍스트를 직접 입력하지 않고 터치 한번으로 해결할 수 있게 UX를 구성한다. 특히 Settings > View type options > Forms 의 Advance forms automatically 옵션은 꼭 체크해 준다. (기본 설정 FALSE) 엔드유저가 데이터를 입력하면 포커스가 자동으로 다음 칸으로 이동되므로 불필요한 터치를 줄여 준다.

h) rework가 필요한 목록, 즉 불량 목록의 경우, 앞서 Slice에서 표시될 데이터를 필터링 해두었다. View의 설정을 table로 하고(deck도 무방하다.) 목록이 최신 목록의 위에 위치하도록 Sort by 옵션을 _RowNumber를 기준으로 Descending으로 설정한다.

i) 테이블의 열 순서를 엔드유저의 필요에 맞게 수정해준다. 그리고 목록의 항목을 터치하게 되면 Rework 일보를 작성하는 화면으로 넘어 가도록 reworkView > Behavior에서 Event Actions를 수정해준다. (Row Selected: Edit) 그러면 따로 view를 또 생성하지 않아도 자동으로 앱시트에서 Form view를 생성한다.



왼쪽 메뉴 패널의 하단에서 SYSTEM GENERATED라는 자동 생성된 뷰의 항목을 볼 수 있다. 방금 생성된 Form의 Display name을 수정해 준다.

j) 이제 위의 Rework일보라고 적힌 Form을 확인해 본다. 처음에 Formula로 설정한 값들이 잘 작동하는지 확인한다.

처음에 설계한 대로 작업 시작? / 작업 종료? 버튼을 누를 때마다
NOW() 함수로 시간을 계산하여 ReworkStartTime / ReworkEndTime 에 입력된다.
그리고 ReworkEndTime이 입력된 시점에 reworkDuration이 계산되어 화면에 표시된다.

k) 이제 마지막 작업 이력 view만 생성하면 된다. reworkView와 마찬가지로 View type을 table로 설정한 후, 표시될 열을 정리해 준다. reworkView에서는 Behavior의 Event Actions(Row Selected: Edit)으로 설정했지만 이번에는 View details로 설정하여(기본 설정) 엔드유저가 저장된 데이터값을 확인할 수 있게 한다.

l) 이전에 Slice를 만들 때 설정해도 되었지만, Appsheet의 Slice Actions도 수정해 준다. 기본적으로 설정되어 있는 Actions는 Add, Edit, Delete 3가지인데 내가 만든 앱의 경우, 엔드유저가 reworkSlice에 연결된 Form 등에서 Add를 할 경우, 작업 이력이 존재하지 않는 rework 데이터가 만들어질 수 있다.(추가설명: 작업 데이터과 rework 데이터가 모두 한 테이블에 저장되고 rework 데이터는 작업 데이터에 종속되기 때문에, 최초의 레코드 생성은 반드시 작업이력에서 이루어져야 한다. 작업 이력이 없는 제품을 rework한다는 것은 불가능하기 때문에) 따라서 Slice Actions에서 Add를 삭제해 준다. 다른 Slice도 필요하다면 수정해 준다.

6단계: 디버깅

데이터를 실제로 입력하여 문제가 없는지 확인한다. 그리고 엔드유저가 개발 의도대로 사용하지 않았을 경우에 생길 수 있는 문제점도 전부 체크해 준다.

7단계: Deploy

이제 앱을 배포할 준비가 거의 다 되었다. 앱시트 홈 화면 왼쪽 패널 맨위에 로켓 모양의 아이콘이 Deploy 버튼이다. 해당 버튼을 누르면 Deploy, Versions, Monitor, Collaborate & Publish 설정을 할 수 있다. 먼저 Deployment Check를 해본다.

위의 단계대로 작업을 진행했다면 필연적으로 여러가지 Warning과 Error가 뜰 것이다.(Error만 없다면 Deploy가 가능하지만 가급적 최소 요건은 맞춰 주면 좋다.)

하나 하나 살펴보면, 아래와 같은 메세지를 확인할 수 있다.


App description의 경우 Settings > Information에서 경고 문구의 내용들을 수정해 주면 된다. Short Description과 Industry를 정하라고 하니, 해당 항목을 수정해준다.

Custom Logo의 경우, 로고가 기본설정된 로고일 경우 뜨는 경고 메세지이다. Settings > Theme & Brand에서 해당 내용을 수정해준다. 로고 파일을 Url로 추가해주거나 Google Drive 등에 업로드해서 불러온다.
마지막으로 Account에서 발생한 Error는 User signin 기능이 Free plan에서 지원하지 않는다는 내용이다. Security > Require sign-in 항목에서 Require user signin? 을 FALSE로 수정하고, 아래의 보안 항목 관련 확인란에(데이터가 공개된다는 것을 인지하였는가? 보안이 되지 않는 앱이라는 것을 인지하였는가?) 체크를 해주면 된다.

이후, Deplyment Check를 다시 진행하면 모두 Passed 된 것을 확인할 수 있다.

총평:

실제로 앱 개발에 걸린 시간은 3시간 내외로, (오히려 이 글을 작성하는 시간이 훨씬 오래 걸렸다.) 빠른 시간 안에 MVP를 만들어 테스트를 진행할 수 있다는 점에서 Appsheet 활용 가능성은 무궁무진하다. 이런 MVP 기능의 앱을 만들어서 실제로 테스트를 해보고 필요 데이터를 추가하고 정리하는 등 과정을 거치면 이후에 데이터 솔루션을 직접 제작하든, 외주 제작을 하든 강력한 레퍼런스로 활용할 수 있다. 제작 기간과 디버깅 기간도 훨씬 단축될 것이다. 사실 엔터프라이즈급 데이터가 아니라면, Appsheet를 구독하여 사용하는 것도 비용 절감 차원에서는 하나의 옵션이 될 수 있겠다고 생각한다.(프리미엄 플랜에서는 Automation 등 다른 기능 여러가지 기능도 지원한다.) 여러가지 LOW-CODE 툴 중에서도 커스텀 제약이 많은 편이긴 하지만, 반대로 말하면 커스텀 제약이 많기 때문에 개발 기간도 그만큼 짧아진다. 개발 시간 단축이 목표라면 Appsheet 만한 선택지가 많지는 않을 것이라고 생각한다.
