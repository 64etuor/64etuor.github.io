MissingGreenlet — SQLAlchemy async에서 가장 흔한 함정

MissingGreenlet — SQLAlchemy async에서 가장 흔한 함정

sqlalchemy.exc.MissingGreenlet: greenlet_spawn has not been called;
can't call await_only() here. Was IO attempted in an unexpected place?

SQLAlchemy의 AsyncSession을 쓰기 시작하면 거의 반드시 마주치는 에러다. 메시지만 봐서는 greenlet이 뭔지, spawn이 왜 안 됐는지 감이 안 온다. 하지만 이 에러의 본질은 단순하다: async 컨텍스트 바깥에서 암묵적 I/O가 발생했다.

greenlet 브릿지 — SQLAlchemy가 async를 구현하는 방법

SQLAlchemy ORM은 전체가 동기 코드로 작성되어 있다. async를 지원하기 위해 전체를 재작성하는 대신, greenlet 라이브러리를 브릿지로 사용한다. 핵심 구조는 세 가지 함수로 요약된다.

# sqlalchemy/util/_concurrency_py3k.py (단순화)

class _AsyncIoGreenlet(greenlet):
	def __init__(self, fn, driver):
		greenlet.__init__(self, fn, driver)
		self.driver = driver  # 부모 async greenlet 참조

def await_only(awaitable):
	"""동기 코드 안에서 async 작업을 대기한다."""
	current = getcurrent()
	return current.driver.switch(awaitable)  # ← 여기서 MissingGreenlet 발생

async def greenlet_spawn(fn, *args, **kwargs):
	"""동기 함수를 greenlet 안에서 실행하며, await 호출을 중계한다."""
	context = _AsyncIoGreenlet(fn, getcurrent())
	result = context.switch(*args, **kwargs)
	while not context.dead:
		value = await result        # 이벤트 루프에서 실제 I/O 수행
		result = context.switch(value)  # 결과를 동기 코드로 반환
	return result

실행 흐름을 따라가면 이렇다:

[사용자 async 코드]
	await session.execute(stmt)
		↓
[greenlet_spawn] — _AsyncIoGreenlet 생성
		↓
	[동기 ORM 코드가 greenlet 안에서 실행]
		↓ (DB I/O 필요)
	await_only(coro) → greenlet_spawn으로 스위치
		↓
	value = await coro  ← 이벤트 루프가 실제 I/O 처리
		↓
	context.switch(value) → 동기 코드 재개
		↓
	return result → greenlet 종료

await session.execute()가 반환되면 greenlet은 죽는다. 이후 ORM 객체의 속성에 접근할 때 내부적으로 await_only()가 호출되지만, 이미 greenlet 컨텍스트가 없으므로 current.driver가 존재하지 않는다. 이것이 MissingGreenlet이다.

터지는 6가지 시나리오

1. relationship lazy loading

가장 흔한 케이스. relationship의 기본 lazy 전략은 "select"이고, 속성 접근 시점에 SELECT를 발행한다.

async with async_session() as session:
	result = await session.execute(select(PirateCrew))
	crew = result.scalars().first()
	members = crew.members  # 💥 MissingGreenlet

crew.members에 접근하는 순간 ORM은 lazy load를 시도한다. 하지만 이 코드는 greenlet 안이 아니다.

2. commit 후 만료된 속성 접근

Session.commit()은 기본적으로 세션 내 모든 객체의 속성을 만료(expire)시킨다. 이후 속성에 접근하면 DB에서 다시 읽으려 한다.

async with async_session() as session:
	user = (await session.execute(select(User).where(User.id == 1))).scalar_one()
	user.name = "Updated"
	await session.commit()      # 모든 속성 만료
	print(user.name)            # 💥 만료된 속성 재로드 시도

3. rollback 후 객체 접근

rollback()도 세션 내 모든 객체를 만료시킨다.

async with async_session() as session:
	profiles = (await session.execute(select(Profile))).scalars().all()
	try:
		session.add(conflicting_object)
		await session.flush()
	except IntegrityError:
		await session.rollback()  # 전체 만료

	for p in profiles:
		print(p.id)              # 💥 만료된 속성

4. Pydantic 직렬화

Pydantic의 model_validate는 ORM 객체의 모든 필드를 순회한다. 로드되지 않은 relationship이 있으면 lazy load가 트리거된다.

async with async_session() as session:
	user = (await session.execute(select(User))).scalar_one()
	return UserSchema.model_validate(user)  # 💥 user.posts 접근

5. onupdate 컬럼 (2.0.36 regression)

onupdate가 설정된 컬럼은 UPDATE 후 서버에서 값이 결정된다. commit 후 해당 속성에 접근하면 재로드가 필요하다.

class User(Base):
	updated_at = Column(DateTime, default=func.now(), onupdate=func.now())

async with async_session() as session:
	user = await session.get(User, 1)
	user.name = "Bob"
	await session.commit()
	print(user.updated_at)      # 💥 서버 측 onupdate 값 미반환

6. 세션 미종료

세션을 명시적으로 닫지 않으면, 가비지 컬렉션 시점에 커넥션 풀 정리 과정에서 greenlet 에러가 발생할 수 있다.

async def get_user():
	session = AsyncSession(engine)
	result = await session.execute(select(User))
	user = result.scalar_one()
	# session.close() 없이 반환 → GC 시점에 💥
	return user

해결 전략

eager loading으로 암묵적 I/O 제거

가장 근본적인 해결. 쿼리 시점에 필요한 데이터를 모두 가져온다.

# 쿼리 단위 적용
stmt = select(PirateCrew).options(selectinload(PirateCrew.members))

# 중첩 관계
stmt = select(User).options(
	selectinload(User.orders).selectinload(Order.items)
)

# 모델 단위 적용 (전역)
class User(Base):
	posts: Mapped[list["Post"]] = relationship(lazy="selectin")

selectinload는 별도의 SELECT ... WHERE id IN (...) 쿼리를 발행한다. joinedload와 달리 원본 쿼리를 변형하지 않으므로 페이지네이션과 충돌하지 않는다.

expire_on_commit=False

commit/rollback 후 속성 만료를 막는다. 시나리오 2, 3, 5에 직접 대응한다.

async_session = async_sessionmaker(engine, expire_on_commit=False)

주의: 이 옵션을 켜면 commit 후에도 메모리의 값이 DB와 다를 수 있다. 단일 요청 내에서 세션을 쓰고 버리는 웹 앱 패턴에서는 안전하지만, 장기 세션에서는 stale 데이터 위험이 있다.

AsyncAttrs 믹스인

SQLAlchemy 2.0.13+에서 제공. lazy load를 명시적 await로 전환한다.

from sqlalchemy.ext.asyncio import AsyncAttrs
from sqlalchemy.orm import DeclarativeBase

class Base(AsyncAttrs, DeclarativeBase):
	pass

# 사용
async with async_session() as session:
	crew = (await session.execute(select(PirateCrew))).scalars().first()
	members = await crew.awaitable_attrs.members  # 명시적 async lazy load

에러는 막아주지만, N+1 문제는 여전하다. 루프 안에서 awaitable_attrs를 쓰면 쿼리 수가 N개로 늘어난다. eager loading과 병행해야 한다.

run_sync로 동기 코드 실행

레거시 동기 코드를 async 세션 안에서 그대로 돌릴 수 있다. 내부적으로 greenlet을 생성하므로 lazy loading이 작동한다.

def process_user_sync(session):
	"""greenlet 안에서 실행 — lazy loading 정상 작동"""
	for user in session.execute(select(User)).scalars():
		for post in user.posts:  # lazy load OK
			print(post.title)

async with async_session() as session:
	await session.run_sync(process_user_sync)

단일 속성 접근만 필요하면 한 줄로도 된다:

posts = await session.run_sync(lambda s: user.posts)

lazy="raise"로 방어적 설계

에러 메시지를 개선하는 전략. MissingGreenlet 대신 명확한 InvalidRequestError가 발생한다.

class User(Base):
	posts: Mapped[list["Post"]] = relationship(lazy="raise")

# user.posts 접근 시:
# sqlalchemy.exc.InvalidRequestError:
# 'User.posts' is not available due to lazy='raise'

어떤 relationship에서 eager loading을 빠뜨렸는지 즉시 알 수 있다.

어떤 전략을 쓸 것인가

상황 권장 전략
API 응답에 관계 데이터 포함 selectinload 쿼리 옵션
모든 쿼리에서 항상 필요한 관계 모델에 lazy="selectin"
commit 후 속성 접근 필요 expire_on_commit=False
레거시 동기 코드 재사용 run_sync()
lazy load 누락 조기 발견 lazy="raise" + 테스트
점진적 마이그레이션 AsyncAttrs 믹스인

핵심은 하나다. async 컨텍스트에서 암묵적 I/O는 불가능하다. SQLAlchemy의 동기 ORM이 "알아서" 해주던 lazy loading, 속성 재로드, 세션 정리가 async에서는 전부 명시적으로 처리해야 한다. MissingGreenlet은 이 원칙을 어긴 지점을 알려주는 신호다.
