FastAPI를 위한 Elasticsearch 대안 — 검색 엔진 선택 가이드

FastAPI를 위한 Elasticsearch 대안 — 검색 엔진 선택 가이드

Elasticsearch 단일 노드의 기본 JVM 힙은 1GB다. 실제 운영에선 최소 4GB를 권장하고, 인덱싱과 검색을 동시에 처리하려면 8GB 이상을 잡아야 한다. 사이드 프로젝트 서버 전체 RAM이 4GB인 상황에서, 검색 기능 하나를 위해 서버 메모리의 절반을 Elasticsearch에 바치는 건 과하다.

FastAPI 프로젝트에서 전문 검색(full-text search)이 필요할 때, ES만이 답은 아니다. 문제는 "어떤 대안이 있는가"가 아니라 "내 프로젝트 규모와 요구사항에 어떤 엔진이 맞는가"다.

FastAPI와 궁합이 좋다는 것의 의미

대안을 비교하기 전에, FastAPI 맥락에서 검색 엔진을 고를 때 실제로 무엇이 중요한지 정리해야 한다.

  • async 지원: FastAPI는 ASGI 기반이다. 검색 클라이언트가 async/await를 지원하지 않으면 이벤트 루프를 블로킹하게 된다. run_in_executor로 우회할 수 있지만, 네이티브 async SDK가 있는 쪽이 낫다.
  • Pydantic 호환: 검색 결과를 Pydantic 모델로 직접 파싱할 수 있는가. dict 응답이면 model_validate로 변환해야 하고, 자체 모델 시스템이 있으면 매핑 레이어가 필요하다.
  • 배포 복잡도: Docker Compose에 서비스 하나를 추가하는 것과, JVM 기반 클러스터를 관리하는 것은 운영 비용이 다르다.
  • 리소스 풋프린트: 검색 엔진이 차지하는 RAM과 디스크. 특히 소규모 프로젝트에서 결정적이다.

이 네 가지 축으로 각 대안을 평가한다.

Meilisearch — 가장 빠른 전환

# docker-compose.yml
services:
  meilisearch:
    image: getmeili/meilisearch:v1.12
    ports:
      - "7700:7700"
    environment:
      - MEILI_MASTER_KEY=your-master-key
    volumes:
      - meili_data:/meili_data

이것으로 검색 엔진이 뜬다. 초기 메모리 사용량은 약 30~50MB. Rust로 작성되어 있고, 단일 바이너리로 동작한다.

FastAPI 통합 코드:

# pip install meilisearch-python-sdk
from meilisearch_async_client import Client
from pydantic import BaseModel
from fastapi import FastAPI, Depends
from contextlib import asynccontextmanager


class Product(BaseModel):
	id: int
	name: str
	description: str
	price: float


@asynccontextmanager
async def lifespan(app: FastAPI):
	app.state.meili = Client("http://localhost:7700", "your-master-key")
	index = app.state.meili.index("products")
	await index.update_filterable_attributes(["price"])
	await index.update_sortable_attributes(["price"])
	yield
	await app.state.meili.aclose()


app = FastAPI(lifespan=lifespan)


@app.get("/search")
async def search_products(q: str, max_price: float | None = None):
	index = app.state.meili.index("products")
	filters = f"price <= {max_price}" if max_price else None
	results = await index.search(q, filter=filters, limit=20)
	return [Product.model_validate(hit) for hit in results.hits]


@app.post("/products")
async def add_products(products: list[Product]):
	index = app.state.meili.index("products")
	await index.add_documents([p.model_dump() for p in products])
	return {"status": "indexed"}

meilisearch-python-sdk는 네이티브 async를 지원한다(meilisearch_async_client). 공식 SDK인 meilisearch도 있지만, 이쪽은 동기 전용이므로 FastAPI에서는 async 버전을 써야 한다.

Meilisearch의 특징:

  • 오타 허용(typo tolerance) 내장 — 별도 설정 없이 "pyhton"으로 검색해도 "python" 결과가 나온다
  • 패싯 필터링과 정렬이 선언적 — filterable_attributes, sortable_attributes를 미리 지정
  • 인덱싱이 비동기 태스크 큐로 동작 — add_documents는 즉시 반환하고 백그라운드에서 처리
  • 한국어 토크나이저 내장 — CJK 언어 자동 감지로 별도 플러그인 불필요

한계:

  • 싱글 노드 전용 — 샤딩이나 레플리카 없음. 수억 건 규모엔 부적합
  • 집계(aggregation) 기능 없음 — 로그 분석이나 통계 용도로는 쓸 수 없다
  • 인덱스당 문서 수가 수백만 건을 넘으면 인덱싱 속도가 느려짐

Typesense — 타입 안전한 즉시 검색

from contextlib import asynccontextmanager

# Typesense는 공식 Python SDK가 동기 전용이다.
# FastAPI에서는 run_in_executor로 감싸거나 httpx로 직접 호출한다.

client = typesense.Client({
	"api_key": "your-api-key",
	"nodes": [{"host": "localhost", "port": "8108", "protocol": "http"}],
	"connection_timeout_seconds": 2,
})


# 컬렉션 스키마 — ES와 달리 필드 타입을 명시적으로 선언
schema = {
	"name": "products",
	"fields": [
		{"name": "name", "type": "string"},
		{"name": "description", "type": "string"},
		{"name": "price", "type": "float", "facet": True},
		{"name": "category", "type": "string", "facet": True},

Typesense는 C++로 작성되어 있고, 메모리 사용량은 Meilisearch와 비슷한 수준이다. 핵심 차이는 스키마를 명시적으로 선언한다는 것이다. ES의 dynamic mapping과 Meilisearch의 스키마리스 방식과 대조적이다.

FastAPI에서 async로 쓰려면 httpx 래퍼를 만들어야 한다:

import httpx
from pydantic import BaseModel


class TypesenseAsync:
	def __init__(self, host: str, api_key: str):
		self.base_url = host
		self.headers = {"X-TYPESENSE-API-KEY": api_key}
		self.client = httpx.AsyncClient(
			base_url=self.base_url, headers=self.headers
		)

	async def search(self, collection: str, params: dict) -> dict:
		resp = await self.client.get(
			f"/collections/{collection}/documents/search",
			params=params,
		)
		resp.raise_for_status()
		return resp.json()

	async def upsert(self, collection: str, document: dict) -> dict:
		resp = await self.client.post(
			f"/collections/{collection}/documents",
			json=document,
			params={"action": "upsert"},
		)
		resp.raise_for_status()
		return resp.json()

	async def close(self):
		await self.client.aclose()


# FastAPI 엔드포인트
ts = TypesenseAsync("http://localhost:8108", "your-api-key")


class SearchResult(BaseModel):
	name: str
	description: str
	price: float
	category: str


@app.get("/search")
async def search(q: str, category: str | None = None):
	params = {
		"q": q,
		"query_by": "name,description",
		"per_page": 20,
	}
	if category:
		params["filter_by"] = f"category:={category}"
	result = await ts.search("products", params)
	return [
		SearchResult.model_validate(hit["document"])
		for hit in result["hits"]
	]

기준 Meilisearch Typesense

고가용성 싱글 노드 Raft 기반 클러스터링 지원

클러스터링이 필요하면 Typesense, 빠른 프로토타이핑이면 Meilisearch. 이것이 가장 실질적인 분기점이다.

PostgreSQL FTS — 인프라를 추가하지 않는 선택

이미 PostgreSQL을 쓰고 있다면, 검색 엔진을 별도로 띄우지 않아도 된다. PostgreSQL의 Full-Text Search는 tsvector와 tsquery로 동작한다.

-- 검색용 컬럼과 인덱스 추가
ALTER TABLE products ADD COLUMN search_vector tsvector
	GENERATED ALWAYS AS (
		setweight(to_tsvector('simple', coalesce(name, '')), 'A') ||
		setweight(to_tsvector('simple', coalesce(description, '')), 'B')
	) STORED;

CREATE INDEX idx_products_search ON products USING GIN (search_vector);

'simple' 사전을 쓴 이유가 있다. PostgreSQL의 한국어 전용 사전(korean)은 기본 설치에 포함되지 않는다. 'simple'은 언어 불문 공백 기반 토크나이징을 하므로, 한국어에서도 기본적인 검색이 동작한다. 형태소 분석이 필요하면 pg_bigm 확장이나 pgroonga를 설치해야 한다.

FastAPI + SQLAlchemy async 통합:

from sqlalchemy import func, select, text
from sqlalchemy.ext.asyncio import AsyncSession, create_async_engine
from fastapi import FastAPI, Depends


engine = create_async_engine("postgresql+asyncpg://user:pass@localhost/db")


async def get_session():
	async with AsyncSession(engine) as session:
		yield session


@app.get("/search")
async def search_products(
	q: str,
	session: AsyncSession = Depends(get_session),
):
	# plainto_tsquery는 사용자 입력을 안전하게 tsquery로 변환
	query = select(Product).where(
		Product.search_vector.op("@@")(func.plainto_tsquery("simple", q))
	).order_by(
		func.ts_rank(Product.search_vector, func.plainto_tsquery("simple", q)).desc()
	).limit(20)

	result = await session.execute(query)
	return result.scalars().all()

asyncpg와 직접 쓰면 SQLAlchemy 오버헤드 없이 더 가볍다:

import asyncpg
from fastapi import FastAPI


pool: asyncpg.Pool


@app.on_event("startup")
async def startup():
	global pool
	pool = await asyncpg.create_pool("postgresql://user:pass@localhost/db")


@app.get("/search")
async def search(q: str):
	async with pool.acquire() as conn:
		rows = await conn.fetch(
			"""
			SELECT id, name, description, price,
				ts_rank(search_vector, plainto_tsquery('simple', $1)) AS rank
			FROM products
			WHERE search_vector @@ plainto_tsquery('simple', $1)
			ORDER BY rank DESC
			LIMIT 20
			""",
			q,
		)
		return [dict(r) for r in rows]

PostgreSQL FTS가 충분한 경우:

  • 문서 수가 수십만 건 이하
  • 오타 허용(fuzzy matching)이 필수가 아닌 경우
  • 이미 PostgreSQL을 메인 DB로 쓰고 있고, 별도 서비스를 띄우고 싶지 않은 경우
  • 트랜잭션 일관성이 중요한 경우 — 데이터 변경과 검색 인덱스가 같은 트랜잭션 안에 있다

부족한 경우:

  • 오타 허용, 동의어 처리, 자동완성 같은 "검색 UX" 기능이 필요하면 전용 검색 엔진이 낫다
  • pg_trgm 확장으로 trigram 기반 유사 검색을 추가할 수 있지만, Meilisearch의 오타 허용과는 품질 차이가 있다
  • 수백만 건 이상에서 복잡한 패싯 검색을 하면 성능이 떨어진다

pg_trgm으로 퍼지 검색 보강

오타 허용이 어느 정도 필요하다면, pg_trgm 확장을 추가할 수 있다:

CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_products_name_trgm ON products USING GIN (name gin_trgm_ops);

@app.get("/suggest")
async def suggest(q: str, session: AsyncSession = Depends(get_session)):
	# similarity 함수로 유사도 기반 검색
	query = select(Product).where(
		func.similarity(Product.name, q) > 0.3
	).order_by(
		func.similarity(Product.name, q).desc()
	).limit(10)
	result = await session.execute(query)
	return result.scalars().all()

자동완성이나 "혹시 이것을 찾으셨나요?" 수준은 커버되지만, Meilisearch나 Typesense의 즉시 검색 경험에는 못 미친다.

OpenSearch — ES API를 버릴 수 없을 때

기존 ES 쿼리가 많거나, Kibana 대시보드를 유지해야 하거나, 로그 집계가 필요하면 OpenSearch가 현실적 선택이다. AWS가 ES를 포크한 프로젝트로, ES 7.x API와 대부분 호환된다.

# pip install opensearch-py
from opensearchpy import AsyncOpenSearch
from fastapi import FastAPI


client = AsyncOpenSearch(
	hosts=[{"host": "localhost", "port": 9200}],
	use_ssl=False,
)


@app.get("/search")
async def search(q: str):
	body = {
		"query": {
			"multi_match": {
				"query": q,
				"fields": ["name^2", "description"],
				"fuzziness": "AUTO",
			}
		},
		"size": 20,
	}
	result = await client.search(index="products", body=body)
	return [hit["_source"] for hit in result["hits"]["hits"]]


@app.get("/aggregate")
async def price_stats(category: str):
	body = {
		"query": {"term": {"category.keyword": category}},
		"aggs": {
			"price_stats": {"stats": {"field": "price"}},
			"price_histogram": {
				"histogram": {"field": "price", "interval": 10000},
			},
		},
		"size": 0,
	}
	result = await client.search(index="products", body=body)
	return result["aggregations"]

opensearch-py는 AsyncOpenSearch 클래스로 네이티브 async를 지원한다. ES에서 마이그레이션할 때, 대부분의 쿼리는 클라이언트 import만 바꾸면 동작한다.

ES 대비 OpenSearch의 차이:

  • 라이선스: Apache 2.0 (ES는 SSPL/Elastic License)
  • Security 플러그인 내장 — ES는 X-Pack 유료 기능이었던 것들이 기본 포함
  • OpenSearch Dashboards = Kibana 포크
  • 리소스 사용량은 ES와 동일하게 높다 — JVM 기반이므로 경량화 목적이면 의미 없음

OpenSearch는 "ES에서 벗어나는 대안"이 아니라, "ES의 라이선스 문제를 해결한 포크"다. 메모리와 운영 복잡도는 그대로이므로, 리소스 절약이 목적이면 다른 대안을 봐야 한다.

Tantivy — 서버 없는 임베디드 검색

# pip install tantivy
import tantivy

# 스키마 정의
schema_builder = tantivy.SchemaBuilder()
schema_builder.add_text_field("name", stored=True, tokenizer_name="default")
schema_builder.add_text_field("description", stored=True, tokenizer_name="default")
schema_builder.add_float_field("price", stored=True, fast=True)
schema = schema_builder.build()

# 인덱스 생성 (메모리 또는 디스크)
index = tantivy.Index(schema, path="/tmp/tantivy_index")

Tantivy는 Rust로 작성된 검색 라이브러리로, Apache Lucene의 Rust 대안이다. tantivy-py로 Python 바인딩을 제공한다. 별도 서버 프로세스 없이, FastAPI 프로세스 안에서 직접 동작한다.

from fastapi import FastAPI
from contextlib import asynccontextmanager
import tantivy
import asyncio
from functools import partial


@asynccontextmanager
async def lifespan(app: FastAPI):
	schema_builder = tantivy.SchemaBuilder()
	schema_builder.add_text_field("name", stored=True)
	schema_builder.add_text_field("description", stored=True)
	schema = schema_builder.build()
	app.state.index = tantivy.Index(schema, path="./search_index")
	yield


app = FastAPI(lifespan=lifespan)


@app.post("/index")
async def index_docs(docs: list[dict]):
	writer = app.state.index.writer()
	for doc in docs:
		writer.add_document(tantivy.Document(**doc))
	# commit은 디스크 I/O이므로 executor에서 실행
	await asyncio.get_event_loop().run_in_executor(None, writer.commit)
	return {"indexed": len(docs)}


@app.get("/search")
async def search(q: str):
	app.state.index.reload()
	searcher = app.state.index.searcher()
	query = app.state.index.parse_query(q, ["name", "description"])
	# search도 CPU 바운드이므로 executor로
	search_fn = partial(searcher.search, query, 20)
	results = await asyncio.get_event_loop().run_in_executor(None, search_fn)
	return [
		{
			"score": score,
			"doc": searcher.doc(addr),
		}
		for score, addr in results.hits
	]

Tantivy는 async SDK가 없다. 네이티브 Rust 바인딩이므로 GIL을 해제하긴 하지만, run_in_executor로 감싸야 이벤트 루프를 안전하게 유지할 수 있다.

Tantivy가 적합한 경우:

  • 외부 서비스 의존 없이 검색을 프로세스 내에 임베딩하고 싶을 때
  • CLI 도구, 데스크톱 앱, 단일 서버 배포처럼 아키텍처가 단순한 경우
  • 문서 수가 수백만 건까지이고, 수평 확장이 필요 없는 경우

부적합한 경우:

  • 다중 FastAPI 워커가 같은 인덱스에 쓸 때 동시성 제어가 까다롭다
  • 오타 허용, 패싯, 하이라이팅은 직접 구현해야 한다
  • 한국어 토크나이저를 직접 구성해야 한다 (lindera 토크나이저 설정 필요)

벡터 검색이 필요할 때 — Qdrant

키워드 매칭이 아니라 의미 기반 검색이 필요하면 카테고리가 달라진다. "노트북 추천"으로 검색했을 때 "랩탑 비교"도 결과에 나와야 하는 경우다.

# pip install qdrant-client sentence-transformers
from qdrant_client import AsyncQdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct
from sentence_transformers import SentenceTransformer


encoder = SentenceTransformer("all-MiniLM-L6-v2")  # 384차원
qdrant = AsyncQdrantClient("http://localhost:6333")


@app.on_event("startup")
async def setup():
	await qdrant.create_collection(
		collection_name="products",
		vectors_config=VectorParams(size=384, distance=Distance.COSINE),
	)


@app.post("/index")
async def index_products(products: list[Product]):
	points = []
	for i, p in enumerate(products):
		vector = encoder.encode(f"{p.name} {p.description}").tolist()
		points.append(PointStruct(
			id=p.id,
			vector=vector,
			payload=p.model_dump(),
		))
	await qdrant.upsert(collection_name="products", points=points)
	return {"indexed": len(points)}


@app.get("/search")
async def semantic_search(q: str):
	query_vector = encoder.encode(q).tolist()
	results = await qdrant.search(
		collection_name="products",
		query_vector=query_vector,
		limit=20,
	)
	return [
		{"score": hit.score, "product": hit.payload}
		for hit in results
	]

Qdrant는 Rust 기반, 네이티브 async Python SDK를 제공한다. 메모리 사용량은 벡터 차원과 문서 수에 비례하지만, 동일 규모에서 ES보다 적다.

주의할 점: 벡터 검색은 임베딩 모델에 의존한다. sentence-transformers로 인코딩하는 시간이 검색 자체보다 오래 걸릴 수 있다. 프로덕션에서는 임베딩을 인덱싱 시점에 미리 계산하고, 쿼리 임베딩만 실시간으로 생성하는 패턴을 쓴다.

선택 매트릭스

"검색 엔진이 필요하다"
│
├─ 이미 PostgreSQL을 쓰고 있다
│   ├─ 오타 허용/자동완성 필요 없음 → PostgreSQL FTS
│   └─ 오타 허용/자동완성 필요 → Meilisearch 추가
│
├─ 별도 검색 서비스를 도입한다
│   ├─ 키워드 검색
│   │   ├─ 단일 서버, 빠른 셋업 → Meilisearch
│   │   ├─ 클러스터링 필요 → Typesense
│   │   └─ 집계/로그 분석 필요 → OpenSearch
│   │
│   └─ 의미 기반 검색 → Qdrant (+ 키워드 검색 엔진 병행)
│
├─ 외부 서비스 없이 프로세스 내 임베딩 → Tantivy
│
└─ 기존 ES 쿼리/대시보드 유지 필수 → OpenSearch
엔진 메모리 (1M 문서 기준) async SDK 한국어 클러스터링 주 용도
Meilisearch ~200MB 커뮤니티 SDK 내장 CJK 없음 즉시 검색, 이커머스
Typesense ~200MB httpx 래퍼 필요 내장 Raft 지원 즉시 검색, HA 필요 시
PostgreSQL FTS DB에 포함 asyncpg 네이티브 pg_bigm 필요 PostgreSQL HA 소규모, 일관성 중시
OpenSearch ~4GB+ 공식 async 플러그인 ES 호환 클러스터 로그, 집계, ES 마이그레이션
Tantivy ~100MB 없음 (executor) lindera 필요 없음 임베디드, CLI, 단일 서버
Qdrant ~300MB 공식 async 임베딩 모델 의존 내장 의미 검색, 추천

어디서부터 시작할 것인가

PostgreSQL을 이미 쓰고 있다면 FTS부터 시도하라. 별도 인프라 없이 ALTER TABLE 하나로 검색이 붙는다. 부족하다고 느끼는 지점이 생기면 — 오타 허용이 아쉽거나, 자동완성 응답이 느리거나 — 그때 Meilisearch를 Docker 한 줄로 올리면 된다. ES가 필요한 진짜 순간은 생각보다 늦게 온다.
