SSOT가 깨지는 자리 — 데이터 아키텍처 관점

같은 이메일이 세 곳에서 달랐다

어느 금요일 저녁, CS팀이 티켓을 올렸습니다. "회원이 설정 화면에서 이메일을 바꿨는데, 추천 메일은 예전 주소로 계속 나가요." 원인은 단순했습니다. 인증 서비스의 PostgreSQL은 새 이메일을 갖고 있었고, 추천 서비스가 보는 Redis 캐시는 6시간 묵은 값을, 마케팅 자동화가 쓰는 BigQuery는 전날 새벽 배치 이후로 아무것도 갱신되지 않은 상태였습니다. 세 개의 저장소가 각자 자기가 진실이라고 주장하고 있었고, 사용자 입장에선 그 셋 다 자기 시스템이었습니다.

이 장면이 SSOT(Single Source of Truth)가 데이터 아키텍처에서 다뤄지는 진짜 이유입니다. 문서에 "유저 테이블은 auth-db가 소유한다"고 적어둔다고 끝나지 않습니다. 같은 엔터티의 복제본이 OLTP DB, 데이터 웨어하우스, 캐시, 검색 인덱스, 그리고 이벤트를 구독하는 N개의 다운스트림 서비스에 퍼져 있는 게 현실의 분산 시스템입니다.

왜 엔터티는 한 벌로 못 사는가

서비스가 복잡해질수록 저장소는 용도별로 찢어집니다. 트랜잭션은 Postgres에서 하고, 집계는 BigQuery에서, 자동완성은 Elasticsearch에서, 프로필 조회는 Redis에서 받습니다. 각 저장소는 서로 다른 물리 포맷과 읽기 패턴에 최적화돼 있기 때문에, "하나의 DB만 쓴다"는 선택지는 규모가 커지면 사실상 사라집니다.

문제는 이 분화 자체가 아니라, 쓰기가 여러 저장소로 향한다는 점입니다. 쓰기가 하나여도 읽기 복제본이 여러 곳이면 복제 지연만 신경 쓰면 됩니다. 그러나 애플리케이션 코드가 "Postgres에 저장하고, Redis에 갱신하고, Kafka로 이벤트를 쏜다"고 세 번 쓰는 순간 SSOT는 기술 부채가 됩니다.

이중 쓰기는 언제나 조용히 깨진다

전형적인 안티패턴은 이렇게 생겼습니다.

def update_email(user_id, new_email):
	db.execute("UPDATE users SET email=%s WHERE id=%s", new_email, user_id)
	cache.set(f"user:{user_id}:email", new_email)
	kafka.produce("user.email_changed", {"user_id": user_id, "email": new_email})

세 줄 전부 성공해야 상태가 일관됩니다. 하지만 DB 커밋 뒤에 Redis 네트워크가 끊기면 캐시는 낡은 값을 유지합니다. Kafka produce만 실패하면 DW와 추천 서비스는 변경을 영원히 못 봅니다. 재시도 로직을 넣어도 본질적인 문제, 즉 두 개의 서로 다른 시스템에 원자적으로 쓸 방법이 없다는 사실은 바뀌지 않습니다. 분산 트랜잭션(2PC)은 운영 비용이 크고, 장애 상황에서 블로킹을 만들기 때문에 대부분의 현대 스택에서는 선택지에서 빠집니다.

결과적으로 이중 쓰기는 "대체로 잘 되다가 가끔 이상해지는" 시스템을 만듭니다. 버그 리포트는 재현이 안 되고, 매번 수작업으로 캐시를 날리거나 재집계 잡을 돌리게 됩니다.

아웃박스 패턴 + CDC — 쓰기를 하나로

해결의 방향은 단순합니다. 쓰기를 한 군데로 수렴시키고, 나머지는 그 한 군데에서 파생되도록 만드는 것입니다. outbox pattern과 CDC의 조합이 그 표준적인 구현입니다.

아웃박스 패턴에서는 비즈니스 테이블을 업데이트할 때 같은 트랜잭션에서 outbox 테이블에 이벤트 레코드도 함께 씁니다. DB 트랜잭션의 원자성이 보장되기 때문에 "상태 변경은 됐는데 이벤트가 안 나갔다"는 케이스가 사라집니다.

BEGIN;
UPDATE users SET email = 'new@example.com', updated_at = now() WHERE id = 42;
INSERT INTO outbox (aggregate_type, aggregate_id, event_type, payload, created_at)
VALUES ('user', '42', 'email_changed',
	'{"user_id":42,"email":"new@example.com"}'::jsonb, now());
COMMIT;

그 뒤를 Debezium 같은 CDC 커넥터가 받습니다. Postgres의 WAL(또는 MySQL binlog)을 구독해서 outbox 테이블 변경을 Kafka 토픽으로 흘려보냅니다.

{
	"name": "users-outbox-connector",
	"config": {
		"connector.class": "io.debezium.connector.postgresql.PostgresConnector",
		"database.hostname": "auth-db",
		"database.dbname": "auth",
		"plugin.name": "pgoutput",
		"publication.name": "outbox_pub",
		"table.include.list": "public.outbox",
		"transforms": "outbox",
		"transforms.outbox.type": "io.debezium.transforms.outbox.EventRouter",
		"transforms.outbox.route.topic.replacement": "events.${routedByValue}"
	}
}

<이중 쓰기는 실패 지점이 N개, Outbox+CDC는 하나의 TX에서 출발해 N개로 팬아웃된다>

이 구조의 핵심은 Postgres가 유일한 쓰기 목적지라는 점입니다. Redis, Elasticsearch, BigQuery는 전부 Kafka 토픽을 구독해 자기 스키마로 투영(projection)합니다. 다운스트림 중 하나가 죽어도 토픽은 남아 있고, 복구 후 오프셋을 이어 소비하면 결국 따라잡힙니다(eventual consistency). 중요한 건 발산이 아니라 수렴이 기본값이 됐다는 것입니다.

이벤트 소싱 — 상태를 이벤트로 대체한다

아웃박스가 "상태 테이블 + 파생 이벤트"라면, event sourcing은 한 발 더 나갑니다. 상태 테이블을 아예 만들지 않고, 엔터티의 변화를 append-only 이벤트 로그로만 기록합니다. 현재 상태는 이벤트를 재생(replay)해서 얻습니다.

UserCreated(id=42, email=old@example.com, ts=2026-04-01T10:00:00Z)
EmailChanged(id=42, email=new@example.com, ts=2026-04-20T19:23:11Z)

이 모델에서 SSOT는 이벤트 로그 자체입니다. 어떤 읽기 모델(프로필 캐시, 검색 인덱스, OLAP 큐브)이 깨져도, 로그를 처음부터 다시 돌려 재구축할 수 있습니다. 감사(audit)와 시간여행(temporal query)이 공짜로 따라옵니다.

대가는 만만치 않습니다. 스키마 변경은 과거 이벤트를 그대로 둔 채 업캐스팅(upcasting)으로 처리해야 하고, 재생 성능을 위해 스냅샷 전략을 설계해야 하며, 개발자가 CRUD 머리를 도메인 이벤트 머리로 바꾸는 데 시간이 걸립니다. 모든 도메인이 이벤트 소싱에 맞지 않습니다. 주문, 결제, 포지션 같이 변화 이력 자체가 비즈니스 가치인 영역에서 가장 강력합니다.

데이터 컨트랙트 — 조직 차원의 SSOT

기술적으로 Kafka 토픽을 SSOT로 만들어도, 조직이 그걸 깰 수 있습니다. 프로듀서 팀이 공지 없이 필드를 바꾸면 컨슈머 대여섯 팀이 새벽에 깨어납니다. 이걸 막는 장치가 data contract입니다.

데이터 컨트랙트는 토픽/테이블의 스키마, 의미론, SLA, 소유자, 하위 호환성 정책을 명시한 계약입니다. 스키마 레지스트리(Confluent Schema Registry, AWS Glue Schema Registry 등)에 등록하고 CI에서 호환성을 강제하는 게 실무적인 형태입니다.

dataset: events.user.email_changed
owner: team-auth
version: 2.1.0
schema:
	type: record
	fields:
		- name: user_id
			type: long
		- name: email
			type: string
		- name: changed_at
			type: { type: long, logicalType: timestamp-millis }
sla:
	freshness_seconds: 30
	availability: 99.9
compatibility: BACKWARD
pii:
	- email

이 문서가 있으면 "이메일 변경 이벤트의 진실이 무엇인가"라는 질문에 대한 답이 단일해집니다. 프로듀서는 계약을 어기지 않고서는 배포할 수 없고, 컨슈머는 계약에 적힌 필드만 믿고 소비합니다. 기술적 SSOT가 프로토콜이라면, 데이터 컨트랙트는 그 프로토콜을 조직이 배신하지 않겠다는 합의입니다.

트레이드오프는 사라지지 않는다

SSOT 아키텍처를 제대로 세우면 얻는 게 분명합니다.

  • 장애 후 복구가 재생으로 끝난다 — 수동 수리 잡이 사라진다
  • 새 컨슈머(새 ML 모델, 새 검색 인덱스)를 과거 데이터부터 붙일 수 있다
  • 감사와 디버깅이 가능해진다 — "그 순간 시스템이 뭘 믿고 있었나"에 답할 수 있다

대신 지불하는 것도 명확합니다.

  • 운영 부담 증가: Kafka/Debezium/스키마 레지스트리를 굴리고, DLQ, 컨슈머 랙, 커넥터 헬스체크를 모니터링해야 한다
  • 최종 일관성의 수용: 캐시와 검색 인덱스는 몇 초~몇 분 뒤처진다. UX와 제품 요구사항이 이를 견딜 수 있어야 한다
  • 재생 복잡도: 긴 히스토리를 재생할 때 시간이 걸리고, 부작용(외부 API 호출 등)이 있는 이벤트는 재생 시 반드시 멱등해야 한다
  • 학습 곡선: 팀이 "DB 한 개에 UPDATE 때리는" 세계관에서 벗어나야 한다

언제 이 세금을 낼 가치가 있는가

단일 서비스, 단일 DB, 하루 수천 쓰기 수준이라면 SSOT를 위한 인프라 투자는 과잉 설계입니다. 캐시는 TTL로 털어내고, DW는 밤중에 덤프로 쏟아붓는 걸로 충분합니다. 이중 쓰기가 가끔 깨져도 비용보다 피해가 적습니다.

SSOT 아키텍처로 넘어가야 할 신호는 세 가지입니다. 첫째, 같은 엔터티를 쓰는 다운스트림이 3개 이상이고 각각 다른 팀이 소유한다. 둘째, "왜 이 두 화면의 숫자가 다르냐"는 질문이 반복적으로 엔지니어링 시간을 잡아먹는다. 셋째, 규제·감사·금액이 걸려 있어서 "그 시점에 시스템이 뭘 믿고 있었는지"를 증명해야 한다. 이 중 두 개 이상이 맞으면, 아웃박스와 CDC를 먼저 깔고, 변화 이력이 비즈니스인 도메인에 한해 이벤트 소싱을 얹고, 조직이 여럿 엮인 경계에는 데이터 컨트랙트를 걸어두는 순서가 합리적입니다.

SSOT는 기술 선택이 아니라 "어디를 진실로 정하고, 거기서 나머지가 파생되도록 강제하겠다"는 결정입니다. 그 결정이 없으면 어떤 도구도 진실을 만들어주지 않습니다.
