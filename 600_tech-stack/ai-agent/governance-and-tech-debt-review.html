Governance and Tech Debt Review

Governance and Tech Debt Review: Definitive Recommendation

Date: 2026-03-12

Corpus reviewed: 27 markdown documents in /Users/davidpark/Projects/Mirrorless_Mono/docs/plans/

Initiatives represented: 13 workstreams spanning product UX, platform architecture, incident response, and reliability hardening

External guidance consulted: OpenAI, Microsoft, Google ADK, AWS, NIST AI RMF, Atlassian, Thoughtworks, Microsoft Well-Architected

Executive Recommendation

Do not keep governance review and tech debt review as one undifferentiated process.

Also do not split them into fully separate document systems or separate boards.

The best fit for this corpus is a unified intake with split review tracks:

  1. Feature / governance track -- for user-facing behavior, UX, and product scope decisions

  2. Reliability / tech debt track -- for defects, regressions, operational safety, and engineering friction

  3. Hybrid architecture track -- for work that changes both product behavior and system reliability

This recommendation is driven by three converging lines of evidence:

  • The shape of the documents themselves (Section 1): the plans share a common structure but differ in triggers, reviewers, urgency, and evidence expectations

  • Eight concrete governance gaps (Section 2): the corpus already shows where undifferentiated review fails

  • External best practices (Section 3): industry guidance from NIST, AWS, Microsoft, Thoughtworks, and others confirms that governance must be cross-cutting but review must be fit-for-purpose

The concrete answer: divide the review tracks, not the documentation system.

1. What the Corpus Shows

Across 27 documents, the work clusters into three practical groups.

A. Feature-heavy / governance-heavy initiatives (5)

Primarily about user-facing behavior, UX choices, or product-level scope decisions:

  • header-profile-badge-* -- UI placement, animation, visibility

  • experimental-feature-flags-* -- user feedback-driven (MIR-KSK-294), approved status, UI placement decisions

  • retake-warning-popup-* -- preventing accidental user data loss through confirmation UX

  • touch-control-drawer-* -- touch targets, visual specs, guarded interactions (long-press session completion)

  • face-blur-image-* -- privacy-sensitive media processing with user-visible output

These need reviewers who can judge product fit, UX consistency, and scope discipline.

B. Reliability / tech debt initiatives (5)

Primarily about defects, regressions, operational safety, or engineering friction:

  • crm-fastapi-live-regression-* -- audit-oriented package: approved constraints, timebox, verification commands, execution log

  • crash-recovery-hardening-* -- 22 potential crash recovery vulnerabilities, atomic writes, re-queuing, startup ordering

  • cicd-redesign-* -- broken CI topology and deploy gating

  • sharp-thumbnail-* -- silent correctness bug

  • lut-pipeline-race-condition-* -- surgical fix for two root causes

These need reviewers who can judge failure modes, rollback posture, regression coverage, and operational impact.

C. Hybrid architecture initiatives (3)

The reason a hard split is the wrong answer:

  • kiosk-auth-offline-* -- security policy + offline resilience + queue architecture + token scope + user-visible QR login

  • qr-login-persistence-* -- reliability gap (refresh/crash state loss) that shows up as a user-facing badge/profile regression

  • auto-update-architecture-* -- operational safety + rollout policy + rollback strategy + admin-facing UI/API surface

These need both governance-style and reliability-style review. If the two tracks were fully separated, these initiatives would either be reviewed twice with duplicate overhead or reviewed once through the wrong lens.

Why one process is not enough

| Dimension | Feature track | Reliability track |

|-----------|--------------|-------------------|

| Trigger | Proactive: roadmap, user feedback, planned extension | Reactive: regression, code review finding, CI failure, production defect |

| Reviewer fit | Product/UX reviewer: animation, affordances, scope | Infra/reliability reviewer: CI topology, rollback flows, race conditions |

| Urgency | Planned cadence, paired design/plan cycles | May need immediate action (e.g., CRM regression 12-hour plan) |

| Evidence | Smoke tests, UX validation | Root-cause analysis, verification commands, execution logs, schema notes |

A single review checklist either under-specifies evidence for debt work or over-burdens routine feature work.

Why a full split also fails

  • Artifact shape is shared. Most initiatives follow: design doc, implementation plan, file-change scope, verification expectations. Breaking this into separate repositories or templates destroys useful consistency.

  • Hybrid work is common, not exceptional. Three of the most architecturally important initiatives are hybrid -- this is a real category, not edge-case noise.

  • Low-friction documentation is a strength. The corpus is co-located in one directory with consistent naming. Splitting too hard adds admin work faster than it adds clarity.

2. Eight Governance Gaps as Evidence

The following gaps were identified by cataloging all 27 documents against governance best practices. They provide the concrete "why" for differentiating review tracks and adding operating mechanisms.

Gap 1: Ownership is mostly implicit

Most docs do not name a responsible team, DRI, approver, or operational owner. The few owner references are implementation settings, not accountability metadata.

Implication: Without explicit ownership, review assignments default to whoever is available rather than whoever is qualified. This is especially costly for hybrid work where both product and reliability perspectives are needed.

Gap 2: Documentation standards are inconsistent

Some design docs include Date, Status, Issue, Target, or explicit constraints; many do not. The corpus mixes design docs, plans, execution logs, and schema notes without a required template or common metadata standard.

Implication: Inconsistent metadata makes track routing manual and error-prone. A reviewer cannot quickly determine whether a document needs feature review, reliability review, or both.

Gap 3: Rollout policy is underdefined

Outside the auto-update work, most features describe implementation but not activation policy, blast-radius control, phased rollout, canary strategy, or approval gates. Even the auto-update design chooses full simultaneous rollout.

Implication: This is the gap where governance and tech debt most visibly collide. A feature with no rollout policy creates tech debt (cleanup, rollback complexity) when it fails.

Gap 4: Auditability is uneven

The CRM regression documents are the strongest example of audit trail discipline (execution logs, schema notes, verification commands), but that standard is not carried across most other feature docs.

Implication: Reliability review needs audit artifacts. Feature review needs scope artifacts. A unified process that does not distinguish these produces inconsistent quality.

Gap 5: Observability governance is thin

Logging appears in several plans, but there is little definition of metrics, alert thresholds, dashboards, incident response, or runbooks. Recovery behavior is described more often than operational response.

Implication: Tech debt review should explicitly assess observability posture. Feature review should not be burdened with this.

Gap 6: Security and privacy process is under-specified

Kiosk auth, QR persistence, and face blur add sensitive token/media handling, but the docs rarely define key-rotation policy, encryption-at-rest expectations, incident procedures, consent/privacy classification, or deletion SLAs.

Implication: Security review is a cross-cutting concern that should be triggered by track routing metadata, not left to chance.

Gap 7: Data lifecycle controls are local, not policy-driven

Some docs mention cleanup, TTLs, or file deletion, but there is no consistent cross-cutting policy for retention windows, secure deletion, orphan cleanup, or backfill/migration validation.

Implication: Data lifecycle gaps create tech debt that accumulates invisibly until it causes an incident. The reliability track should explicitly assess this.

Gap 8: Experiment governance is incomplete

Experimental feature flags get a registry and UI, but there is no documented graduation, sunset, approval, or telemetry policy for deciding whether a flag stays experimental, becomes default, or is removed.

Implication: Flags without graduation policy are tech debt by definition. Feature review should cover the experiment lifecycle; reliability review should assess cleanup cost.

3. External Best Practices Grounding

Each element of the recommendation is grounded in published industry guidance.

Cross-cutting governance (NIST)

NIST AI RMF defines Govern as a cross-cutting function infused throughout risk management, not a one-time checkpoint. (NIST AI RMF Core)

Application: Governance metadata (track routing, ownership, verification type) should be embedded in every design doc, not maintained in a separate system.

Centralized governance with federation path (AWS)

AWS describes a centralized generative AI operating model as a hub where reusable components, orchestration, and monitoring live in one place, promoting consistency and control. It also describes a federated model as decentralized development with centralized integration and monitoring. (AWS Bedrock operating models)

Application: Start centralized (one review process, one document location, one template family). Evolve to federated only when multiple domain teams genuinely need autonomy.

Architecture review board with debt ownership (AWS, Microsoft)

AWS recommends an ARB with executive sponsorship, defined stakeholders, and a single source of truth for guidance. (AWS ARB guidance) Microsoft Well-Architected recommends version-controlled ADRs and treating tech debt as a regularly recurring backlog item. (Microsoft Well-Architected)

Application: One lightweight architecture and debt council with two jobs: (1) review important architecture and policy changes, (2) continuously prioritize debt items created by those decisions.

Debt is multi-dimensional (Thoughtworks)

Thoughtworks emphasizes that debt is not only code debt; it includes architectural, technical decision, test, documentation, infrastructure, process, culture, UX, and security debt. (Thoughtworks technical debt guide)

Application: The reliability/tech debt track must assess debt across all dimensions, not just code quality. The eight governance gaps above span architecture debt, process debt, documentation debt, security debt, and observability debt.

Federated computational governance (Thoughtworks)

Thoughtworks' data mesh framing: give domains autonomy, but encode global rules and standards into the platform. (Thoughtworks on data mesh)

Application: The three-track model encodes global rules (routing metadata, shared templates) while giving each track autonomy over its review criteria.

RACI as minimum viable governance (Atlassian)

Atlassian's RACI guidance: Responsible (does the work), Accountable (final owner), Consulted (gives input), Informed (receives updates). Single accountable owner per task. (Atlassian RACI chart)

Application: Every initiative should have explicit RACI, especially hybrid work where the accountable owner must be named to prevent responsibility diffusion.

Human-in-the-loop for high-risk actions (Google, Microsoft)

Google ADK's Human in the Loop with PolicyEngine pattern and Microsoft's HITL guidance recommend explicit approval gates for destructive or irreversible actions. (Google ADK multi-agent systems, Microsoft agent orchestration patterns)

Application: Policy-gated approval for: schema changes, production deploys, secrets access, irreversible migrations, cross-domain writes. These gates should be encoded in the track-specific review checklists.

Observability around handoffs and decisions (Microsoft, AWS)

Microsoft recommends instrumenting operations and handoffs, integration testing multi-agent workflows, and adding HITL gates. AWS recommends EventBridge-style deterministic, auditable routing. (Microsoft agent patterns, AWS orchestration models)

Application: Every delegation, tool call, approval, retry/rollback, and final decision should be logged with correlation IDs. This is the observability standard the reliability track should enforce.

4. Recommended Operating Model

Single intake, three review tracks

Adopt a single intake and document location, routing each initiative into one of three review tracks:

  1. Feature / governance -- "Should we do this and is the user-facing behavior right?"

  2. Reliability / tech debt -- "Does this reduce risk, remove a defect, or improve operational correctness?"

  3. Hybrid architecture -- "This changes product behavior and system reliability simultaneously."

Shared intake requirements

Every initiative declares routing fields at the top of the design doc:


type: feature | tech-debt | hybrid

trigger: roadmap | user-feedback | incident | code-review | operational-friction

owner: <team/person>

reviewers: <roles>

verification: smoke | test-suite | audit-log | rollback-check | mixed

This keeps the storage model unified while making routing explicit. It directly addresses Gap 2 (inconsistent documentation standards) and Gap 1 (implicit ownership).

Track 1: Feature / governance review

Primary criteria:

  • User value and UX consistency

  • Scope discipline / non-goals

  • API or schema compatibility

  • Experiment lifecycle (graduation, sunset, telemetry) -- addresses Gap 8

  • Privacy and consent classification where sensitive data is involved -- addresses Gap 6

Typical examples: header profile badge, retake warning popup, touch control drawer, experimental feature flags

Suggested reviewers: product/design owner, feature owner or tech lead

Track 2: Reliability / tech debt review

Primary criteria:

  • Root cause clarity

  • Regression prevention

  • Rollback or containment posture

  • Verification strength (execution logs, schema notes, test suites)

  • Operational blast radius

  • Observability posture (metrics, alerts, dashboards, runbooks) -- addresses Gap 5

  • Data lifecycle compliance (retention, cleanup, deletion) -- addresses Gap 7

  • Rollout policy (blast radius, canary, activation gates) -- addresses Gap 3

Typical examples: CRM live regression, crash recovery hardening, CI/CD redesign, sharp thumbnail bug, LUT race condition fix

Suggested reviewers: system owner, infra/reliability owner

Track 3: Hybrid architecture review

Primary criteria:

  • Pass both feature/governance and reliability/debt checks

  • Require named sign-off from both perspectives

  • Keep one document set, not two parallel plans

  • Explicit RACI naming the accountable owner

Typical examples: kiosk auth + offline resilience, QR login persistence, auto-update architecture

Suggested reviewers: tech lead / architect, product or UX representative, reliability or platform owner

5. Concrete Operating Mechanisms (the HOW)

Six mechanisms transform the three-track model from a conceptual recommendation into an operational system.

Mechanism 1: Agent/capability registry

Mirrorless already uses SSOT registries for feature flags. Apply the same idea to all capabilities and agents:

| Field | Purpose |

|-------|---------|

| Capability name | Identification |

| Purpose | Why it exists |

| Tools/data access | What it can touch |

| Owner | DRI (addresses Gap 1) |

| Approval level | Who must sign off |

| Required evals | What must pass before deploy |

| Rollback behavior | How to undo |

| Last review date | When it was last assessed |

This is the missing bridge between the existing ServiceRegistry pattern and a real governed platform. It directly addresses Gap 1 (implicit ownership) and provides the data needed for track routing.

Mechanism 2: Architecture Decision Records (ADRs)

The plans already function like design memos, but the project needs tighter, version-controlled decision records for:

  • Orchestration model choices

  • Agent and service boundaries

  • Policy boundaries and approval thresholds

  • Event schema choices

  • Memory/state retention choices

  • Track routing precedents (when a hybrid decision sets a pattern)

ADRs directly address Gap 2 (inconsistent standards) and Gap 4 (uneven auditability) by creating a durable record of why decisions were made, not just what was built.

Industry backing: Microsoft Well-Architected recommends version-controlled ADRs as a core practice. (Microsoft Well-Architected)

Mechanism 3: RACI matrix

At minimum, define RACI for these lifecycle stages:

| Stage | R | A | C | I |

|-------|---|---|---|---|

| Agent spec/prompt ownership | | | | |

| Tool access approval | | | | |

| Model/config changes | | | | |

| Production deployment | | | | |

| Evaluation sign-off | | | | |

| Incident response | | | | |

| Rollback execution | | | | |

| Tech debt remediation | | | | |

Without this, hybrid work creates gray zones where no one is clearly accountable. The CRM regression set is the positive example -- it has implicit RACI through its constraints and execution log. The rest of the corpus does not.

Industry backing: Atlassian recommends a single accountable owner per task. (Atlassian RACI)

Mechanism 4: Policy-gated human-in-the-loop (HITL)

The following actions should require explicit approval, enforced by review track checklists:

  • Destructive filesystem actions

  • Schema changes (database migrations)

  • Production deploys

  • Secrets access or rotation

  • Irreversible migrations

  • Cross-domain writes

This directly addresses Gap 3 (underdefined rollout policy) and Gap 6 (under-specified security process).

Industry backing: Google ADK's PolicyEngine pattern and Microsoft's HITL guidance. (Google ADK, Microsoft)

Mechanism 5: Standardized observability

Extend the existing execution log discipline (strongest in CRM regression docs) to all tracks:

  • Every delegation, tool call, and event emission logged with correlation IDs

  • Every approval and rejection recorded

  • Every retry and rollback recorded

  • Every final decision recorded with rationale

The reliability track should enforce this as a review criterion. The feature track should require it for any initiative that touches shared state or infrastructure.

Industry backing: Microsoft recommends instrumenting handoffs and decisions. AWS recommends auditable routing. (Microsoft, AWS)

Mechanism 6: Lightweight architecture and debt review cadence

Run a weekly or biweekly review with a narrow charter:

  1. Approve or reject major capability additions

  2. Review failed runs and policy exceptions

  3. Review new debt introduced by shortcuts

  4. Decide whether debt stays in team backlogs or escalates

  5. Assess experiment flag lifecycle (graduation, sunset, removal) -- addresses Gap 8

Do not create two heavy boards. Create one lightweight council with two standing agenda items: architecture decisions and debt prioritization.

Industry backing: AWS recommends an ARB with defined stakeholders and a single source of truth. Thoughtworks emphasizes that debt spans architecture, process, documentation, security, and more. (AWS ARB, Thoughtworks)

6. Implementation Path

Ordered from highest-impact/lowest-cost to longer-term investments.

Step 1: Add routing metadata to design doc template (NOW)

Add the five routing fields (type, trigger, owner, reviewers, verification) to the design doc template. Apply retroactively to in-flight work.

Effort: 1 day. Addresses: Gaps 1, 2.

Step 2: Formalize the capability registry (NOW)

Create a single file or table listing all services, agents, and capabilities with owner, approval level, and rollback behavior. Start with the services already documented in the 27 plans.

Effort: 2-3 days. Addresses: Gap 1. Grounded in: existing ServiceRegistry and feature flag registry patterns already in the codebase.

Step 3: Start the ADR practice (NEXT 2 WEEKS)

Begin recording architecture decisions as lightweight ADRs. Backfill the most important decisions from the existing corpus (orchestration model, persistence strategy, offline-first design, CI/CD topology).

Effort: 1-2 days for first 5 ADRs, then ongoing. Addresses: Gaps 2, 4.

Step 4: Add RACI for agent lifecycle and production operations (NEXT 2 WEEKS)

Fill in the RACI matrix for the eight lifecycle stages listed in Mechanism 3. Start with the stages that have the most ambiguity today (incident response, rollback execution, debt remediation).

Effort: 1 day workshop. Addresses: Gap 1 fully.

Step 5: Add policy-gated HITL for sensitive actions (NEXT MONTH)

Define and enforce approval gates for schema changes, production deploys, secrets access, and irreversible migrations. Encode these in the reliability track review checklist.

Effort: 3-5 days. Addresses: Gaps 3, 6.

Step 6: Standardize observability with correlation IDs and audit logs (NEXT MONTH)

Extend the CRM regression execution log standard to all work that touches shared state, infrastructure, or cross-domain boundaries. Add correlation IDs to delegations, events, and approvals.

Effort: 5-7 days. Addresses: Gaps 4, 5.

Step 7: Establish the review cadence and reassess federation (NEXT QUARTER)

Launch the lightweight architecture and debt review meeting. After 2-3 months of operation, assess whether the centralized model should evolve toward federated governance based on whether multiple domain teams genuinely need autonomy.

Effort: Ongoing. Addresses: Gap 8 and long-term governance maturity. Grounded in: AWS federated operating model guidance -- federate only when domains actually need it.

8. Bottom Line

The Mirrorless plans corpus is already strong on disciplined delivery: explicit phases, verification commands, rollback planning, and execution logs. What it lacks is differentiated review, explicit ownership, and operating mechanisms that prevent governance gaps from becoming tech debt.

The three-track model is the smallest process change that matches the evidence:

  • One place to write plans

  • One basic template family with routing metadata

  • Three review tracks with different criteria, reviewers, and evidence expectations

  • Six operating mechanisms that make the tracks actionable

  • Seven implementation steps ordered by impact and effort

That preserves speed while preventing two real failure modes:

  1. Tech debt getting reviewed as if it were just another feature

  2. Hybrid architecture work getting trapped between two separate processes

The path forward is not more process. It is the right process in the right places, grounded in what the documents already show and what the industry has learned.
