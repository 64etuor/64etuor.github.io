AI 에이전트 시대의 문서화 전략과 하네스 설계

AI 에이전트 시대의 문서화 전략과 하네스 설계

AI 에이전트가 코드를 읽고 쓰는 시대에, "문서"의 역할이 근본적으로 달라지고 있다. 사람이 읽는 문서와 LLM이 읽는 문서는 요구사항이 다르고, 문서의 최신성이 곧 에이전트의 신뢰성이 된다. 이 글은 실전에서 부딪히는 문서화 문제들과 그 해법을 정리하고, 각 주장의 한계도 함께 다룬다.

1. 글로벌 스킬 vs 로컬 스킬: 이중 배치 패턴

에이전트 스킬을 어디에 둘 것인가는 단순한 정리 문제가 아니다.

플러그인 하나를 등록하는 데 파일 10군데를 수정해야 하는 프로젝트가 있다고 하자. 에이전트가 10개 파일을 전부 읽고 맥락을 이해하는 데는 토큰 비용이 든다. 해법은 각 디렉토리에 로컬 instruction을 배치하는 것이다.

스킬 유형 위치 용도
글로벌 스킬 ~/.claude/skills/ 프로젝트 무관 범용 워크플로우
로컬 스킬 소스 레포 내 각 디렉토리 프로젝트 특화 작업 이행 가이드
배치 원칙

스킬의 위치는 "누가, 어디서 읽느냐"로 결정한다. 범용 패턴은 글로벌, 특정 디렉토리 작업 가이드는 해당 디렉토리 안에.

이중 배치의 함정

두 곳에 스킬이 있으면 어느 쪽이 최신인지 관리해야 한다. 이것은 §2에서 비판하는 "구버전 문서" 문제를 스킬 레이어에서 그대로 재현하는 것이다. Single Source of Truth 원칙과 충돌하므로, 글로벌과 로컬 스킬의 책임 범위가 겹치지 않도록 설계해야 한다. 글로벌은 "어떻게(how)", 로컬은 "여기서 뭘(what here)" — 이 경계가 무너지면 이중 배치의 이점이 사라진다.

2. 구버전 문서와 할루시네이션

LLM 에이전트는 소스코드를 직접 파싱하기보다 오버뷰 문서를 먼저 읽는 경향이 있다. llm.txt, overview.md, README.md 같은 파일이 디렉토리에 존재하면, 에이전트는 소스를 스캔하기 전에 이 문서부터 열어본다.

문서가 현재 코드와 동기화되어 있지 않으면, 에이전트는 구버전 정보를 사실로 받아들이고 그 위에 작업을 쌓는다. 결과는 할루시네이션이다.

구버전 문서가 정말 "없는 것보다" 나쁜가?

항상 그렇지는 않다. 코드로 검증 가능한 사실(함수 시그니처, API 엔드포인트, 파일 구조 등)이 문서에 틀리게 적혀 있어도, 좋은 LLM은 코드를 대조해서 코드 쪽을 우선한다.

진짜 위험한 건 코드만 봐서는 검증할 수 없는 맥락 정보가 구버전인 경우다:

  • "왜 이 아키텍처를 선택했는가" (의사결정 근거)
  • "이 방식은 ~한 이유로 폐기되었다" (금지 사항)
  • "이 모듈은 ~팀이 소유한다" (조직 맥락)

이런 정보가 틀려있으면 LLM이 코드를 아무리 읽어도 검증할 방법이 없다.

실전 규칙

유지보수하지 않을 overview 문서는 삭제하거나 DEPRECATED 표시. 더 좋은 방법은 문서 생성/수정 시점을 명시하는 것이다 — frontmatter에 modified 날짜를 넣거나, git blame으로 추적 가능하게. LLM이 "이 문서는 6개월 전에 마지막으로 수정되었다"를 알면 스스로 신뢰도를 조절할 수 있다.

3. 하네스 최신성 문제

하네스(harness)는 에이전트가 반복적인 실수를 하지 않도록 규칙과 패턴을 기록한 시스템이다. 효과는 기록의 최신성에 전적으로 의존한다.

구버전 하네스가 주는 거짓된 안전감은 하네스가 아예 없는 것보다 위험할 수 있다. "하네스가 있으니 괜찮겠지"라는 전제 하에 리뷰를 생략하기 때문이다.

다만 이것도 정도의 문제다. 90%가 여전히 유효한 하네스라면 없는 것보다는 낫다. 핵심은 하네스의 어느 부분이 구버전인지 파악 가능해야 한다는 것이다. 이를 위해:

  • 하네스 규칙마다 추가 날짜와 근거를 기록
  • 주기적 lint로 하네스 자체를 감사 (코드의 현재 상태와 대조)
  • 규칙이 더 이상 코드에서 관련 패턴을 찾을 수 없으면 자동 플래그

4. Plan → Markdown → Execution 체이닝

에이전트 워크플로우 패턴 중 하나:

  1. Plan 모드로 설계 문서(MD)를 생성
  2. 생성된 MD를 다음 세션의 instruction으로 투입
  3. 에이전트가 MD를 읽고 실행

이 패턴의 장점은 설계 의사결정이 문서에 남는다는 것이다. 코드 주석에 의사결정 이슈 번호를 달아서 추적할 수도 있다:

// Decision #AI-042: 동기 처리 대신 이벤트 큐 선택
// 근거: 오프라인 복원 시 순서 보장 필요
await taskQueue.enqueue(operation);

이 패턴이 깨지는 경우

Plan 모드의 출력은 설계 문서이지 실행 명령이 아니다. 추상도가 다르기 때문에 "그대로 넣으면 체이닝된다"는 경험은 Plan의 구체성 수준에 따라 크게 달라진다.

더 큰 문제는 실행 중 Plan이 틀렸다고 판명되는 경우다. Plan을 수정하고 재실행해야 하는데, 이 피드백 루프가 자동화되어 있지 않으면 사람이 개입해야 한다. Plan → Execute가 단방향이 아니라 Plan ↔ Execute 양방향 루프여야 실전에서 작동한다.

또한 의사결정 주석이 쌓이면 노이즈가 되는 시점이 온다. 주석 자체에도 생명주기 관리가 필요하다 — 결정이 뒤집혔거나 더 이상 관련 없는 주석은 정리해야 한다.

5. 정보의 입력과 검색은 다른 문제다

문서를 "잘 쓰는 것"과 "잘 찾는 것"은 서로 다른 구조를 요구한다.

검색 방식의 특성

검색 방식 작동 원리 강점 약점
Regex 텍스트 패턴 매칭 비정형 텍스트에서 정확한 패턴 추출 의미적 유사성 못 잡음
BM25 단어 빈도(TF) × 희귀도(IDF) 빠르고 인프라 가벼움, 어떤 문서든 작동 동의어/맥락 한계
RAG 임베딩 벡터 공간 의미 유사도 유사 개념 연결 정밀도 한계, 청킹 품질에 민감
그래프 엔티티-관계 구조 탐색 관계 추론, 다홉 질의 구축 비용 높음

흔한 오해: "Regex는 정형 문서를 요구한다." 사실 정규식은 본질적으로 비정형 텍스트에서 패턴을 찾기 위해 만들어진 도구다. 정형 문서가 있으면 더 쉬울 뿐, 전제 조건은 아니다. BM25도 마찬가지 — 특별한 문서 구조를 요구하지 않으며 어떤 자연어 텍스트든 작동한다.

마크다운에서 구조적으로 유의미한 파싱 단위는 사실상 헤딩 기반 섹션 분리 정도가 한계다. tree-sitter 같은 파서로 마크다운을 나누려 해도 코드만큼 깔끔하게 되지 않는다.

LLM의 비정형 문서 처리 능력과 그 한계

LLM은 LSP나 파서 없이도 함수 간 관계를 파악하고, 비정형 소스코드를 이해해서 설계하고 수정한다. 소스코드의 구성이 프로젝트마다 제각각인데도 작동한다.

그러나 이 능력은 컨텍스트 윈도우에 들어오는 범위에서만 성립한다. 파일이 수천 개인 레포에서 LLM이 전수 탐색하는 것은 불가능하다. 스케일이 커지면 결국 인덱싱 → 관련 파일 선별 → LLM 투입이라는 파이프라인이 필요하고, 이때 위 검색 방식들이 LLM의 "눈" 역할을 한다.

실전적 결론

소규모(~수십 파일): LLM이 직접 탐색해도 충분. 구조화 투자 ROI 낮음.
중규모(~수백 파일): index.md 같은 카탈로그 + 헤딩 기반 구조화면 충분.
대규모(~수천 파일 이상): BM25/임베딩 기반 검색 인프라 필요. 문서 구조화의 ROI가 급격히 올라감.

6. 터미널 멀티플렉서와 에이전트 환경

에이전트를 병렬로 실행하거나 장시간 세션을 유지할 때, 터미널 멀티플렉서가 필요해진다.

도구 플랫폼 비고
tmux Linux/macOS 사실상 표준. 에이전트 도구와 궁합 좋음
cmux macOS AI 에이전트 전용 네이티브 터미널
psmux Windows (PowerShell) tmux 대안으로 언급되나, 프로젝트 활성도 확인 필요

Windows 환경에서 Claude Code를 쓸 때는 bash 셸로 전환하는 것이 UNIX 계열 도구 호환성 면에서 유리하다. WSL2나 Git Bash가 대안.

7. 토큰 경제학: 소프트웨어의 재료비

AI 에이전트 시대에 소프트웨어 개발 비용은 두 가지 경로로 영향을 받는다:

경로 1 (전통): 유가 → 물가 → 인건비 → 개발 단가
경로 2 (신규): 유가 → 발전 비용 → IDC 전기료 → 토큰값 → 개발 재료비

코딩에 물리적 재료비(토큰 비용)가 직접 연결되기 시작했다는 관찰 자체는 맞다. 하지만 이것만으로 "개발 단가 폭등"을 예측하는 것은 추세의 한쪽만 보는 것이다.

비용을 낮추는 반대 방향의 힘

요인 효과
모델 경량화 같은 품질을 더 적은 파라미터로. 추론 비용 지속 하락
토큰 단가 하락 2024 → 2026 기준 주요 모델 토큰 가격 약 1/10 이하로 하락
하드웨어 효율 GPU/TPU 세대별 FLOPS/$ 개선. 에너지 효율도 동반 상승
캐싱/배칭 prompt caching, speculative decoding 등으로 실질 비용 감소
오픈소스 모델 로컬 추론 시 토큰 비용 = 전기료만. 클라우드 마진 제거

현재까지의 추세를 보면, 에너지 비용 상승분을 효율화가 상쇄하고도 남는 상태다. 토큰 단가는 장기적으로 0에 수렴하는 경향을 보이고 있다.

다만 이 추세가 영원히 지속된다는 보장은 없고, 에너지 가격이 소프트웨어 개발 비용에 직접 연결되는 경로가 생겼다는 구조적 변화 자체는 유의미하다. 과거에는 없던 비용 변수가 추가된 것이다.

실전적 의미

토큰 비용이 하락 추세라고 해서 무시해도 된다는 뜻은 아니다. 에이전트를 무제한으로 돌리면 비용은 여전히 크다. 토큰 효율적인 하네스/스킬 설계는 비용 최적화이자 동시에 품질 최적화다 — 불필요한 컨텍스트가 줄면 에이전트의 정확도도 올라간다.
