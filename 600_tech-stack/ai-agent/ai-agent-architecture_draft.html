AI Agent Architecture_draft

AI Agent Architecture

A comprehensive architecture for multi-agent systems, derived from production patterns observed in a hybrid-stack monorepo (Electron/React/FastAPI/SQLite/Firebase) that manages kiosk devices, image processing pipelines, CRM workflows, and offline-resilient operations. Governance and technical debt management are treated as first-class architectural concerns, not afterthoughts.

Evidence basis: 27 design documents, implementation plans, execution logs, and schema notes under /docs/plans/ in the Mirrorless_Mono repository, cross-referenced with current guidance from OpenAI, Microsoft, Google ADK, AWS, NIST, Atlassian, Thoughtworks, and Microsoft Well-Architected.

1. Agent Types and Roles

1.1 Classification Framework

Agents are classified along two axes: autonomy level (how much human oversight they require) and domain scope (how broad their operational boundary is).

| Agent Type | Autonomy | Scope | Examples |

|------------|----------|-------|----------|

| Lifecycle Agent | Fully autonomous | Narrow | UpdateService, HealthCheckService, RecoveryService |

| Pipeline Agent | Fully autonomous | Medium | ImageProcessingService, PhotoSyncService, UploadTaskService |

| Coordinator Agent | Semi-autonomous | Broad | ServiceRegistry, AuthManager, SessionLifecycleManager |

| Gateway Agent | Reactive | Narrow | TokenInterceptor, NetworkStatusService, IpcManager |

| Operator Agent | Human-triggered | Broad | TouchControlDrawer actions, Admin UI, manual update triggers |

1.2 Specific Agent Roles

Lifecycle Agents own a bounded state machine and run without intervention. They initialize, perform their work on a schedule or in response to events, and shut down cleanly. The UpdateService exemplifies this: it cycles through idle -> checking -> available -> downloading -> downloaded -> installing -> idle, with scheduled checks every 6 hours and installation gated behind a maintenance window (02:00-05:00). Lifecycle agents must be idempotent -- crashes at any state transition must recover to a consistent state on restart.

Pipeline Agents process work items through a sequence of transformations. The image processing pipeline demonstrates the canonical pattern: capture -> thumbnail generation -> face detection -> blur compositing -> local storage -> Firebase upload -> backend PATCH sync. Each stage is independently retriable. A pipeline agent never blocks the critical path -- if face detection fails, the pipeline continues without blur images. If thumbnail generation fails, the original buffer is used as fallback. This staged media pipeline -- local capture, local processing, queued upload, backend patch sync -- decouples capture and derivative generation from network completion, with sync logic mapping uploaded artifacts back onto backend schema fields later. Evidence: sharp-thumbnail-design.md, face-blur-image-design.md.

Coordinator Agents manage the lifecycle of other agents and mediate access to shared resources. The ServiceRegistry is the central coordinator: it initializes all services in dependency order (initialize() first, then startBackgroundWorkers() after recovery completes), holds singleton references, and manages shutdown via dispose handlers. AuthManager coordinates two sub-stores (DeviceTokenStore and UserSessionAuth) and arbitrates token refresh with single-flight deduplication.

Gateway Agents sit at system boundaries and translate between domains. TokenInterceptor wraps every outbound HTTP request: it checks network status, attaches device and user tokens, handles 401 responses with automatic refresh, and routes offline errors to callers. NetworkStatusService uses hybrid detection (OS-level events + health pings + consecutive failure counting) to maintain a three-state model: ONLINE | OFFLINE | DEGRADED.

Operator Agents execute user-initiated commands with appropriate safeguards. The TouchControlDrawer maps touch gestures to system actions, with session completion protected by a 1.5-second long-press requirement to prevent accidental termination.

1.3 The Registry Pattern

Every agent registers as a singleton through a central ServiceRegistry. This is a deliberate architectural choice over dependency injection containers.

Rationale:

  • Singletons are appropriate when agents manage exclusive hardware resources (cameras, SQLite database) or represent global state (auth tokens, network status).

  • The registry provides explicit initialization ordering, which is critical when agents have startup dependencies.

  • Dispose handlers registered during initialization ensure deterministic shutdown.

Decision: Use a singleton registry with explicit lifecycle phases, not a DI container. DI containers obscure initialization order and make it difficult to reason about startup failures. In agent systems where boot sequence matters (recovery must complete before background workers start), explicit ordering is worth the verbosity.

Known risk: The ServiceRegistry is becoming a hotspot. Centralized discovery simplifies wiring, but it concentrates lifecycle coupling, ordering concerns, and test complexity. Mitigation: treat the registry as infrastructure, not business logic. Agent initialization should be self-contained; the registry only controls sequencing and provides lookup. Evidence: crash-recovery-hardening identified that ordering bugs in the registry caused background workers to race with recovery logic.

2.2 Architecture Pattern Comparison

The following table compares candidate orchestration patterns against external guidance (OpenAI, Microsoft, Google ADK, AWS) and the Mirrorless plan corpus.

| Pattern | What it is | Best fit | Main upside | Main risk | Match to Mirrorless plans |

|---------|-----------|----------|-------------|-----------|---------------------------|

| Central manager / orchestrator | One agent controls workflow and delegates to specialist agents as tools | Early-stage systems, strong control needs, single user-facing entrypoint | Clear control, easier policy enforcement, easier auditability | Bottlenecking and over-centralization | Strong match -- ServiceRegistry, IpcManager, singleton services |

| Sequential pipeline | Fixed ordered stages; each step builds on the last | Deterministic workflows, maker-checker flows, staged refinement | Predictable behavior and easier testing | Slow if stages are independent; upstream failures cascade | Strong match -- most plans use explicit phases with verification gates |

| Event-driven / choreography | Components react to events and emit new events with little or no central coordinator | Long-running jobs, async background work, decoupled agents | Loose coupling, scalability, resilience | Lower end-to-end visibility without strong observability | Partial match -- TaskQueue, circuit breakers, online/offline events exist as infrastructure patterns |

| Concurrent fan-out/fan-in | Multiple agents analyze or execute in parallel, then aggregate | Independent analysis, scoring, review, latency-sensitive tasks | Better coverage and speed | Conflict resolution, cost spikes, shared-state hazards | Partial match -- thumbnail and face-detect are independent; no formal fan-out yet |

| Decentralized handoff | Peer agents transfer control between each other | Triage, specialist routing, conversational support | Flexible specialization, lower manager overhead | Harder governance, harder state and policy consistency | Weak match -- plans favor explicit ordering, registration, clear ownership |

| Microkernel / plugin host | Stable kernel exposes extension points; plugins implement capabilities | Platforms with many optional capabilities or domain plugins | Extensibility, clear boundaries, reuse | Versioning, plugin compatibility, policy drift | Emerging -- registry and handler registration exist, but no formal capability contracts or lifecycle hooks |

Decision: Central orchestrator plus event-driven workers. This matches the existing codebase patterns and external guidance (OpenAI's manager pattern, Microsoft's sequential orchestration, Google's coordinator/dispatcher, AWS's orchestrated workflow model). Decentralized handoff is rejected as a first architecture because the plans show no appetite for autonomous peer-to-peer coordination and explicitly favor ordered workflows with formal recovery paths.

Concretely:

  • Keep one top-level coordinator agent responsible for planning, routing, synthesis, and user interaction.

  • Implement specialist agents as tools or registered capabilities, not as free-roaming peers.

  • Use sequential pipelines for deterministic work: planning, code generation, testing, review, deployment preparation.

  • Use concurrent fan-out/fan-in only where work is truly independent: research, static analysis, alternative solution generation, or multi-perspective review.

  • Use event-driven workers for long-running or resumable tasks: repository indexing, background evaluation, replay, telemetry aggregation, or verification jobs.

3. Communication Patterns

3.1 Primary Patterns

The architecture uses four distinct communication patterns, chosen based on the coupling requirements between agents.

Pattern A: Direct Method Call (Synchronous, Tightly Coupled)

Used between agents that share a process boundary and have strong ordering requirements.


ServiceRegistry.initialize()

-> RecoveryService.runStartupRecovery()

-> HealthCheckService.runBootCheck()

-> ServiceRegistry.startBackgroundWorkers()

When to use: Startup sequences, transactional operations, synchronous validation chains. The crash recovery hardening design explicitly splits initialization into two phases to ensure recovery runs before any background workers that might compete for the same resources.

Pattern B: Event Emission (Asynchronous, Loosely Coupled)

Used when multiple consumers need to react to a state change without the producer knowing about them.


NetworkStatusService emits 'online' | 'offline' | 'degraded'

-> AuthManager pauses/resumes token refresh

-> BackendSyncService enables/disables circuit breaker

-> UploadTaskService triggers immediate queue processing

-> UI renders NetworkIndicator

The UpdateService extends EventEmitter to broadcast download-progress and state-change events. The IpcManager forwards these events to all renderer windows without the UpdateService knowing about the UI layer.

When to use: State changes that affect multiple independent subsystems. Network status is the canonical example -- every agent that makes HTTP requests needs to know about connectivity, but NetworkStatusService should not know about any of them.

Pattern C: IPC Channel (Cross-Process, Schema-Typed)

Used for Electron main-renderer communication. Every IPC channel is defined in a typed enum (IpcChannel) with corresponding request/response types. This is pattern #1 from the architecture inventory: stateful capabilities are pushed into main-process services while the renderer stays focused on UI, hooks, and stores.


Renderer -> IPC invoke -> Main Process Handler -> Service -> Response

The pattern enforces a handler abstraction (IpcInvokeHandler) where each handler class maps multiple related channels to a single service. The UpdateHandler handles Update_getStatus, Update_checkNow, Update_installNow, and Update_getHistory through a switch on the channel parameter.

For push notifications (main -> renderer), the pattern uses webContents.send() through a forwarding layer:


UpdateService.emit('state-change')

-> IpcManager listener

-> BrowserWindow.getAllWindows().forEach(win => win.webContents.send(channel, data))

When to use: Any communication that crosses process boundaries. Always use typed channels -- untyped IPC is a source of silent failures.

Pattern D: Task Queue (Asynchronous, Durable, Retriable)

Used for operations that must survive process crashes and be retried on failure.


Interface:

enqueue(task) -> taskId

process(handler) -> void

retry(taskId) -> void

getStatus(taskId) -> TaskStatus

  

States: pending -> processing -> done | failed

The current implementation uses SQLite as the queue backend (SQLiteTaskQueue), with an abstract TaskQueue<T> interface that allows future migration to Redis or Kafka. Upload tasks are the primary consumer: each task stores local file paths, and the upload handler reads files, uploads to Firebase, and updates the task record with download URLs.

Decision: Use SQLite for task queues, not an in-memory queue. In-memory queues lose tasks on crash. The kiosk offline architecture requires that photos captured during network outages survive restarts and are uploaded when connectivity returns. SQLite provides crash-safe persistence with zero operational overhead.

3.2 Communication Anti-Patterns to Avoid

Bidirectional dependencies between agents. If Agent A calls Agent B and Agent B calls Agent A, extract the shared concern into a third agent or use events. The AuthManager/NetworkStatusService relationship is unidirectional: NetworkStatusService emits events, AuthManager subscribes. AuthManager never calls back into NetworkStatusService.

Synchronous cross-network calls in event handlers. When NetworkStatusService detects ONLINE, it emits an event. Subscribers must not make blocking network calls in the event handler -- they should schedule work (e.g., trigger a queue processing cycle) instead.

Implicit ordering through event timing. If Agent A must complete before Agent B starts, use explicit initialization ordering (Pattern A), not event ordering. Events do not guarantee delivery order across subscribers.

4. Orchestration Strategy

4.1 Centralized Orchestration with Decentralized Execution

Decision: Centralized startup orchestration, decentralized runtime execution.

At startup, a single orchestrator (MirrorlessApplication) controls the exact sequence:


1. Database initialization (migrations run)

2. ServiceRegistry.initialize() -- all services get their dependencies

3. RecoveryService.runStartupRecovery() -- fix any crash artifacts

4. HealthCheckService.runBootCheck() -- verify post-update integrity

5. If health check fails -> RollbackManager.rollback() -> exit

6. ServiceRegistry.startBackgroundWorkers() -- timers, queues, watchers

7. IpcManager.registerHandlers() -- enable renderer communication

8. Window creation -> renderer loads -> app ready

This strict ordering solves real problems observed in the codebase:

  • Recovery must complete before background workers start, otherwise upload workers might process orphaned tasks that recovery would have cleaned up.

  • Health checks must run before IPC handlers are registered, because a failed health check triggers rollback and process exit.

  • Database migrations must complete before any service accesses the database.

At runtime, agents operate independently. The upload queue processes tasks on its own schedule. NetworkStatusService polls health endpoints independently. The UpdateService checks for updates on its own timer. No central coordinator orchestrates runtime behavior -- agents react to events and process their own work.

4.2 The Two-Phase Initialization Pattern

Every agent that has both setup work and ongoing background work must split these into two methods:


interface LifecycleAgent {

initialize(): Promise<void>; // Phase 1: setup, no side effects

startBackgroundWorkers(): void; // Phase 2: timers, watchers, queues

stop(): void; // Shutdown: cancel timers, flush queues

}

Phase 1 (initialize) prepares state: reads configuration, ensures database tables exist, validates credentials. It must be safe to call and then abandon -- if a later agent's initialization fails, Phase 1 work should not leave orphaned resources.

Phase 2 (startBackgroundWorkers) starts timers, queue processors, and file watchers. It runs only after all agents have completed Phase 1 and recovery has finished.

This pattern is derived from the crash recovery hardening design, which identified that the original single-phase initialize() allowed background workers to race with recovery logic.

4.3 State Machine Discipline

Every agent with non-trivial behavior must define its states explicitly. The UpdateService state machine is the model:


idle -> checking -> available -> downloading -> downloaded -> installing -> idle

-> not-available -> idle

-> error -> idle (retry scheduled)

Rules:

  • State transitions must be logged with before/after values.

  • Invalid transitions must throw, not silently no-op.

  • External callers query state but never set it directly.

  • State persists to SQLite when it must survive restarts (e.g., pending_update flag).

The network status agent uses a three-state model (ONLINE | OFFLINE | DEGRADED) with hybrid detection to avoid flapping: three consecutive HTTP failures trigger OFFLINE, but a single success triggers ONLINE. This asymmetric threshold prevents rapid state oscillation on unstable networks.

Known risk: Auth and security state machines are growing more complex. Device tokens, kiosk-scoped user tokens, offline license snapshots, grace periods, refresh flows, and recovery paths improve resilience but expand the number of edge states the system must handle correctly. Mitigation: maintain explicit state diagrams per agent; review state explosion during architecture reviews.

4.4 Feature Flag Gating

New agent capabilities should be gated behind a feature flag system. The experimental feature flag architecture provides the pattern:


// Registry: single source of truth for all flags

const EXPERIMENTAL_FLAGS = {

beforePhotosOnMain: {

label: 'Before photos on main screen',

defaultEnabled: false,

relatedIssue: 'MIR-KSK-294',

},

} as const;

  

// Store: runtime overrides persisted to localStorage

// overrides[key] ?? EXPERIMENTAL_FLAGS[key].defaultEnabled

  

// Hook: one-liner consumption

const isEnabled = useExperimentalFlag('beforePhotosOnMain');

Decision: Registry-based flags with override persistence, not server-side flags. For offline-capable systems (kiosks, mobile, edge devices), feature flags must work without network access. The registry pattern with localStorage persistence provides instant reads, offline support, and a single file to modify when adding new flags.

Governance gap: The flag registry has no documented graduation, sunset, approval, or telemetry policy for deciding whether a flag stays experimental, becomes default, or is removed. See Section 7 (Governance) for the recommended experiment lifecycle.

5. Fault Tolerance

5.1 Defensive Recovery Hierarchy

Fault tolerance operates at four levels, from least to most disruptive:

Level 0: Graceful Degradation (no recovery needed)

Operations that can fail without affecting the system. Thumbnail generation falls back to the original buffer. Face detection returns an empty array. Feature flags default to their registry value if localStorage is corrupted.


try {

thumbnailBuffer = await imageProcessingService.generateThumbnail(buffer);

} catch (error) {

log.warn('Thumbnail generation failed, using original', { error });

thumbnailBuffer = buffer; // graceful fallback

}

Rule: Non-critical processing must never block the critical path. Image processing failures must not prevent photo capture. Upload failures must not prevent photo storage. Blur generation failures must not prevent thumbnail generation.

Level 1: Retry with Backoff

For transient failures (network timeouts, temporary server errors). The TaskQueue abstraction encodes retry policy in the handler:


interface TaskHandler<T> {

execute(task: T): Promise<void>;

shouldRetry(error: Error, retryCount: number): boolean;

getBackoff(retryCount: number): number;

}

Upload tasks use exponential backoff. Token refresh uses single-flight deduplication with a 30-second grace period -- the expiring token remains valid for 30 seconds after its nominal expiry, giving the refresh request time to complete without failing in-flight requests.

Level 2: State Recovery on Restart

For crash-induced inconsistencies. The RecoveryService runs at startup and fixes:

  • Sessions with status='done' but zero upload tasks (outbox pattern -- DB committed but uploads never enqueued).

  • Stale upload locks (reduced from 10 minutes to 60 seconds -- 2x the handler timeout).

  • Archive consistency for all non-expired sessions.


Recovery sequence:

1. Find sessions where DB committed but uploads missing

2. Re-invoke enqueueSessionHairHistoryUploads() (idempotent)

3. Release stale upload locks

4. Verify archive consistency

Atomic pointer writes protect against partial-write corruption: write to temp file, then fs.rename() (atomic on POSIX). On recovery, check for .tmp fallback files.

Known risk: Recovery logic is compensating for non-atomic workflows. Multiple design docs add requeueing, temp-file fallbacks, rollback managers, cleanup registries, and retry guards, which suggests core commit/enqueue/update flows are still fragile under crash or partial failure. This is a recognized tech debt area -- see Section 8 for prioritization.

Level 3: Version Rollback

For update-induced failures. The HealthCheckService runs a 30-second boot verification after every update:


Boot check:

1. Verify SQLite connectivity

2. Verify core service initialization

3. Verify renderer heartbeat (IPC round-trip)

-> All pass: clear pending_update flag, record "verified"

-> Any fail: RollbackManager.rollback()

Rollback extracts the previous version from backup, records the failure, adds the broken version to a blocklist, and relaunches. After 3 consecutive rollback failures, the system locks on the current version and alerts operators.

5.2 Circuit Breaker Pattern

The circuit breaker prevents wasted resources during outages:


ONLINE state:

-> All network operations proceed normally

-> Sync/upload loops run on schedule

  

OFFLINE state (3 consecutive failures):

-> Circuit breaker ON

-> Sync/upload loops skip immediately (no 30-second timeout waste)

-> Tasks enqueue locally in SQLite

-> Token refresh pauses

  

ONLINE recovery (any single success):

-> Circuit breaker OFF

-> Pending queue processes immediately

-> Token refresh resumes

Decision: Use consecutive-failure counting, not time-based circuit breaking. Time-based breakers (e.g., "stay open for 60 seconds") waste time when the network recovers quickly and miss extended outages. Failure counting adapts to actual conditions.

5.3 Offline-First Data Architecture

For systems that must operate without network access:

  1. Signed snapshots for validation data. License status is cached as a signed JWT. The kiosk verifies the signature locally and applies a 7-day grace period after expiry. This eliminates the need for network access to validate business rules.

  2. Profile caching with TTL. User profiles are cached in SQLite with a 7-day TTL. During offline operation, cached profiles display in the UI. Stale entries are cleaned automatically.

  3. Auth-aware queue. The task queue checks NetworkStatusService.isOnline before processing. Offline tasks accumulate in SQLite and drain automatically on reconnection.

  4. Hybrid state persistence. QR login state uses Zustand persist (localStorage) for instant UI hydration and SQLite for secure token storage. On restart, an IPC validation reconciles the two: if localStorage says "logged in" but SQLite has no active session, localStorage is cleared.

5.4 Idempotency Requirements

Every operation that might be retried must be idempotent:

  • Upload task re-enqueue checks for existing tasks before creating duplicates.

  • Token refresh uses single-flight (one in-flight refresh promise shared by all callers).

  • Database migrations use CREATE TABLE IF NOT EXISTS and ALTER TABLE ADD COLUMN (which fail gracefully if the column exists).

  • Photo sync PATCH operations use field-level partial updates (only non-null fields are written).

6. Scalability

6.1 Horizontal Scaling: Package-per-Workflow

The CI/CD architecture provides the scaling model for the agent system itself. A monolithic CI workflow was replaced with package-per-workflow:


PR opened

+-- packages/admin/** changed -> ci-frontend.yml: lint -> typecheck -> build

+-- packages/mirrorless/** changed -> ci-mirrorless.yml: lint -> typecheck -> test -> e2e

+-- packages/mirrorless-backend/** changed -> ci-backend.yml: ruff lint -> pytest

  

Merge to dev

+-- packages/mirrorless-backend/** changed -> deploy-backend.yml: ci-backend -> deploy -> health check

The same principle applies to agent architecture: agents should be independently deployable and testable. The workflow_call mechanism (backend CI is callable from deploy workflow) maps directly to agent composition -- a coordinator agent can invoke a lifecycle agent as a sub-step.

6.2 Queue-Based Scaling

The TaskQueue<T> abstraction is the primary scaling mechanism. The current implementation uses SQLite, which handles the throughput of a single kiosk device. The interface is designed for drop-in replacement:


// Current: SQLiteTaskQueue (single device)

// Future: RedisTaskQueue (multi-device, shared state)

// Future: KafkaProducer (high-throughput, event streaming)

Decision: Abstract the queue interface now, implement SQLite, defer broker migration. Premature optimization toward Kafka or Redis adds operational complexity with no current benefit. The abstraction costs nothing and preserves the migration path.

6.3 Processing Pipeline Scaling

The image processing pipeline demonstrates the scaling pattern for compute-intensive agents:


Sequential (current, single device):

capture -> thumbnail(Sharp) -> face-detect(face-api.js) -> blur(Sharp) -> store -> upload

  

Parallel (future, multi-device or server-side):

capture -> [thumbnail, face-detect] in parallel -> blur (depends on detect) -> store -> upload

Face detection and thumbnail generation are independent operations that can run in parallel. The current implementation runs them sequentially because single-device throughput is sufficient, but the pipeline's stage-based architecture supports parallelization without structural changes.

6.4 State Partitioning

State is partitioned by concern and persistence requirement:

| State | Storage | Scope | Rationale |

|-------|---------|-------|-----------|

| Auth tokens (user) | Memory only | Session | Evaporates on crash, no disk leakage |

| Auth tokens (device) | SQLite + safeStorage encryption | Permanent | Survives restarts, encrypted at rest |

| UI preferences | localStorage (Zustand persist) | Permanent | Instant hydration, no IPC overhead |

| Task queue | SQLite | Permanent | Crash-safe, queryable, retriable |

| Update state | SQLite | Permanent | Survives restarts, tracks history |

| Cache (profiles, licenses) | SQLite with TTL | Ephemeral | Automatic cleanup, offline fallback |

| Pipeline state (WebGL, LUT) | Module-level Map | Process | Fast synchronous access, rebuilt on restart |

Decision: Memory for secrets, SQLite for durability, localStorage for UI speed. This three-tier storage model matches the access patterns: secrets must never touch disk unnecessarily, durable state must survive crashes, and UI state must load without async IPC round-trips.

6.5 Scaling Constraints and Escape Hatches

The architecture is designed for single-device kiosk operation. When scaling beyond that:

  1. Multi-device coordination requires replacing SQLite task queues with a shared broker (Redis/Kafka) and adding device-to-device discovery.

  2. Server-side processing requires extracting ImageProcessingService into a microservice with a job queue (the generateBlurredImage method already returns a BlurResult | null that maps cleanly to an async job result).

  3. Multi-tenant isolation requires adding tenant scoping to all queue operations and storage paths (the current shopId-prefixed Firebase paths provide the foundation).

  4. Rate limiting requires adding per-device quotas to the API gateway (the current device_and_user auth level provides the identity basis).

None of these migrations require architectural changes to the agent model. The singleton registry, event-driven communication, queue abstraction, and two-phase initialization all transfer directly to distributed deployments. The abstractions were chosen specifically to avoid premature distribution while preserving the path to it.

7. Governance

7.1 Governance Model

Decision: Centralized governance now, designed to evolve into federated governance later.

Centralized governance is the right starting point because:

  • One team owns the platform.

  • Domains are still small.

  • Governance maturity is low.

  • The cost of bad autonomous behavior is high (unattended kiosks).

  • The plan corpus already assumes centralized service registration, shared recovery logic, common verification practices, and approval-style design docs.

Federated governance (where domain teams own domain-specific agents while a central platform team owns shared infrastructure) should be adopted only when multiple domain teams actually need to own separate agent portfolios. Evidence: AWS describes both centralized and federated operating models, recommending centralized for early-stage systems and federated for multi-domain organizations with mature shared components. NIST AI RMF defines governance as a cross-cutting function infused throughout risk management, not a one-time checkpoint.

Alternative rejected: Distributed governance with local autonomy. Too loose for this project. The plans show no appetite for fully autonomous agent clusters. Risks: inconsistent policies, duplicated capabilities, weak cross-system accountability, and difficult incident reconstruction.

7.2 Review Tracks

Work entering the system follows a single intake with split review tracks, reflecting three distinct categories observed in the plan corpus:

| Track | When to use | Primary criteria | Reviewer profile |

|-------|------------|-----------------|------------------|

| Feature / governance | Main question is "should we do this and is the user-facing behavior right?" | User value, UX consistency, scope discipline, API/schema compatibility | Product/design owner, feature owner or tech lead |

| Reliability / tech debt | Main question is "does this reduce risk, remove a defect, or improve operational correctness?" | Root cause clarity, regression prevention, rollback posture, verification strength, blast radius | System owner, infra/reliability owner |

| Hybrid architecture | Initiative changes product behavior and system reliability at the same time | Must pass both feature and reliability checks; requires sign-off from both perspectives | Tech lead/architect, product rep, reliability/platform owner |

Evidence for three tracks: The plan corpus contains clear feature-heavy work (header-profile-badge, retake-warning-popup, touch-control-drawer), clear reliability work (crm-fastapi-live-regression, crash-recovery-hardening, cicd-redesign, lut-pipeline-race-condition), and hybrid architecture work (kiosk-auth-offline, qr-login-persistence, auto-update-architecture). A single review track under-specifies evidence for debt work or over-burdens routine feature work. A full split creates routing churn for hybrid items.

Every initiative must declare routing metadata:


type: feature | tech-debt | hybrid

trigger: roadmap | user-feedback | incident | code-review | operational-friction

owner: <team/person>

reviewers: <roles>

verification: smoke | test-suite | audit-log | rollback-check | mixed

7.3 Agent Capability Registry

Extend the existing SSOT registry pattern (already used for feature flags) to agents:

| Field | Purpose |

|-------|---------|

| capability_name | Unique identifier |

| purpose | One-line description of what the agent does |

| tools_data_access | What tools, APIs, or data stores the agent can access |

| owner | Accountable individual or team |

| approval_level | What level of review is required to modify this agent |

| required_evals | What evaluations must pass before deployment |

| rollback_behavior | How to undo if the agent causes harm |

| last_review_date | When this capability was last reviewed |

This is the missing bridge between the existing ServiceRegistry mindset and a real agent platform.

7.4 Architecture Decision Records (ADRs)

The plan corpus already functions like design memos, but the project needs tighter, version-controlled decision records for:

  • Orchestration model choices

  • Agent boundary definitions

  • Policy boundary definitions

  • Approval thresholds

  • Event schema choices

  • Memory/state retention choices

ADRs should be stored in /docs/decisions/ and referenced from the relevant design docs. Evidence: Microsoft Well-Architected recommends version-controlled ADRs and treating technical debt as a regularly recurring backlog item.

7.5 RACI Matrix

Define accountability for the full agent lifecycle:

| Responsibility | R (does it) | A (owns it) | C (input) | I (updates) |

|---------------|-------------|-------------|-----------|-------------|

| Agent spec/prompt ownership | Feature owner | Tech lead | Product, design | Team |

| Tool access approval | Tech lead | Architect | Security | Team |

| Model/config changes | Engineer | Tech lead | Architect | Team |

| Production deployment | Engineer | Tech lead | QA | Stakeholders |

| Evaluation sign-off | QA/test engineer | Tech lead | Feature owner | Team |

| Incident response | On-call engineer | System owner | Architect | Stakeholders |

| Rollback execution | On-call engineer | System owner | Tech lead | Stakeholders |

| Tech debt remediation | Assigned engineer | Tech lead | Architect | Team |

Without this, a multi-agent system will quickly create gray zones. Evidence: Atlassian guidance recommends a single accountable owner per task.

7.6 Human-in-the-Loop Gates

High-risk actions require explicit human approval before execution:

  • Destructive filesystem actions

  • Schema changes (backend migrations, local migrations, prepared statement changes)

  • Production deploys

  • Secrets access

  • Irreversible migrations

  • Cross-domain writes

Evidence: Google ADK's Human in the Loop with PolicyEngine and Microsoft's HITL guidance both recommend policy-gated approval for sensitive operations.

7.7 Observability Requirements

Every agent interaction must be logged with correlation IDs:

  • Every delegation from coordinator to specialist

  • Every tool call and its result

  • Every event emission and consumption

  • Every approval request and response

  • Every retry, circuit-breaker activation, and rollback

  • Every final decision and its rationale

This extends the existing discipline of execution logs and verification commands (strongest in the CRM regression work) to agent runtime. It is especially important when event-driven or concurrent behavior is introduced, as end-to-end visibility drops without explicit instrumentation.

7.8 Architecture and Debt Review Cadence

Run a weekly or biweekly review with a narrow charter:

  1. Approve or reject major agent capability additions.

  2. Review failed runs and policy exceptions.

  3. Review new debt introduced by shortcuts.

  4. Decide whether debt stays in team backlogs or escalates.

This is more practical than a heavyweight architecture review board and aligns with the project's existing bias toward explicit reviews, rollback planning, and execution logs. Evidence: AWS recommends an ARB with executive sponsorship and a single source of truth for guidance; Thoughtworks emphasizes that debt includes architectural, technical decision, test, documentation, infrastructure, process, and security debt.

7.9 Experiment Lifecycle

Feature flags must follow a defined lifecycle to prevent flag accumulation:


proposed -> approved -> enabled (experiment) -> graduated (default) -> removed

-> sunset (disabled) -> removed

Each flag must have:

  • An expiry date or review trigger

  • A graduation criterion (what data proves the flag should become default)

  • A sunset criterion (what data proves the flag should be removed)

  • Telemetry requirements (what must be measured while the flag is active)

8. Technical Debt

8.1 Known Debt Areas

Eight areas of technical debt were identified from the plan corpus. Each is evidenced by multiple design documents and has a concrete mitigation path.

1. Fragmented persistence and reconciliation logic

Session/auth/config state is spread across memory, SQLite, encrypted preferences, and localStorage, creating dual-write and restore/validate complexity. Evidence: kiosk-auth-offline, qr-login-persistence, experimental-feature-flags.

Mitigation: Standardize on the three-tier model (memory for secrets, SQLite for durability, localStorage for UI speed) and eliminate ad-hoc reconciliation by using a single source of truth per data category with one-way sync.

2. Recovery logic compensating for non-atomic workflows

Multiple docs add requeueing, temp-file fallbacks, rollback managers, cleanup registries, and retry guards. This indicates core commit/enqueue/update flows are still fragile under crash or partial failure. Evidence: crash-recovery-hardening, kiosk-auth-offline, auto-update-architecture, crm-fastapi-live-regression.

Mitigation: Make the primary write paths atomic (write-then-rename, transactional enqueue) so recovery logic handles only genuine crash-during-commit scenarios, not normal-path fragility.

3. High cross-layer change tax

Single features routinely require coordinated updates across backend schema, Electron DB, IPC types, preload, services, hooks, stores, and UI. Evidence: face-blur-image, auto-update-architecture, qr-login-persistence, header-profile-badge.

Mitigation: Introduce code generation for IPC type definitions and database schema mappings. Reduce the number of manual touchpoints per feature.

4. Schema and mapping growth

Features like face blur add backend migrations, local migrations, prepared statement changes, queue payload changes, and multi-field sync maps, increasing drift risk and maintenance cost.

Mitigation: Adopt a schema-first design process where the schema change is reviewed and approved before implementation begins. Track schema-to-code mapping in a single manifest.

5. ServiceRegistry as a hotspot

Centralized discovery simplifies wiring but concentrates lifecycle coupling, ordering concerns, and test complexity.

Mitigation: Keep the registry thin -- it should only sequence initialization and provide lookup. Move business logic out of registry callbacks. Test agents in isolation by mocking the registry interface.

6. Retrofitted verification

CI/CD redesign explicitly fixed missing backend CI and silent failures; many plans still rely on manual smoke checks or visual verification for final confidence. Evidence: cicd-redesign, multiple plans with "verify by manual check" as the only gate.

Mitigation: Add automated verification for every agent capability. At minimum: unit tests for state machines, integration tests for queue processing, and health-check tests for startup sequences.

7. Tactical workarounds around async/lifecycle fragility

LUT caching, Sharp fallback-to-original behavior, and recovery hardening are pragmatic but indicate accumulated complexity rather than structural simplification. Evidence: lut-pipeline-race-condition, sharp-thumbnail, crash-recovery-hardening.

Mitigation: Schedule periodic simplification reviews. When three or more workarounds accumulate around the same subsystem, treat it as a structural refactoring candidate.

8. Growing auth/security state machine complexity

Device tokens, kiosk-scoped user tokens, offline license snapshots, grace periods, refresh flows, and recovery paths expand the number of edge states the system must handle. Evidence: kiosk-auth-offline, qr-login-persistence.

Mitigation: Maintain explicit state diagrams for every auth flow. Add property-based tests that exercise random state transitions. Review state explosion risk during architecture reviews.

8.2 Debt Prioritization

Debt items should be prioritized by blast radius and frequency of impact:

| Priority | Debt area | Rationale |

|----------|-----------|-----------|

| P1 | Non-atomic workflows (#2) | Root cause of recovery complexity; fixing this reduces debt in areas #1 and #7 |

| P1 | Retrofitted verification (#6) | Silent failures compound all other debt; automated gates prevent regression |

| P2 | Fragmented persistence (#1) | Reconciliation bugs are a recurring source of user-visible defects |

| P2 | Auth state complexity (#8) | Security-sensitive; edge-state bugs have high blast radius |

| P3 | Cross-layer change tax (#3) | Slows delivery but does not cause runtime failures |

| P3 | Schema growth (#4) | Maintenance cost, not runtime risk |

| P3 | ServiceRegistry hotspot (#5) | Test friction, not production risk |

| P3 | Async workarounds (#7) | Localized; address during subsystem refactoring |

9. Summary of Key Architectural Decisions

| Decision | Choice | Alternative Rejected | Rationale |

|----------|--------|---------------------|-----------|

| Orchestration model | Central orchestrator + event-driven workers | Decentralized handoff, fully autonomous swarm | Matches existing patterns; endorsed by OpenAI, Microsoft, Google ADK, AWS guidance |

| Agent lifecycle | Singleton registry with two-phase init | DI container | Explicit ordering, debuggable startup |

| Runtime coordination | Event-driven, decentralized | Central message bus | Simpler, no single point of failure |

| Task durability | SQLite queue with abstract interface | In-memory queue, Redis | Crash-safe, zero ops overhead, migration path preserved |

| Auth token storage | Memory (user) + encrypted SQLite (device) | All in SQLite | Minimize secret exposure surface |

| Network detection | Hybrid (OS event + health ping + failure count) | OS event only | OS events are unreliable for actual API reachability |

| Feature flags | Local registry + localStorage persist | Server-side flags | Offline-first requirement |

| Update rollback | Automatic with health check + blocklist | Manual rollback | Unattended kiosk operation |

| Error handling | Four-level hierarchy (degrade -> retry -> recover -> rollback) | Uniform retry | Different failure modes need different responses |

| CI/CD | Package-per-workflow with path filters | Monolithic CI | Independent validation, no cross-stack failures |

| Offline data | Signed snapshots + TTL cache + auth-aware queue | Fail-closed on offline | Kiosk must operate during network outages |

| Governance model | Centralized now, federated later | Distributed local autonomy | Small team, low governance maturity, high cost of bad autonomous behavior |

| Review process | Single intake, three review tracks | Unified single track; fully separate systems | Hybrid work is common, not exceptional; different triggers need different reviewers |

| HITL gates | Policy-gated approval for destructive/irreversible actions | Fully autonomous execution | Kiosk operates unattended; schema/deploy errors are expensive to fix remotely |

10. Implementation Path

The recommended sequence for adopting this architecture:

  1. Adopt central orchestrator model. Formalize the existing ServiceRegistry pattern as the agent coordination layer. Do not introduce decentralized peers.

  2. Create the capability registry. Extend the SSOT pattern from feature flags to all agent capabilities. Define owner, approval level, rollback behavior, and required evaluations for each.

  3. Add ADRs for agent decisions. Version-control all architectural choices in /docs/decisions/.

  4. Establish RACI for agent lifecycle. Define accountability for spec ownership, tool access, deployment, incident response, rollback, and debt remediation.

  5. Add HITL gates for high-risk actions. Schema changes, production deploys, secrets access, and irreversible migrations require explicit approval.

  6. Instrument with correlation IDs. Log every delegation, tool call, event, approval, retry, and decision.

  7. Introduce event-driven workers for background tasks. Use the existing TaskQueue and event-emission patterns for long-running or resumable work.

  8. Address P1 debt. Make primary write paths atomic; add automated verification gates.

  9. Establish review cadence. Weekly or biweekly review of capability additions, failed runs, policy exceptions, and new debt.

  10. Reassess federation only after multiple domains genuinely need autonomy.
