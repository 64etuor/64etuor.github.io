Decision Tree Architect - Claude Code Harness 해부

Decision Tree Architect: Claude Code Harness 해부

이 하네스가 하는 일

산발적인 프로젝트 의사결정 항목들 — 제약조건, 논의 메모, 아이디어 — 을 넣으면, 우선순위가 매겨진 의사결정 트리와 회의 안건이 나온다.

핵심 질문은 "뭐가 중요한가?"가 아니라 "뭘 먼저 결정해야 다음이 풀리는가?"다. 의존성 기반 우선순위 정렬. 팀 리드, PM, 창업자가 구조화된 회의를 준비할 때 쓸 수 있도록 설계됐다.

산발적 입력 → 정규화된 DAG → 우선순위 스코어링 → 트리 + 회의안건 → 감사 → 피드백 루프

아키텍처: 5명의 전문 에이전트 팀

하네스의 골격은 5개의 특화 에이전트가 파이프라인으로 연결된 구조다. 각 에이전트는 .claude/agents/ 하위에 마크다운 스펙으로 정의되어 있고, 입출력이 JSON 스키마로 엄격히 규정되어 있다.

에이전트 역할 입력 출력
input-normalizer 입력 분류 + DAG 구축 사용자 원문 (00_input.md) 01_canonical_graph.json
priority-ranker 5축 스코어링 + 버킷 분류 01_canonical_graph.json 02_ranked_decisions.json
tree-renderer DAG → 트리 프로젝션 02_ranked_decisions.json 03_tree_view.json
facilitator 회의 안건 + 타임박스 생성 02 + 03 04_facilitation_plan.json
auditor 스키마/원칙 위반 검증 01~04 전체 05_audit_report.json

여기에 2개의 스킬 (dag-construction, priority-scoring)이 input-normalizer와 priority-ranker에게 도메인 지식을 주입한다.

실행 파이프라인

에이전트들이 순서대로 실행되지만, 무조건 직렬은 아니다. 의존 관계가 없는 단계는 병렬 실행한다.

Task 1: input-normalizer → 01_canonical_graph.json
    ↓
★ CHECKPOINT ★  (unknowns/assumptions 있으면 사용자 확인, 없으면 자동 통과)
    ↓
Task 2: priority-ranker → 02_ranked_decisions.json
    ↓
Task 3a: tree-renderer ──┐
                         ├─ (병렬)
Task 3b: facilitator ────┘
    ↓
Task 4: auditor → 05_audit_report.json
    ↓
통합: DECISION_TREE.md + tree_viewer.html 생성

핵심 제약: Task 1과 2는 절대 동시에 실행하지 않는다. 체크포인트를 건너뛰면 검증되지 않은 가정 위에서 우선순위를 매기게 되기 때문이다.

핵심 개념: Canonical DAG

이 하네스의 단일 진실 원천(Single Source of Truth)은 01_canonical_graph.json에 담긴 Canonical DAG다. 트리 뷰, 회의 안건, 감사 보고서는 전부 이 DAG의 프로젝션(투영)이다.

DAG 구성 요소

  • Locked Decisions (L1, L2...): 이미 확정된 결정. 재논의 대상 아님.
  • Candidate Decisions (D1, D2...): 열린 결정. 각각 2개 이상의 선택지를 가짐.
  • Edges (4종):
    • constrains — 확정 결정이 후보 결정의 선택지를 제한
    • depends_on — 선행 결정 없이는 의미 없음
    • enables — A를 결정하면 B가 가능해짐
    • conflicts_with — A와 B는 공존 불가 (대칭)

13개 결정 축 (Axis)

각 결정 노드는 반드시 하나의 축에만 속한다 (원칙 5: One node = one axis).

goal_and_constraint, scope_and_sequencing, customer_and_market, product_and_service, architecture_and_structure, build_buy_partner, pricing_and_business_model, execution_and_rollout, measurement_and_validation, operating_model, organization_and_governance, risk_and_compliance, other

우선순위 스코어링: Gate + Score + Cap

단순 점수가 아니라 3단계 필터 구조다.

Stage 1: Hard Gate

  • Pruned — 확정 결정/제약조건에 의해 이미 소멸
  • Blocked — 선행 결정 미해결로 평가 불가

Gate를 통과한 항목만 스코어링 대상이 된다.

Stage 2: 5축 스코어링 (축당 0-2점, 최대 10점)

축 측정 대상
information_gain 이 결정이 다른 몇 개의 질문을 제거하는가?
irreversibility_risk 번복 난이도 × 영향 범위
dependency_enablement 몇 개의 차단된 결정을 해제하는가?
deadline_pressure 지금 결정 안 하면 일정이 깨지는가?
evidence_readiness 결정에 필요한 정보가 충분한가?

Stage 3: Meeting Cap

  • 기본값: 상위 3개가 priority 버킷
  • 보호 규칙: irreversibility_risk=2 또는 deadline_pressure=2인 항목은 점수 무관하게 priority에 포함 (최대 5개)
  • 타이브레이커: 4단계 결정론적 정렬 (irreversibility_risk → dependency_enablement → recommended_order → ID 오름차순)

같은 입력은 항상 같은 결과를 낸다. 타이브레이커 4단계까지 간 경우 경고 플래그가 남는다.

5개 버킷 체계

버킷 정의
priority 이번 회의에서 반드시 결정
later_active 곧 논의할 항목 (다음 라운드)
blocked 선행 결정 미해결
deferred 의도적 보류 (deferrable=true + evidence_readiness≤1 + deadline_pressure=0)
pruned 확정 결정/제약에 의해 완전 소멸

DAG → 트리 변환

DAG는 한 노드가 여러 부모를 가질 수 있지만, 사용자가 읽는 트리는 단일 부모 구조여야 한다. 변환 시 잃어버리는 의존 관계는 hidden_dependency_edges 배열에 기록된다.

부모 선택 전략: recommended_order가 가장 가까운 상위 노드, 동점이면 information_gain이 높은 쪽.

12개 핵심 원칙

이 하네스의 모든 에이전트가 따르는 원칙들이다. 원칙 간 충돌 시 우선순위가 있다: 원칙 4 > 11 > 12 > 1 > 나머지.

  1. 상위 결정 먼저 (중요도가 아니라 의존 순서)
  2. 번복 가능한 결정은 우선순위 낮춤
  3. 각 결정은 정확히 하나의 버킷에만
  4. 확인된 사실만으로 pruning (추정으로 제거 금지)
  5. 한 노드 = 한 축
  6. 결정과 태스크를 분리
  7. 내부 관계(DAG)와 출력 관계(트리)를 분리
  8. 선택지 효과는 options 안에서만 기록
  9. 추론 과정은 내부용 — 출력에는 근거만
  10. 불충분한 정보는 구조화해서 보존 (assumptions/unknowns/open_questions)
  11. 모순을 감지하고 기록 (임의 해결 금지)
  12. Canonical DAG가 진실 원천

원칙 4가 최상위인 이유: 추정 기반 pruning은 회의에서 논의해야 할 항목을 사전에 제거해버리는 가장 위험한 오류이기 때문이다.

학습 시스템

하네스는 오류로부터 배운다. learning-log.md에 개선 이력이 누적되고, prompts/learning-rules.md에 체계화된 오류 분류 → 근본 원인 진단 → 수정 위치 매핑이 정의되어 있다.

오류 유형 (E1~E7)

분류 오류, 구조 오류, 스코어링 오류, 조기 pruning, 누락, 렌더링 오류, 일관성 오류

근본 원인 → 수정 위치 매핑

근본 원인 수정 파일
규칙 누락 prompts/core-prompt.md
규칙 모호 prompts/core-prompt.md + 판정 기준 추가
규칙 위반 .claude/agents/*.md 가드레일
지식 부족 .claude/skills/*/skill.md
스키마 한계 schemas/*.json → 에이전트/스킬/프롬프트 전파
규칙 충돌 상위 문서 (core-prompt > rules > agents)
전파 누락 중복 규칙 문서 동기화

출력물

한 라운드가 끝나면 _workspace/{날짜}_{주제}/round-N/ 디렉토리에 다음이 생성된다:

파일 소비자 용도
DECISION_TREE.md 사람 마크다운 트리 + 안건 + 상세
tree_viewer.html 사람 인터랙티브 시각화
01~05 JSON 파일 에이전트/스크립트 파이프라인 중간 산출물

피드백 라운드를 돌리면 회의 결과를 반영해서 DAG를 업데이트하고, 다음 라운드의 우선순위 결정을 자동 생성한다.

검증 레이어

품질 보증이 중첩으로 깔려 있다.

  1. JSON 스키마 검증 — 모든 출력이 schemas/*.json에 대해 검증
  2. ID 참조 무결성 — 존재하지 않는 노드 참조 탐지
  3. 버킷 일관성 — pruned_items 안의 노드가 실제로 current_bucket="pruned"인지
  4. 원칙 위반 검사 — 축 혼합, 조기 확정, 조기 pruning, 태스크→결정 오분류
  5. 순환 탐지 — DAG 구축 시 사이클 감지 + 엣지 강도 기반 해소
  6. DAG-트리 매핑 일관성 — 트리에 priority/later_active 외의 노드가 있으면 위반
  7. Auditor — 위의 전부를 한 번 더 확인하고, 실패 시 해당 에이전트에 수정 요청 (최대 2회)

골든 마스터 테스트 픽스처(kiosk-mvp, saas-launch)로 회귀 테스트도 가능하다.

설계에서 배울 점

1. 프로젝션 패턴

내부 표현(DAG)과 사용자 표현(트리)을 분리하고, 변환 시 손실되는 정보를 명시적으로 기록한다. 정보 손실을 숨기지 않고 hidden_dependency_edges로 추적하는 접근이 인상적이다.

2. 결정론적 스코어링

Gate → Score → Cap → Tiebreaker 4단계를 거치면 같은 입력에 항상 같은 출력이 나온다. AI 에이전트의 비결정성을 구조로 제어하는 방법.

3. 체크포인트 패턴

비용이 큰 작업(우선순위 매기기) 전에 저비용 검증(가정 확인)을 삽입한다. 에이전트 파이프라인에서의 "measure twice, cut once".

4. 학습 루프 구조화

에러가 나면 "어디를 고쳐야 하는가"의 매핑이 이미 정의되어 있다. 하네스 자체가 자기 개선 프로세스를 내장한 구조.

파일 구조 요약

decision-tree/
├── .claude/
│   ├── CLAUDE.md                   ← 하네스 운영 스펙 (880줄)
│   ├── agents/                     ← 5개 에이전트 스펙
│   └── skills/                     ← 2개 도메인 지식 스킬
├── prompts/
│   ├── core-prompt.md              ← 12개 핵심 원칙
│   ├── learning-rules.md           ← 오류 분류 + 진단 체계
│   └── rules/                      ← 버킷/트리변환/타이브레이커 규칙
├── schemas/                        ← 7개 JSON 스키마
├── scripts/
│   ├── validate-output.js          ← 스키마 + 일관성 검증
│   └── generate-tree-viewer.js     ← HTML 뷰어 생성
├── tests/fixtures/                 ← 골든 마스터 테스트
├── docs/                           ← 가이드 문서
└── _workspace/                     ← 런타임 출력 (라운드별)
한 줄 요약

의사결정의 혼돈을 DAG로 정규화하고, 의존성 기반 우선순위로 "지금 결정해야 할 3가지"를 뽑아주는 Claude Code 멀티에이전트 하네스.
