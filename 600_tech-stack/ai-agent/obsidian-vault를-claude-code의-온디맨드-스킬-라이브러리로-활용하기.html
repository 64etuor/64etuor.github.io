1장. 왜 Obsidian Vault가 AI 스킬 베이스가 될 수 있는가

AI 도구가 발전할수록 역설적인 문제가 생긴다. 정보가 많아질수록 AI에게 무엇을 줘야 할지 모르게 된다. 이 장은 그 혼란의 원인을 정확히 짚고, 왜 Obsidian vault를 구조화된 스킬 베이스로 만들어야 하는지 납득하기 위한 출발점이다.

1.1 AI에게 폴더를 통째로 주면 생기는 문제

가장 직관적인 접근은 이렇다. Obsidian vault 폴더를 Claude Code에 마운트하고, "여기 내 지식이 다 있으니 알아서 참고해"라고 말하는 것. 실제로 많은 사람이 이렇게 시작한다. 그리고 얼마 지나지 않아 뭔가 이상함을 느낀다.

1.1.1 컨텍스트 윈도우 한계와 토큰 비용의 현실

현재 Claude의 컨텍스트 윈도우는 최대 200K 토큰이다. 넉넉해 보이지만, 마크다운 파일 하나가 평균 500~2,000토큰임을 감안하면 vault에 파일이 200개만 넘어도 전체를 한 번에 넣는 것은 불가능하다. 수백 개의 파일을 가진 vault라면 물리적으로 다 들어가지 않는다.

억지로 넣는다 해도 비용 문제가 따라온다. API를 직접 사용하거나 토큰 기반 과금 모델을 쓴다면, 매 요청마다 수만 토큰을 소모하는 구조는 지속 가능하지 않다.

1.1.2 관련 없는 문서가 응답 품질을 떨어뜨리는 메커니즘

컨텍스트 윈도우에 관련 없는 문서가 섞이면 AI의 응답 품질이 저하된다. 이것은 직관에 반하지만 연구로 검증된 현상이다. LLM은 컨텍스트 전체에 걸쳐 균등하게 주의(attention)를 분산하지 않는다. 관련 없는 내용이 많을수록 실제로 필요한 정보에 대한 집중도가 희석된다.

쉽게 말하면, 코드 리뷰를 요청했는데 컨텍스트에 레시피 메모, 여행 계획, 독서 기록이 함께 들어있다면 AI는 그 모든 것을 처리하느라 정작 코딩 컨벤션 파일에 집중하지 못한다.

1.1.3 실제 실패 사례: vault 전체 주입 vs 선택 주입 비교

다음은 동일한 요청에 대한 두 가지 결과다.

시도 A: vault 전체 주입:

  • 요청: "내 글쓰기 스타일에 맞게 이 초안을 다듬어줘"
  • 컨텍스트: vault 전체 (240개 파일, 약 180K 토큰)
  • 결과: 글쓰기 스타일 가이드 파일이 있었지만 AI는 이를 제대로 참조하지 못했고, 일반적인 수준의 교정만 수행했다. vault 안의 기술 문서, 프로젝트 메모 등이 노이즈로 작용했다.

시도 B: 선택 주입:

  • 요청: 동일
  • 컨텍스트: writing-style.md, audience-persona.md 2개 파일만 선택 주입
  • 결과: 특정 어투 패턴, 문장 길이 선호도, 독자 페르소나가 명확히 반영된 교정이 이루어졌다.

같은 정보가 vault 안에 있었지만 결과는 전혀 달랐다. 문제는 정보의 존재가 아니라 정보의 정밀한 전달 이었다.

1.2 취사선택(Selective Loading)이 왜 핵심인가

문제의 해결 방향은 명확하다. AI에게 더 많은 것을 주는 것이 아니라, 더 정확한 것을 주는 것이다.

1.2.1 정밀한 3개 문서 > 노이즈 낀 30개 문서인 이유

컨텍스트의 질은 양보다 관련성에 의해 결정된다. 작업과 직접 연관된 문서 3개를 정확히 선별해서 주는 것이, 관련 있을 수도 있는 문서 30개를 뭉텅이로 주는 것보다 일관되게 좋은 결과를 만든다.

이유는 단순하다. AI가 처리해야 할 정보의 신호 대비 잡음(signal-to-noise ratio)이 높아지기 때문이다. 관련 문서만 있으면 AI는 그 안에서 답을 찾는 데 집중할 수 있다.

1.2.2 RAG와 Selective Loading의 차이: 왜 RAG만으로는 부족한가

RAG(Retrieval-Augmented Generation)는 이미 널리 쓰이는 접근이다. 대규모 문서를 벡터로 임베딩하고 의미적으로 유사한 청크를 검색해서 주입한다. 강력한 방법이지만 개인 vault 활용에는 맞지 않는 부분이 있다.

RAG는 문서를 청크 단위로 쪼갠다. 그런데 스킬 파일은 청크가 아니라 파일 단위로 의미가 완결된다. 코딩 컨벤션 파일의 일부만 잘라서 주입하면 전체 맥락이 사라진다. 또한 RAG 파이프라인 구축에는 임베딩 모델, 벡터 DB, 검색 로직이 필요하다. 개인 워크플로우에 도입하기에 오버엔지니어링이다.

Selective Loading은 더 단순하다. 메타데이터를 읽고, 작업과 매칭되는 파일 전체를 통째로 주입한다. 파이프라인이 없다. 파일 자체가 단위다.

1.3 Obsidian이 다른 도구보다 유리한 이유

취사선택 전략이 유효하다면, 다음 질문은 왜 하필 Obsidian이냐는 것이다.

1.3.1 Markdown + YAML Frontmatter가 LLM 친화적인 이유

LLM은 구조화된 텍스트를 잘 처리한다. Markdown은 헤딩, 리스트, 코드블록이 명시적인 기호로 표현되어 LLM이 문서의 구조를 파악하기 쉽다. 여기에 YAML frontmatter는 파일 상단에 기계가 읽기 좋은 메타데이터를 담는 표준 방식이다.

아래는 스킬 파일의 frontmatter 예시다.

---
description: Python 비동기 처리 시 에러 핸들링 패턴 정리
type: reference
domain: backend-engineering
use_when:
  - Python async/await 코드 작성 시
  - 에러 핸들링 구조 설계 시
load_priority: on-demand
last_reviewed: 2026-03-15
---

이 구조는 AI가 파일을 열기 전에 핵심 정보를 파악할 수 있게 한다. 파일 내용을 전부 읽지 않아도 이 파일이 무엇인지, 언제 써야 하는지 알 수 있다.

1.3.2 로컬 파일시스템 접근의 단순성과 이식성

Obsidian vault는 그냥 폴더다. 별도의 API가 없다. 데이터베이스도 없다. 모든 파일이 로컬 디스크의 .md 파일이다. Claude Code 같은 에이전트가 로컬 파일시스템에 접근하는 것은 가장 단순하고 신뢰할 수 있는 통합 방식이다.

클라우드 서비스에 종속되지 않고, 인터넷 연결 없이도 작동하며, 어떤 도구로든 읽고 쓸 수 있다. 이식성이 완벽하다.

1.3.3 Notion, Confluence 대비 구조적 이점

항목 Obsidian Notion Confluence
파일 포맷 로컬 .md 파일 독점 DB 독점 DB
YAML 메타데이터 네이티브 지원 프로퍼티로 부분 지원 미지원
AI 에이전트 직접 접근 파일시스템으로 즉시 가능 API 인증 필요 API 인증 필요
오프라인 작동 완전 지원 제한적 미지원
데이터 이식성 완전 (표준 Markdown) 낮음 (독점 포맷) 낮음 (독점 포맷)

Notion과 Confluence는 강력하지만 AI 에이전트가 직접 파일시스템처럼 다룰 수 없다. 항상 API를 거쳐야 하고, 그 과정에서 YAML frontmatter 같은 메타데이터 구조를 자유롭게 설계하기 어렵다.

1.4 이 가이드를 읽고 나면 무엇이 달라지는가

1.4.1 Before: '폴더 던지기' 워크플로우

이 흐름에서 vault는 사실상 활용되지 않는다. 있으나 마나 한 상태다.

1.4.2 After: 메타데이터 기반 정밀 로딩 워크플로우

vault가 수동적인 저장소에서 능동적인 스킬 레이어로 바뀐다. AI는 작업할 때마다 내 지식과 선호를 자동으로 참조한다. 이것이 이 가이드가 만들려는 상태다.

2장. 설계 철학: description 중심의 메타데이터 전략

구현보다 철학이 먼저다. YAML 필드를 어떻게 채울지 알기 전에, 왜 이 구조가 작동하는지를 이해해야 한다. 이 장을 건너뛰면 3장의 스키마 설계가 단순한 따라하기가 된다. 이 장을 이해하면 스키마의 모든 선택에 이유가 생긴다.

2.1 메타데이터는 AI를 위한 '문서의 명함'이다

2.1.1 AI가 파일을 읽기 전에 무엇을 판단하는가

Claude Code가 vault 폴더에 접근했을 때 가장 먼저 하는 일은 파일 목록을 확인하는 것이다. 이 시점에서 AI는 파일 이름과 경로만 알고 있다. coding-conventions.md 라는 이름은 내용을 짐작하게 하지만, note-2024-11.md 나 temp.md 는 아무것도 알려주지 않는다.

파일 이름만으로 판단이 어렵다면 AI는 파일을 열어서 읽어야 한다. 파일이 수백 개라면 이 과정 자체가 컨텍스트를 소모한다. 그리고 결국 모든 파일을 읽기 전에는 무엇이 필요한지 알 수 없는 순환에 빠진다.

YAML frontmatter는 이 순환을 끊는다. 파일의 첫 몇 줄만 읽어도 이 파일이 무엇인지, 언제 필요한지, 얼마나 중요한지를 알 수 있다. AI는 전체 내용을 읽기 전에 이미 로드 여부를 결정할 수 있다.

2.1.2 First-pass filter로서의 메타데이터 작동 원리

메타데이터 기반 탐색은 두 단계로 작동한다.

1단계: 메타데이터 스캔 (저비용):
AI가 vault 전체 파일의 frontmatter만 읽는다. 파일당 수십 토큰이면 충분하다. 100개 파일의 frontmatter를 읽어도 전체 비용은 수천 토큰 수준이다.

2단계: 선택적 전체 로드 (고비용, 최소화):
1단계에서 현재 작업과 매칭된 파일 2~3개만 전체 내용을 로드한다. 나머지 97개 파일은 건드리지 않는다.

이것이 first-pass filter다. 메타데이터가 필터의 기준이 되고, 전체 로드는 통과한 파일에만 일어난다.

2.1.3 사람이 이름표를 붙이는 것과 AI가 메타데이터를 읽는 것의 유사성

도서관에서 책의 목차와 뒷면 소개글을 읽고 빌릴 책을 고르는 과정을 생각해보자. 모든 책을 처음부터 끝까지 읽고 나서 선택하는 사람은 없다. YAML frontmatter는 책의 뒷면 소개글 이다. 잘 쓰인 소개글은 독자가 10초 안에 이 책이 자신에게 필요한지 판단하게 한다.

AI도 마찬가지다. description이 잘 쓰인 파일은 AI가 즉시 필요 여부를 판단한다. description이 없거나 모호한 파일은 AI가 내용을 열어봐야 하고, 그마저도 확신하지 못한 채 로드 여부를 결정한다.

2.2 description 한 줄의 힘: 좋은 설명과 나쁜 설명의 차이

2.2.1 나쁜 description의 4가지 패턴

패턴 1: 모호함: 무엇에 관한 파일인지 알 수 없다.

  • 나쁜 예: description: 메모
  • 나쁜 예: description: 작업 관련 내용

패턴 2: 중복: 다른 파일의 description과 구별이 안 된다.

  • 나쁜 예 A: description: 글쓰기 관련 가이드
  • 나쁜 예 B: description: 글 작성 시 참고 문서
  • AI는 둘 중 어느 파일을 로드해야 할지 알 수 없다.

패턴 3: 과포함: 너무 많은 것을 담으려 해서 초점이 없다.

  • 나쁜 예: description: 코딩, 글쓰기, 회의, 프로젝트 관리에 관한 여러 가지 메모와 참고 자료

패턴 4: 과소포함: 실제 용도를 설명하지 않는다.

  • 나쁜 예: description: Python 파일
  • 나쁜 예: description: 2024년 자료

2.2.2 좋은 description의 구조: 동사 + 대상 + 맥락

좋은 description은 세 가지 요소를 담는다.

  • 동사: 이 파일이 무엇을 하는가 (정의한다, 정리한다, 안내한다, 체크한다)
  • 대상: 무엇에 관한 것인가 (Python 에러 핸들링, 블로그 글쓰기 어투, SEO 체크리스트)
  • 맥락: 언제 또는 어떤 상황에서 쓰는가 (비동기 코드 작성 시, 기술 블로그 발행 전)

이 세 요소가 갖춰진 description은 AI가 매칭 여부를 즉시 판단할 수 있다.

2.2.3 대조 예시 10쌍으로 감각 익히기

나쁜 description 좋은 description
메모 일상적인 아이디어를 빠르게 기록하는 캡처 전용 파일
코딩 관련 Python 프로젝트에서 사용하는 네이밍 컨벤션과 코드 스타일 규칙 정의
글쓰기 가이드 기술 블로그 작성 시 유지할 문체, 문장 길이, 어투 패턴 정리
회의 자료 주간 팀 미팅 결과를 액션 아이템 중심으로 구조화하는 템플릿
참고 문서 REST API 설계 시 엔드포인트 네이밍과 응답 포맷 표준 정의
체크리스트 블로그 글 발행 전 SEO, 가독성, 링크를 점검하는 최종 체크리스트
프로젝트 관련 새 프로젝트 킥오프 시 확인할 요구사항 수집 및 범위 정의 프로세스
독서 기록 읽은 책에서 발췌한 핵심 인사이트와 적용 아이디어 정리
AI 프롬프트 Claude에게 코드 리뷰를 요청할 때 사용하는 구조화된 프롬프트 템플릿
개인 정보 이력서와 포트폴리오에 사용하는 자기소개 문구와 경력 요약

2.3 MECE 원칙을 메타데이터에 적용하기

2.3.1 description이 겹치면 AI가 혼란에 빠지는 이유

MECE(Mutually Exclusive, Collectively Exhaustive)는 컨설팅에서 온 개념이지만 메타데이터 설계에 그대로 적용된다. 서로 겹치지 않고(ME), 빠짐없이 커버하는(CE) 구조다.

description이 겹치는 파일이 두 개 있으면 AI는 둘 다 로드하거나 둘 다 무시하는 양극단으로 흐른다. "글쓰기 가이드"라는 description을 가진 파일이 세 개라면 AI는 어느 것이 현재 작업에 더 적합한지 판단할 근거가 없다. 결과적으로 셋 다 로드해서 컨텍스트를 낭비하거나, 임의로 하나만 선택해서 나머지를 놓친다.

2.3.2 vault 전체 도메인 분류 체계를 먼저 정의해야 하는 이유

개별 파일의 description을 쓰기 전에 vault 전체의 도메인 분류 체계를 먼저 설계해야 한다. 분류 체계 없이 파일마다 description을 쓰면 비슷한 파일들이 제각각 다른 언어로 자신을 설명하게 된다.

도메인 분류 체계는 vault의 지도다. "내 vault에는 어떤 종류의 지식이 있는가"를 먼저 정의하면, 각 파일의 description은 그 지도 위의 정확한 좌표가 된다.

2.3.3 분류 체계 설계 워크숍: 내 vault를 5개 도메인으로 나눠보기

vault를 5개 내외의 도메인으로 나누는 연습이다. 아래는 예시이며, 자신의 vault 성격에 맞게 조정한다.

예시: IT 실무자의 vault 도메인 분류:

domain:
  - writing        # 글쓰기, 블로그, 문서 작성 관련
  - engineering    # 코드, 아키텍처, 개발 패턴 관련
  - thinking       # 의사결정, 프레임워크, 사고 도구 관련
  - workflow       # 반복 작업 프로세스, 체크리스트 관련
  - persona        # AI에게 주입할 페르소나, 역할 정의 관련

이 5개 도메인이 정해지면, vault 안의 모든 파일은 반드시 이 중 하나에 속해야 한다. 어디에도 속하지 않는 파일이 생기면 도메인 분류 자체를 재검토하는 신호다.

2.4 스킬 파일 vs 참고 문서 vs 컨텍스트 문서: 문서 역할 구분

2.4.1 type: skill: AI가 따라야 할 행동 지침

스킬 파일은 AI에게 "이렇게 행동해라"를 지시하는 파일이다. 작업 방식, 글쓰기 규칙, 코딩 컨벤션, 응답 포맷 지침이 여기에 해당한다. AI는 이 파일을 로드하면 내용을 따라야 한다는 것을 안다.

  • 예: writing-style.md: "문장은 2줄 이내로 유지하고, 수동태를 쓰지 않는다"
  • 예: code-review-rules.md: "변수명 리뷰 시 이 기준을 적용한다"

2.4.2 type: reference: AI가 참고할 지식 베이스

참고 문서는 "이런 정보가 있다"를 알려주는 파일이다. AI에게 지시를 내리지 않고, 작업에 필요한 배경 지식이나 사실 정보를 제공한다. AI는 이 파일을 읽고 내용을 판단의 근거로 활용한다.

  • 예: async-error-patterns.md: Python 비동기 에러 패턴 분류와 설명
  • 예: seo-principles.md: 검색 최적화 원칙 정리

2.4.3 type: context: 작업 배경을 제공하는 상황 문서

컨텍스트 문서는 AI에게 "지금 상황은 이렇다"를 알려주는 파일이다. 프로젝트 배경, 독자 페르소나, 팀 상황처럼 작업의 맥락을 설정한다. 기술적 지식이 아니라 상황적 이해를 준다.

  • 예: audience-persona.md: "이 블로그의 독자는 3년차 이상 개발자다"
  • 예: project-background.md: "이 프로젝트는 B2B SaaS이고 주요 고객은 중견 제조업체다"

2.4.4 세 유형을 혼동했을 때 생기는 문제 사례

가장 흔한 혼동은 reference를 skill처럼 쓰는 경우다. Python 에러 패턴을 정리한 파일에 "항상 이 패턴을 사용하라"는 지시를 섞으면, AI는 단순 참고 자료인지 강제 규칙인지 구분하지 못한다. 유형을 명확히 분리하면 AI의 파일 활용 방식도 정확해진다.

2.5 load_priority 개념: 항상 로드 vs 온디맨드 vs 잠금

2.5.1 always: 모든 세션에 주입할 핵심 페르소나·룰 파일

always 로 표시된 파일은 어떤 작업을 하든 세션 시작 시 자동으로 컨텍스트에 포함된다. 코드 리뷰를 하든 블로그 글을 쓰든, AI는 항상 이 파일들을 먼저 읽고 시작한다.

always 에 적합한 파일:

  • AI의 기본 응답 방식과 어투를 정의하는 페르소나 파일
  • 모든 작업에 공통으로 적용되는 핵심 규칙
  • vault 탐색 방법을 설명하는 메타 지시 파일

always 파일은 적을수록 좋다. 3~5개가 이상적이다. 이 목록이 길어지면 온디맨드의 의미가 사라진다.

2.5.2 on-demand: 작업 트리거에 따라 선택적으로 로드

on-demand 는 기본값이다. 대부분의 스킬 파일은 이 우선순위를 가진다. AI가 현재 작업을 분석하고, use_when 필드와 매칭될 때만 로드한다.

on-demand 파일은 vault의 핵심 자산이다. 수십, 수백 개가 있어도 작업과 무관한 파일은 컨텍스트에 들어오지 않으므로 부담이 없다.

2.5.3 locked: AI에게 절대 노출하지 않을 민감한 문서

locked 로 표시된 파일은 AI가 접근하지 않는다. 개인 일기, 비공개 재무 정보, 아직 완성되지 않은 초안처럼 AI 컨텍스트에 포함되면 안 되는 파일에 사용한다.

CLAUDE.md에 "load_priority가 locked인 파일은 절대 읽지 말 것"이라는 지시를 명시적으로 포함시켜야 실제로 작동한다.

2.6 메타데이터 설계는 한 번으로 끝나지 않는다

2.6.1 vault가 성장하면서 메타데이터가 부패하는 3가지 방식

방식 1: 내용 변화: 파일 내용을 업데이트했지만 description은 그대로 둔다. AI는 예전 description을 믿고 파일을 로드했다가 기대와 다른 내용을 만난다.

방식 2: 맥락 변화: 처음에는 맞았던 use_when 조건이 워크플로우가 바뀌면서 더 이상 현실을 반영하지 못한다.

방식 3: 중복 누적: 비슷한 파일이 계속 추가되면서 description이 서로 겹치기 시작한다. MECE가 깨진다.

이 세 가지 부패는 조용히 일어난다. AI의 응답이 서서히 부정확해지지만 원인을 파악하기 어렵다.

2.6.2 리팩토링 시점을 인식하는 신호들

다음 신호가 나타나면 메타데이터 리팩토링을 시작할 때다.

  • AI가 엉뚱한 파일을 로드하기 시작한다
  • 같은 작업을 요청했는데 매번 다른 파일이 선택된다
  • vault에 파일을 추가했는데 AI가 그것을 전혀 활용하지 않는다
  • description을 보고도 어떤 파일인지 스스로 확신이 서지 않는다

마지막 신호가 가장 중요하다. 파일 작성자인 본인이 description만 읽고 파일 내용을 정확히 예측할 수 없다면, AI도 마찬가지다.

3장. YAML 메타데이터 스키마 설계하기

2장에서 왜 메타데이터가 필요한지를 이해했다면, 이 장에서는 그것을 실제 YAML 필드로 구현한다. 각 필드는 목적이 있고, 그 목적에 맞는 작성 규칙이 있다. 스키마를 먼저 완성해두면 이후의 모든 파일이 일관된 언어로 AI와 소통할 수 있다.

3.1 필수 필드 정의: description, type, domain

3.1.1 description: 작성 규칙과 길이 제한

description은 스키마의 핵심이다. 모든 파일에 반드시 있어야 하며, 다음 규칙을 따른다.

작성 규칙:

  • 1~2문장을 넘지 않는다. 길어질수록 AI의 매칭 정확도가 떨어진다.
  • 2장에서 정의한 구조를 따른다: 동사 + 대상 + 맥락
  • 파일 이름을 반복하지 않는다. writing-style.md 의 description에 "글쓰기 스타일 파일"이라고 쓰는 것은 의미 없다.
  • 현재 시제로 작성한다. "정리했다"가 아니라 "정리한다" 또는 "정의한다".

길이 기준:

  • 권장: 15~40자
  • 허용: 최대 80자
  • 80자를 넘어가면 파일이 너무 많은 역할을 하고 있다는 신호다. 파일 분리를 고려한다.

3.1.2 type: 허용 값 정의

type 필드는 2장에서 정의한 세 가지 역할에 meta 를 추가한 4개의 허용 값만 사용한다.

type: skill      # AI가 따라야 할 행동 지침
type: reference  # AI가 참고할 지식 베이스
type: context    # 작업 배경을 제공하는 상황 문서
type: meta       # vault 구조와 탐색 방법을 설명하는 시스템 파일

meta 타입은 CLAUDE.md, index.md처럼 AI가 vault를 탐색하는 방법 자체를 설명하는 파일에 사용한다. 이 파일들은 작업의 내용이 아니라 시스템의 작동 방식을 정의한다.

이 4개 이외의 값은 사용하지 않는다. 허용 값을 자유롭게 추가하면 AI가 type 필드를 해석하는 일관성이 깨진다.

3.1.3 domain: 사전 정의된 분류 체계에서만 선택하게 강제하는 이유

domain 필드는 2.3.3에서 설계한 도메인 분류 체계의 값만 사용한다. 즉흥적으로 새 도메인 값을 만들지 않는다.

왜 강제하는가: domain 값이 파일마다 제각각이면 AI가 도메인 기반 필터링을 할 수 없다. "engineering 도메인 파일 중에서 찾아라"는 지시가 작동하려면 모든 engineering 관련 파일이 정확히 domain: engineering 이라고 쓰여 있어야 한다.

도메인 목록은 CLAUDE.md에 명시적으로 선언해두고, 새 파일을 만들 때 반드시 그 목록에서 선택한다. 목록에 없는 도메인이 필요하다고 느껴지면 파일을 먼저 만들지 말고, CLAUDE.md의 도메인 목록을 업데이트한 후 진행한다.

3.2 use_when 필드: AI에게 '언제 이 파일을 써라'를 알려주기

3.2.1 작업 시나리오 기반으로 use_when 작성하는 패턴

use_when 은 AI에게 로드 트리거를 명시적으로 알려주는 필드다. description이 "이 파일이 무엇인가"를 답한다면, use_when 은 "이 파일을 언제 써야 하는가"를 답한다.

작업 시나리오 기반으로 작성하는 것이 핵심이다. 추상적인 키워드가 아니라, 실제로 사용자가 요청할 법한 작업 상황을 문장으로 기술한다.

나쁜 예 (추상적 키워드):

use_when:
  - 글쓰기
  - 블로그

좋은 예 (작업 시나리오):

use_when:
  - 기술 블로그 초안을 작성하거나 다듬을 때
  - 글의 어투와 문체를 검토할 때
  - 독자를 고려한 문장 수정을 요청받을 때

3.2.2 복수 트리거 작성법: 배열 형태로 여러 조건 지정

하나의 파일이 여러 상황에 유효하다면 배열로 여러 트리거를 지정한다. 단, 트리거가 5개를 넘어가면 파일의 역할이 너무 넓다는 신호다.

use_when:
  - Python 코드에서 예외 처리 구조를 설계할 때
  - async/await 함수의 에러 흐름을 검토할 때
  - try/except 패턴의 적절성을 판단할 때

트리거 간에는 의미적 연관성이 있어야 한다. 연관성 없는 트리거가 섞여 있다면 파일을 분리하는 것이 맞다.

3.2.3 use_when이 너무 넓거나 좁을 때의 문제

너무 넓은 경우:

use_when:
    - 무언가를 작성할 때

이 트리거는 거의 모든 작업에 매칭된다. AI는 이 파일을 항상 로드하게 되고, 사실상 load_priority: always 와 같아진다. always 로 올릴 만큼 중요한 파일이 아니라면 트리거를 더 구체화해야 한다.

너무 좁은 경우:

use_when:
    - 2024년 4분기 결산 보고서를 작성할 때

이 트리거는 그 특정 작업이 끝나면 영구히 매칭되지 않는다. 일반화 가능한 시나리오로 작성하거나, 그것이 불가능하다면 스킬 파일이 아닌 context 타입으로 분류하는 것이 맞다.

3.3 tags vs domain: 두 필드를 중복 없이 설계하는 법

3.3.1 tags = 내용 키워드 (사람용), domain = 라우팅 분류 (AI용)

Obsidian의 기존 tags 시스템은 그대로 유지한다. 단, 역할을 명확히 분리한다.

필드 목적 사용자 예시
tags 내용 기반 검색과 연결 사람 #python, #error-handling, #async
domain AI 라우팅을 위한 카테고리 분류 AI engineering

tags 는 세분화된 키워드다. Obsidian 그래프 뷰에서 파일 간 연결을 만들고, 사람이 검색할 때 쓴다. domain 은 큰 분류다. AI가 "어느 영역의 파일인가"를 파악하는 데 쓴다.

같은 정보를 두 필드에 중복으로 쓰지 않는다. tags: [engineering] 이라고 쓰면서 domain: engineering 도 쓰는 것은 중복이다. tags는 구체적인 내용 키워드로만 채운다.

3.3.2 두 필드가 겹쳤을 때 AI가 혼동하는 이유와 해결책

tags와 domain이 같은 정보를 담으면 AI는 어느 필드를 라우팅 기준으로 삼아야 할지 불분명해진다. CLAUDE.md에 "domain 필드를 라우팅 기준으로 사용하고 tags는 무시한다"고 명시하거나, 처음부터 두 필드의 역할을 분리해서 혼동의 여지를 없앤다.

3.4 load_priority 필드 구현

3.4.1 YAML에서 always / on-demand / locked 값으로 구현하기

load_priority: always      # 모든 세션에 자동 주입
load_priority: on-demand   # use_when 매칭 시에만 로드 (기본값)
load_priority: locked      # AI 접근 금지

on-demand 가 기본값이므로, 이 값은 명시하지 않아도 된다. 그러나 명시적으로 적어두는 것이 가독성에 좋다. 파일을 훑어볼 때 의도가 분명히 보인다.

3.4.2 CLAUDE.md에서 이 값을 참조하는 지시 패턴

CLAUDE.md에 다음과 같은 지시를 포함시킨다.

## 파일 로딩 규칙

1. load_priority가 \`always\`인 파일은 작업 시작 전에 반드시 모두 읽는다.
2. load_priority가 \`on-demand\`인 파일은 use_when 조건이 현재 작업과 매칭될 때만 읽는다. 매칭된 파일은 최대 3개로 제한한다.
3. load_priority가 \`locked\`인 파일은 어떤 상황에서도 읽지 않는다. 해당 파일의 존재를 언급하지도 않는다.

이 지시가 있어야 AI가 load_priority 필드를 실제 행동 기준으로 사용한다.

3.4.3 locked 파일을.gitignore처럼 처리하는 방법

.gitignore 가 특정 파일을 git 추적에서 제외하듯, locked 파일 목록을 CLAUDE.md에 명시적으로 선언할 수 있다.

## 접근 금지 파일 및 폴더

다음 경로는 어떤 상황에서도 읽거나 참조하지 않는다.
- /private/
- /journal/
- load_priority가 locked인 모든 파일

폴더 단위로 선언하면 개별 파일마다 locked 를 붙이지 않아도 된다. 민감한 파일을 /private/ 폴더에 모아두고 폴더 전체를 금지하는 것이 더 안전하다.

3.5 version과 last_reviewed: 스킬 신선도 관리

3.5.1 version 필드로 스킬 변경 이력 추적하기

version 필드는 스킬 파일이 얼마나 성숙한지, 그리고 마지막으로 언제 의도적으로 검토되었는지를 나타낸다.

version: 1.0   # 초안, 아직 검증되지 않은 스킬
version: 1.3   # 여러 번 수정을 거쳐 안정된 스킬
version: 2.0   # 큰 방향 변경이 있었던 스킬

버전을 올리는 기준은 단순하다. 파일 내용을 의도적으로 검토하고 수정했을 때 버전을 올린다. 오탈자 수정은 버전을 올리지 않는다. 방향이나 내용이 바뀌었을 때 올린다.

3.5.2 last_reviewed 기준 90일 이상 된 파일 자동 경고 설정

last_reviewed 는 ISO 8601 날짜 형식으로 작성한다.

last_reviewed: 2026-03-15

90일이 지난 파일은 내용이 현실과 달라졌을 가능성이 있다. 7장에서 다루는 월간 리뷰 루틴에서 Dataview 쿼리로 이 날짜 기준 오래된 파일을 자동으로 목록화한다.

3.6 스키마 템플릿 완성본 및 Templater 연동

3.6.1 모든 필드를 포함한 최종 YAML 템플릿

지금까지 정의한 모든 필드를 담은 완성된 스키마 템플릿이다.

---
description: ""
type: skill         # skill | reference | context | meta
domain: ""          # 사전 정의된 도메인 목록에서 선택
use_when:
  - ""
load_priority: on-demand   # always | on-demand | locked
tags: []
version: "1.0"
last_reviewed: ""
---

필수 필드: description, type, domain
권장 필드: use_when, load_priority, version, last_reviewed
선택 필드: tags

type: meta 인 시스템 파일(CLAUDE.md, index.md)은 use_when 과 load_priority 를 생략해도 된다. 이 파일들은 스킬이 아니라 시스템 자체이기 때문이다.

3.6.2 Obsidian Templater로 새 파일 생성 시 자동 삽입 설정

Obsidian의 Templater 플러그인을 사용하면 새 파일을 만들 때 위 템플릿이 자동으로 삽입된다.

설정 방법:

  1. Obsidian 설정 → Templater 플러그인 설치 및 활성화
  2. 템플릿 폴더를 지정한다 (예: /templates/)
  3. /templates/skill-template.md 파일을 만들고 위 YAML 템플릿을 붙여넣는다
  4. Templater 설정에서 "Trigger Templater on new file creation"을 활성화한다
  5. 이후 새 파일을 만들면 템플릿 선택 프롬프트가 뜬다

3.6.3 type별로 다른 템플릿 분기 적용하기

skill, reference, context, meta 각각에 맞는 템플릿을 별도로 만들면 더 효율적이다.

/templates/
  ├── skill-template.md       # use_when 필드 포함
  ├── reference-template.md   # use_when 생략, tags 강조
  ├── context-template.md     # 프로젝트/상황 설명 섹션 포함
  └── meta-template.md        # 시스템 파일용 최소 frontmatter

type에 따라 필요한 필드가 다르기 때문에, 하나의 범용 템플릿보다 분기된 템플릿이 실수를 줄인다.

3.7 기존 문서에 메타데이터 일괄 추가하기

3.7.1 우선순위 매트릭스: 어떤 파일부터 메타데이터를 붙일 것인가

vault에 이미 수백 개의 파일이 있다면 모든 파일에 한 번에 메타데이터를 붙이려 하지 않는다. 다음 매트릭스로 우선순위를 정한다.

우선순위 파일 유형 이유
1순위 가장 자주 참조하는 파일 10~20개 즉각적인 효과, 투자 대비 효용 최대
2순위 skill 타입으로 분류될 파일 전체 AI 행동에 직접 영향을 미침
3순위 reference 타입 파일 지식 베이스로 활용 가능
4순위 context 타입 파일 프로젝트 진행 중인 것부터
보류 1년 이상 열어보지 않은 파일 메타데이터 전에 필요 여부 판단

3.7.2 Claude Code를 활용해 기존 문서 description 초안 자동 생성하기

이미 있는 파일들의 description을 직접 쓰는 것이 부담스럽다면, Claude Code 자체를 활용해 초안을 만들 수 있다. 역설적이지만 가장 효율적인 방법이다.

CLAUDE.md가 아직 없는 상태에서 다음과 같이 요청한다.

이 vault의 마크다운 파일들을 읽고, 각 파일에 맞는 YAML frontmatter를 제안해줘. description은 1~2문장, type은 skill/reference/context/meta 중 하나, domain은 내가 지금 정의할 분류 체계를 따라줘.

AI가 제안한 description을 그대로 쓰지 않는다. 초안으로 사용하고 반드시 직접 검토해서 수정한다. AI가 파일의 맥락을 완전히 이해하지 못할 수 있기 때문이다. 그러나 빈 칸에서 시작하는 것보다 초안을 수정하는 것이 훨씬 빠르다.

3.7.3 점진적 마이그레이션 4주 플랜

주차 목표 작업 내용
1주차 기반 설계 도메인 분류 체계 확정, CLAUDE.md 초안 작성, 템플릿 파일 생성
2주차 핵심 파일 처리 1순위 파일 10~20개에 메타데이터 수동 작성 및 검증
3주차 스킬 파일 완성 Claude Code로 나머지 skill 타입 파일 description 초안 생성 후 검토
4주차 나머지 처리 reference, context 파일 처리. 전체 연동 테스트 및 CLAUDE.md 보완

4주 후에도 메타데이터가 없는 파일이 남아있는 것은 정상이다. 활용 빈도가 낮은 파일은 실제로 필요할 때 추가해도 늦지 않다.

4장. Vault 구조화: AI가 탐색하기 좋은 폴더와 인덱스 설계

YAML 스키마가 개별 파일의 언어라면, 폴더 구조와 인덱스는 vault 전체의 지도다. 아무리 잘 쓴 메타데이터도 AI가 파일을 찾아가는 경로가 혼란스러우면 제 기능을 못한다. 이 장은 AI가 최소한의 비용으로 최대한 정확하게 탐색할 수 있는 vault 구조를 설계하는 방법을 다룬다.

4.1 AI의 탐색 방식 이해하기

4.1.1 Claude Code가 디렉토리를 읽는 순서와 깊이 한계

Claude Code가 vault 폴더에 접근하면 가장 먼저 루트 디렉토리의 파일 목록을 확인한다. 이 시점에서 AI는 폴더명과 파일명만 본다. 내용은 아직 읽지 않는다.

탐색은 기본적으로 너비 우선(breadth-first)으로 진행된다. 루트를 먼저 파악하고, 그 안의 폴더들을 확인하고, 각 폴더 안을 살펴본다. 이 과정에서 CLAUDE.md가 있으면 가장 먼저 읽는다. CLAUDE.md가 이후 탐색의 방향을 결정하기 때문이다.

깊이 한계도 존재한다. 폴더가 4~5단계 이상 중첩되면 탐색 비용이 증가하고, AI가 깊은 곳에 있는 파일을 자발적으로 탐색하지 않을 가능성이 높아진다. vault의 폴더 구조는 3단계 이내로 유지하는 것이 안전하다.

권장 구조 (3단계 이내):
vault/
  ├── CLAUDE.md
  ├── 01-core/
  │   ├── index.md
  │   └── persona.md
  ├── 02-skills/
  │   ├── index.md
  │   ├── writing/
  │   │   ├── index.md
  │   │   └── blog-style.md
  │   └── engineering/
  │       ├── index.md
  │       └── code-review.md
  └── 03-reference/
      ├── index.md
      └── python-patterns.md

4.1.2 폴더 depth가 깊어질수록 탐색 비용이 증가하는 이유

AI가 파일을 찾으려면 경로를 따라가야 한다. 경로의 각 단계마다 디렉토리 목록을 읽는 작업이 필요하다. 이것은 토큰을 소모하지 않지만 도구 호출(tool call)을 소모한다.

더 중요한 문제는 가시성이다. 깊이 중첩된 파일은 AI의 초기 탐색에서 발견되지 않을 수 있다. CLAUDE.md나 index.md에 명시적으로 언급되지 않은 파일은 AI가 그 존재를 알 방법이 없다. 폴더 구조가 단순할수록 AI의 자율 탐색이 더 잘 작동한다.

4.2 폴더 네이밍 전략

4.2.1 AI가 오해하는 폴더명 패턴

다음 폴더명은 사람에게도 모호하고 AI에게는 더욱 모호하다.

  • Misc, Etc, Other: 무엇이 들어있는지 알 수 없다. AI는 이 폴더를 탐색해야 할지 말지 판단할 수 없다.
  • Archive, Old, Backup: 오래된 정보가 들어있다는 신호다. AI가 이 폴더를 탐색하더라도 description이 없으면 어떤 파일이 여전히 유효한지 알 수 없다.
  • Temp, WIP, Draft: 미완성 파일이 섞여있다는 신호다. AI가 미완성 스킬을 로드하면 품질에 문제가 생긴다.

이런 폴더가 이미 존재한다면 정리하거나, CLAUDE.md에서 접근 금지 목록에 추가한다.

4.2.2 도메인 기반 네이밍 컨벤션 예시

폴더명은 3장에서 정의한 도메인 분류 체계와 일치시킨다. 폴더명을 보면 어떤 도메인의 파일이 있는지 즉시 알 수 있어야 한다.

/writing-skills/        # domain: writing인 파일들
/code-patterns/         # domain: engineering인 파일들
/thinking-tools/        # domain: thinking인 파일들
/workflow-templates/    # domain: workflow인 파일들
/personas/              # domain: persona인 파일들

폴더명과 domain 값을 일치시키면 AI가 "domain: writing 관련 파일을 찾아라"는 지시를 받았을 때 /writing-skills/ 폴더로 바로 이동할 수 있다.

4.2.3 prefix 숫자 정렬로 중요도 시각화하기

폴더 앞에 숫자 prefix를 붙이면 두 가지 효과가 생긴다. 첫째, 파일 탐색기에서 항상 같은 순서로 보인다. 둘째, AI에게 "낮은 번호가 더 중요하다"는 암묵적 신호를 준다.

01-core/          # always 로드 파일들. 가장 중요.
02-skills/        # on-demand 스킬 파일들
03-reference/     # 참고 문서들
04-context/       # 프로젝트별 컨텍스트 파일들
99-archive/       # 보관용. AI 탐색 대상 아님.

CLAUDE.md에 "번호가 낮은 폴더일수록 우선순위가 높다"고 명시해두면 AI가 이 구조를 탐색 전략에 반영한다.

4.3 폴더별 index.md 작성법

4.3.1 index.md의 권장 포맷

각 폴더에 index.md 를 두면 AI가 폴더 안의 모든 파일을 열지 않아도 해당 폴더의 내용을 파악할 수 있다. index.md는 폴더의 목차이자 안내문이다.

권장 포맷은 다음 세 섹션으로 구성된다.

---
description: writing-skills 폴더의 목차. 글쓰기 관련 스킬 파일 목록과 각 파일의 용도를 정리한다.
type: meta
domain: writing
load_priority: on-demand
---

# Writing Skills 폴더 인덱스

이 폴더는 글쓰기 작업 시 AI에게 주입할 스킬 파일을 담고 있다.
블로그, 기술 문서, 이메일 등 텍스트 기반 작업에서 참조한다.

## 파일 목록

| 파일명 | 한 줄 설명 | load_priority |
|---|---|---|
| blog-style.md | 기술 블로그 작성 시 유지할 문체와 어투 규칙 | on-demand |
| audience-persona.md | 이 블로그의 독자 페르소나 정의 | on-demand |
| seo-checklist.md | 블로그 발행 전 SEO 항목 점검 체크리스트 | on-demand |
| email-template.md | 외부 커뮤니케이션 이메일 작성 템플릿 | on-demand |

## 이 폴더를 탐색해야 할 상황

- 블로그 초안 작성 또는 교정 요청 시
- 독자를 고려한 문체 조정이 필요할 때
- 글 발행 전 체크리스트가 필요할 때

이 포맷의 핵심은 파일 목록 표다. AI가 index.md 하나만 읽어도 이 폴더에 어떤 파일이 있고 각각 어떤 역할인지 파악할 수 있다.

4.3.2 Dataview 쿼리로 index.md를 자동 생성·갱신하기

폴더에 파일이 자주 추가된다면 index.md의 파일 목록을 수동으로 유지하는 것이 번거롭다. Obsidian의 Dataview 플러그인을 사용하면 자동으로 목록을 생성할 수 있다.

\`\`\`dataview
TABLE description, load_priority
FROM "02-skills/writing"
WHERE type != "meta"
SORT file.name ASC
\`\`\`

이 쿼리는 해당 폴더의 모든 파일에서 description 과 load_priority 필드를 읽어 자동으로 표를 만든다. 새 파일을 추가하면 index.md의 목록이 자동으로 업데이트된다.

단, Dataview 쿼리는 Obsidian 앱 안에서만 렌더링된다. Claude Code가 raw markdown을 읽을 때는 쿼리 구문이 그대로 보인다. 이 경우 AI가 Dataview 쿼리를 해석하지 못하므로, AI용 index.md에는 정적 표를 별도로 유지하거나 Dataview 쿼리 아래에 정적 표를 병기한다.

4.4 vault 루트의 CLAUDE.md: 전체 지도 역할

4.4.1 CLAUDE.md에 들어갈 3가지 필수 섹션

CLAUDE.md는 AI가 vault에 접근했을 때 가장 먼저 읽는 파일이다. 이 파일 하나가 AI의 전체 탐색 전략을 결정한다. 세 가지 섹션이 반드시 포함되어야 한다.

섹션 1: vault 소개: 이 vault가 무엇이고, 어떤 사람이 만든 것이며, 어떤 목적으로 AI가 활용해야 하는지를 설명한다.

섹션 2: 탐색 지시: AI가 파일을 어떻게 탐색하고 선택해야 하는지 구체적인 절차를 명시한다. 어떤 순서로 읽고, 어떤 기준으로 선택하고, 몇 개까지 로드할 것인지를 포함한다.

섹션 3: 금지 사항: 읽으면 안 되는 폴더와 파일을 명시한다. 이 섹션이 없으면 locked 파일 설정이 실질적으로 작동하지 않는다.

4.4.3 CLAUDE.md가 너무 길어지지 않도록 유지하는 원칙

CLAUDE.md는 지도다. 지도는 간결해야 유용하다. 모든 세부 규칙을 이 파일에 담으려 하면 AI가 핵심 지시를 파악하기 어려워진다.

다음 원칙으로 길이를 관리한다.

  • CLAUDE.md는 구조와 절차만 담는다. 스킬의 세부 내용은 각 스킬 파일에 있어야 한다. "블로그 글은 이렇게 써라"는 내용이 CLAUDE.md에 있으면 안 된다. 그것은 blog-style.md 에 있어야 한다.
  • 500단어를 넘어가면 분리를 고려한다. 특정 섹션이 길어진다면 별도 파일로 분리하고 CLAUDE.md에서 링크로 참조한다.
  • 6개월에 한 번 전체를 재검토한다. 추가만 하다 보면 쌓인다. 지시 중 실제로 AI 행동에 영향을 미치는 것과 그렇지 않은 것을 구분하고 불필요한 내용을 제거한다.

5장. Claude Code와 연동하기: CLAUDE.md 및 MCP 설정

vault 구조와 메타데이터가 준비되었다면 이제 실제로 Claude Code에 연결할 차례다. 이 장은 설계한 vault가 실제 AI 세션에서 작동하도록 만드는 기술적 구현을 다룬다. 설정은 복잡하지 않다. 핵심은 Claude Code가 vault를 어떻게 읽고, 어떤 기준으로 파일을 선택하며, 그 과정을 어떻게 투명하게 만들 것인가다.

5.1 CLAUDE.md에 탐색 지시 심기

5.1.1 탐색 순서 지시: index.md 먼저, 그 다음 description 매칭

AI가 vault를 탐색할 때 따를 순서를 CLAUDE.md에 명시적으로 적어야 한다. 지시가 없으면 AI는 매 세션마다 다른 방식으로 탐색한다. 일관성이 없어지고, 같은 작업을 요청해도 매번 다른 파일이 선택된다.

탐색 순서의 핵심 원칙은 두 단계다. 첫째, 폴더의 index.md를 먼저 읽어 해당 폴더에 어떤 파일이 있는지 파악한다. 둘째, index.md에서 파악한 파일 목록을 바탕으로 각 파일의 description과 use_when을 매칭한다. 이 순서를 지키면 AI가 불필요하게 모든 파일을 직접 열어보는 낭비를 막을 수 있다.

CLAUDE.md에 다음과 같이 명시한다.

## 탐색 순서

1. CLAUDE.md (이 파일)를 읽는다.
2. 01-core/ 폴더의 파일을 모두 읽는다.
3. 작업 도메인에 해당하는 폴더의 index.md를 읽는다.
4. index.md에 나열된 파일의 description과 use_when을
   현재 작업과 대조한다.
5. 매칭된 파일만 전체 내용을 읽는다.

5.1.2 선택 개수 제한 지시: '관련 파일 최대 3개만 로드하라'

제한을 명시하지 않으면 AI는 관련성이 낮은 파일까지 로드하는 경향이 있다. "혹시 도움이 될 수도 있으니"라는 판단으로 파일 수가 늘어난다. 이것은 1장에서 설명한 노이즈 문제로 이어진다.

CLAUDE.md에 명시적으로 상한을 정한다.

## 파일 로딩 제한

on-demand 파일은 현재 작업과 가장 관련성 높은 것부터
최대 3개까지만 로드한다. 관련성이 확실하지 않은 파일은
로드하지 않는다. 적게 로드하는 것이 많이 로드하는 것보다 낫다.

"적게 로드하는 것이 많이 로드하는 것보다 낫다"는 문장이 중요하다. AI는 기본적으로 더 많은 정보를 제공하려는 경향이 있다. 이 지시가 그 경향을 명시적으로 억제한다.

5.1.3 선택 이유 출력 지시: 왜 이 파일을 골랐는지 명시하게 하기

AI가 어떤 파일을 선택했는지 투명하게 출력하도록 지시하면 두 가지 이점이 생긴다. 첫째, 스킬이 올바르게 로드되고 있는지 확인할 수 있다. 둘째, 예상과 다른 파일이 선택되었을 때 description이나 use_when을 개선하는 단서를 얻는다.

## 파일 선택 투명성

파일을 로드하기 전에 다음 형식으로 선택 결과를 출력한다.

[스킬 로드] blog-style.md: 기술 블로그 초안 작성 요청과 매칭
[스킬 로드] audience-persona.md: 독자 페르소나 고려 필요

선택하지 않은 이유도 필요 시 간략히 설명한다.

이 출력은 응답 본문 앞에 한 줄씩 표시된다. 작업이 익숙해진 후 이 출력이 불필요하다고 느껴지면 CLAUDE.md에서 해당 지시를 제거하면 된다.

5.2 로컬 폴더 마운트

5.2.1 --add-dir 옵션으로 vault 경로 연결하는 CLI 명령어

Claude Code CLI에서 vault 폴더를 컨텍스트에 포함시키는 가장 직접적인 방법은 --add-dir 옵션이다.

claude --add-dir /Users/dk/obsidian-vault

이 명령어로 Claude Code를 실행하면 지정한 경로의 파일시스템에 접근할 수 있는 상태로 세션이 시작된다. CLAUDE.md가 vault 루트에 있으면 Claude Code가 세션 시작 시 자동으로 이를 감지하고 읽는다.

여러 폴더를 동시에 추가할 수도 있다.

claude --add-dir /Users/dk/obsidian-vault --add-dir /Users/dk/projects/current

작업 중인 프로젝트 폴더와 vault를 함께 마운트하면 코드와 스킬을 동시에 참조할 수 있다.

5.2.2.claude/settings.json으로 영구 경로 등록하기

매번 CLI 옵션을 입력하는 것이 번거롭다면 설정 파일에 영구 등록한다. 홈 디렉토리의 .claude/settings.json 파일에 다음과 같이 추가한다.

{
  "additionalDirectories": [
    "/Users/dk/obsidian-vault"
  ]
}

이 설정을 추가하면 이후 claude 명령어만 입력해도 vault가 자동으로 마운트된다. 특정 프로젝트에서만 vault를 사용하고 싶다면 프로젝트 루트의 .claude/settings.json 에 동일한 설정을 추가한다. 로컬 설정이 글로벌 설정보다 우선한다.

5.2.3 권한 설정: read-only vs read-write 구분의 중요성

vault를 마운트할 때 Claude Code에 쓰기 권한을 부여할지 신중하게 결정해야 한다.

read-only 권장 상황:

  • 스킬 파일을 참조만 하고 수정은 직접 하는 경우
  • vault 내용의 무결성을 보장하고 싶은 경우
  • AI가 실수로 파일을 덮어쓰는 것을 방지하고 싶은 경우

read-write 허용 상황:

  • AI가 새 스킬 파일을 작성하거나 기존 파일을 업데이트하도록 허용하는 경우
  • vault 자체를 AI와 함께 발전시키는 워크플로우를 운영하는 경우

read-write를 허용한다면 CLAUDE.md에 수정 가능한 폴더와 절대 수정하면 안 되는 폴더를 명시적으로 구분해둔다.

읽기 전용: /01-core/, /02-skills/, /03-reference/
접근 금지: /private/, /99-archive/

5.3 always 로드 파일 자동 주입 설정

5.3.1 CLAUDE.md에서 always 파일 목록을 명시적으로 나열하는 패턴

load_priority: always 인 파일이 실제로 자동 주입되려면 CLAUDE.md에 그 목록이 명시되어 있어야 한다. 파일의 frontmatter에 always 를 써두는 것만으로는 부족하다. AI가 그 값을 참조해서 행동하도록 CLAUDE.md에서 직접 지시해야 한다.

## 항상 로드할 파일 (always)

작업 시작 시 다음 파일을 반드시 읽는다.
다른 어떤 파일보다 먼저 읽어야 한다.

- /01-core/persona.md
- /01-core/core-rules.md
- /01-core/response-format.md

이 목록은 3~5개를 넘지 않도록 관리한다. always 파일이 많아질수록 온디맨드 로딩의 의미가 퇴색되고, 매 세션마다 컨텍스트의 상당 부분이 고정된 파일로 채워진다.

5.3.2 always 파일이 많아질 때 컨텍스트 압박을 줄이는 요약 전략

always 파일은 짧을수록 좋다. 페르소나 파일이 2,000 토큰짜리 장문이라면 매 세션마다 그만큼이 기본으로 소모된다. always 파일의 권장 길이는 파일당 300토큰(약 200단어) 이내다.

세부 내용은 별도 파일에 두고, always 파일에는 핵심 원칙만 남긴다. "상세 글쓰기 규칙은 /02-skills/writing/blog-style.md 를 참조"처럼 포인터 역할만 해도 충분하다. AI는 필요할 때 그 파일을 추가로 로드할 수 있다.

5.4 on-demand 로딩 트리거 설계

5.4.1 작업 키워드 → use_when 매칭 프롬프트 패턴

AI가 사용자의 요청에서 작업 유형을 파악하고 관련 스킬을 찾는 과정은 자동으로 일어나지 않는다. CLAUDE.md에 이 과정을 명시적으로 지시해야 한다.

## on-demand 파일 로딩 절차

사용자의 요청을 받으면 다음 순서로 관련 스킬을 탐색한다.

1. 요청의 핵심 작업을 파악한다.
   (예: "블로그 초안 작성", "코드 리뷰", "회의록 정리")

2. 해당 작업과 관련된 도메인 폴더의 index.md를 읽는다.

3. 각 파일의 use_when 항목을 현재 작업과 대조한다.
   "현재 작업이 이 use_when 조건에 해당하는가?"를 판단한다.

4. 매칭된 파일을 최대 3개 선택하고 전체 내용을 읽는다.

5. 스킬을 적용해서 작업을 수행한다.

5.4.2 Claude가 스스로 관련 스킬이 있는지 먼저 확인하도록 유도하기

단순히 스킬을 찾는 것에서 한 단계 더 나아가, AI가 작업을 시작하기 전에 스스로 "이 작업에 활용할 수 있는 스킬이 있는가"를 먼저 점검하도록 유도할 수 있다.

## 스킬 점검 우선 원칙

어떤 작업을 시작하기 전에 vault에 관련 스킬이 있는지
먼저 확인한다. 스킬 없이 일반적인 방식으로 작업하는 것보다
스킬을 찾아 적용하는 것이 항상 우선이다.

관련 스킬을 찾지 못했을 경우에만 일반적인 방식으로 진행하고,
이를 사용자에게 알린다.

이 지시가 있으면 AI는 매 작업마다 vault를 먼저 확인하는 습관을 갖는다. vault가 단순한 저장소가 아닌 실질적인 작업 파트너가 되는 핵심 지시다.

5.4.3 매칭 실패 시 fallback 처리

매칭되는 스킬 파일이 없을 때 AI가 어떻게 행동할지도 지정해야 한다. 정의하지 않으면 AI는 침묵하거나 임의로 관련 없는 파일을 로드하려 시도한다.

## 스킬 없음 처리

현재 작업과 매칭되는 스킬 파일이 없을 경우:

1. 사용자에게 관련 스킬 파일이 없음을 한 줄로 알린다.
   (예: "이 작업에 맞는 스킬 파일을 찾지 못했습니다.
   일반적인 방식으로 진행합니다.")

2. 스킬 없이 작업을 수행한다.

3. 작업 완료 후, 이 유형의 작업을 위한 스킬 파일을
   만들면 유용할 것 같다고 제안한다.

세 번째 항목이 중요하다. AI가 스킬 부재를 발견하고 새 스킬 파일 생성을 제안하면, vault가 사용하면서 자연스럽게 성장하는 구조가 만들어진다.

5.5 OpenClaw 및 다른 로컬 AI 클라이언트에도 적용하기

5.5.1 MCP filesystem 서버를 통한 vault 연결 방법

Claude Code 이외의 클라이언트에서도 같은 vault를 활용하려면 MCP(Model Context Protocol) filesystem 서버를 사용한다. MCP는 AI 클라이언트가 로컬 파일시스템에 표준화된 방식으로 접근할 수 있게 하는 프로토콜이다.

# MCP filesystem 서버 설치
npm install -g @modelcontextprotocol/server-filesystem

# vault 경로를 허용 목록에 추가하여 서버 실행
mcp-server-filesystem /Users/dk/obsidian-vault

MCP를 지원하는 클라이언트(Cursor, Continue, OpenClaw 등)는 이 서버를 통해 vault에 접근할 수 있다. 클라이언트 설정 파일에 MCP 서버 주소를 등록하면 Claude Code와 동일한 vault를 여러 도구에서 공유할 수 있다.

5.5.2 클라이언트별 CLAUDE.md 대응 파일 위치 차이 정리

각 AI 클라이언트는 프로젝트 지시 파일의 이름과 위치가 다르다. vault 루트에 클라이언트별 지시 파일을 함께 두면 동일한 vault를 여러 도구에서 활용할 수 있다.

클라이언트 지시 파일 이름 위치
Claude Code CLAUDE.md 프로젝트 루트 또는 vault 루트
Cursor .cursorrules 프로젝트 루트
Continue .continuerules 프로젝트 루트
Cline / OpenClaw .clinerules 프로젝트 루트

내용은 모두 동일하게 유지하되, 각 클라이언트의 파일 이름 규칙에 맞게 복사해서 관리한다. vault 루트에 다음과 같이 배치한다.

vault/
  ├── CLAUDE.md          # Claude Code용
  ├── .cursorrules       # Cursor용
  ├── .clinerules        # Cline / OpenClaw용
  └── ...

내용을 동기화해서 유지하는 것이 번거롭다면, CLAUDE.md를 마스터로 두고 나머지 파일에서 "CLAUDE.md의 내용을 따른다"고 참조하거나, 심볼릭 링크로 연결한다.

- 성능: N+1 쿼리 가능성 검토 필요 ✗
- 테스트: 이 함수의 단위 테스트 존재 여부 확인 필요 ✗

스킬이 적용된 리뷰는 "이 vault 주인이 정의한 기준"으로 코드를 평가한다. 팀 컨벤션, 개인 코딩 스타일, 자주 겪는 에러 패턴이 모두 반영된 개인화된 리뷰가 된다.

이 세트 구조를 index.md에 반영하면 AI가 단순히 개별 파일을 매칭하는 것을 넘어, 작업 유형에 맞는 파일 묶음을 함께 로드하도록 유도할 수 있다.

7장. 유지보수와 스킬 품질 관리

시스템을 만드는 것보다 유지하는 것이 더 어렵다. vault는 살아있는 문서 집합이다. 시간이 지나면서 내용이 바뀌고, 작업 방식이 달라지고, 새 파일이 쌓인다. 관리하지 않으면 처음에 잘 작동하던 스킬 시스템이 서서히 신뢰를 잃는다. 이 장은 그 부패를 감지하고 복원하는 루틴을 설계한다.

7.1 description 부패 감지

7.1.1 부패 신호 3가지: 내용 변화, 사용 맥락 변화, 중복 발생

메타데이터 부패는 갑자기 일어나지 않는다. 조용히, 천천히 진행된다. 세 가지 경로로 발생한다.

부패 경로 1: 내용 변화:
파일 본문을 수정했지만 frontmatter는 그대로 뒀을 때 발생한다. 예를 들어 blog-style.md 의 문체 규칙을 전면 개정했는데 description은 여전히 예전 규칙을 설명하고 있는 경우다. AI는 description을 믿고 파일을 로드하지만, 실제 내용은 description과 다르다.

부패 경로 2: 사용 맥락 변화:
워크플로우가 바뀌어서 use_when 조건이 더 이상 현실을 반영하지 못하는 경우다. 예를 들어 이전에는 블로그를 직접 썼는데 이제는 초안만 쓰고 편집자에게 넘기는 구조로 바뀌었다면, SEO 체크리스트의 use_when 도 바뀌어야 한다.

부패 경로 3: 중복 누적:
비슷한 역할의 파일이 시간을 두고 추가되면서 description이 겹치기 시작한다. 처음에는 writing-rules.md 하나였는데, 나중에 blog-guidelines.md, content-style.md 가 생기면서 세 파일의 description이 서로 구별되지 않게 된다. AI는 셋 다 로드하거나 임의로 하나를 선택한다.

7.1.2 last_reviewed 기준 점검 트리거 설정

부패를 사전에 방지하는 가장 단순한 방법은 last_reviewed 날짜를 기준으로 점검 주기를 강제하는 것이다.

점검 기준:

- 30일 이내: 정상. 점검 불필요.
- 31~90일: 주의. 내용 변경이 있었는지 확인 권장.
- 91일 이상: 경고. 반드시 검토 후 last_reviewed 갱신.
- 180일 이상: 위험. 파일 유효성 자체를 재판단.

이 기준을 7.3에서 다루는 월간 리뷰 루틴과 연결하면, 매달 경고 상태 이상의 파일을 자동으로 파악할 수 있다.

7.2 중복 스킬 정리와 deprecated 처리

7.2.1 비슷한 파일 통합 기준: 80% 이상 겹치면 합쳐라

두 파일을 나란히 놓고 내용이 80% 이상 겹친다고 느껴지면 통합을 고려한다. 기준은 단순하다. "이 두 파일이 같은 작업에 동시에 로드될 때 중복 지시가 발생하는가?" 그렇다면 합친다.

통합 절차:

1. 두 파일을 열고 각각의 고유한 내용을 파악한다.
2. 하나의 파일로 합치되, 더 최신 버전을 기준으로 삼는다.
3. 합쳐진 파일의 description과 use_when을 새로 작성한다.
4. 기존 파일 중 하나는 삭제하거나 deprecated 처리한다.
5. index.md의 파일 목록을 업데이트한다.

통합 후 반드시 version 을 올리고 last_reviewed 를 오늘 날짜로 갱신한다.

7.2.2 load_priority: locked으로 파일을 '은퇴'시키는 절차

바로 삭제하기 아까운 파일, 혹은 삭제했다가 나중에 필요할 수도 있는 파일은 locked 로 은퇴시킨다. 삭제 없이 AI의 탐색 범위에서 제외할 수 있다.

은퇴 처리 절차:

1. 파일의 load_priority를 on-demand에서 locked으로 변경한다.
2. description 앞에 [DEPRECATED] 표시를 추가한다.
3. 파일 본문 최상단에 은퇴 이유와 날짜를 기록한다.
4. index.md에서 해당 파일 항목을 제거하거나 취소선으로 표시한다.

예시:
---
description: "[DEPRECATED] 구버전 블로그 스타일 가이드. blog-style-v2.md로 대체됨."
load_priority: locked
last_reviewed: 2026-04-01
---

> 이 파일은 2026-04-01부로 비활성화되었습니다.
> blog-style-v2.md로 대체되었습니다.

이 방식은 이력을 보존하면서 AI의 탐색 범위에서 제외한다. 6개월 후에도 참조할 일이 없다면 그때 완전히 삭제한다.

7.3.2 Dataview로 last_reviewed 90일 초과 파일 자동 목록화 쿼리

Obsidian Dataview 플러그인을 사용하면 last_reviewed 날짜 기준으로 점검이 필요한 파일을 자동으로 목록화할 수 있다. vault 루트 또는 01-core/ 폴더에 review-dashboard.md 파일을 만들고 다음 쿼리를 삽입한다.

\`\`\`dataview
TABLE last_reviewed, description, load_priority
FROM "02-skills" OR "03-reference"
WHERE last_reviewed < date(today) - dur(90 days)
AND load_priority != "locked"
SORT last_reviewed ASC
\`\`\`

이 쿼리는 90일 이상 검토되지 않은 파일을 오래된 순서로 나열한다. locked 파일은 제외한다. 매월 리뷰 시작 시 이 대시보드를 먼저 열면 어디서부터 시작할지 즉시 파악된다.

8장. 고급 확장: 다른 AI 에이전트 및 자동화와의 통합

이 장은 vault를 Claude Code 하나에서 벗어나 더 넓은 생태계로 연결하는 방법을 다룬다. 같은 스킬 베이스를 여러 AI 클라이언트가 공유하고, 특정 상황이 발생하면 스킬이 자동으로 주입되는 구조를 만든다. 여기까지 오면 vault는 단순한 개인 지식 저장소를 넘어 자신의 AI 작업 환경 전체를 제어하는 허브가 된다.

8.1 MCP 서버로 vault를 API화하기

8.1.1 obsidian-mcp 서버 설치 및 설정 방법

MCP(Model Context Protocol)는 Anthropic이 제안한 AI 클라이언트-서버 통신 표준이다. vault를 MCP 서버로 노출하면 MCP를 지원하는 모든 클라이언트가 동일한 vault에 표준화된 방식으로 접근할 수 있다.

설치는 두 가지 방법을 선택할 수 있다.

방법 1: 공식 MCP filesystem 서버 사용:

# 설치
npm install -g @modelcontextprotocol/server-filesystem

# 실행 (vault 경로를 허용 목록에 추가)
npx @modelcontextprotocol/server-filesystem /Users/dk/obsidian-vault

방법 2: obsidian-mcp 플러그인 사용:
Obsidian 커뮤니티 플러그인 중 MCP 서버 기능을 제공하는 플러그인을 설치하면 Obsidian 앱이 실행 중일 때 자동으로 MCP 서버가 활성화된다. 별도의 터미널 프로세스 없이 vault에 접근할 수 있다.

클라이언트 측 설정 파일(예: Claude Desktop의 claude_desktop_config.json)에 서버 정보를 등록한다.

{
  "mcpServers": {
    "obsidian-vault": {
      "command": "npx",
      "args": [
        "@modelcontextprotocol/server-filesystem",
        "/Users/dk/obsidian-vault"
      ]
    }
  }
}

8.1.2 여러 AI 클라이언트가 동일한 vault를 공유하는 아키텍처

MCP 서버 방식의 핵심 이점은 단일 vault를 여러 클라이언트가 동시에 참조할 수 있다는 점이다. 스킬 파일을 클라이언트마다 복사하거나 동기화할 필요가 없다.

네 개의 클라이언트가 모두 같은 vault의 스킬 파일을 참조한다. blog-style.md 를 수정하면 다음 세션부터 모든 클라이언트에 즉시 반영된다. 스킬 업데이트가 한 곳에서 이루어진다.

8.1.3 멀티 에이전트 환경에서 스킬 충돌 방지 전략

여러 클라이언트가 같은 vault를 사용하면 예상치 못한 충돌이 생길 수 있다. 특히 read-write 권한을 허용한 경우 두 클라이언트가 동시에 같은 파일을 수정하려 하면 덮어쓰기가 발생한다.

충돌을 방지하는 세 가지 전략:

전략 1: 클라이언트별 쓰기 영역 분리:
각 클라이언트가 쓸 수 있는 폴더를 다르게 지정한다. Claude Code는 /02-skills/drafts/ 에만 쓸 수 있고, Cursor는 /04-context/projects/ 에만 쓸 수 있는 식이다. CLAUDE.md와 각 클라이언트의 지시 파일에 쓰기 가능 경로를 명시한다.

전략 2: 스킬 파일은 read-only, 작업 산출물은 read-write:
스킬 파일(01-core/, 02-skills/, 03-reference/)은 모든 클라이언트에게 읽기 전용으로 제공한다. AI가 생성한 결과물을 저장하는 폴더(05-outputs/)만 쓰기 허용한다.

전략 3: 수정 이력 추적:
vault를 git으로 버전 관리하면 어떤 클라이언트가 언제 무엇을 수정했는지 추적할 수 있다. 충돌이 발생해도 이전 상태로 롤백이 가능하다.

# vault 루트에서 git 초기화
cd /Users/dk/obsidian-vault
git init
git add .
git commit -m "initial vault snapshot"

이후 월간 리뷰 때마다 커밋하면 vault의 변화 이력이 git log에 남는다.

8.2 n8n / Make로 스킬 로딩 자동화하기

8.2.1 캘린더 이벤트 → 관련 스킬 자동 주입 파이프라인 설계

지금까지는 AI에게 직접 요청할 때 스킬이 로드되었다. 자동화 파이프라인을 구성하면 특정 이벤트가 발생했을 때 관련 스킬이 미리 준비된 상태로 AI 세션이 시작된다.

예시 시나리오: 캘린더에 "주간 팀 미팅"이 등록되면, 미팅 30분 전에 회의 관련 스킬 세트가 자동으로 로드된 AI 세션 링크가 슬랙으로 전송된다.

n8n 워크플로우 구성:

이 파이프라인이 작동하면 미팅이 끝난 후 회의록을 붙여넣기만 해도 스킬이 이미 로드된 AI가 즉시 구조화된 정리를 수행한다.

8.2.2 슬랙 메시지 키워드 → 스킬 트리거 자동화 예시

슬랙에서 특정 키워드가 포함된 메시지를 보내면 관련 스킬이 준비된 AI 세션으로 연결되는 워크플로우다. 팀 환경에서 특히 유용하다.

Make(구 Integromat) 시나리오 구성:

이 자동화의 핵심은 vault의 스킬 파일이 단일 진실 공급원(single source of truth)이라는 점이다. n8n이나 Make 워크플로우 안에 프롬프트를 하드코딩하지 않는다. 스킬 파일의 내용이 바뀌면 자동화 파이프라인도 자동으로 업데이트된다. 관리 포인트가 vault 하나로 집중된다.

마치며

이 가이드는 하나의 간단한 아이디어에서 출발했다. YAML description 한 줄을 잘 써두면 AI가 필요한 파일을 스스로 찾아 쓸 수 있다는 것. 그 아이디어를 끝까지 밀어붙이면 어디까지 갈 수 있는지를 8개 장에 걸쳐 추적했다.

핵심을 한 줄로 요약하면 이렇다.

vault의 품질은 description의 품질이다.

스키마가 아무리 정교하고, 폴더 구조가 아무리 깔끔하고, CLAUDE.md가 아무리 잘 쓰여 있어도, 개별 파일의 description이 모호하면 AI는 길을 잃는다. 반대로 description이 정밀하면 나머지는 자연스럽게 따라온다.

시작은 간단하다. 지금 가장 자주 쓰는 파일 10개를 골라 description을 붙이는 것으로 충분하다. 그것만으로도 vault가 AI와 소통하기 시작한다. 나머지는 쓰면서 쌓인다.
