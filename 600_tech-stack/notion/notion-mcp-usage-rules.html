Notion MCP Usage Rules

Notion MCP Usage Rules (v2025-11)

Purpose: Ensure AI agents correctly create, read, and update Notion pages/databases through structured validation → execution → verification patterns.

Critical Rules (Top 10)

  1. Tool Names: Official MCP tools are notion-create-pages, notion-fetch, notion-update-page, notion-search. Some clients may abbreviate to fetch/search.

  2. Parent Format (MOST IMPORTANT): Always use object format with type + ID field. NEVER pass parent as a plain string.

    ✅ { "type": "data_source_id", "data_source_id": "<UUID>" }
❌ "<UUID>"

  3. Schema Validation: Always fetch schema with notion-fetch before creation to get field types, options, and identify read-only properties.

  4. Property Value Validation: Only use values defined in schema for status/select/multi_select/people/relation/date.

  5. Date Properties: Use extended notation (date:field:start, date:field:end, date:field:is_datetime). Due to MCP bugs, apply two-step pattern: create → immediate update.

  6. Batch Creation: notion-create-pages requires pages: [...] array. Never omit this parameter.

  7. Relation Connections: Use { "relation": [{ "id": "<page_id>" }] } format. NEVER use URL strings.

  8. Rate Limits:

    • Overall: 180 RPM (~3 RPS)
    • Search: 30 RPM
    • Use serial execution + exponential backoff on failures

  9. Security: Only use official endpoints (https://mcp.notion.com/mcp). Require user approval for destructive operations.

  10. API Version Awareness: Since 2025-09-03, use data_source_id (not deprecated database_id).

Standard Execution Pipeline

Every create/update operation MUST follow these 9 steps:

1. Normalize Input → 2. Fetch Schema → 3. Map Properties → 4. Validate → 5. Create
   ↓
2. Update (date fix) → 7. Set Relations → 8. Verify → 9. Handle Rate Limits

Detailed Steps

1. Normalize Input

  • Standardize user-provided values (status, dates, people, etc.)

2. Fetch Schema

const schema = await notion-fetch({ id: "<data_source_id>" });
// Extract: field types, allowed options, required fields, read-only properties

3. Map Properties

  • select/status: Only schema-defined options
  • multi_select: No duplicates, case-sensitive
  • relation: Array of valid page IDs from related data source
  • date: ISO-8601 format
  • title/rich_text: Check length limits

4. Pre-Validation

  • Detect missing required fields, typos, invalid options
  • Return specific error messages for correction

5. Create (Primary)

await notion-create-pages({
  pages: [{
    content: "# Title\n\nContent...",
    properties: {
      "Name": "Page Title",
      "Status": "Not Started",
      "date:Due Date:start": "2025-11-01",
      "date:Due Date:is_datetime": 0
    }
  }],
  parent: {
    type: "data_source_id",
    data_source_id: "<UUID>"
  }
});

6. Update (Date Correction)

// MCP bug workaround: start/end often ignored in create
await notion-update-page({
  data: {
    page_id: createdPageId,
    command: "update_properties",
    properties: {
      "date:Due Date:start": "2025-11-01",
      "date:Due Date:end": "2025-11-05"
    }
  }
});

7. Set Relations

// CORRECT: Use relation format
await notion-update-page({
  data: {
    page_id: subTaskId,
    command: "update_properties",
    properties: {
      "Parent Task": {
        relation: [{ id: parentPageId }]
      }
    }
  }
});

8. Post-Verification

const result = await notion-fetch({ id: createdPageId });
// Verify: properties match, relations connected, content exists

9. Rate Limit Handling

  • 429/5xx errors: Exponential backoff (max 3 retries)
  • Respect search 30 RPM limit

Property Type Rules

Read-Only Properties (NEVER include in requests)

  • created_by, created_time, last_edited_by, last_edited_time
  • formula, rollup

Type-Specific Rules

Type Format Validation
status/select "Status": "In Progress" Must be in schema options
multi_select "Tags": ["tag1", "tag2"] No duplicates, case-sensitive
relation "relation": [{"id": "<page_id>"}] Target data source must be shared
people "people": [{"id": "<user_id>"}] Max 25 users per request
date "date:Field:start": "2025-11-01" ISO-8601, use update for ranges
files Full array replacement Partial removal not supported

Common MCP Bugs & Workarounds

Bug 1: Object Serialization

Error: Expected object, received string

 

Cause: Parent parameter serialized as string instead of object

 

Fix:

✅ parent: { type: "data_source_id", data_source_id: "<UUID>" }
❌ parent: "<UUID>"

Bug 2: Date Extension Fields Ignored

Error: Date start/end not set after creation

 

Cause: MCP create ignores extended date notation

 

Fix: Use two-step pattern (create → update)

Bug 3: Read-Only Properties Auto-Inserted

Error: Property X not found

 

Cause: MCP caches view-specific read-only columns

 

Fix: Use whitelist approach - only send writable properties

Bug 4: Missing Required Parameter

Error: Required parameter 'pages' is missing

 

Fix: Always include pages: [...] array in notion-create-pages

Error Handling Patterns

// Standard error handling structure
try {
  const result = await notion-create-pages({ /*...*/ });
  
  // Immediate verification
  const verified = await notion-fetch({ id: result.pages[0].id });
  
  // Check for missing data (indicates bugs or permission issues)
  if (!verified.properties["Expected Field"]) {
    // Apply corrective update
    await notion-update-page({ /*...*/ });
  }
  
} catch (error) {
  if (error.code === 429) {
    // Rate limit: exponential backoff
    await sleep(Math.pow(2, retryCount) * 1000);
    retry();
  } else if (error.message.includes("Property X not found")) {
    // Read-only property issue: filter request
    const filtered = filterWritableProperties(properties, schema);
    retry(filtered);
  } else {
    // Log and report specific error
    throw new Error(`Notion MCP Error: ${error.message}`);
  }
}

Production Checklist

Before Creating Pages/Rows

  •  Fetch current data source schema (notion-fetch)
  •  Extract writable properties only (exclude formula, rollup, created_*, etc.)
  •  Validate all property values against schema options
  •  Prepare parent as object with type + ID
  •  Prepare relation IDs (verify data source is shared)
  •  Format dates as ISO-8601
  •  Include pages: [...] array parameter

For Subtasks/Related Pages

  •  Obtain parent page ID
  •  Verify related data sources are shared with integration
  •  Use two-step pattern: create → update relation
  •  Confirm relation field name matches schema exactly

Performance & Reliability

  •  Serialize bulk operations (avoid parallel creates)
  •  Implement exponential backoff for retries
  •  Respect search rate limit (30 RPM)
  •  Verify all created pages with notion-fetch
  •  Log all operations for debugging

Complete Example: Epic Creation

// Step 1: Fetch schema
const schema = await notion-fetch({
  id: '29a2b332-7e20-8148-ba33-000bdba85db4'
});

// Step 2: Validate options
const validStatuses = schema.properties["Status"].status.options.map(o => o.name);
if (!validStatuses.includes("Not Started")) {
  throw new Error("Status 'Not Started' not in schema");
}

// Step 3: Create page
const epic = await notion-create-pages({
  pages: [{
    content: '# Camera & Media Capture\n\n## Overview\nCamera control, real-time streaming...',
    properties: {
      'Name': 'Camera & Media Capture',
      'Issue Type': 'Epic',
      'Status': 'Not Started',
      'date:Start Date:start': '2025-10-28',
      'date:Start Date:is_datetime': 0
    }
  }],
  parent: {
    type: 'data_source_id',
    data_source_id: '29a2b332-7e20-8148-ba33-000bdba85db4'
  }
});

// Step 4: Fix dates (MCP bug workaround)
await notion-update-page({
  data: {
    page_id: epic.pages[0].id,
    command: 'update_properties',
    properties: {
      'date:Start Date:start': '2025-10-28',
      'date:Start Date:is_datetime': 0
    }
  }
});

// Step 5: Verify
const verified = await notion-fetch({ id: epic.pages[0].id });
console.log('Created:', verified.properties["Name"].title[0].plain_text);

Security & Permissions

Endpoint Validation:

  • ✅ Only use: https://mcp.notion.com/mcp or https://mcp.notion.com/sse
  • ❌ Never use: Custom or unverified endpoints

Operation Approval:

  • Show preview before: bulk operations, moves, deletions
  • Require explicit user confirmation

API Key Management:

  • Use minimum required permissions
  • Rotate keys regularly
  • Log all API operations

Data Source Sharing:

  • Relation targets must be shared with integration
  • Verify permissions before relation operations
  • Empty relation values indicate permission issues

Quick Reference: Common Patterns

Pattern 1: Create Page in Database

await notion-create-pages({
  pages: [{ content: "...", properties: {...} }],
  parent: { type: "data_source_id", data_source_id: "<UUID>" }
});

Pattern 2: Create Subtask with Parent Relation

// Create subtask
const sub = await notion-create-pages({...});

// Link to parent
await notion-update-page({
  data: {
    page_id: sub.pages[0].id,
    command: "update_properties",
    properties: {
      "Parent Task": { relation: [{ id: parentId }] }
    }
  }
});

Pattern 3: Batch Creation with Rate Limiting

for (const item of items) {
  await notion-create-pages({...});
  await sleep(350); // ~170 RPM (safe margin under 180 RPM)
}

Pattern 4: Schema-Safe Property Mapping

const schema = await notion-fetch({ id: dataSourceId });
const writableProps = Object.entries(properties)
  .filter(([key]) => !schema.properties[key]?.read_only)
  .reduce((obj, [k, v]) => ({ ...obj, [k]: v }), {});

AI Agent Instructions

When using this guide:

  1. Always fetch schema first - Never assume property names or types
  2. Use two-step pattern for dates - Create, then immediately update
  3. Serialize bulk operations - Never create multiple pages in parallel
  4. Verify after every operation - Use notion-fetch to confirm success
  5. Handle errors specifically - Each error type has a documented fix
  6. Log everything - Include operation type, parameters, and results
  7. Fail fast - Stop and report on first validation error rather than attempting creation
